# 📌 1.1 — JavaScript Architecture
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** JavaScript Architecture is the entire machinery — from the specification that defines the language rules, to the engines that execute your code, to the runtime environments that provide APIs like `setTimeout`, `fetch`, or `fs.readFile`. It's the full picture of *how your `.js` file goes from text to running program*.

**Technical Definition:** JavaScript's architecture is a layered system comprising:
1. **ECMAScript specification** — the abstract language definition
2. **JavaScript engine** — the parser, compiler, and garbage collector (V8, SpiderMonkey, etc.)
3. **Runtime environment** — the host that embeds the engine and provides platform APIs (Browser, Node.js, Deno, Bun)

**One-line mental model:**
> *ECMAScript is the blueprint, the engine is the factory, and the runtime is the city the factory operates in.*

**Where does this live?**
- The **specification** lives as a document (tc39.es/ecma262)
- The **engine** lives as native C++/Rust code compiled into the browser or Node.js binary
- The **runtime** is the full executable (Chrome, Node, Deno) that wires the engine to OS-level capabilities

## ② DRILL DOWN

### 🔹 A. ECMAScript Specification

ECMAScript (ECMA-262) is the **standard** that defines:
- Syntax (what `let x = 5;` means)
- Semantics (what happens when you run `x + y`)
- Built-in objects (`Object`, `Array`, `Promise`, `Map`, etc.)
- Execution model (execution contexts, scope chains, the job queue)

It does **NOT** define:
- `console.log` (that's a host API)
- `setTimeout` / `setInterval` (that's the runtime)
- `document`, `window` (that's the browser)
- `require`, `fs`, `process` (that's Node.js)

**Key TC39 Process (how features get added):**

| Stage | Name | Meaning |
|-------|------|---------|
| 0 | Strawperson | Just an idea |
| 1 | Proposal | Formal proposal with API sketch |
| 2 | Draft | Formal spec text drafted |
| 3 | Candidate | Spec complete, needs implementation feedback |
| 4 | Finished | Ready for inclusion in the next edition |

```javascript
// This is pure ECMAScript — works in any compliant engine:
const arr = [1, 2, 3];
const doubled = arr.map(x => x * 2); // [2, 4, 6]

// This is NOT ECMAScript — it's a browser runtime API:
document.getElementById("app"); // browser-only

// This is NOT ECMAScript — it's a Node.js runtime API:
const fs = require('fs'); // Node-only
```

### 🔹 B. JavaScript Engines

A JavaScript engine is a **program** (written in C++ or Rust) that:
1. **Parses** your source code into an Abstract Syntax Tree (AST)
2. **Compiles** the AST into bytecode (and later into optimized machine code)
3. **Executes** the bytecode/machine code
4. **Manages memory** via garbage collection

**The Big Four Engines:**

| Engine | Creator | Used In | Language |
|--------|---------|---------|----------|
| **V8** | Google | Chrome, Node.js, Deno, Edge | C++ |
| **SpiderMonkey** | Mozilla | Firefox | C++/Rust |
| **JavaScriptCore (JSC / Nitro)** | Apple | Safari, Bun | C++ |
| **Chakra** | Microsoft | Old Edge (pre-Chromium) | C++ |

#### 🔸 V8 (Google)

V8 is the most widely-used engine. It powers Chrome, Node.js, Deno, Electron, and Cloudflare Workers.

**V8's Pipeline (2024+):**

```
Source Code
    │
    ▼
┌─────────┐
│ Scanner  │  ← Tokenizes source into tokens
└────┬─────┘
     ▼
┌─────────┐
│ Parser   │  ← Builds AST (Abstract Syntax Tree)
└────┬─────┘
     ▼
┌───────────┐
│ Ignition   │  ← Bytecode interpreter (generates bytecode, executes it)
└────┬───────┘
     │
     │  (Hot code detected via profiling)
     ▼
┌────────────┐
│ TurboFan    │  ← Optimizing JIT compiler (generates machine code)
└────┬────────┘
     │
     │  (If assumptions violated → DEOPTIMIZATION)
     ▼
  Back to Ignition bytecode
```

**Key Insight:** V8 does NOT interpret line-by-line like old JS engines. It compiles to **bytecode first** (Ignition), then **JIT-compiles hot functions** to optimized machine code (TurboFan). If assumptions break (e.g., you change a variable's type), it **deoptimizes** back to bytecode.

```javascript
// V8 optimizes this because `add` always receives numbers:
function add(a, b) {
    return a + b;
}
add(1, 2);     // V8 profiles: "a and b are always Smi (small integers)"
add(3, 4);     // Still Smi → TurboFan generates optimized machine code for integer add
add("x", "y"); // TYPE CHANGE! → V8 deoptimizes, falls back to Ignition bytecode
```

#### 🔸 SpiderMonkey (Mozilla)

Firefox's engine. It was the **first-ever** JavaScript engine, written by Brendan Eich in 1995.

Pipeline: Parser → Bytecode → Baseline JIT → IonMonkey (optimizing JIT) → Warp (newer optimizer)

#### 🔸 JavaScriptCore (Apple)

Safari's engine. Also called "Nitro." Used by Bun (the new JS runtime).

Pipeline: Parser → Bytecode (LLInt) → Baseline JIT → DFG (Data Flow Graph) JIT → FTL (Faster Than Light) JIT

**Four tiers of JIT** — the most tiered of any JS engine.

#### 🔸 Chakra (Microsoft)

Was used in the old Edge browser (pre-2020). Now dead — Microsoft switched Edge to Chromium/V8. ChakraCore was open-sourced but is largely unmaintained.

### 🔹 D. Parsing → AST → Bytecode → JIT Optimization

Let's trace what happens when V8 executes this code:

```javascript
function square(n) {
    return n * n;
}
console.log(square(5));
```

**Step 1 — Scanning/Tokenization:**
```
function  →  KEYWORD
square    →  IDENTIFIER
(         →  LPAREN
n         →  IDENTIFIER
)         →  RPAREN
{         →  LBRACE
return    →  KEYWORD
n         →  IDENTIFIER
*         →  OPERATOR
n         →  IDENTIFIER
;         →  SEMICOLON
}         →  RBRACE
```

**Step 2 — Parsing into AST:**
```
Program
└── FunctionDeclaration (name: "square")
    ├── params: [Identifier("n")]
    └── body: BlockStatement
        └── ReturnStatement
            └── BinaryExpression (operator: *)
                ├── left: Identifier("n")
                └── right: Identifier("n")
```

**Step 3 — Ignition compiles AST → Bytecode:**
```
// Simplified bytecode (actual V8 bytecode is similar):
LdaNamedProperty     a0, [0]     // Load parameter 'n'
Star                 r0          // Store in register r0
Mul                  r0, a0      // Multiply r0 * a0
Return                           // Return accumulator
```

**Step 4 — If `square()` is called frequently ("hot"):**

TurboFan kicks in:
- Sees `n` is always a `Smi` (Small Integer)
- Generates native x86/ARM machine code for integer multiplication
- Installs this optimized code, bypassing Ignition

**Step 5 — Deoptimization (if assumptions break):**
```javascript
square(5);       // Smi → optimized
square(10);      // Smi → still optimized
square("hello"); // STRING! → Assumption violated → Deoptimize back to Ignition
```

### 🔹 E. Interpreted vs JIT-Compiled Execution

**Common misconception: "JavaScript is interpreted."**

This was true in the 1990s. Modern JavaScript is **JIT (Just-In-Time) compiled**.

| Execution Model | Description | JS Engines |
|----------------|-------------|------------|
| Pure Interpreter | Reads and executes line by line. No compilation. | 1995 Netscape, old browsers |
| AOT Compiler | Compiles everything before running (like C/Java) | Not used by JS |
| JIT Compiler | Compiles at runtime, optimizes hot paths | V8, SpiderMonkey, JSC — ALL modern engines |

**V8's approach: Bytecode interpreter + JIT compiler (tiered compilation)**

```
Cold code → Ignition (bytecode interpreter) → Fast startup
Hot code  → TurboFan (JIT optimizer)         → Fast execution
```

**Why not AOT-compile everything?**
- JavaScript is dynamically typed. You don't know `x + y` means number addition or string concatenation until runtime.
- AOT would be slow because you'd have to compile code paths you may never take.
- JIT is perfect: it watches what types actually flow through, then compiles for those specific types.

### 🔹 F. Hidden Classes and Inline Caching

These are V8's two most important optimizations. Without them, JavaScript would be 10–100x slower.

#### Hidden Classes (also called "Maps" or "Shapes")

In statically-typed languages, the compiler knows the memory layout of objects at compile time. In JavaScript, objects are dynamic — you can add/remove properties anytime. So how does V8 make property access fast?

**Hidden Classes:** V8 creates an internal "shape" (hidden class) for each object layout. Objects with the same properties in the same order share the same hidden class.

```javascript
function Point(x, y) {
    this.x = x;    // Transition: HiddenClass0 → HiddenClass1 (has 'x')
    this.y = y;    // Transition: HiddenClass1 → HiddenClass2 (has 'x', 'y')
}

const p1 = new Point(1, 2);
const p2 = new Point(3, 4);
// p1 and p2 share the SAME hidden class (HiddenClass2)
// V8 knows: x is at offset 0, y is at offset 4
// Property access is as fast as a C struct!
```

**Memory Diagram:**
```
p1 ──────────►  ┌─────────────────────┐
                │ Hidden Class: HC2    │
                │ x: offset 0 → 1     │
                │ y: offset 4 → 2     │
                └─────────────────────┘

p2 ──────────►  ┌─────────────────────┐
                │ Hidden Class: HC2    │  ← SAME hidden class!
                │ x: offset 0 → 3     │
                │ y: offset 4 → 4     │
                └─────────────────────┘
```

**What breaks hidden classes:**
```javascript
const p1 = new Point(1, 2);
const p2 = new Point(3, 4);
p2.z = 5;  // p2 now transitions to HiddenClass3
           // p1 and p2 NO LONGER share a hidden class
           // This is slower — V8 can't optimize property access uniformly
```

**RULE:** Always initialize properties in the **same order** in constructors. Never add properties dynamically after construction.

#### Inline Caching (IC)

When V8 encounters `obj.x`, the first time it does a full lookup (check the hidden class, find the offset of `x`). This is slow. With Inline Caching, V8 **remembers** the result:

```
First access to obj.x:
  → Look up hidden class → Find offset of 'x' → Cache: "If HC2, x is at offset 0"

Subsequent accesses:
  → Check hidden class == HC2? Yes → Read offset 0 directly (one memory read!)
```

**IC States:**

| State | Meaning |
|-------|---------|
| Uninitialized | Never accessed |
| Monomorphic | Always same hidden class → FASTEST |
| Polymorphic | 2-4 different hidden classes → Still ok |
| Megamorphic | 5+ different hidden classes → SLOW (falls back to dictionary lookup) |

```javascript
// Monomorphic — FAST:
function getX(point) { return point.x; }
getX({ x: 1, y: 2 });  // All same shape
getX({ x: 3, y: 4 });  // All same shape
getX({ x: 5, y: 6 });  // All same shape → monomorphic IC

// Megamorphic — SLOW:
function getX(obj) { return obj.x; }
getX({ x: 1 });              // Shape 1
getX({ x: 1, y: 2 });        // Shape 2
getX({ x: 1, y: 2, z: 3 });  // Shape 3
getX({ a: 0, x: 1 });        // Shape 4
getX({ b: 0, x: 1 });        // Shape 5 → megamorphic → SLOW
```

### 🔹 G. Startup Execution Flow

When a JavaScript program starts (in browser or Node.js), here's the exact sequence:

```
1. Engine receives source code (from <script> tag or `node app.js`)
2. Scanner tokenizes the source
3. Parser builds the AST
   └── Lazy parsing: Only fully parses functions that are immediately invoked
   └── Pre-parsing: Quickly scans other functions (just checks for syntax errors)
4. Ignition generates bytecode from the AST
5. Global Execution Context is created:
   ├── Creation Phase:
   │   ├── Create Variable Environment (var declarations → undefined)
   │   ├── Create Lexical Environment (let/const → TDZ)
   │   ├── Create `this` binding (window / global / undefined)
   │   └── Process function declarations (fully hoisted)
   ├── Execution Phase:
   │   └── Execute code line by line
6. When a function is called → new Function Execution Context is pushed onto Call Stack
7. Event loop starts (in browser: after initial script; in Node: after module loading)
```

**Lazy Parsing (important optimization):**
```javascript
// V8 does NOT fully parse this function at startup:
function rarelyUsed() {
    // ... 1000 lines of code ...
}

// V8 only FULLY parses this when it's first called:
rarelyUsed(); // NOW V8 fully parses, compiles bytecode, executes
```

This is why large JavaScript files start up faster than you'd expect — V8 only compiles what it needs, when it needs it.

### 🔹 H. Strict Mode

**What is it?** A restricted variant of JavaScript that eliminates some silent errors, fixes some engine optimizations, and forbids some syntax.

```javascript
"use strict";

// 1. No accidental globals
x = 5;  // ReferenceError! (without strict mode, this creates a global)

// 2. Assignment to non-writable property throws
const obj = {};
Object.defineProperty(obj, 'x', { value: 1, writable: false });
obj.x = 2;  // TypeError in strict mode (silent fail in sloppy mode)

// 3. Deleting undeletable properties throws
delete Object.prototype;  // TypeError in strict mode

// 4. Duplicate parameter names are forbidden
function f(a, a) {}  // SyntaxError in strict mode

// 5. `this` is undefined in plain function calls (not window)
function f() {
    console.log(this);  // undefined (in sloppy mode, this would be `window`)
}
f();

// 6. `with` statement is forbidden
with (obj) {}  // SyntaxError in strict mode

// 7. `eval` doesn't introduce variables into surrounding scope
eval("var x = 5;");
console.log(x);  // ReferenceError in strict mode (works in sloppy mode)

// 8. Octal literals are forbidden
const n = 010;  // SyntaxError (in sloppy mode, this is 8)

// 9. `arguments` and `eval` can't be reassigned
arguments = [];  // SyntaxError
```

**Why Strict Mode helps V8 optimize:**
- No `with` statements → scope chain is statically analyzable
- No `eval` variable injection → variable lookup is predictable
- No `arguments.callee` → enables tail call optimization
- `this` is `undefined` in free functions → no boxing needed

**Where is strict mode automatic?**
- Inside ES Modules (`import`/`export`) — always strict
- Inside `class` bodies — always strict
- Never inside `<script>` tags by default (you must opt in)

## ③ WHY IT EXISTS

**What problem does it solve?**

In 1995, Brendan Eich created JavaScript in **10 days** for Netscape Navigator. It was designed to be a simple scripting language — not a systems language. There was no concept of "architecture" because the whole engine was a naive tree-walking interpreter.

As the web grew, JavaScript had to run **real applications** (Gmail, Google Maps, Facebook). The original architecture couldn't keep up. The breakthrough was V8 (2008), which introduced JIT compilation and made JavaScript fast enough for serious use.

The architecture exists because:
1. **Dynamic typing** means you can't know types at compile time → JIT compilation watches types at runtime
2. **The web platform** needed backward compatibility forever → ECMAScript evolves via a careful proposal process
3. **Different environments** (browser vs server) need different APIs → engine/runtime separation

**How does it compare to Java/C++/Python?**

| Aspect | JavaScript | Java | C++ | Python |
|--------|-----------|------|-----|--------|
| Specification | ECMAScript | JLS (Java Language Spec) | ISO C++ | PEP / CPython reference |
| Compilation | JIT (at runtime) | AOT to bytecode, then JIT (JVM) | AOT to machine code | Interpreted (CPython) |
| Typing | Dynamic | Static | Static | Dynamic |
| Memory | Garbage collected | Garbage collected | Manual (RAII) | Garbage collected (ref counting) |
| Engines | V8, SpiderMonkey, JSC | HotSpot, GraalVM | gcc, clang, MSVC | CPython, PyPy |

## ④ INTERNAL WORKING

### What does the parser/compiler do?

**Phase 1 — Scanning:** Converts raw UTF-16 source text into a stream of tokens. The scanner handles:
- Keywords (`function`, `let`, `return`)
- Identifiers (`myVar`, `console`)
- Literals (`42`, `"hello"`, `true`)
- Operators (`+`, `===`, `=>`)
- Whitespace and comments (discarded)

**Phase 2 — Parsing:** Converts tokens into an AST. V8 uses two parsers:
- **Pre-parser (lazy):** Quickly scans function bodies for syntax errors without building full AST nodes. Used for functions not immediately called.
- **Full parser (eager):** Builds the complete AST for code that needs immediate execution.

**Phase 3 — Bytecode Generation (Ignition):**
The AST is walked and Ignition generates register-based bytecode. Each bytecode instruction is a single operation (load, store, add, call, etc.).

**Phase 4 — Profiling:**
Ignition collects **type feedback** as it executes. For each operation, it records:
- What types were the operands? (Smi, HeapNumber, String, Object)
- What hidden classes were the objects?
- How many times was this code path taken?

**Phase 5 — Optimization (TurboFan):**
When a function is "hot" (called many times), TurboFan:
1. Takes the bytecode + type feedback
2. Builds a **Sea of Nodes** IR (intermediate representation)
3. Performs optimizations: inlining, dead code elimination, escape analysis, bounds check elimination
4. Generates optimized machine code specific to the observed types

**Phase 6 — Deoptimization:**
If the assumptions TurboFan made are violated:
```javascript
function add(a, b) { return a + b; }
// TurboFan compiled for integer addition
add(1, 2);  // Fine
add(1.5, 2.5);  // DEOPT! Now it's floats, not Smi
// V8 throws away the optimized code, falls back to Ignition bytecode
// Re-profiles, eventually re-optimizes for the new type feedback
```

### Where does the engine/runtime live in memory?

```
Process Memory Layout (simplified):
┌──────────────────────────────────────┐
│            Stack Memory              │  ← Call stack, local variables (primitives),
│  (fixed size, ~1-8MB)               │     return addresses, execution context pointers
├──────────────────────────────────────┤
│            Heap Memory               │  ← Objects, arrays, strings, closures,
│  (dynamic, can grow)                │     functions, hidden classes, bytecode,
│                                      │     optimized machine code
│  ┌────────────┐  ┌───────────────┐  │
│  │ Young Gen  │  │  Old Gen      │  │  ← Generational GC
│  │ (Scavenger)│  │ (Mark-Sweep)  │  │
│  └────────────┘  └───────────────┘  │
│  ┌────────────┐  ┌───────────────┐  │
│  │ Code Space │  │  Map Space    │  │  ← JIT code and hidden classes
│  └────────────┘  └───────────────┘  │
├──────────────────────────────────────┤
│          Native Code                 │  ← V8 C++ code, libuv, browser internals
└──────────────────────────────────────┘
```

## ⑤ ADVANTAGES

1. **JIT compilation gives you the best of both worlds:** Fast startup (interpreter) + fast execution (optimized machine code). This is why Node.js can compete with Go/Java for I/O-bound workloads.

2. **Hidden classes + inline caching** make dynamic property access nearly as fast as static languages. Without these, every `obj.x` would require a hash table lookup.

3. **Lazy parsing** means large codebases start fast — V8 only parses what it executes. A 10MB bundle doesn't compile 10MB upfront.

4. **The engine/runtime separation** means the same ECMAScript knowledge works everywhere — browser, server (Node), edge (Cloudflare Workers), mobile (React Native), desktop (Electron).

5. **The TC39 proposal process** ensures backward compatibility. Code written in 1997 still runs in 2026 browsers. No other language of this age has maintained such strict backward compatibility.

6. **Generational GC** is tuned for web workloads — short-lived objects (DOM events, HTTP responses) are collected very quickly in the young generation.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Deoptimization is invisible and costly:**
   ```javascript
   // This function gets deoptimized EVERY time types change:
   function process(input) { return input.value + 1; }
   process({ value: 1 });      // Optimized for {value: Smi}
   process({ value: "hello" }); // DEOPT! Different type
   process({ value: 1, extra: true }); // DEOPT! Different shape
   ```
   **Trap:** In production, polymorphic functions (receiving different shapes) are silently 10x slower.

2. **Hidden class fragmentation:**
   ```javascript
   // BAD — each object gets a DIFFERENT hidden class:
   function makeUser(name, age) {
       const user = {};
       if (age) user.age = age;  // Some objects have 'age', some don't
       user.name = name;         // Property order differs!
       return user;
   }
   
   // GOOD — all objects get the SAME hidden class:
   function makeUser(name, age) {
       return { name: name, age: age || null };  // Always same shape
   }
   ```

3. **`eval` and `with` kill optimization:** V8 cannot optimize any function that uses `eval` or `with` because the scope chain becomes unpredictable.

4. **Single-threaded limitation:** The JS engine runs on one thread. CPU-intensive work blocks everything — rendering, events, I/O callbacks. This is the #1 cause of jank in web apps and event loop starvation in Node.js.

5. **Specification complexity:** ECMAScript is now a 900+ page spec. Edge cases in type coercion, prototype chains, and async behavior create subtle bugs that even senior engineers miss.

6. **Memory leaks are easy:** GC only collects unreachable objects. Closures, global variables, detached DOM nodes, and event listeners can silently keep objects alive forever.

```javascript
// Feature detection — write code that works across runtimes:

const isDeno = typeof Deno !== 'undefined';

// Universal fetch (before Node 18):

```

### Real Production Pattern: V8 Optimization-Friendly Code

```javascript
// In a hot loop serving millions of requests (e.g., a Node.js API):

// ❌ BAD — polymorphic, deoptimizes V8:
function processItem(item) {
    return item.value * 2;
}
processItem({ value: 1 });           // Shape A
processItem({ value: 2, id: "a" });  // Shape B → polymorphic → slow

// ✅ GOOD — monomorphic, V8 stays optimized:
class Item {
    constructor(value, id = null) {
        this.value = value;
        this.id = id;
    }
}
function processItem(item) {
    return item.value * 2;
}
processItem(new Item(1));        // Always same hidden class
processItem(new Item(2, "a"));   // Always same hidden class → monomorphic → fast
```

### Real Production Pattern: Lazy Loading / Code Splitting

```javascript
// In a production React/Next.js app:
// Leverages the fact that V8 lazily parses non-executed code

// Heavy analytics module — only parsed and compiled when user clicks:
button.addEventListener('click', async () => {
    const { initAnalytics } = await import('./analytics.js'); // Dynamic import
    initAnalytics(); // V8 only now fully parses analytics.js
});
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: Is JavaScript interpreted or compiled?**
**A:** Neither in the traditional sense. Modern JavaScript engines use **JIT (Just-In-Time) compilation**. Code is first parsed into bytecode (interpreted), then hot code paths are compiled into optimized machine code at runtime by an optimizing compiler (TurboFan in V8).

**Q2: What is the difference between JavaScript and ECMAScript?**
**A:** ECMAScript is the **specification** (the rules). JavaScript is an **implementation** of that specification plus runtime APIs. `console.log`, `setTimeout`, `fetch`, and the DOM are NOT part of ECMAScript — they're provided by the runtime (browser or Node.js).

**Q3: Name three JavaScript engines and where they're used.**
**A:** V8 (Chrome, Node.js, Deno, Edge), SpiderMonkey (Firefox), JavaScriptCore/Nitro (Safari, Bun).

**Q4: What is strict mode and how do you enable it?**
**A:** Strict mode is a restricted variant of JS that catches common mistakes and prevents unsafe actions. Enable it with `"use strict";` at the top of a file or function. ES Modules and class bodies are always in strict mode.

**Q5: What is the difference between a JavaScript engine and a JavaScript runtime?**
**A:** The **engine** (V8, SpiderMonkey) executes ECMAScript — it parses, compiles, and runs your code. The **runtime** (Chrome, Node.js) wraps the engine and adds platform APIs like the DOM, `setTimeout`, `fs`, HTTP, and the event loop.

---

### 🟡 Intermediate (5)

**Q6: What are hidden classes in V8?**
**A:** Hidden classes (called "Maps" internally in V8) are internal data structures that describe the shape/layout of an object — which properties it has and at what memory offsets. Objects with the same properties in the same order share a hidden class, enabling fast property access via fixed offsets instead of hash lookups.

**Q7: What is deoptimization in V8?**
**A:** When TurboFan (V8's optimizing compiler) compiles a function based on observed type patterns, and later the function receives different types, the assumptions are violated. V8 discards the optimized machine code and falls back to Ignition bytecode. This is called **deoptimization** and can cause significant performance cliffs.

**Q8: What is lazy parsing and why does V8 do it?**
**A:** Lazy parsing (pre-parsing) means V8 only checks function bodies for syntax errors without fully parsing them into AST nodes. It defers full parsing until the function is actually called. This dramatically reduces startup time for large applications since most code paths aren't executed on startup.

**Q9: Why does strict mode help engine optimization?**
**A:** Strict mode eliminates `with` (unpredictable scope), prevents `eval` from injecting variables, forbids `arguments.callee`, and makes `this` undefined in free function calls. All of these make the scope chain and variable resolution statically analyzable, allowing the JIT compiler to generate faster code.

**Q10: What is the TC39 proposal process?**
**A:** TC39 is the committee that evolves ECMAScript. New features go through 5 stages (0–4): Strawperson → Proposal → Draft → Candidate → Finished. A feature reaches Stage 4 only when it has multiple engine implementations and full spec text. This ensures stability and backward compatibility.

---

### 🔴 Advanced / Tricky (5)

**Q11: What is inline caching and how does it affect performance?**
**A:** Inline caching (IC) is V8's optimization for property access. The first time `obj.x` is accessed, V8 does a full hidden class lookup. It then caches the result: "if the hidden class is HC2, property 'x' is at offset 0." Subsequent accesses skip the lookup entirely. IC has states: monomorphic (1 shape, fastest), polymorphic (2–4 shapes, ok), megamorphic (5+ shapes, slow — falls back to generic lookup).

**Q12: Can `console.log` behave differently across runtimes? Why?**
**A:** Yes. `console` is NOT part of ECMAScript — it's a host API. Browser consoles display live object references (which can change), while Node.js serializes at the time of the call. In Cloudflare Workers, `console.log` writes to a different output stream. The behavior, formatting, and even availability of `console` methods are runtime-dependent.

**Q13: If V8 uses JIT compilation, why can't it do AOT like C++?**
**A:** JavaScript is dynamically typed. The expression `a + b` could mean integer addition, float addition, string concatenation, or even a custom `valueOf` call — this is unknowable before runtime. AOT would require compiling ALL possible paths, creating huge and slow code. JIT waits to see actual types, then compiles specialized fast paths. The cost is deoptimization when types change.

**Q14: What happens if you add a property to an object after construction?**
**A:** V8 creates a **new hidden class** for the modified object via a hidden class transition. The original object's hidden class had N property slots; the new one has N+1. This means the modified object no longer shares its hidden class with objects created by the same constructor, breaking monomorphic inline caches and causing performance degradation.

**Q15: How does V8's generational garbage collector affect web application performance?**
**A:** V8 uses generational GC with two spaces: Young Generation (collected via Scavenger, very fast) and Old Generation (collected via Mark-Sweep/Mark-Compact, slower). Short-lived objects (event handlers, temporary computations) die in the young gen cheaply. But if objects survive multiple GC cycles, they're promoted to old gen. Old gen collections can cause noticeable pauses (GC jank). This is why memory leaks that keep objects alive hurt twice: they consume memory AND make GC pauses longer.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step what happens internally when you open a web page with a `<script>` tag.**

**A:**
1. Browser's network layer fetches the HTML document
2. HTML parser encounters `<script src="app.js">`
3. HTML parsing **pauses** (unless `async` or `defer`)
4. Browser fetches `app.js` from the network
5. Source code is handed to the JavaScript engine (V8 in Chrome)
6. **Scanner** tokenizes the source into tokens
7. **Pre-parser** does a quick scan for syntax errors and identifies function boundaries
8. **Parser** builds an AST for the top-level code and immediately-invoked functions
9. **Ignition** generates bytecode from the AST
10. **Global Execution Context** is created:
    - Creation Phase: `var` declarations → `undefined`, `function` declarations → hoisted, `let`/`const` → TDZ, `this` → `window`
    - Execution Phase: bytecode executes top-to-bottom
11. As functions are called, new **Function Execution Contexts** are pushed onto the **Call Stack**
12. When the initial script finishes executing, the **Event Loop** begins processing:
    - Microtask queue (Promises, MutationObserver)
    - Macrotask queue (setTimeout, I/O)
    - Rendering pipeline (requestAnimationFrame, layout, paint)
13. Meanwhile, Ignition collects **type feedback** during execution
14. Hot functions are sent to **TurboFan** for optimization
15. Optimized machine code replaces bytecode for hot paths
16. HTML parser resumes (if it was waiting)

---

**Q17: Explain what happens internally in V8 when you call a function with different argument types across multiple calls.**

**A:**
```javascript
function compute(x) { return x * 2; }
compute(5);        // Call 1
compute(3.14);     // Call 2
compute("hello");  // Call 3
```

**Call 1: `compute(5)`**
- Ignition executes the bytecode for `x * 2`
- `x` is a **Smi** (Small Integer — V8's optimized integer representation, stored directly in the pointer with a tag bit)
- Type feedback records: "parameter `x` was Smi"
- Ignition performs integer multiplication: `5 * 2 = 10`
- If `compute` has been called enough times, V8 marks it as **hot**
- TurboFan compiles `compute` with the assumption: **x is always Smi**
- Generated machine code is an efficient `IMUL` instruction (or shift-left by 1)

**Call 2: `compute(3.14)`**
- `3.14` is a **HeapNumber** (64-bit IEEE 754 double, allocated on the heap)
- This violates TurboFan's assumption that `x` is Smi
- **DEOPTIMIZATION** occurs:
  1. V8 discards the optimized machine code
  2. V8 creates a **deoptimization entry** that maps machine code state back to bytecode state
  3. Execution resumes in Ignition (bytecode interpreter)
  4. Type feedback is updated: "parameter `x` can be Smi OR HeapNumber"
- If `compute` is still hot, TurboFan recompiles with a broader assumption: **x is Number (Smi or HeapNumber)**
- Generated machine code now uses floating-point `FMUL` instruction

**Call 3: `compute("hello")`**
- `"hello"` is a **String** (pointer to a string object on the heap)
- `"hello" * 2` → `NaN` (string-to-number coercion fails)
- This again violates TurboFan's assumption
- **DEOPTIMIZATION** again:
  1. Optimized code is discarded
  2. Type feedback: "parameter `x` can be Smi, HeapNumber, or String"
  3. If the function keeps receiving different types, V8 may give up optimizing and stay in Ignition
  4. The inline cache for `x * 2` becomes **megamorphic** — V8 uses a generic, slow multiply operation

**Performance Impact:**
- Call 1 alone: ~1 nanosecond (optimized integer multiply)
- After Call 3: ~50-100 nanoseconds (generic multiply with type checks)
- That's a **50-100x slowdown** from mixing types
