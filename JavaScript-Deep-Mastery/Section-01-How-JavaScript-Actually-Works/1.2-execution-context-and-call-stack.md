# 📌 1.2 — Execution Context & Call Stack
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** An Execution Context is a container that holds all the information needed to run a piece of JavaScript code — what variables exist, what `this` refers to, and what scope chain to follow. The Call Stack is a data structure that keeps track of which execution context is currently running, stacking them like plates when functions call other functions.

**Technical Definition:** An execution context is an abstract concept from the ECMAScript specification (§9.4) that encapsulates the state of running code. It contains:
- **Variable Environment** — bindings for `var` declarations
- **Lexical Environment** — bindings for `let`, `const`, function declarations
- **ThisBinding** — the value of `this`
- **Outer Environment Reference** — link to the parent scope (for scope chain)

The **Call Stack** (also called the Execution Context Stack) is a LIFO (Last-In, First-Out) data structure that tracks the currently executing context and all suspended contexts waiting for a return.

**One-line mental model:**
> *An execution context is "everything a piece of code needs to run." The call stack is "the queue of who's waiting for whom."*

**Where does this live?**
- Execution contexts are **conceptual structures** defined by the spec — but at runtime, they manifest as **stack frames** on the call stack (stored in stack memory)
- The call stack itself lives in **stack memory** (a fixed-size memory region, typically 1–8 MB)
- Variable bindings for primitives → stack memory
- Variable bindings for objects → reference on stack, object on heap

## ② DRILL DOWN

### 🔹 A. Three Types of Execution Contexts

#### 1. Global Execution Context (GEC)

Created when the JavaScript program starts. There's **exactly one** per program.

```javascript
// This code runs in the Global Execution Context
var name = "Alice";
let age = 30;
function greet() { return "Hello"; }

// In browser: `this` === `window`
// In Node.js: `this` === `module.exports` (not `global`!)
// In strict mode: `this` === `undefined` (in function scope, not global)
```

**GEC Creation:**
```
Global Execution Context:
┌─────────────────────────────────────────────┐
│  Variable Environment:                       │
│    name → undefined        (var, hoisted)    │
│                                              │
│  Lexical Environment:                        │
│    age → <TDZ>             (let, TDZ)        │
│    greet → function(){}    (function, hoisted)│
│                                              │
│  ThisBinding: window (browser) / {}  (Node)  │
│  Outer: null (no parent — this IS the top)   │
└─────────────────────────────────────────────┘
```

#### 2. Function Execution Context (FEC)

Created every time a function is **invoked** (not declared). Each call creates a new, independent context.

```javascript
function multiply(a, b) {
    let result = a * b;
    return result;
}

multiply(3, 4);  // Creates FEC #1
multiply(5, 6);  // Creates FEC #2 (completely independent of #1)
```

**FEC for `multiply(3, 4)`:**
```
Function Execution Context (multiply):
┌─────────────────────────────────────────────┐
│  Variable Environment:                       │
│    (nothing — no var declarations)           │
│                                              │
│  Lexical Environment:                        │
│    a → 3                                     │
│    b → 4                                     │
│    result → <TDZ> then 12                    │
│                                              │
│  ThisBinding: window (sloppy) / undefined    │
│              (strict)                        │
│  Outer: → Global Lexical Environment         │
└─────────────────────────────────────────────┘
```

#### 3. Eval Execution Context

Created when `eval()` is called. **Never use this in production.**

```javascript
eval("var x = 10;");  // Creates an Eval Execution Context
// In sloppy mode: x leaks into the surrounding scope
// In strict mode: x stays within the eval context
```

**Why eval is dangerous:**
- Breaks V8 optimizations (scope becomes unpredictable)
- Potential security vulnerability (code injection)
- Makes debugging nearly impossible
- V8 marks any function containing `eval` as "not optimizable"

### 🔹 B. Creation Phase vs Execution Phase

Every execution context goes through **two phases**:

#### Phase 1: Creation Phase (before any code runs)

```javascript
console.log(x);     // undefined (not ReferenceError!)
console.log(y);     // ReferenceError: Cannot access 'y' before initialization
console.log(greet); // function greet() { return "hi"; }

var x = 10;
let y = 20;
function greet() { return "hi"; }
```

During creation phase:
1. **`var` declarations** → allocated in Variable Environment, initialized to `undefined`
2. **`let`/`const` declarations** → allocated in Lexical Environment, marked as `<uninitialized>` (TDZ)
3. **Function declarations** → fully hoisted — the entire function body is available
4. **`this`** binding is determined
5. **Outer Environment Reference** is set (scope chain link)

```
CREATION PHASE DIAGRAM:
┌──────────────────────────────────────────────┐
│  Variable Environment:                        │
│    x → undefined           ← var hoisted     │
│                                               │
│  Lexical Environment:                         │
│    y → <uninitialized>     ← let in TDZ      │
│    greet → function() {}   ← fully hoisted   │
│                                               │
│  this → window                                │
└──────────────────────────────────────────────┘
```

#### Phase 2: Execution Phase (code runs line by line)

```
Line 1: console.log(x)     → logs undefined (x exists but = undefined)
Line 2: console.log(y)     → ReferenceError! (y is in TDZ)
Line 3: console.log(greet) → logs the function (fully hoisted)
Line 5: x = 10             → x updated from undefined to 10
Line 6: y = 20             → y exits TDZ, set to 20
Line 7: greet already defined (no-op — already done in creation phase)
```

### 🔹 C. Variable Environment vs Lexical Environment

These are the two record-keeping structures inside every execution context.

**Variable Environment:**
- Stores `var` declarations and `function` declarations
- `var` bindings are initialized to `undefined` during creation
- In ES5, Variable Environment and Lexical Environment were the same thing

**Lexical Environment:**
- Stores `let`, `const`, and `function` declarations (in ES6+)
- `let`/`const` bindings are "uninitialized" during creation (TDZ)
- Has a reference to the **outer lexical environment** (this creates the scope chain)

```javascript
function outer() {
    var a = 1;       // Variable Environment of outer
    let b = 2;       // Lexical Environment of outer
    
    {
        var c = 3;   // Variable Environment of outer (var ignores blocks!)
        let d = 4;   // NEW Lexical Environment for this block
        const e = 5; // Same new Lexical Environment
    }
    
    console.log(a);  // 1
    console.log(b);  // 2
    console.log(c);  // 3 (var hoists to function scope)
    console.log(d);  // ReferenceError (block-scoped, dead)
}
```

**Memory diagram showing the block scope difference:**
```
outer() Execution Context:
┌──────────────────────────────────────────────┐
│  Variable Environment:                        │
│    a → 1                                      │
│    c → 3       ← var is function-scoped!      │
│                                               │
│  Lexical Environment (function scope):        │
│    b → 2                                      │
│    Outer → Global Env                         │
│    ┌─────────────────────────────────────┐    │
│    │ Lexical Environment (block scope):  │    │
│    │   d → 4                             │    │
│    │   e → 5                             │    │
│    │   Outer → outer's Lexical Env       │    │
│    └─────────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

### 🔹 D. Scope Chain

The scope chain is formed by the chain of `Outer Environment References` from the current context back to the global context.

```javascript
const global = "I'm global";

function outer() {
    const outerVar = "I'm outer";
    
    function inner() {
        const innerVar = "I'm inner";
        console.log(innerVar);  // Found in inner's environment
        console.log(outerVar);  // Not in inner → follows Outer → found in outer's env
        console.log(global);    // Not in inner or outer → follows Outer → found in global
        console.log(missing);   // Not found anywhere → ReferenceError
    }
    
    inner();
}
outer();
```

**Scope Chain Lookup Algorithm:**
```
1. Look in the current Lexical Environment
2. If not found → follow the Outer Reference to the parent Lexical Environment
3. Repeat until found or reach Global Environment
4. If not found in Global → ReferenceError (in strict mode)
                          → Creates global variable (in sloppy mode, for assignments)
```

**ASCII diagram of the scope chain:**
```
inner's Env ──Outer──► outer's Env ──Outer──► Global Env ──Outer──► null
│                      │                      │
│ innerVar: "I'm       │ outerVar: "I'm      │ global: "I'm
│           inner"     │           outer"     │         global"
```

**Critical Detail:** The scope chain is determined by **where functions are defined** (lexical scoping), NOT where they are called (dynamic scoping). This is what makes closures possible.

```javascript
function createGreeter() {
    const greeting = "Hello";  // Defined in createGreeter's scope
    
    return function() {
        console.log(greeting); // Looks up scope chain → finds "Hello"
    };
}

const greet = createGreeter();
greet(); // "Hello" — even though createGreeter has returned!
// The inner function's Outer Reference still points to createGreeter's Lexical Environment
```

### 🔹 E. `this` Binding

`this` is determined during the **creation phase** of each execution context. The rules are:

| Context | `this` value (sloppy mode) | `this` value (strict mode) |
|---------|---------------------------|---------------------------|
| Global | `window` (browser) / `global` (Node) | `window` / `global` |
| Function call (`fn()`) | `window` / `global` | `undefined` |
| Method call (`obj.fn()`) | `obj` | `obj` |
| `new` constructor | New empty object | New empty object |
| Arrow function | Inherited from enclosing context | Inherited from enclosing context |
| `call`/`apply`/`bind` | Explicit first argument | Explicit first argument |
| Event handler | The DOM element | The DOM element |

```javascript
// 1. Global this
console.log(this); // window (browser) / {} (Node module)

// 2. Function call — the biggest trap
function showThis() {
    console.log(this);
}
showThis();            // window (sloppy) / undefined (strict)

// 3. Method call
const obj = {
    name: "Alice",
    greet() { console.log(this.name); }
};
obj.greet();           // "Alice" — this is obj

// 4. The classic trap — losing `this`
const fn = obj.greet;
fn();                  // undefined — this is window/undefined (not obj!)

// 5. new binding
function Person(name) {
    this.name = name;  // this = newly created object
}
const p = new Person("Bob"); // p.name === "Bob"

// 6. Arrow function — lexical this
const obj2 = {
    name: "Carol",
    greet: () => console.log(this.name),      // this = enclosing scope (NOT obj2!)
    greetCorrect() {
        const arrow = () => console.log(this.name); // this = obj2 (inherited from greetCorrect)
        arrow();
    }
};
obj2.greet();         // undefined — arrow inherited from global/module scope
obj2.greetCorrect();  // "Carol" — arrow inherited from greetCorrect's this
```

### 🔹 F. Call Stack Internals

The call stack is a **LIFO** (Last-In, First-Out) data structure that the JavaScript engine uses to manage execution contexts.

```javascript
function first() {
    console.log("first");
    second();
    console.log("first done");
}

function second() {
    console.log("second");
    third();
    console.log("second done");
}

function third() {
    console.log("third");
}

first();
```

**Call Stack Progression:**

```
Step 1: Program starts
┌─────────────────┐
│ Global Context   │  ← current
└─────────────────┘

Step 2: first() called
┌─────────────────┐
│ first() Context  │  ← current
├─────────────────┤
│ Global Context   │
└─────────────────┘

Step 3: second() called inside first()
┌─────────────────┐
│ second() Context │  ← current
├─────────────────┤
│ first() Context  │
├─────────────────┤
│ Global Context   │
└─────────────────┘

Step 4: third() called inside second()
┌─────────────────┐
│ third() Context  │  ← current
├─────────────────┤
│ second() Context │
├─────────────────┤
│ first() Context  │
├─────────────────┤
│ Global Context   │
└─────────────────┘

Step 5: third() returns
┌─────────────────┐
│ second() Context │  ← current (resumes after third() call)
├─────────────────┤
│ first() Context  │
├─────────────────┤
│ Global Context   │
└─────────────────┘

Step 6: second() returns
┌─────────────────┐
│ first() Context  │  ← current (resumes after second() call)
├─────────────────┤
│ Global Context   │
└─────────────────┘

Step 7: first() returns
┌─────────────────┐
│ Global Context   │  ← current
└─────────────────┘

Output: "first", "second", "third", "second done", "first done"
```

### 🔹 G. Stack Overflow

The call stack has a **fixed size** (typically ~10,000–25,000 frames, depending on the engine and frame size). If you exceed it, you get a `RangeError: Maximum call stack size exceeded`.

```javascript
// Classic stack overflow — infinite recursion
function infinite() {
    infinite();  // Each call adds a frame, never returns
}
infinite(); // RangeError: Maximum call stack size exceeded

// Subtle stack overflow — large but not infinite recursion
function factorial(n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}
factorial(100000);  // Stack overflow! Too many frames

// Fix with iteration:
function factorialIterative(n) {
    let result = 1n; // BigInt for large numbers
    for (let i = 2n; i <= n; i++) {
        result *= i;
    }
    return result;
}
factorialIterative(100000n); // Works! No stack frames accumulated
```

**How V8 handles stack overflow:**
1. Each function call pushes a **stack frame** containing:
   - Return address
   - Parameters
   - Local variables
   - Saved registers
2. V8 checks the **stack pointer** against the **stack limit** before each call
3. If the stack pointer would exceed the limit → throws `RangeError`
4. The stack is unwound (all frames are popped), and the error propagates

**Tail Call Optimization (TCO):**
ES6 specified TCO — if the last thing a function does is call another function, the engine can reuse the current stack frame. However, only **JavaScriptCore (Safari)** implements this. V8 and SpiderMonkey do NOT implement TCO.

```javascript
// Tail position — theoretically optimizable:
function factorial(n, acc = 1) {
    if (n <= 1) return acc;
    return factorial(n - 1, n * acc); // Tail call — last thing is a function call
}

// Non-tail position — NOT optimizable:
function factorial(n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1); // Not tail call — multiplication happens AFTER the call
}
```

## ③ WHY IT EXISTS

**What problem does it solve?**

JavaScript needed a way to:
1. **Track what code is running** — especially when functions call other functions
2. **Manage variable scope** — each function should have its own isolated variables
3. **Handle nested calls** — the engine needs to remember where to return after a function finishes
4. **Support closures** — inner functions need access to outer variables even after the outer function returns

Without execution contexts: There would be no way to have local variables. Everything would be global. Functions couldn't call functions. There would be no scope isolation.

**Historical context:**
- In the 1990s, JavaScript had only `var` and function scope. Execution contexts were simpler (no TDZ, no block scope).
- ES3 (1999) formalized execution contexts with VariableObject and ActivationObject.
- ES5 (2009) refined the model with Lexical Environment and Variable Environment.
- ES6 (2015) made it more complex: block scoping (`let`/`const`) required nested Lexical Environments within the same function.

**Comparison with other languages:**

| Language | Call Stack | Scope Model | Variable Lookup |
|----------|-----------|-------------|-----------------|
| JavaScript | Fixed-size, no TCO (except Safari) | Lexical scoping, scope chain | Chain of Lexical Environments |
| Java | Fixed-size per thread, no TCO | Block scoping (simpler — static types) | Direct offset from stack frame |
| C++ | Fixed-size, supports TCO | Block scoping | Direct offset (compile-time known) |
| Python | Fixed-size (default 1000 frames) | LEGB rule (Local, Enclosing, Global, Built-in) | Dictionary lookup per scope |

## ④ INTERNAL WORKING

### What does V8 actually do?

When V8 creates an execution context, it doesn't literally create the spec's abstract "Lexical Environment" object. Instead, it:

1. **Allocates a stack frame** on the native call stack
2. **Stores local variables** (that don't escape to closures) directly in the stack frame registers
3. **Creates a Context object** on the heap ONLY if the function has closures or `eval`
4. **Links** the Context object to the parent Context (this IS the scope chain)

```
Stack Frame (for a simple function with no closures):
┌────────────────────────┐
│ Return address          │
│ Saved frame pointer     │
│ Param 1: a = 3         │
│ Param 2: b = 4         │
│ Local: result = 12     │  ← All on the stack! Fast!
└────────────────────────┘

Context Object (for a function with closures — on the heap):
┌────────────────────────────────┐
│ Context {                      │
│   parent: → outer Context      │  ← Scope chain link
│   slot[0]: closedOverVar = 42  │  ← Variables captured by closure
│ }                              │
└────────────────────────────────┘
```

**Key optimization:** V8's escape analysis determines if variables are captured by closures. If not, they stay on the stack (fast). If yes, they're "context-allocated" on the heap (slightly slower, but necessary for closure correctness).

```javascript
function noEscape() {
    let x = 5;    // x stays on the stack — fast
    return x + 1;
}

function escapes() {
    let x = 5;    // x is heap-allocated in a Context object
    return () => x; // Because this inner function captures x
}
```

### Call Stack in the CPU

The JavaScript call stack maps to the actual CPU call stack:
- Each function call → `CALL` instruction on x86/ARM
- Return address pushed onto hardware stack
- Local variables stored in CPU registers or stack slots
- The stack grows downward in memory (on most architectures)

```
High memory
┌───────────────────┐
│   ...              │
├───────────────────┤
│ Global frame       │
├───────────────────┤
│ first() frame      │
├───────────────────┤
│ second() frame     │
├───────────────────┤
│ third() frame      │  ← Stack Pointer (SP)
├───────────────────┤
│                    │
│   (unused stack)   │
│                    │
├───────────────────┤
│ Stack Guard Page   │  ← Stack Limit (touching this → stack overflow)
└───────────────────┘
Low memory
```

## ⑤ ADVANTAGES

1. **Predictable variable isolation:** Each function call gets its own execution context. Variables in one function cannot accidentally interfere with another. This is why recursion works correctly — each recursive call has its own `n`, `result`, etc.

2. **Lexical scoping enables closures:** The scope chain (Outer Environment Reference) is set at function **definition time**, not call time. This makes closures natural and reliable — one of JavaScript's most powerful features.

3. **Efficient memory use:** V8's escape analysis keeps most local variables on the stack (fast allocation, automatic deallocation). Only closure-captured variables go to the heap.

4. **Debugging clarity:** Stack traces map directly to the call stack. When an error occurs, `Error.stack` gives you the exact chain of function calls that led to the error.

5. **Security through isolation:** Each execution context has its own scope. Without this, user input in one function could corrupt variables in another.

## ⑥ DISADVANTAGES / TRAPS

1. **Stack overflow in recursion:**
   ```javascript
   // Deep recursion crashes — there's no automatic trampoline or TCO (in V8)
   function deepRecurse(n) {
       if (n === 0) return;
       deepRecurse(n - 1);
   }
   deepRecurse(100000); // RangeError: Maximum call stack size exceeded
   ```

2. **`this` binding confusion — the #1 source of JavaScript bugs:**
   ```javascript
   class Timer {
       constructor() {
           this.seconds = 0;
       }
       start() {
           setInterval(function() {
               this.seconds++; // BUG! `this` is `window`, not the Timer instance
           }, 1000);
       }
   }
   
   // Fix 1: Arrow function
   start() {
       setInterval(() => {
           this.seconds++; // Correct — arrow inherits `this` from start()
       }, 1000);
   }
   
   // Fix 2: bind
   start() {
       setInterval(function() {
           this.seconds++;
       }.bind(this), 1000);
   }
   ```

3. **Creation phase hoisting surprises:**
   ```javascript
   var x = 1;
   function test() {
       console.log(x); // undefined, NOT 1!
       var x = 2;       // This `var x` is hoisted inside test's context
   }
   test();
   // The local `var x` shadows the global `x`, and during creation phase it's `undefined`
   ```

4. **Memory leaks from execution context retention:**
   ```javascript
   function createHandler() {
       const hugeData = new Array(1000000).fill("x"); // 1M elements
       return function handler() {
           // handler captures hugeData's execution context
           // Even if handler never uses hugeData, some engines keep the entire context alive
           console.log("handled");
       };
   }
   const fn = createHandler(); // hugeData may never be garbage collected!
   ```

5. **Async code breaks stack trace continuity:**
   ```javascript
   async function a() { await b(); }
   async function b() { await c(); }
   async function c() { throw new Error("oops"); }
   
   a().catch(e => console.log(e.stack));
   // Stack trace may not show a() → b() → c() clearly
   // Because each `await` suspends and resumes in a new microtask
   ```

## ⑦ PRODUCTION USAGE

### Pattern 1: Understanding Call Stack in Error Monitoring

```javascript
// Production error tracking (Sentry, Datadog pattern):
class AppError extends Error {
    constructor(message, context = {}) {
        super(message);
        this.name = 'AppError';
        this.context = context;
        // Error.captureStackTrace removes the AppError constructor from the trace
        // so the stack starts from where the error was thrown
        if (Error.captureStackTrace) {
            Error.captureStackTrace(this, AppError);
        }
    }
}

// Usage in production:
async function handleRequest(req) {
    try {
        const user = await fetchUser(req.userId);
        if (!user) throw new AppError('User not found', { userId: req.userId });
        return user;
    } catch (error) {
        // error.stack contains the full call chain
        logger.error({ stack: error.stack, context: error.context });
        throw error;
    }
}
```

### Pattern 2: Avoiding Stack Overflow in Production

```javascript
// Processing large arrays — trampoline pattern to avoid stack overflow:
function trampoline(fn) {
    return function(...args) {
        let result = fn(...args);
        while (typeof result === 'function') {
            result = result();
        }
        return result;
    };
}

const sumArray = trampoline(function sum(arr, index = 0, acc = 0) {
    if (index >= arr.length) return acc;
    return () => sum(arr, index + 1, acc + arr[index]); // Return thunk, not recursive call
});

const hugeArray = new Array(1000000).fill(1);
console.log(sumArray(hugeArray)); // 1000000 — no stack overflow!
```

### Pattern 3: `this` Binding in React/Event Handlers

```javascript
// Modern React — arrow functions in class components:
class SearchComponent extends React.Component {
    state = { query: '' };

    // Arrow function auto-binds `this` (uses enclosing context's `this`)
    handleSearch = (event) => {
        this.setState({ query: event.target.value });
        // `this` correctly refers to the SearchComponent instance
        // because arrow functions capture `this` from the class field context
    };

    render() {
        return <input onChange={this.handleSearch} />;
    }
}
```

### Pattern 4: Execution Context in Module Systems

```javascript
// Node.js CommonJS — each module gets its own execution context:
// file: module-a.js
console.log(this);         // {} (module.exports)
console.log(typeof window); // "undefined" — not in global scope

// Node wraps each module in a function:
// (function(exports, require, module, __filename, __dirname) {
//     // Your module code here
// });
// This creates a Function Execution Context for each module
// That's why `var` in a module doesn't become global
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is an execution context?**
**A:** An execution context is the environment in which JavaScript code is evaluated and executed. It contains the variable bindings (Variable Environment and Lexical Environment), the `this` binding, and a reference to the outer scope. There are three types: Global, Function, and Eval execution contexts.

**Q2: What is the call stack?**
**A:** The call stack is a LIFO data structure that tracks the currently executing function and all functions waiting for their callees to return. When a function is called, its execution context is pushed onto the stack. When it returns, it's popped off.

**Q3: What happens during the creation phase of an execution context?**
**A:** During the creation phase: (1) `var` variables are allocated and initialized to `undefined`, (2) `let`/`const` variables are allocated but left uninitialized (TDZ), (3) function declarations are fully hoisted with their bodies, (4) `this` binding is determined, (5) the outer scope reference is set.

**Q4: What causes a stack overflow?**
**A:** A stack overflow occurs when the call stack exceeds its maximum size, typically due to infinite or very deep recursion. Each function call consumes stack space, and when the stack pointer exceeds the stack limit, the engine throws `RangeError: Maximum call stack size exceeded`.

**Q5: Why does `console.log(x)` print `undefined` instead of throwing an error when `var x = 5` is declared later?**
**A:** Because of hoisting. During the creation phase, `var x` is allocated in the Variable Environment and initialized to `undefined`. The assignment `x = 5` happens during the execution phase, which hasn't reached that line yet. So `x` exists but equals `undefined`.

### 🟡 Intermediate (5)

**Q6: What is the difference between Variable Environment and Lexical Environment?**
**A:** Variable Environment stores `var` declarations (function-scoped, initialized to `undefined`). Lexical Environment stores `let`/`const` declarations (block-scoped, subject to TDZ). In ES6+, blocks create new Lexical Environments but share the same Variable Environment as their enclosing function.

**Q7: How does the scope chain work?**
**A:** Each Lexical Environment has an Outer Reference pointing to its parent Lexical Environment. When a variable is accessed, the engine looks in the current Lexical Environment first. If not found, it follows the Outer Reference to the parent, repeating until it reaches the Global Environment. If still not found, it throws a `ReferenceError`.

**Q8: Why is `this` different in arrow functions vs regular functions?**
**A:** Regular functions determine `this` at **call time** based on how they're invoked (global, method, new, explicit binding). Arrow functions capture `this` from their **enclosing execution context** at definition time. Arrow functions don't have their own `this` binding — they close over the surrounding `this`.

**Q9: What is the Temporal Dead Zone (TDZ)?**
**A:** The TDZ is the period between entering a scope where a `let`/`const` variable is declared and the line where it's actually initialized. During this zone, accessing the variable throws a `ReferenceError`. It exists to catch bugs where developers use variables before they're assigned.

**Q10: Can you have multiple execution contexts for the same function?**
**A:** Yes! Every function **call** creates a new, independent execution context. If `factorial(5)` recursively calls `factorial(4)`, there are two separate execution contexts for `factorial` on the call stack simultaneously, each with their own local variables.

### 🔴 Advanced / Tricky (5)

**Q11: What will this code output and why?**
```javascript
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
```
**A:** Output: `3, 3, 3`. Because `var i` is in the function/global scope (not block-scoped). All three closures share the same `i`. By the time the setTimeout callbacks execute (after the loop finishes), `i` is 3. Fix: use `let` (creates a new Lexical Environment per iteration) or an IIFE.

**Q12: How does V8 decide whether to stack-allocate or heap-allocate a variable?**
**A:** V8 performs **escape analysis**. If a variable is only used within its declaring function and is NOT captured by any inner function (closure), it's stack-allocated. If it IS captured by a closure, V8 creates a **Context object** on the heap to store it, because the closure may outlive the function's stack frame.

**Q13: What is context allocation and how does it affect performance?**
**A:** Context allocation is when V8 allocates variables on the heap (in a Context object) instead of the stack, because they're captured by closures. This is slower because: (1) heap allocation involves the GC, (2) variable access requires an extra indirection through the context pointer, (3) the Context object stays alive as long as any closure referencing it is alive, potentially causing memory retention.

**Q14: Explain the output:**
```javascript
function foo() {
    console.log(this);
}
const obj = { foo };
const bar = obj.foo;
obj.foo();  // ?
bar();      // ?
new foo();  // ?
```
**A:** `obj.foo()` → `this` is `obj` (implicit binding — method call). `bar()` → `this` is `window` (sloppy mode) or `undefined` (strict mode) — the method reference is detached. `new foo()` → `this` is a new empty object created by `new`.

**Q15: Why can't V8 implement Tail Call Optimization (TCO)?**
**A:** While ES6 specifies TCO, V8 chose not to implement it because: (1) it breaks `Error.stack` traces — tail-called frames are removed, making debugging harder, (2) it changes observable behavior in DevTools, (3) the V8 team argued that explicit opt-in (like a `return continue` syntax) would be better than implicit TCO. Only JavaScriptCore (Safari) implements TCO.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Walk through exactly what happens in V8 when `let x = 5` is encountered inside a function.**

**A:**
1. **Parsing:** The parser sees `let x = 5` and creates a `VariableDeclaration` AST node with kind `let`, declarator `x`, and initializer `NumericLiteral(5)`.
2. **Scope analysis:** The compiler's scope analyzer marks `x` as a let-binding in the current block scope. It checks for redeclaration errors. It performs escape analysis — if `x` is not captured by any closure, it will be register-allocated.
3. **Bytecode generation (Ignition):** Ignition generates:
   - A slot in the stack frame for `x` (or in a Context object if captured by closure)
   - `x` is marked as `<the_hole>` (V8's internal representation of TDZ)
   - When execution reaches `let x = 5`:
     - `LdaSmi [5]` — load small integer 5 into the accumulator
     - `Star r0` — store into register r0 (the slot for x)
     - The TDZ mark is removed
4. **If `x` is accessed before the `let` line:** V8 checks if the value is `<the_hole>`. If yes, it throws `ReferenceError: Cannot access 'x' before initialization`.
5. **Memory:** If stack-allocated, `x` occupies a register in the stack frame. If context-allocated, `x` is a slot in a heap-allocated Context object.

**Q17: Explain exactly what happens to the call stack when an async function awaits.**

**A:**
```javascript
async function fetchData() {
    console.log("start");
    const data = await fetch("/api/data"); // await point
    console.log("done");
    return data;
}
fetchData();
console.log("after call");
```

1. `fetchData()` is called → a new execution context is pushed onto the call stack
2. `console.log("start")` executes → "start" is logged
3. `fetch("/api/data")` is called → returns a **Promise** → the fetch starts in the browser's network layer
4. `await` is encountered:
   a. The **current execution context for fetchData is SUSPENDED** (saved to the heap, not the stack)
   b. The function returns an **implicit Promise** to the caller
   c. fetchData's frame is **popped from the call stack**
   d. A `.then()` handler is attached to the fetch Promise internally
5. Execution continues after `fetchData()` → `console.log("after call")` runs → "after call" is logged
6. The call stack is now **empty** (back to the event loop)
7. When the fetch Promise resolves:
   a. A **microtask** is queued to resume fetchData
   b. The event loop picks up the microtask
   c. fetchData's **saved execution context is restored** (pushed back onto the stack)
   d. Execution resumes after the `await` — `data` receives the resolved value
   e. `console.log("done")` executes → "done" is logged
   f. fetchData returns → its implicit Promise resolves

**Output:** "start", "after call", "done"

**Key insight:** `await` does NOT block the call stack. It **suspends** the async function (saving state to the heap), **pops** it from the stack, and **resumes** it later via the microtask queue. The call stack is free to process other work in between.
