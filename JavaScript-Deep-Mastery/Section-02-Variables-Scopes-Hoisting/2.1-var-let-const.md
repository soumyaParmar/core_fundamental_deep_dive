# 📌 2.1 — var, let, const
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** `var`, `let`, and `const` are the three ways to declare variables in JavaScript. They differ in **scope** (where the variable is accessible), **hoisting** (what happens before the declaration line), **reassignment** (can you change the value), and **redeclaration** (can you declare it again in the same scope).

**Technical Definition:**
- `var` — creates a function-scoped or globally-scoped binding, hoisted with initialization to `undefined`, allows reassignment and redeclaration
- `let` — creates a block-scoped binding, hoisted but NOT initialized (TDZ), allows reassignment, no redeclaration
- `const` — creates a block-scoped binding, hoisted but NOT initialized (TDZ), NO reassignment, no redeclaration. The binding is immutable, not the value.

**One-line mental model:**
> *Use `const` by default, `let` when you need to reassign, and never use `var` in modern JavaScript.*

**Where does this live?**
- `var` bindings → Variable Environment of the enclosing function (or Global object for top-level)
- `let`/`const` bindings → Lexical Environment of the enclosing block
- Values: Primitives → stack; Objects → heap (regardless of var/let/const)

## ② DRILL DOWN

### 🔹 A. Complete Comparison Table

| Feature | `var` | `let` | `const` |
|---------|-------|-------|---------|
| Scope | Function (or Global) | Block | Block |
| Hoisting | Yes, initialized to `undefined` | Yes, but NOT initialized (TDZ) | Yes, but NOT initialized (TDZ) |
| Reassignment | ✅ Yes | ✅ Yes | ❌ No (TypeError) |
| Redeclaration | ✅ Yes (silent) | ❌ No (SyntaxError) | ❌ No (SyntaxError) |
| Creates window property (global) | ✅ Yes | ❌ No | ❌ No |
| TDZ (Temporal Dead Zone) | ❌ No | ✅ Yes | ✅ Yes |
| Required initializer | ❌ No | ❌ No | ✅ Yes |
| ES version | ES1 (1997) | ES6 (2015) | ES6 (2015) |

### 🔹 B. Scope Behavior (The Critical Difference)

```javascript
// === var is FUNCTION-scoped ===
function testVar() {
    if (true) {
        var x = 10;  // x belongs to testVar's scope, NOT the if-block
    }
    console.log(x); // 10 — x is accessible outside the block!
}

// === let is BLOCK-scoped ===
function testLet() {
    if (true) {
        let y = 10;  // y belongs to the if-block's scope
    }
    console.log(y); // ReferenceError: y is not defined
}

// === const is BLOCK-scoped (same as let) ===
function testConst() {
    if (true) {
        const z = 10;
    }
    console.log(z); // ReferenceError: z is not defined
}
```

**var in for loops — the classic problem:**
```javascript
// var shares ONE variable across all iterations:
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
}
// Output: 3, 3, 3 (all closures share the same i, which is 3 after the loop)

// let creates a NEW variable per iteration:
for (let i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2 (each closure captures its own i)
```

**Memory perspective of let in for loops:**
```
Iteration 0: Block Scope { i: 0 } ← closure 0 captures this
Iteration 1: Block Scope { i: 1 } ← closure 1 captures this
Iteration 2: Block Scope { i: 2 } ← closure 2 captures this

With var: Function Scope { i: 3 } ← ALL closures capture this one variable
```

### 🔹 C. Hoisting Deep Dive

```javascript
console.log(a); // undefined — var is hoisted AND initialized
console.log(b); // ReferenceError — let is hoisted but NOT initialized (TDZ)
console.log(c); // ReferenceError — const is hoisted but NOT initialized (TDZ)

var a = 1;
let b = 2;
const c = 3;
```

**What the engine sees (conceptually):**
```javascript
// Creation phase:
var a = undefined;   // Hoisted and initialized
// let b → TDZ (exists but uninitialized)
// const c → TDZ (exists but uninitialized)

// Execution phase:
console.log(a); // undefined
console.log(b); // ReferenceError (TDZ)
console.log(c); // ReferenceError (TDZ)
a = 1;
b = 2;          // TDZ ends for b
c = 3;          // TDZ ends for c
```

**Proof that let/const ARE hoisted (not just undeclared):**
```javascript
let x = "outer";
{
    console.log(x); // ReferenceError! NOT "outer"
    let x = "inner";
}
// If let weren't hoisted, console.log(x) would print "outer" (from the outer scope).
// The error proves that the inner `let x` was hoisted — JavaScript KNOWS about it
// but it's in the TDZ, so access throws.
```

### 🔹 D. Reassignment and Redeclaration

```javascript
// === var: reassign ✅, redeclare ✅ ===
var x = 1;
var x = 2;  // No error — redeclaration is allowed (silently)
x = 3;      // Reassignment works

// === let: reassign ✅, redeclare ❌ ===
let y = 1;
let y = 2;  // SyntaxError: Identifier 'y' has already been declared
y = 3;      // Reassignment works

// === const: reassign ❌, redeclare ❌ ===
const z = 1;
const z = 2; // SyntaxError
z = 3;       // TypeError: Assignment to constant variable

// BUT: const with objects — the VALUE is mutable:
const obj = { x: 1 };
obj.x = 99;         // ✅ Works! Mutating the object is fine
obj.y = 100;         // ✅ Works! Adding properties is fine
obj = { x: 2 };     // ❌ TypeError — reassigning the binding is not
```

### 🔹 E. Global Scope Differences

```javascript
// In a browser's global scope:
var a = 1;
let b = 2;
const c = 3;

console.log(window.a); // 1 — var creates a property on window
console.log(window.b); // undefined — let does NOT create a window property
console.log(window.c); // undefined — const does NOT create a window property

// In Node.js top-level (CommonJS module):
var d = 4;
console.log(global.d); // undefined — Node wraps modules in a function
                        // so var is function-scoped to the module wrapper
```

### 🔹 F. Every Edge Case and Gotcha

```javascript
// 1. var hoisting in different positions:
function test() {
    console.log(x); // undefined (hoisted)
    if (false) {
        var x = 5;   // Never executes, but STILL hoisted!
    }
    console.log(x); // undefined (declared but never assigned)
}

// 2. const must have an initializer:
const x;  // SyntaxError: Missing initializer in const declaration
let y;    // OK — y is undefined
var z;    // OK — z is undefined

// 3. TDZ with typeof:
console.log(typeof undeclared); // "undefined" — no error for truly undeclared
console.log(typeof tdzVar);    // ReferenceError! TDZ beats typeof
let tdzVar = 1;

// 4. TDZ in function default parameters:
function f(a = b, b = 1) {
    // When evaluating `a = b`, b is in TDZ!
    // ReferenceError: Cannot access 'b' before initialization
}
f();

// But this works:
function g(a = 1, b = a) {
    // a is already initialized when b uses it
    console.log(a, b); // 1, 1
}
g();

// 5. var in switch statements:
switch (true) {
    case true:
        var x = 1;  // x is function-scoped — visible everywhere in the function
        break;
}
console.log(x); // 1

// 6. let in switch statements:
switch (action) {
    case 'A':
        let result = 1; // result is scoped to the entire switch block!
        break;
    case 'B':
        let result = 2; // SyntaxError! result already declared in this block
        break;
}

// Fix: use explicit blocks
switch (action) {
    case 'A': {
        let result = 1; // scoped to this block
        break;
    }
    case 'B': {
        let result = 2; // different block — OK
        break;
    }
}

// 7. Re-declaring var in the same scope is fine:
var x = 1;
var x = 2; // No error, x is now 2

// 8. for...in and for...of with let:
const obj = { a: 1, b: 2, c: 3 };
for (let key in obj) {
    setTimeout(() => console.log(key), 100);
}
// Output: "a", "b", "c" — let creates new binding per iteration

// 9. var in catch was historically special:
try { throw 1; } catch (e) {
    var x = e; // x is function-scoped, NOT catch-scoped
}
console.log(x); // 1 — var escapes the catch block
console.log(e); // ReferenceError — e is catch-scoped
```

## ③ WHY IT EXISTS

**Historical Context:**
- **1995–2015:** Only `var` existed. It was function-scoped, which caused countless bugs with loops, closures, and unintentional variable sharing.
- **2015 (ES6):** `let` and `const` were introduced to fix these problems. `let` provides block scoping (like most other languages). `const` provides binding immutability (a strong signal of intent).

**Why three keywords instead of one?**
- `var` — kept for backward compatibility (millions of existing scripts)
- `let` — block scoping without binding immutability (for variables that need reassignment)
- `const` — block scoping WITH binding immutability (communicates "this won't change")

**Comparison with other languages:**

| Language | Variable Keywords | Default Behavior |
|----------|------------------|-----------------|
| JavaScript | var, let, const | No default — must choose |
| Java | (type name) with `final` | Mutable by default, `final` for const |
| Python | just name = value | Mutable by default, no const |
| Rust | let, let mut, const | Immutable by default (`let`) |
| Go | var, :=, const | Mutable by default |
| Swift | var, let | var = mutable, let = immutable |

JavaScript's `const` is closest to Swift's `let` — binding immutability, not value immutability.

## ④ INTERNAL WORKING

### How V8 Represents var vs let vs const

**var — Variable Environment:**
```javascript
function example() {
    var x = 1;
    if (true) {
        var y = 2;  // Hoisted to function scope
    }
}
```

V8 allocates both `x` and `y` in the function's stack frame during creation phase. There's no separate block scope for the `if` block with `var`.

```
Function Stack Frame:
┌────────────────────┐
│ x: undefined → 1   │  ← var, function-scoped
│ y: undefined → 2   │  ← var, hoisted out of if-block
└────────────────────┘
```

**let/const — Lexical Environment (may involve Context):**
```javascript
function example() {
    let a = 1;
    {
        let b = 2;  // Block-scoped
        const c = 3;
    }
    // b and c are dead here
}
```

V8 creates separate scopes:
```
Function Scope:
┌────────────────────────────────────────────┐
│ a: the_hole → 1                            │
│ ┌────────────────────────────────────────┐ │
│ │ Block Scope:                           │ │
│ │ b: the_hole → 2                        │ │
│ │ c: the_hole → 3                        │ │
│ └────────────────────────────────────────┘ │
│ // After block ends: b and c are gone      │
└────────────────────────────────────────────┘
```

### Bytecode Differences

```javascript
// var x = 5;
// Bytecode:
LdaSmi [5]
StaGlobal [x]  // or StaLookupSlot for function-local var

// let y = 5;
// Bytecode:
LdaSmi [5]
Star r0        // Store in register (with TDZ check before first access)

// const z = 5;
// Bytecode:
LdaSmi [5]
Star r0        // Same bytecode as let — constness enforced at parse time
```

**Key insight:** In V8, `let` and `const` generate the **same bytecode** at runtime. The difference is enforced by the parser: `const` reassignment is a **SyntaxError** caught during parsing, not a runtime check. This means `const` has zero runtime overhead compared to `let`.

## ⑤ ADVANTAGES

1. **`const` communicates intent:** When you read `const x = ...`, you immediately know `x` won't be reassigned later. This dramatically improves code readability in large codebases.

2. **Block scoping prevents variable leaking:** `let`/`const` variables are confined to their block, preventing accidental access from outer scopes.

3. **TDZ catches bugs early:** Accessing a variable before its declaration throws immediately, rather than silently giving `undefined` (which propagates through the code and causes confusing errors later).

4. **No redeclaration prevents shadowing accidents:** `let x; let x;` is a SyntaxError, catching cases where you accidentally reuse a name.

5. **`const` helps V8 optimize:** While V8 generates the same bytecode for `let` and `const`, the parser's constness guarantee helps TurboFan make stronger optimization assumptions (e.g., the value can be inlined as a constant).

---

## ⑥ DISADVANTAGES / TRAPS

1. **`const` with objects is misleading:**
   ```javascript
   const user = { name: "John" };
   user.name = "Jane"; // Works! Developers expect const to prevent this.
   // Rule: const protects the BINDING, not the VALUE
   ```

2. **`var` still appears in legacy code and can surprise you:**
   ```javascript
   for (var i = 0; i < 5; i++) {
       // i is shared across all iterations and leaks out of the loop
   }
   console.log(i); // 5 — still accessible!
   ```

3. **TDZ creates a "dead zone" where `typeof` throws:**
   ```javascript
   // In polyfill/feature-detection code, this can break:
   if (typeof someNewAPI !== 'undefined') { /* use it */ }
   // This pattern works for truly undeclared globals
   // But BREAKS if there's a `let someNewAPI` later in the same block
   ```

4. **`let` in for-loops creates per-iteration scope (slight overhead):**
   ```javascript
   // Each iteration creates a new context object on the heap
   for (let i = 0; i < 1000000; i++) {
       // If there's a closure capturing i, V8 allocates a new Context per iteration
   }
   // This is negligible for most code but matters in extreme hot loops
   ```

5. **`const` requires initialization — can't defer:**
   ```javascript
   const result; // SyntaxError
   if (condition) {
       result = computeA(); // Can't do this
   } else {
       result = computeB(); // Can't do this
   }
   
   // Must use let or ternary:
   const result = condition ? computeA() : computeB();
   ```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the differences between var, let, and const?**
**A:** `var` is function-scoped, hoisted with `undefined`, and allows redeclaration. `let` is block-scoped, hoisted but uninitialized (TDZ), allows reassignment but not redeclaration. `const` is like `let` but also prevents reassignment. It requires an initializer.

**Q2: What is the Temporal Dead Zone?**
**A:** The TDZ is the region between entering a block scope and the point where a `let` or `const` variable is declared. Accessing the variable during TDZ throws a `ReferenceError`, even though the binding exists (hoisted).

**Q3: Can you change a property of a const object?**
**A:** Yes. `const` prevents reassigning the variable, not mutating the object. `const obj = {x: 1}; obj.x = 2;` works fine. To prevent mutation, use `Object.freeze()`.

**Q4: What is the output?**
```javascript
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
```
**A:** `3, 3, 3`. Because `var i` is function-scoped, all three callbacks share the same `i`, which is `3` after the loop ends.

**Q5: Does `const` mean immutable?**
**A:** No. `const` means the binding (variable) cannot be reassigned. The value it points to can still be mutable (for objects, arrays, etc.).

### 🟡 Intermediate (5)

**Q6: Are let and const hoisted?**
**A:** Yes, they ARE hoisted — the binding is created when the scope is entered. But unlike `var`, they are not initialized. They sit in the TDZ until the declaration line is reached. This is provable: if `let` weren't hoisted, accessing a same-named outer variable inside a block before the inner `let` would work — but it doesn't.

**Q7: Why does `var` create a property on `window` but `let` doesn't?**
**A:** `var` declarations at the global level are added to the global object's property list (the Global Environment Record's Object Record). `let`/`const` at the global level go into the Declarative Record, which is NOT the same as the global object. This is a spec distinction to keep `let`/`const` from polluting `window`.

**Q8: What happens if you use `var` and `let` with the same name?**
```javascript
var x = 1;
let x = 2;
```
**A:** `SyntaxError: Identifier 'x' has already been declared`. You cannot have a `let`/`const` and a `var` with the same name in the same scope.

**Q9: What's the output?**
```javascript
let x = 1;
{
    console.log(x);
    let x = 2;
}
```
**A:** `ReferenceError`. The inner `let x` is hoisted to the block scope, creating a TDZ. The `console.log(x)` accesses the inner `x` (which shadows the outer one), but it's in the TDZ.

**Q10: Is there any runtime difference between let and const in V8?**
**A:** No. V8 generates identical bytecode for `let` and `const`. The constness of `const` is enforced at parse time (SyntaxError if you try to reassign). At runtime, there is zero overhead difference.

### 🔴 Advanced (5)

**Q11: How does V8 implement the per-iteration scope for `let` in for loops?**
**A:** V8 creates a new block-scope Context object at the start of each iteration. The loop variable's value is copied from the previous iteration's context into the new one. This is done via `CreateBlockContext` bytecode. If no closure captures the loop variable, V8 can optimize this away and use a simple register.

**Q12: Explain why this throws in strict mode but not sloppy mode:**
```javascript
x = 5;
console.log(x);
```
**A:** In sloppy mode, assigning to an undeclared variable creates an implicit global (`window.x = 5`). In strict mode, this throws a `ReferenceError` because implicit global creation is forbidden. This is one of the key reasons strict mode (and `let`/`const`) were introduced.

**Q13: What's the memory layout difference between `var` and `let` in V8?**
**A:** `var` bindings are allocated in the function's Variable Environment (typically as registers in the stack frame). `let`/`const` bindings are allocated in the Lexical Environment, which may be a separate block-scope Context on the heap if the variable is captured by a closure. If not captured, both `var` and `let` are register-allocated and have identical memory layout.

**Q14: Can const be optimized better than let by TurboFan?**
**A:** In theory, yes. TurboFan can rely on the guarantee that a `const` binding is never reassigned, enabling more aggressive constant folding and inlining. In practice, TurboFan can often determine that a `let` variable is never reassigned (through SSA analysis), achieving similar optimization. The benefit of `const` is more for developer intent than for measurable performance.

**Q15: What happens with var in eval?**
```javascript
eval("var x = 10;");
console.log(x);
```
**A:** In sloppy mode, `eval` introduces `x` into the surrounding scope — so `console.log(x)` prints `10`. In strict mode, `eval` creates its own scope — `x` stays inside `eval`, and `console.log(x)` throws `ReferenceError`. This is why `eval` + `var` is dangerous and why V8 can't optimize functions containing `eval`.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step by step what V8 does differently for `var x = 1` vs `let x = 1` at the bytecode level.**

**A:**
**`var x = 1` (in a function):**
1. Creation phase: `x` is allocated a register slot, initialized to `undefined`
2. Bytecode when declaration is reached: `LdaSmi [1]; Star r0;`
3. Before the declaration line: `Ldar r0` reads `undefined` (no error)
4. No TDZ check needed

**`let x = 1` (in a function):**
1. Creation phase: `x` is allocated a register slot, initialized to `the_hole`
2. Before the declaration line: any `Ldar r0` is preceded by `ThrowReferenceErrorIfHole r0`
3. Bytecode when declaration is reached: `LdaSmi [1]; Star r0;` (replaces the_hole with 1)
4. After declaration: `Ldar r0` reads `1` (TDZ check still exists in bytecode but V8 may optimize it away)

The key difference is the initial value (`undefined` vs `the_hole`) and the TDZ check (`ThrowReferenceErrorIfHole`) that `let` generates.

**Q17: Explain what happens to `var i` inside a for loop at the scope/memory level.**

**A:**
```javascript
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
```

1. `var i` is hoisted to the function (or global) scope — there is ONE binding for `i`
2. Loop starts: `i = 0`
3. First iteration: `setTimeout` creates a closure. The closure's `[[Environment]]` points to the function scope's Lexical Environment, which contains `i`. The closure does NOT get its own copy of `i`.
4. `i++` → `i = 1`. Second iteration: another closure, same environment, same `i`.
5. `i++` → `i = 2`. Third iteration: another closure, same environment, same `i`.
6. `i++` → `i = 3`. Condition `i < 3` is false → loop ends. `i` is now `3`.
7. Call stack is empty → event loop processes setTimeout callbacks
8. Each closure reads `i` from the shared function scope → reads `3`
9. Output: `3, 3, 3`

With `let i`:
1. `let i = 0` creates a block-scoped binding
2. V8 creates a new Context object per iteration with its own `i` copy
3. Iteration 0: Context { i: 0 } → closure captures this
4. Iteration 1: Context { i: 1 } → closure captures this
5. Iteration 2: Context { i: 2 } → closure captures this
6. Each callback reads its own `i` → Output: `0, 1, 2`

The trade-off: `let` creates 3 Context objects on the heap; `var` creates zero. But correctness matters more than this negligible overhead.
