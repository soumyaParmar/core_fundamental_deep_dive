# 📌 2.2 — Hoisting Internals
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Hoisting is JavaScript's behavior of moving declarations to the top of their scope during the creation phase — before any code executes. It means you can use certain things (like functions and `var` variables) before the line where they're declared. But different declarations are hoisted differently.

**Technical Definition:** Hoisting is a consequence of how the JavaScript engine processes code in two phases: the **creation phase** (where it scans for declarations and sets up bindings) and the **execution phase** (where code runs line by line). During the creation phase, `var` declarations are initialized to `undefined`, function declarations are fully initialized, and `let`/`const` bindings are allocated but left uninitialized (TDZ). Class declarations are hoisted but in TDZ.

**One-line mental model:**
> *Hoisting is not "code moving to the top." It's the engine setting up all bindings during the creation phase before execution begins.*

**Where does this live?**
- In the **creation phase** of the Execution Context — Variable Environment and Lexical Environment setup
- In V8: during parsing and scope analysis, before bytecode execution

## ② DRILL DOWN

### 🔹 A. Function Hoisting

Function **declarations** are fully hoisted — both the name AND the body are available from the start of the scope.

```javascript
// You can call a function BEFORE its declaration:
greet(); // "Hello!" — works because function declarations are fully hoisted

function greet() {
    console.log("Hello!");
}
```

**What the engine does during creation phase:**
```
Creation Phase:
  greet → function() { console.log("Hello!"); }  ← fully initialized!

Execution Phase:
  greet() → calls the function → "Hello!"
```

**Function expressions are NOT fully hoisted:**
```javascript
greet(); // TypeError: greet is not a function

var greet = function() {
    console.log("Hello!");
};
// `var greet` is hoisted as undefined, but the function assignment happens at execution time
// So greet is undefined when you try to call it → TypeError
```

```javascript
greet(); // ReferenceError: Cannot access 'greet' before initialization

const greet = function() {
    console.log("Hello!");
};
// const greet is hoisted into TDZ → ReferenceError
```

**Function declaration vs function expression hoisting:**

| Type | Hoisting Behavior | Usable Before Declaration? |
|------|------------------|--------------------------|
| `function foo() {}` (declaration) | Fully hoisted (name + body) | ✅ Yes |
| `var foo = function() {}` (expression + var) | `var foo` hoisted as `undefined` | ❌ TypeError (not a function) |
| `const foo = function() {}` (expression + const) | `const foo` in TDZ | ❌ ReferenceError (TDZ) |
| `const foo = () => {}` (arrow + const) | `const foo` in TDZ | ❌ ReferenceError (TDZ) |

### 🔹 B. Variable Hoisting

```javascript
console.log(x); // undefined (var hoisted, initialized to undefined)
var x = 5;
console.log(x); // 5

console.log(y); // ReferenceError (let hoisted, but in TDZ)
let y = 10;

console.log(z); // ReferenceError (const hoisted, but in TDZ)
const z = 15;
```

**var hoisting in detail:**
```javascript
// What you write:
function example() {
    console.log(a);
    var a = 10;
    console.log(a);
}

// What the engine processes:
function example() {
    var a = undefined; // ← Creation phase sets this up
    console.log(a);    // undefined
    a = 10;            // ← Execution phase assignment
    console.log(a);    // 10
}
```

**var hoisting across blocks:**
```javascript
function test() {
    console.log(x); // undefined (hoisted from inside the if block!)
    
    if (false) {
        var x = 5; // This NEVER executes, but x is still hoisted
    }
    
    console.log(x); // undefined (declared but never assigned)
}
```

This is because `var` is **function-scoped** — it hoists to the top of the enclosing function, regardless of blocks.

### 🔹 C. Class Hoisting

Classes ARE hoisted but sit in the **TDZ** (like `let`/`const`):

```javascript
// Class declarations:
const obj = new Animal(); // ReferenceError: Cannot access 'Animal' before initialization

class Animal {
    constructor(name) {
        this.name = name;
    }
}

// Proof that classes are hoisted (not just undeclared):
let x = "outer";
{
    console.log(x); // If classes weren't hoisted, this would print "outer"
    // But actually: ReferenceError because the inner class x shadows it (TDZ)
    class x {}
}
```

**Class expressions follow the variable's hoisting rule:**
```javascript
const Dog = class {}; // const Dog → TDZ
var Cat = class {};   // var Cat → undefined (not a constructor)
```

### 🔹 D. TDZ Behavior (Detailed)

The TDZ starts at the **beginning of the scope** and ends at the **declaration line**:

```javascript
// TDZ exists from block entry to declaration
{
    // ─── TDZ for x starts ───────────────────────────────
    
    // Any access to x in this zone throws:
    // console.log(x);      // ReferenceError
    // x = 5;               // ReferenceError
    // typeof x;            // ReferenceError (unlike undeclared vars!)
    // if (x) {}            // ReferenceError
    
    // ─── TDZ for x ends ─────────────────────────────────
    let x = 10;  // TDZ ends HERE — x is now initialized
    
    console.log(x); // 10 ✅
}
```

**TDZ with destructuring:**
```javascript
// TDZ applies to destructured variables too:
console.log(a); // ReferenceError
const { a, b } = { a: 1, b: 2 };
```

**TDZ is temporal, not spatial:**
```javascript
// This works even though `x` appears before the let declaration
// because the FUNCTION is not CALLED until after x is declared
function useX() {
    console.log(x); // 42 ✅ — called after x is initialized
}

let x = 42;
useX(); // Works! The function body isn't executed until here
```

**TDZ in complex scenarios:**
```javascript
// 1. TDZ in conditional expressions:
let x = x; // ReferenceError! The RHS `x` is in the TDZ of the LHS `let x`

// 2. TDZ with default parameter values:
function f(a = b, b = 1) {} // ReferenceError: b is in TDZ when a's default is evaluated
f();

// 3. TDZ in for-of/for-in headers:
for (let x of [1, 2, 3]) {
    // x is initialized per iteration — no TDZ issue inside the loop body
}

// 4. No TDZ for var in loops:
for (var x of [1, 2, 3]) {} // var x is hoisted, no TDZ
```

### 🔹 E. Engine Preprocessing Phase

During the creation phase, the engine processes ALL declarations in the current scope. This is done by the **parser's scope analyzer**, NOT during execution:

```
Scope Analysis Algorithm:
1. Enter a new scope (global, function, or block for let/const)
2. Scan ALL statements in this scope:
   a. var declaration → add to Variable Environment, init = undefined
   b. let/const declaration → add to Lexical Environment, init = <the_hole> (TDZ)
   c. function declaration → add to Variable/Lexical Environment, init = full function
   d. class declaration → add to Lexical Environment, init = <the_hole> (TDZ)
3. Check for conflicts:
   - let/const + let/const same name → SyntaxError
   - let/const + var same name → SyntaxError
   - var + var same name → OK (redeclaration allowed)
4. Perform escape analysis (which variables are captured by closures)
5. Allocate registers (stack) or context slots (heap) for each variable
```

### 🔹 F. Hoisting Priority and Conflicts

When a function declaration and a `var` have the same name:

```javascript
console.log(foo); // function foo() { return 2; }

var foo = 1;
function foo() { return 2; }

console.log(foo); // 1

// What happens:
// Creation phase: 
//   1. var foo → undefined
//   2. function foo → function (OVERWRITES the var binding)
//   Result: foo = function foo() { return 2; }
// Execution phase:
//   Line 1: console.log(foo) → logs the function
//   Line 3: foo = 1 → foo is now 1 (assignment, not declaration)
//   Line 7: console.log(foo) → logs 1
```

**Multiple function declarations with the same name:**
```javascript
function foo() { return 1; }
function foo() { return 2; }
function foo() { return 3; }

console.log(foo()); // 3 — last declaration wins
```

## ③ WHY IT EXISTS

**What problem does hoisting solve?**

1. **Mutual recursion:** Functions can call each other regardless of declaration order:
```javascript
function isEven(n) {
    if (n === 0) return true;
    return isOdd(n - 1); // isOdd is hoisted — available here
}

function isOdd(n) {
    if (n === 0) return false;
    return isEven(n - 1); // isEven is hoisted — available here
}
```

2. **Code organization:** You can put your main logic at the top and helper functions at the bottom — a common pattern in many languages:
```javascript
// Main logic first (readable):
main();

// Implementation details below:
function main() {
    const data = loadData();
    process(data);
}

function loadData() { /* ... */ }
function process(data) { /* ... */ }
```

3. **Historical simplicity:** In the 10-day creation of JavaScript, single-pass execution with hoisting was simpler to implement than multi-pass compilation.

**Why TDZ was added later:**

`var` hoisting's behavior of initializing to `undefined` masks bugs. The TDZ for `let`/`const` was specifically designed to catch "use before declaration" errors that `var` silently ignores.

## ④ INTERNAL WORKING

### V8's Scope Analysis Phase

During parsing, V8 runs a scope analyzer that:

1. **Pre-scan the function body** — identify all declarations (var, let, const, function, class)
2. **Determine binding types:**
   - `var` → Variable Environment slot (function-scoped)
   - `let`/`const` → Lexical Environment slot (block-scoped)
   - `function` → Variable/Lexical Environment (function-scoped, fully initialized)
3. **Assign storage locations:**
   - Non-captured variables → CPU registers (stack)
   - Captured variables → Context object slots (heap)
4. **Generate TDZ checks** for let/const bindings

### Bytecode for Hoisting

```javascript
console.log(x);
var x = 5;
```

**V8 Ignition bytecode:**
```
// Note: x is already allocated as register r0, initialized to undefined
LdaGlobal [console]      // Load console
LdaNamedProperty [log]   // Load console.log
Ldar r0                  // Load r0 (x = undefined)
CallProperty1 ...        // console.log(undefined)
LdaSmi [5]               // Load 5
Star r0                  // Store 5 into r0 (x = 5 now)
```

For `let`:
```javascript
console.log(x);
let x = 5;
```

**V8 Ignition bytecode:**
```
// r0 is initialized to the_hole
Ldar r0                          // Load r0
ThrowReferenceErrorIfHole [x]    // If r0 === the_hole → throw ReferenceError!
// ↑ This is the TDZ check
CallProperty1 ...                // Never reached

LdaSmi [5]                       // Load 5
Star r0                          // Store 5 into r0 (TDZ ends)
```

### Memory State at Each Phase

```
function example() {
    console.log(a, b);
    var a = 1;
    let b = 2;
    function c() {}
}

CREATION PHASE:
┌──────────────────────────────────────┐
│ Variable Environment:                 │
│   a: undefined                        │
│   c: function c() {}                  │
│                                       │
│ Lexical Environment:                  │
│   b: <the_hole> (TDZ)               │
└──────────────────────────────────────┘

EXECUTION PHASE (line by line):
Line 2: console.log(a, b)
  a → undefined ✅
  b → the_hole → ReferenceError ❌ (execution stops here)

If b weren't in TDZ:
Line 3: a = 1 → a is now 1
Line 4: b = 2 → b exits TDZ, is now 2
```

## ⑤ ADVANTAGES

1. **Function hoisting enables natural code organization** — main functions at the top, helpers at the bottom. This is the most practical benefit and the original reason hoisting was designed.

2. **Mutual recursion works naturally** — functions can reference each other without forward declarations (unlike C/C++ which require prototypes).

3. **TDZ catches real bugs** — using a variable before it's declared is almost always a mistake. TDZ makes this a hard error instead of a silent `undefined`.

4. **Predictable scoping** — once you understand the rules, hoisting behavior is completely deterministic and predictable.

---

## ⑥ DISADVANTAGES / TRAPS

1. **var hoisting creates confusing behavior:**
   ```javascript
   var x = 1;
   function foo() {
       console.log(x); // undefined, NOT 1!
       var x = 2;
   }
   foo();
   // The inner `var x` is hoisted inside foo, shadowing the outer x
   // During creation phase: local x = undefined
   ```

2. **Function hoisting in blocks is inconsistent:**
   ```javascript
   // In strict mode: function declarations in blocks are block-scoped
   "use strict";
   {
       function foo() { return 1; }
   }
   foo(); // ReferenceError in strict mode
   
   // In sloppy mode: behavior varies between engines!
   // V8: hoists the function name but not the body (legacy web compat)
   // This is one of the most inconsistent areas of JavaScript
   ```

3. **Hoisting can mask variable shadowing bugs:**
   ```javascript
   let count = 10;
   function process() {
       // Developer forgot they have an outer `count`
       // and accidentally creates a new local one
       if (true) {
           var count = 0; // Hoisted to function scope, shadows outer count
       }
       console.log(count); // 0 (not 10!)
   }
   ```

4. **Named function expressions hoist differently:**
   ```javascript
   var foo = function bar() {
       console.log(typeof bar); // "function" — bar is accessible INSIDE
   };
   console.log(typeof bar); // "undefined" — bar is NOT accessible outside
   // The name `bar` is only available inside the function body
   ```

## ⑦ PRODUCTION USAGE

### Best Practice: Declare Before Use

```javascript
// Production code should never rely on hoisting for variables:

// ✅ Good — declare at the top of scope:
function processOrder(order) {
    const tax = calculateTax(order.total);
    const shipping = calculateShipping(order.address);
    const total = order.total + tax + shipping;
    return { ...order, tax, shipping, total };
}

// ❌ Bad — relying on var hoisting:
function processOrder(order) {
    total = order.total + tax + shipping; // Using before declare — confusing!
    var tax = calculateTax(order.total);
    var shipping = calculateShipping(order.address);
    var total;
    return { ...order, tax, shipping, total };
}
```

### Function Hoisting is Acceptable

```javascript
// This pattern is acceptable and common in production:
// "Public API at the top, implementation at the bottom"

export function createServer(config) {
    validateConfig(config);       // Defined below — hoisted
    const server = initServer();  // Defined below — hoisted
    setupRoutes(server, config);  // Defined below — hoisted
    return server;
}

// Private helpers below:
function validateConfig(config) { /* ... */ }
function initServer() { /* ... */ }
function setupRoutes(server, config) { /* ... */ }
```

### ESLint Rules for Hoisting Safety

```json
{
    "rules": {
        "no-var": "error",
        "no-use-before-define": ["error", {
            "functions": false,   // Allow function hoisting
            "classes": true,      // Catch class TDZ issues
            "variables": true     // Catch variable TDZ issues
        }]
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is hoisting?**
**A:** Hoisting is JavaScript's behavior of processing declarations during the creation phase before code execution. `var` variables are hoisted and initialized to `undefined`, function declarations are fully hoisted with their body, and `let`/`const`/`class` are hoisted but left uninitialized (TDZ).

**Q2: What's the output?**
```javascript
console.log(foo());
function foo() { return "hello"; }
```
**A:** `"hello"`. Function declarations are fully hoisted — both the name and the body are available before the declaration line.

**Q3: What's the difference between function declaration and function expression hoisting?**
**A:** Function declarations are fully hoisted (usable before declaration). Function expressions follow the hoisting rules of their variable keyword: `var` → `undefined` (TypeError if called), `let`/`const` → TDZ (ReferenceError).

**Q4: What's the output?**
```javascript
console.log(x);
var x = 10;
```
**A:** `undefined`. `var x` is hoisted and initialized to `undefined`. The assignment `x = 10` happens later during execution.

**Q5: What is the TDZ?**
**A:** The Temporal Dead Zone is the period between entering a scope and the declaration line of a `let`/`const`/`class`. Accessing the variable during TDZ throws `ReferenceError`.

### 🟡 Intermediate (5)

**Q6: What's the output?**
```javascript
var a = 1;
function a() { return 2; }
console.log(typeof a);
```
**A:** `"number"`. During creation phase, `var a` → `undefined`, then `function a` → overwrites to function. During execution: `a = 1` overwrites the function. So `typeof a` is `"number"`.

**Q7: What's the output?**
```javascript
console.log(typeof x);
let x = 5;
```
**A:** `ReferenceError`. Even `typeof` throws for TDZ variables. This is different from truly undeclared variables where `typeof undeclared` returns `"undefined"`.

**Q8: Explain this behavior:**
```javascript
let x = "outer";
{
    console.log(x); // ?
    let x = "inner";
}
```
**A:** `ReferenceError`. The inner `let x` is hoisted to the top of the block, creating a TDZ. The `console.log(x)` tries to access the inner `x` (which shadows the outer one), but it's in the TDZ.

**Q9: What's the output?**
```javascript
foo();
var foo = function() { console.log("A"); };
function foo() { console.log("B"); }
foo();
```
**A:** `"B"`, then `"A"`. During creation: `var foo` → `undefined`, then `function foo` → function (wins). So first `foo()` calls the function → "B". Then `foo = function()...` assigns the expression. Second `foo()` calls the new function → "A".

**Q10: Are imports hoisted?**
**A:** Yes! ES module `import` declarations are hoisted to the top of the module and fully initialized before any code runs. This is why you can use imported values before the import statement in the source code (though this is bad practice).

### 🔴 Advanced (5)

**Q11: How does V8 implement TDZ checks internally?**
**A:** V8 uses a sentinel value called `the_hole`. During scope entry, `let`/`const` variables are set to `the_hole`. Before each access, V8 generates a `ThrowReferenceErrorIfHole` bytecode that checks if the value equals `the_hole`. If yes → throw ReferenceError. TurboFan can eliminate these checks when it can prove the access always occurs after initialization.

**Q12: What happens with function declarations inside blocks in different modes?**
**A:** In strict mode, function declarations in blocks are block-scoped (like `let`). In sloppy mode, the behavior is complex and was standardized as "Annex B" in the spec: the function name is hoisted to the function scope as `undefined`, and the function body is assigned when the block is entered. This is a legacy compatibility hack and behavior can vary between engines.

**Q13: Can hoisting cause a SyntaxError?**
**A:** Yes. During the creation phase (scope analysis), if the engine detects a `let`/`const` redeclaration in the same scope, it throws a `SyntaxError` before any code runs:
```javascript
let x = 1;
let x = 2; // SyntaxError — detected during parsing, not execution
```

**Q14: How does hoisting interact with `eval`?**
**A:** In sloppy mode, `eval("var x = 5")` introduces `x` into the surrounding scope at runtime — this is NOT handled during the creation phase. This breaks V8's ability to statically analyze the scope, which is why `eval` prevents optimization. In strict mode, `eval` creates its own scope and doesn't affect the outer scope.

**Q15: What's the output and explain the hoisting chain:**
```javascript
console.log(a);
console.log(b);
console.log(c);
var a = function b() { var c = 10; };
```
**A:** `undefined`, `ReferenceError`. `var a` is hoisted as `undefined`. `b` is the named function expression name — it's only accessible inside the function body, not in the outer scope. `c` is hoisted inside function `b`, not in the outer scope. So: `a` → `undefined`, `b` → `ReferenceError` (not declared in this scope).

### 🧠 "Explain What Happens Internally" (2)

**Q16: Walk through the creation phase for this code:**
```javascript
var x = 1;
let y = 2;
function foo() { return 3; }
class Bar {}
const z = 4;
```

**A:**
1. Engine enters the Global scope
2. Scan all declarations:
   - `var x` → add to Variable Environment, initialize to `undefined`
   - `let y` → add to Lexical Environment, initialize to `the_hole` (TDZ)
   - `function foo` → add to Variable Environment, initialize to `function foo() { return 3; }` (fully hoisted)
   - `class Bar` → add to Lexical Environment, initialize to `the_hole` (TDZ)
   - `const z` → add to Lexical Environment, initialize to `the_hole` (TDZ)
3. State after creation phase:
```
Variable Environment: { x: undefined, foo: function() }
Lexical Environment:  { y: <the_hole>, Bar: <the_hole>, z: <the_hole> }
```
4. Execution phase:
   - `x = 1` → x is now 1
   - `y = 2` → y exits TDZ, is now 2
   - `foo` declaration → no-op (already done)
   - `class Bar {}` → Bar exits TDZ, is now the class constructor
   - `z = 4` → z exits TDZ, is now 4

---

**Q17: Explain why this specific order of hoisting happens:**
```javascript
console.log(foo); // function foo() { return 'B'; }
var foo = 'A';
function foo() { return 'B'; }
var foo = 'C';
console.log(foo); // 'C'
```

**A:**
Creation phase processing order:
1. `var foo` → foo = undefined
2. `function foo() { return 'B'; }` → foo = function (overwrites undefined)
3. `var foo` → ignored! `var` redeclaration doesn't re-initialize if the binding already exists

So after creation phase: `foo = function foo() { return 'B'; }`

Execution phase:
1. `console.log(foo)` → logs the function
2. `foo = 'A'` → foo is now the string 'A'
3. `function foo()` declaration → no-op (already processed in creation phase)
4. `foo = 'C'` → foo is now the string 'C'
5. `console.log(foo)` → logs 'C'

Key rules:
- Function declarations take priority over `var` in the creation phase
- `var` redeclaration doesn't re-initialize (it's a no-op if binding exists)
- But `var` assignments DO execute during the execution phase
- Function declarations are fully processed during creation phase (body + assignment)
