# 📌 2.3 — Scope
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Scope is the region of your code where a variable is accessible. Think of it as a "visibility bubble" — a variable declared inside a function can't be seen outside it, like a room with walls. JavaScript has four types of scopes: global, function, block, and module.

**Technical Definition:** A scope is a Lexical Environment — a specification type containing an Environment Record (which holds identifier bindings) and a reference to an outer Lexical Environment. The chain of outer references forms the **scope chain**, which determines variable resolution at runtime.

**One-line mental model:**
> *Scope = where you can see a variable. Inner scopes can see outer variables, but outer scopes cannot see inner variables.*

**Where does this live?**
- Conceptually: ECMAScript Lexical Environment records
- In V8: stack registers (non-captured variables) or Context objects on the heap (captured variables)
- The scope chain: linked list of Lexical Environment outer references

## ② DRILL DOWN

### 🔹 A. Global Scope

The outermost scope. Variables declared here are accessible everywhere.

```javascript
// Global scope
var globalVar = "I'm global (var)";
let globalLet = "I'm global (let)";
const globalConst = "I'm global (const)";

function test() {
    console.log(globalVar);   // ✅ Accessible
    console.log(globalLet);   // ✅ Accessible
    console.log(globalConst); // ✅ Accessible
}
test();
```

**Global scope quirks:**

```javascript
// In browser:
var x = 1;
console.log(window.x); // 1 — var attaches to window
let y = 2;
console.log(window.y); // undefined — let does NOT attach to window

// In Node.js (CommonJS):
var z = 3;
console.log(global.z); // undefined — modules are wrapped in a function!

// Implicit globals (sloppy mode):
function leak() {
    accidental = "I'm a global!"; // No var/let/const → implicit global
}
leak();
console.log(accidental); // "I'm a global!" — leaks to global scope
// In strict mode: ReferenceError
```

**Two Global Environment Records:**
```
Global Environment:
┌───────────────────────────────────────────┐
│ Object Record (→ window/global)            │
│   var declarations, function declarations  │
│   window.x, window.setTimeout, etc.       │
├───────────────────────────────────────────┤
│ Declarative Record                         │
│   let declarations, const declarations     │
│   NOT on window/global                    │
└───────────────────────────────────────────┘
```

### 🔹 B. Function Scope

Created by function declarations and expressions. `var` is function-scoped.

```javascript
function outer() {
    var x = 1; // Scoped to outer()
    
    function inner() {
        var y = 2; // Scoped to inner()
        console.log(x); // ✅ Can see outer's x (scope chain lookup)
    }
    
    inner();
    console.log(y); // ❌ ReferenceError — y is in inner's scope
}
```

**Every function creates a new scope:**
```javascript
function createCounter() {
    var count = 0; // Private to this function scope
    return {
        increment() { count++; },
        get() { return count; }
    };
}

const counter1 = createCounter(); // count = 0 (own scope)
const counter2 = createCounter(); // count = 0 (DIFFERENT scope)
counter1.increment();
counter1.increment();
console.log(counter1.get()); // 2
console.log(counter2.get()); // 0 — independent scope!
```

### 🔹 C. Block Scope

Created by `{}` blocks (if, for, while, try/catch, or standalone blocks). `let` and `const` are block-scoped.

```javascript
{
    let blockScoped = "inside block";
    var notBlockScoped = "escapes block";
}
console.log(notBlockScoped); // "escapes block" — var ignores blocks
console.log(blockScoped);    // ReferenceError — let respects blocks

// Block scope in if statements:
if (true) {
    let x = 1;
    const y = 2;
    var z = 3;
}
console.log(z); // 3 — var escapes
console.log(x); // ReferenceError
console.log(y); // ReferenceError

// Block scope in for loops:
for (let i = 0; i < 3; i++) {
    // i is scoped to THIS iteration's block
}
console.log(i); // ReferenceError — i doesn't exist here

for (var j = 0; j < 3; j++) {
    // j is function-scoped
}
console.log(j); // 3 — j escapes the loop

// Standalone block scope:
{
    let private = "hidden";
    // useful for limiting variable scope in a script
}
```

**Nested block scopes:**
```javascript
function example() {
    let a = 1;
    {
        let b = 2;
        {
            let c = 3;
            console.log(a, b, c); // 1, 2, 3 — all visible
        }
        console.log(a, b);    // 1, 2 — c is out of scope
        // console.log(c);    // ReferenceError
    }
    console.log(a);          // 1 — b is out of scope
}
```

### 🔹 D. Module Scope

Each ES Module has its own scope. Top-level variables are NOT global.

```javascript
// module-a.js
const SECRET = "hidden"; // Module-scoped — NOT global
export const PUBLIC = "visible";

// module-b.js
import { PUBLIC } from './module-a.js';
console.log(PUBLIC);  // "visible" — explicitly exported/imported
console.log(SECRET);  // ReferenceError — not exported
```

**Module scope vs global scope:**
```javascript
// In a module (type="module" in browser, .mjs in Node):
var x = 1;
console.log(window.x); // undefined — var in module scope is NOT global!

// In a regular script:
var x = 1;
console.log(window.x); // 1 — var in global scope IS a window property
```

### 🔹 E. Lexical Scoping (Static Scoping)

JavaScript uses **lexical scoping** — the scope of a variable is determined by its **position in the source code**, NOT by where it's called (which would be dynamic scoping).

```javascript
const x = "global";

function outer() {
    const x = "outer";
    
    function inner() {
        console.log(x); // "outer" — lexical scope (where inner is DEFINED)
    }
    
    return inner;
}

const fn = outer();
fn(); // "outer" — NOT wherever fn is called

// If JavaScript used dynamic scoping (it doesn't):
// fn() would look up x in the CALLER's scope, and find "global"
```

**Lexical scoping is why closures work:**
```javascript
function makeAdder(base) {
    // base is in makeAdder's scope
    return function(n) {
        return base + n; // Looks up base in the scope where THIS function was DEFINED
        // That's makeAdder's scope → base is available
    };
}

const add5 = makeAdder(5);
const add10 = makeAdder(10);
console.log(add5(3));  // 8  (base=5 from the scope where add5's function was created)
console.log(add10(3)); // 13 (base=10 from the scope where add10's function was created)
```

### 🔹 F. Scope Chain Lookup Algorithm

When the engine needs to resolve a variable, it follows this algorithm:

```
resolveVariable(name):
  1. currentEnv = current Lexical Environment
  2. while (currentEnv !== null):
     a. Check currentEnv's Environment Record for `name`
     b. If found → return the binding
     c. If not found → currentEnv = currentEnv.outer (move up the chain)
  3. If reached null (no more outer scopes):
     a. In strict mode → throw ReferenceError
     b. In sloppy mode (for assignments) → create global variable
     c. In sloppy mode (for reads) → throw ReferenceError
```

```javascript
const a = 1;

function first() {
    const b = 2;
    
    function second() {
        const c = 3;
        
        function third() {
            console.log(a); // 4 lookups: third → second → first → global ✅
            console.log(b); // 3 lookups: third → second → first ✅
            console.log(c); // 2 lookups: third → second ✅
            console.log(d); // 4 lookups: third → second → first → global → null → ReferenceError
        }
        third();
    }
    second();
}
first();
```

**Scope chain diagram:**
```
third's Env ──outer──► second's Env ──outer──► first's Env ──outer──► Global Env ──outer──► null
     │                      │                       │                      │
     c: 3                   c: 3                    b: 2                   a: 1
```

**Performance implication:** Deeply nested scope chains mean more lookups. V8 mitigates this with inline caching — after the first lookup, V8 caches the result and goes directly to the correct scope in subsequent accesses.

### 🔹 G. Variable Shadowing

When an inner scope declares a variable with the same name as an outer scope:

```javascript
let x = "outer";

function test() {
    let x = "inner"; // Shadows the outer x
    console.log(x);  // "inner" — the inner x is found first
}
test();
console.log(x); // "outer" — the outer x is unaffected

// Shadowing with different keywords:
var y = 1;
function test2() {
    let y = 2; // let can shadow var
    console.log(y); // 2
}

// Shadowing in blocks:
let z = "outer";
{
    let z = "block"; // Shadows outer z in this block
    console.log(z);  // "block"
}
console.log(z); // "outer" — outer z is unaffected
```

**Dangerous shadowing:**
```javascript
let user = { name: "Alice" };

function processUser(user) { // Parameter shadows outer 'user'
    user.name = "Bob"; // Modifying the parameter's object
}

processUser(user);
console.log(user.name); // "Bob" — the SAME object was modified
// The parameter 'user' shadowed the outer 'user' by name,
// but since objects are references, they point to the same heap object
```

## ③ WHY IT EXISTS

**What problem does scope solve?**

1. **Name collision prevention:** Without scope, every variable would be global. In a large program with thousands of variables, name conflicts would be constant.

2. **Encapsulation:** Functions can have private variables that external code can't access or corrupt.

3. **Memory management:** When a scope ends (function returns, block exits), its variables can be freed.

4. **Security:** Sandboxing code within scopes prevents unauthorized access to sensitive data.

**Historical context:**
- 1995: JavaScript only had global scope and function scope. Block scope didn't exist.
- 2009: `try/catch` introduced catch-scoped variables (the first block-scoped binding in JS).
- 2015: ES6 introduced `let`/`const` with true block scoping, aligning JS with most other languages.

**Comparison with other languages:**

| Language | Scope Model | Notes |
|----------|-------------|-------|
| JavaScript | Lexical (static) | Function + block scope |
| Python | LEGB (Local, Enclosing, Global, Built-in) | No block scope! `if`/`for` don't create scope |
| Java | Block scope only | Every `{}` creates scope |
| C/C++ | Block scope only | Every `{}` creates scope |
| Bash | Dynamic scope | Variables resolved by caller chain |

**Python's lack of block scope:**
```python
# Python — no block scope!
for i in range(3):
    x = i
print(x)  # 2 — x leaks out of the for loop (like var in JS)
```

## ④ INTERNAL WORKING

### V8's Scope Representation

V8 represents scopes as a chain of Context objects (for closures) or register allocations (for non-escaping variables):

```javascript
function outer() {
    let a = 1;           // Stack register (no closure captures it)
    
    function inner() {
        let b = 2;       // Stack register
        console.log(a);  // a is captured → must be in Context
    }
    
    inner();
}
```

**V8 adjusts after escape analysis:**
```
outer() discovered: inner() captures `a`
  → `a` is context-allocated (heap)
  → `b` is register-allocated (stack) — not captured

Memory layout:
Stack Frame (outer):
┌────────────────────────────┐
│ context_ptr → Context obj  │
└────────────────────────────┘

Heap (Context object):
┌────────────────────────────┐
│ Context {                  │
│   parent: → Global Context │
│   slot[0]: a = 1           │
│ }                          │
└────────────────────────────┘

Stack Frame (inner):
┌────────────────────────────┐
│ b: 2 (register)           │
│ context_ptr → outer's Ctx  │
└────────────────────────────┘
```

### Scope Chain as Context Chain

```
inner's execution:
  Read `b` → register r0 in inner's stack frame (fast!)
  Read `a` → context_ptr → outer's Context → slot[0] (heap read, slightly slower)
  Read `console` → context chain → ... → Global Context → global lookup
```

### V8 Bytecode for Scope Chain Access

```javascript
function outer() {
    let x = 42;
    function inner() {
        return x; // Accessing outer's variable
    }
    return inner();
}
```

**Bytecode for `inner`:**
```
// inner() accessing outer's `x`:
LdaImmutableContextSlot <context>, [0], [1]
// context = current context
// [0] = slot index of x
// [1] = depth (1 means "go up 1 context level")
Return
```

The depth parameter tells V8 how many context levels to traverse. This is computed at parse time (not runtime), making scope chain access deterministic.

## ⑤ ADVANTAGES

1. **Predictability:** Lexical scoping means you can determine a variable's scope by reading the source code — no need to trace execution.

2. **Encapsulation:** Private data inside function or block scopes prevents external interference — the foundation of the module pattern.

3. **Garbage collection:** When a scope is exited and no closures reference it, all its variables become eligible for GC.

4. **Parallel development:** Different developers can use the same variable names in different scopes without conflicts.

5. **Security:** Sensitive data (API keys, tokens) can be confined to specific scopes, reducing exposure.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Closure memory retention:**
   ```javascript
   function setup() {
       const bigData = new Array(10000000);
       return function() {
           return "done"; // Doesn't use bigData, but the scope still retains it
       };
   }
   const fn = setup(); // bigData may not be GC'd (depending on engine)
   ```

2. **Accidental global variables in sloppy mode:**
   ```javascript
   function process() {
       result = compute(); // Forgot `let` — creates global!
   }
   ```

3. **Scope chain performance in deeply nested functions:**
   ```javascript
   // Every variable lookup traverses the scope chain:
   function level1() {
       const x = 1;
       function level2() {
           function level3() {
               function level4() {
                   console.log(x); // 4 levels of scope chain traversal
               }
               level4();
           }
           level3();
       }
       level2();
   }
   // V8 caches scope lookups via inline caching, but initial lookups are slower
   ```

4. **Python developers often confused by block scope:**
   ```javascript
   // Coming from Python where for-loops don't create scope:
   for (let i = 0; i < 3; i++) { }
   console.log(i); // ReferenceError — surprising for Python devs
   ```

## ⑦ PRODUCTION USAGE

### Module Pattern (Pre-ES6)

```javascript
// The classic IIFE module pattern — uses scope for encapsulation:
const UserModule = (function() {
    // Private scope
    let users = [];
    let nextId = 1;
    
    function findById(id) {
        return users.find(u => u.id === id);
    }
    
    // Public API
    return {
        addUser(name) {
            users.push({ id: nextId++, name });
        },
        getUser(id) {
            return findById(id);
        },
        get count() {
            return users.length;
        }
    };
})();

// users, nextId, and findById are completely hidden — encapsulated by scope
UserModule.addUser("Alice");
console.log(UserModule.count); // 1
```

### Modern: Block Scope for Variable Isolation

```javascript
// In a complex handler, use blocks to limit variable lifetime:
async function handleRequest(req) {
    let response;
    
    // Validation block
    {
        const schema = getSchema(req.type);
        const errors = validate(req.body, schema);
        if (errors.length > 0) {
            return { status: 400, errors };
        }
        // schema and errors are garbage-collectible after this block
    }
    
    // Processing block
    {
        const processed = transform(req.body);
        const result = await save(processed);
        response = { status: 200, data: result };
        // processed is garbage-collectible after this block
    }
    
    return response;
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the types of scope in JavaScript?**
**A:** Global scope, function scope, block scope (with let/const), and module scope (ES modules).

**Q2: What is lexical scoping?**
**A:** Lexical scoping means a variable's scope is determined by where it's declared in the source code, not where it's called. Inner functions can access variables from outer functions based on their position in the code.

**Q3: What is the scope chain?**
**A:** The scope chain is the chain of Lexical Environments from the current scope to the global scope. When a variable is accessed, the engine searches the current scope first, then follows the chain outward until the variable is found or the global scope is reached.

**Q4: What is variable shadowing?**
**A:** When an inner scope declares a variable with the same name as a variable in an outer scope, the inner variable "shadows" the outer one — the inner scope accesses its own variable, not the outer one.

**Q5: Does `var` respect block scope?**
**A:** No. `var` is function-scoped. It ignores block boundaries (if, for, while, {}). Only `let` and `const` are block-scoped.

### 🟡 Intermediate (5)

**Q6: Why does `let` in a module NOT create a window property, but `var` in a `<script>` tag does?**
**A:** `var` in the global scope is stored in the Global Environment's Object Record, which IS the global object (`window`). `let`/`const` are stored in the Declarative Record, which is separate from `window`. Modules have their own scope entirely — even `var` in a module doesn't touch `window`.

**Q7: How does the scope chain affect performance?**
**A:** Each variable access traverses the scope chain until the variable is found. Deeply nested scopes mean more traversal. V8 mitigates this with inline caching (caching the scope depth for repeated accesses), but initial lookups in deep chains are slower.

**Q8: What's the output?**
```javascript
function outer() {
    let x = 10;
    function inner() {
        console.log(x);
    }
    x = 20;
    return inner;
}
outer()();
```
**A:** `20`. The closure (`inner`) captures a **reference** to `x`'s binding, not a snapshot of its value. When `inner` executes, it reads the current value of `x`, which was updated to 20.

**Q9: Can you create a private variable in JavaScript without classes?**
**A:** Yes, using closures and function scope:
```javascript
function createPrivate() {
    let secret = 42; // Private — no external access
    return {
        get() { return secret; },
        set(val) { secret = val; }
    };
}
```

**Q10: What happens to scope when a function is passed as a callback?**
**A:** The function carries its scope chain (closure) with it. Even when called in a completely different context, it accesses variables from where it was **defined**, not where it's called. This is lexical scoping in action.

### 🔴 Advanced (5)

**Q11: How does V8 represent the scope chain internally?**
**A:** V8 uses a chain of Context objects on the heap for closures. Each Context has a `parent` pointer to the enclosing Context. Non-captured variables are register-allocated on the stack. The depth of scope chain traversal is computed at parse time and encoded in the bytecode (e.g., `LdaContextSlot depth=2` means "go up 2 contexts").

**Q12: What is the difference between scope and execution context?**
**A:** An **execution context** is the runtime wrapper that includes the scope (Lexical Environment), `this` binding, and Variable Environment. **Scope** (Lexical Environment) is a component of the execution context. Multiple execution contexts can share the same scope (e.g., closures).

**Q13: How does `with` affect the scope chain?**
**A:** `with(obj)` inserts `obj` at the top of the scope chain, making `obj`'s properties accessible as if they were local variables. This makes scope chain analysis impossible at compile time, which is why `with` is forbidden in strict mode and prevents V8 optimization.

**Q14: Explain how scope affects garbage collection.**
**A:** Variables are eligible for GC when their scope is no longer referenced. If a function returns and no closures reference its scope, the scope's variables are collectible. If a closure references the scope, the entire Context object (and all variables in it) is retained until the closure is unreachable.

**Q15: What is scope chain "flattening" in V8?**
**A:** When V8's TurboFan optimizes a function, it can inline the scope chain lookup directly. Instead of traversing context pointers at runtime, TurboFan generates machine code that reads directly from the correct context slot at the known depth. This effectively "flattens" the scope chain into a single memory read.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Trace the scope chain resolution for this code:**
```javascript
let a = 1;
function f1() {
    let b = 2;
    function f2() {
        let c = 3;
        console.log(a + b + c);
    }
    f2();
}
f1();
```

**A:**
1. Global scope is created: `a` binding in Declarative Record
2. `f1()` is called: new execution context, Lexical Environment has `b`, outer → Global
3. `f2()` is called: new execution context, Lexical Environment has `c`, outer → f1's env
4. `console.log(a + b + c)` evaluates:
   - `c`: lookup in f2's env → found (slot 0) → 3
   - `b`: lookup in f2's env → not found → follow outer → f1's env → found (slot 0) → 2
   - `a`: lookup in f2's env → not found → f1's env → not found → Global env → found → 1
   - Result: 6

**V8 bytecode for f2:**
```
LdaCurrentContextSlot [0]           // c (depth 0, slot 0)
Star r0
LdaImmutableContextSlot [0], [1]    // b (depth 1, slot 0)
Add r0
Star r0
LdaImmutableContextSlot [0], [2]    // a (depth 2, slot 0)
Add r0
// Result in accumulator: 6
```

**Q17: Explain why this closure preserves scope:**
```javascript
function createMultiplier(factor) {
    return function(n) {
        return n * factor;
    };
}
const double = createMultiplier(2);
console.log(double(5)); // 10
```

**A:**
1. `createMultiplier(2)` is called → execution context created with `factor = 2`
2. V8's scope analyzer detects that `factor` is captured by the inner function → context-allocated
3. A Context object is created on the heap: `{ parent: Global, factor: 2 }`
4. The inner function object is created with `[[Environment]]` → this Context object
5. `createMultiplier` returns the inner function → its stack frame is popped
6. But the Context object `{ factor: 2 }` is NOT freed — the returned function still references it
7. `double` variable stores the inner function
8. `double(5)` is called → new execution context for the inner function
9. `n = 5` is a parameter (register-allocated on the stack)
10. `n * factor` → n is 5 (stack), factor requires scope chain traversal: read from the Context's slot 0 → 2
11. Result: 5 * 2 = 10
12. The Context `{ factor: 2 }` lives as long as `double` exists. If `double = null`, the Context and `factor` become garbage-collectible.
