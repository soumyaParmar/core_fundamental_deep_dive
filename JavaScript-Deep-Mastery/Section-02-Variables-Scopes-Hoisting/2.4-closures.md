# 📌 2.4 — Closures
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** A closure is when a function "remembers" the variables from the scope where it was created, even after that scope has finished executing. It's like a function carrying a backpack of variables from its birthplace — no matter where you call it, it still has access to those variables.

**Technical Definition:** A closure is the combination of a function and its **lexical environment** (the Lexical Environment in which the function was defined). When a function is created, it captures a reference to the enclosing Lexical Environment via the internal `[[Environment]]` slot. This allows the function to access variables from outer scopes even after those scopes have returned.

**One-line mental model:**
> *A closure = a function + its birth scope. The function carries its scope with it forever.*

**Where does this live?**
- The function object → heap (with a `[[Environment]]` pointer)
- The captured scope → heap (V8 Context object that holds the closed-over variables)
- As long as the closure (function) is reachable, its Context object stays alive in memory

## ② DRILL DOWN

### 🔹 A. Lexical Environment Capture

Every function in JavaScript forms a closure. The question is whether it actually *uses* variables from outer scopes.

```javascript
function outer() {
    const message = "Hello from outer";
    
    function inner() {
        console.log(message); // inner "closes over" message
    }
    
    return inner;
}

const fn = outer(); // outer() has returned — its stack frame is gone
fn(); // "Hello from outer" — but inner still has access to message!
```

**Why does this work?** When `inner` was created, V8 stored a reference from `inner.[[Environment]]` to `outer`'s Context object (which contains `message`). Even though `outer()` returned (its stack frame was popped), the Context object stays alive on the heap because `inner` still references it.

```
After outer() returns:

Stack: (outer's frame is gone)

Heap:
fn ──► inner function object
          │
          └── [[Environment]] ──► Context {
                                    parent: → Global Context
                                    message: "Hello from outer"  ← still alive!
                                  }
```

### 🔹 B. Persistent Scope References

Closures capture the **reference to the variable**, not a **snapshot of its value**:

```javascript
function counter() {
    let count = 0;
    
    return {
        increment() { count++; },      // Closure over count
        decrement() { count--; },      // Same closure — same count!
        getCount()  { return count; }  // Same closure — same count!
    };
}

const c = counter();
c.increment();
c.increment();
c.increment();
console.log(c.getCount()); // 3

c.decrement();
console.log(c.getCount()); // 2
```

**All three methods share the SAME Context object:**
```
c.increment ──► [[Environment]] ──┐
c.decrement ──► [[Environment]] ──┤
c.getCount  ──► [[Environment]] ──┘
                                   ▼
                              Context {
                                count: 2  ← Shared by all three closures
                              }
```

**Closures capture the variable, not the value:**
```javascript
function makeGreeter() {
    let greeting = "Hello";
    
    const greet = () => console.log(greeting);
    
    greeting = "Goodbye"; // Change AFTER creating the closure
    
    return greet;
}

makeGreeter()(); // "Goodbye" — the closure sees the CURRENT value
// Because it holds a reference to the variable binding, not a copy of "Hello"
```

### 🔹 C. Closure Memory Retention

A closure keeps its entire Context object alive, including variables it may NOT directly use:

```javascript
function setup() {
    const smallData = "useful";
    const hugeData = new Array(10000000).fill("x"); // ~80MB
    
    return function() {
        return smallData; // Only uses smallData
        // BUT: does hugeData get garbage collected?
    };
}

const fn = setup();
// Answer: In modern V8, the engine is usually smart enough to NOT retain hugeData
// because the closure doesn't reference it (escape analysis).
// But in older engines or complex scenarios, hugeData might be retained.
```

**When V8 retains the entire Context (dangerous):**
```javascript
function setup() {
    const hugeData = new Array(10000000);
    const smallData = "useful";
    
    // Two closures sharing the same scope
    const useful = () => smallData;
    const dangerous = () => hugeData[0]; // References hugeData
    
    return useful; // Only return useful
    // BUT: useful and dangerous share the same Context
    // If V8 doesn't optimize this, the Context retains hugeData
    // even though the returned function never uses it
}
```

**Fix: Break the scope:**
```javascript
function setup() {
    const hugeData = new Array(10000000);
    const result = processData(hugeData); // Process and discard
    
    const smallData = result.summary;
    
    return function() {
        return smallData; // Only this small data is in scope
    };
    // hugeData is NOT in the returned closure's scope → GC can collect it
}
```

### 🔹 D. Practical Uses of Closures

#### 1. Data Privacy / Encapsulation
```javascript
function createBankAccount(initialBalance) {
    let balance = initialBalance; // Private — no external access
    
    return {
        deposit(amount) {
            if (amount <= 0) throw new Error("Invalid amount");
            balance += amount;
            return balance;
        },
        withdraw(amount) {
            if (amount > balance) throw new Error("Insufficient funds");
            balance -= amount;
            return balance;
        },
        getBalance() {
            return balance;
        }
    };
}

const account = createBankAccount(1000);
account.deposit(500);   // 1500
account.withdraw(200);  // 1300
// account.balance → undefined (private!)
```

#### 2. Function Factories
```javascript
function createMultiplier(factor) {
    return (number) => number * factor;
}

const double = createMultiplier(2);
const triple = createMultiplier(3);
const tenX   = createMultiplier(10);

console.log(double(5));  // 10
console.log(triple(5));  // 15
console.log(tenX(5));    // 50
```

#### 3. Memoization
```javascript
function memoize(fn) {
    const cache = new Map(); // Closed over by the returned function
    
    return function(...args) {
        const key = JSON.stringify(args);
        if (cache.has(key)) {
            return cache.get(key); // Cache hit
        }
        const result = fn.apply(this, args);
        cache.set(key, result);
        return result;
    };
}

const expensiveCalc = memoize((n) => {
    console.log("Computing...");
    return n * n;
});

expensiveCalc(5); // "Computing..." → 25
expensiveCalc(5); // → 25 (no "Computing..." — cached!)
```

#### 4. Event Handlers with State
```javascript
function createClickCounter(buttonId) {
    let clicks = 0;
    
    document.getElementById(buttonId).addEventListener('click', () => {
        clicks++; // Closure captures clicks
        console.log(`Button clicked ${clicks} times`);
    });
}

createClickCounter('myButton');
// Each click increments the shared `clicks` variable
```

#### 5. Currying
```javascript
function curry(fn) {
    return function curried(...args) {
        if (args.length >= fn.length) {
            return fn.apply(this, args);
        }
        return function(...moreArgs) {
            return curried.apply(this, [...args, ...moreArgs]);
        };
    };
}

const add = curry((a, b, c) => a + b + c);
add(1)(2)(3);     // 6
add(1, 2)(3);     // 6
add(1)(2, 3);     // 6
```

#### 6. Iterators
```javascript
function createRange(start, end) {
    let current = start;
    
    return {
        next() {
            if (current <= end) {
                return { value: current++, done: false };
            }
            return { done: true };
        }
    };
}

const range = createRange(1, 5);
console.log(range.next()); // { value: 1, done: false }
console.log(range.next()); // { value: 2, done: false }
// ... each call advances the closed-over `current`
```

### 🔹 E. Memory Leaks Due to Closures

#### Leak 1: Forgotten Event Listeners
```javascript
function setupHandler() {
    const heavyData = new Array(1000000).fill("data");
    
    const handler = () => {
        console.log(heavyData.length);
    };
    
    document.addEventListener('click', handler);
    // heavyData is retained forever because:
    // handler is retained by the event listener
    // handler closes over heavyData
    
    // Fix: remove the listener when done:
    // document.removeEventListener('click', handler);
}
```

#### Leak 2: Closures in setInterval
```javascript
function startPolling() {
    const cache = [];
    
    setInterval(() => {
        const data = fetchData(); // Returns new data each time
        cache.push(data);         // cache grows forever!
        // The closure keeps cache alive, and cache never shrinks
    }, 1000);
    
    // Fix: limit cache size or clear the interval
}
```

#### Leak 3: Shared Scope Retention
```javascript
function setup() {
    let bigArray = new Array(10000000);
    let smallValue = 42;
    
    // These two closures share the same scope
    globalThis.leak = () => smallValue;
    // Even though `leak` only uses smallValue,
    // V8 may keep bigArray alive because it's in the same Context
    
    // Fix: isolate scopes
    globalThis.leak = (function(val) {
        return () => val;
    })(smallValue);
    // Now bigArray is not in the closure's scope
}
```

### 🔹 F. Closures in Loops — The Classic Problem

```javascript
// ❌ Bug with var:
for (var i = 0; i < 5; i++) {
    setTimeout(() => console.log(i), i * 100);
}
// Output: 5, 5, 5, 5, 5
// All closures share the SAME `i` (function-scoped)
// By the time callbacks execute, i = 5

// ✅ Fix 1: Use let (block-scoped, new binding per iteration)
for (let i = 0; i < 5; i++) {
    setTimeout(() => console.log(i), i * 100);
}
// Output: 0, 1, 2, 3, 4

// ✅ Fix 2: IIFE (creates a new scope per iteration)
for (var i = 0; i < 5; i++) {
    (function(j) {
        setTimeout(() => console.log(j), j * 100);
    })(i);
}
// Output: 0, 1, 2, 3, 4

// ✅ Fix 3: bind (creates a new function with bound argument)
for (var i = 0; i < 5; i++) {
    setTimeout(console.log.bind(console, i), i * 100);
}
// Output: 0, 1, 2, 3, 4
```

**Memory diagram of the var problem:**
```
With var i:
All closures → same function scope → i: 5

With let i:
Closure 0 → Block Scope 0 { i: 0 }
Closure 1 → Block Scope 1 { i: 1 }
Closure 2 → Block Scope 2 { i: 2 }
Closure 3 → Block Scope 3 { i: 3 }
Closure 4 → Block Scope 4 { i: 4 }
```

## ③ WHY IT EXISTS

**What problem does it solve?**

Without closures, functions would only have access to:
1. Their own local variables
2. Global variables

This would mean:
- No encapsulation (everything would be global or local)
- No function factories
- No callbacks with state
- No iterators
- No module pattern

Closures are the mechanism that enables functional programming in JavaScript. Every callback, every event handler, every promise chain relies on closures.

**Historical context:**
- Closures come from **lambda calculus** and **Lisp** (1958)
- Brendan Eich was influenced by Scheme (a Lisp dialect) when designing JavaScript
- JavaScript was one of the first mainstream languages with first-class closures
- Java didn't get closures (lambdas) until Java 8 (2014)
- C++ didn't get closures (lambdas) until C++11 (2011)

**Comparison:**

| Language | Closures? | How |
|----------|----------|-----|
| JavaScript | ✅ Native | Functions automatically capture scope |
| Python | ✅ Native | Functions capture enclosing scope (but need `nonlocal` to reassign) |
| Java | ✅ (Java 8+) | Lambdas capture "effectively final" variables only |
| C++ | ✅ (C++11+) | Must explicitly specify capture list `[&x, y]` |
| Go | ✅ Native | Functions automatically capture by reference |
| Rust | ✅ Native | Closures with `Fn`, `FnMut`, `FnOnce` traits (ownership-aware) |

JavaScript's closures are the most "automatic" — they capture by reference with no extra syntax.

## ④ INTERNAL WORKING

### V8's Closure Implementation

When V8 creates a function that is a closure:

1. **Parse-time scope analysis** identifies which variables are "free variables" (used but not declared in the function)
2. **Free variables** must be stored in a **Context object** on the heap (not the stack)
3. The function object gets a `[[Environment]]` pointer to the Context
4. When the closure is called, V8 accesses captured variables through the Context chain

```javascript
function outer(x) {
    return function inner(y) {
        return x + y;
    };
}
const add5 = outer(5);
add5(3); // 8
```

**V8 internals:**

```
1. outer(5) is called:
   - Stack frame created for outer
   - x = 5 → scope analysis shows x is captured by inner → context-allocated
   - Context { parent: Global, x: 5 } created on heap
   - inner function object created with [[Environment]] → Context

2. outer returns inner → stack frame popped
   - Context { x: 5 } stays alive (inner references it)

3. add5(3) is called:
   - Stack frame created for inner (add5)
   - y = 3 → not captured, stays on stack
   - x + y → x: read from Context slot 0 (value: 5)
              y: read from stack register (value: 3)
   - Result: 8

Memory:
add5 ──► Function Object {
           code: (y) => x + y bytecode
           [[Environment]]: ──► Context {
                                  parent: Global Context
                                  x: 5       ← alive because add5 references this Context
                                }
         }
```

### Bytecode for Closure Variable Access

```
// inner function bytecode for `x + y`:
Ldar a0                             // Load parameter y from argument register
Star r0                             // Store y in r0
LdaImmutableContextSlot [0], [0]    // Load x from context (depth 0, slot 0)
Add r0                              // x + y
Return                              // Return result
```

### V8 Context Object Structure

```
Context Object (on heap):
┌──────────────────────────────────┐
│ Map:     → context_map           │
│ Length:  → 2 (header + 1 slot)   │
│ Scope Info: → scope descriptor   │
│ Previous: → parent Context       │  ← Scope chain link
│ Extension: → undefined           │
│ Slot[0]: → 5 (Smi, value of x)  │  ← Closed-over variable
└──────────────────────────────────┘
```

### What Happens When a Closure Is Garbage Collected

```javascript
function outer() {
    const big = new Array(10000000);
    return () => big.length;
}

let fn = outer();   // Context with big is alive
console.log(fn());  // 10000000

fn = null;          // fn is now null
// The closure function object is unreachable
// → The Context object is unreachable
// → big (the 10M array) is unreachable
// → All three are garbage collected on the next GC cycle
```

## ⑤ ADVANTAGES

1. **Data encapsulation without classes:** Closures provide private state before JavaScript had `#privateFields`. The counter, bank account, and module patterns all use closures for privacy.

2. **Stateful functions:** Closures let you create functions that remember state between calls — essential for memoization, counters, iterators, and state machines.

3. **Function factories:** Create specialized functions from generic ones. `createMultiplier(2)` returns a function forever bound to multiply by 2.

4. **Callback state:** When you pass a callback to `setTimeout`, `addEventListener`, or `Promise.then`, the callback can access variables from where it was defined — not where it's called.

5. **Foundation of functional programming:** Currying, partial application, composition, and higher-order functions all rely on closures.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Memory leaks — the #1 closure danger:**
   ```javascript
   // Every closure retains its entire Context
   // In long-running apps (servers, SPAs), forgotten closures accumulate memory
   const handlers = [];
   function addHandler() {
       const data = fetchExpensiveData(); // 10MB
       handlers.push(() => console.log(data.summary));
       // Each handler retains 10MB of data, even though it only needs .summary
   }
   // After 100 calls: 1GB of retained data!
   ```

2. **Stale closures in React:**
   ```javascript
   function Counter() {
       const [count, setCount] = useState(0);
       
       useEffect(() => {
           const id = setInterval(() => {
               console.log(count); // Always 0! Stale closure!
               // This closure captured count=0 and never updates
           }, 1000);
           return () => clearInterval(id);
       }, []); // Empty deps → closure created once, with count=0
       
       // Fix: use functional update or add count to deps
   }
   ```

3. **Performance overhead for deeply nested closures:**
   ```javascript
   function level1() {
       let a = 1;
       return function level2() {
           let b = 2;
           return function level3() {
               return a + b; // Two context traversals
           };
       };
   }
   // Each a access: traverse 2 context levels
   // V8's inline caching helps, but initial access is slower
   ```

4. **Unexpected variable sharing:**
   ```javascript
   function createFunctions() {
       const funcs = [];
       for (var i = 0; i < 3; i++) {
           funcs.push(() => i);
       }
       return funcs;
   }
   const fns = createFunctions();
   console.log(fns[0]()); // 3 — NOT 0!
   console.log(fns[1]()); // 3 — NOT 1!
   console.log(fns[2]()); // 3 — NOT 2!
   ```

5. **Closures prevent GC of outer scope:**
   ```javascript
   function process() {
       const tempBuffer = Buffer.alloc(100 * 1024 * 1024); // 100MB
       // ... process tempBuffer ...
       
       return {
           getResult() { return "done"; },
           // This method doesn't use tempBuffer, but V8 may retain it
           // because getResult and the tempBuffer share the same Context
       };
   }
   ```

## ⑦ PRODUCTION USAGE

### Pattern 1: Module Pattern (pre-ES6)

```javascript
const Logger = (function() {
    let logs = [];
    let level = 'info';
    
    function formatLog(msg, lvl) {
        return `[${new Date().toISOString()}] [${lvl.toUpperCase()}] ${msg}`;
    }
    
    return {
        setLevel(newLevel) { level = newLevel; },
        info(msg) { 
            if (['info', 'debug'].includes(level))
                logs.push(formatLog(msg, 'info'));
        },
        error(msg) {
            logs.push(formatLog(msg, 'error'));
        },
        getLogs() { return [...logs]; }, // Return copy, not reference
        clear() { logs = []; }
    };
})();

Logger.info("Server started");
Logger.error("Connection failed");
console.log(Logger.getLogs()); // Two entries
// Logger.logs → undefined (private!)
// Logger.formatLog → undefined (private!)
```

### Pattern 2: Debounce Implementation

```javascript
function debounce(fn, delay) {
    let timerId = null; // Closed over
    
    return function(...args) {
        clearTimeout(timerId);
        timerId = setTimeout(() => {
            fn.apply(this, args);
        }, delay);
    };
}

// Usage:
const handleSearch = debounce((query) => {
    fetch(`/api/search?q=${query}`);
}, 300);

input.addEventListener('input', (e) => handleSearch(e.target.value));
// timerId persists between calls via closure
```

### Pattern 3: Middleware Factory (Express/Koa)

```javascript
function rateLimit(maxRequests, windowMs) {
    const clients = new Map(); // Closed over — persists across requests
    
    return function middleware(req, res, next) {
        const ip = req.ip;
        const now = Date.now();
        
        if (!clients.has(ip)) {
            clients.set(ip, { count: 1, start: now });
            return next();
        }
        
        const client = clients.get(ip);
        if (now - client.start > windowMs) {
            client.count = 1;
            client.start = now;
            return next();
        }
        
        if (client.count >= maxRequests) {
            return res.status(429).json({ error: 'Rate limit exceeded' });
        }
        
        client.count++;
        next();
    };
}

app.use(rateLimit(100, 60000)); // 100 requests per minute
```

### Pattern 4: React Hook Patterns

```javascript
// Custom hook using closure for persistent state:
function useDebounce(value, delay) {
    const [debouncedValue, setDebouncedValue] = useState(value);
    
    useEffect(() => {
        const handler = setTimeout(() => {
            setDebouncedValue(value);
            // This closure captures the current `value` from the render
        }, delay);
        
        return () => clearTimeout(handler);
        // Cleanup closure also captures the `handler` from this render
    }, [value, delay]);
    
    return debouncedValue;
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is a closure?**
**A:** A closure is a function that has access to variables from its outer (enclosing) scope, even after the outer function has returned. It's the combination of a function and its lexical environment.

**Q2: What's the output?**
```javascript
function outer() {
    let x = 10;
    return function() { return x; };
}
const fn = outer();
console.log(fn());
```
**A:** `10`. The inner function forms a closure over `x`. Even though `outer()` has returned, `x` is still accessible.

**Q3: Do all functions in JavaScript create closures?**
**A:** Technically, yes — every function has a `[[Environment]]` reference to its enclosing scope. But we typically say a function "is a closure" when it actually uses variables from an outer scope.

**Q4: What's the output?**
```javascript
for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 0);
}
```
**A:** `3, 3, 3`. All callbacks share the same `i` (function-scoped `var`). By the time they execute, the loop has finished and `i` is 3.

**Q5: How do you fix the closure-in-loop problem?**
**A:** Use `let` instead of `var` (creates a new binding per iteration), use an IIFE wrapper, or use `.bind()` to fix the value.

### 🟡 Intermediate (5)

**Q6: What's the output?**
```javascript
function create() {
    let x = 1;
    return {
        get() { return x; },
        set(val) { x = val; }
    };
}
const obj = create();
obj.set(42);
console.log(obj.get());
```
**A:** `42`. Both `get` and `set` close over the same `x` variable. `set(42)` changes `x` in the shared Context, and `get()` reads the updated value.

**Q7: Can closures cause memory leaks?**
**A:** Yes. If a closure references a large variable and the closure is retained (e.g., via event listeners, global references, or timers), the large variable can never be garbage collected. This is one of the most common sources of memory leaks in JavaScript applications.

**Q8: What's the difference between a closure and a scope?**
**A:** A scope is the region where a variable is accessible (the Lexical Environment). A closure is a function + its captured scope. The scope exists independently; the closure preserves the scope beyond the normal lifetime of the enclosing function.

**Q9: Explain this code:**
```javascript
function makeCounter() {
    let count = 0;
    return () => ++count;
}
const a = makeCounter();
const b = makeCounter();
console.log(a(), a(), b());
```
**A:** `1, 2, 1`. Each `makeCounter()` call creates a new scope with its own `count`. `a` and `b` have independent closures with separate `count` variables.

**Q10: How does a closure affect garbage collection?**
**A:** A closure keeps its Context object (containing captured variables) alive as long as the closure function is reachable. If the closure is referenced by a global, a timer, or an event listener, the entire Context — including all captured variables — is retained and cannot be GC'd.

### 🔴 Advanced (5)

**Q11: What's the output and why?**
```javascript
function createFunctions() {
    var funcs = [];
    for (var i = 0; i < 3; i++) {
        funcs.push(function() { return i; });
    }
    return funcs;
}
var result = createFunctions();
console.log(result[0](), result[1](), result[2]());
```
**A:** `3, 3, 3`. All three functions close over the same `i` (var is function-scoped). After the loop, `i` is 3. Each function returns the current value of `i` when called.

**Q12: How does V8 implement closures internally?**
**A:** V8 creates a Context object on the heap for every scope that has variables captured by inner functions. The Context holds the captured variables and a `parent` pointer to the enclosing Context. Each function object has a `[[Environment]]` slot pointing to its Context. When a closure accesses a captured variable, V8 follows the Context chain to the appropriate depth and reads the slot.

**Q13: What is a "stale closure" and how does it happen in React?**
**A:** A stale closure occurs when a closure captures a variable at one point in time and continues to reference that old value even after the variable has been updated. In React, this happens with hooks: `useEffect` captures state values from a specific render. If the dependency array doesn't include the state variable, the effect closure uses the old value forever.

**Q14: Can closures cause V8 deoptimization?**
**A:** Closures themselves don't cause deoptimization, but they can prevent optimization. Context-allocated variables require heap reads (slower than register reads). If a closure accesses a variable through a deep context chain, V8 may not be able to inline the access. Additionally, closures that share a Context can prevent dead variable elimination.

**Q15: Write a function that creates a closure which properly cleans up after itself.**
**A:**
```javascript
function createResource() {
    let connection = openConnection();
    let buffer = Buffer.alloc(1024 * 1024);
    
    return {
        use() {
            return connection.query('SELECT *');
        },
        dispose() {
            connection.close();
            connection = null;  // Break reference
            buffer = null;      // Break reference
            // Now connection and buffer can be GC'd
        }
    };
}
const resource = createResource();
resource.use();
resource.dispose(); // Explicit cleanup
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Walk through the complete lifecycle of a closure in V8 memory.**

**A:**
```javascript
function outer() {
    let x = 42;
    return function inner() { return x; };
}
let fn = outer();
console.log(fn()); // 42
fn = null;
```

1. **Parse time:** V8's scope analyzer scans `outer`. Finds `x` is used by `inner` → marks `x` for context allocation.
2. **`outer()` called:**
   - Stack frame pushed for `outer`
   - Context object created on heap: `{ parent: Global, x: <the_hole> }`
   - `x = 42` → Context.slot[0] = Smi(42), TDZ ends
   - `inner` function object created on heap: `{ code: ..., [[Environment]]: → Context }`
3. **`outer()` returns `inner`:**
   - `outer`'s stack frame is popped (local variables gone)
   - But Context `{ x: 42 }` is NOT freed — `inner` (now `fn`) references it
4. **`fn()` called:**
   - Stack frame pushed for `inner`
   - `return x` → V8 follows `[[Environment]]` → Context → slot[0] → Smi(42)
   - Returns 42, stack frame popped
5. **`fn = null`:**
   - `fn` no longer references `inner` function object
   - `inner` function object becomes unreachable
   - Context `{ x: 42 }` becomes unreachable (no one references it)
   - Next GC cycle: inner function, Context, and Smi(42) are all freed

**Q17: Explain what happens when two closures share a scope:**
```javascript
function shared() {
    let a = 1;
    let b = new Array(1000000);
    
    function useA() { return a; }
    function useB() { return b[0]; }
    
    return useA; // Only return useA
}
let fn = shared();
```

**A:**
1. V8 scope analysis: both `a` and `b` are captured by inner functions → context-allocated
2. Context created: `{ parent: Global, a: 1, b: Array(1000000) }`
3. `useA` and `useB` are both created with `[[Environment]]` → same Context
4. `shared()` returns only `useA` → `useB` is unreachable
5. BUT: `useA` still references the Context, which contains BOTH `a` AND `b`!
6. The 1M-element array `b` is retained even though `useA` never accesses it

**In modern V8 (with optimization):**
- V8 may perform "context specialization" — if it can prove `useB` is unreachable, it may create a reduced Context with only `a`
- But this optimization is not guaranteed, especially in complex code
- The safe pattern: explicitly null out large variables you don't need: `b = null;` before returning

**Memory state:**
```
fn ──► useA function ──► [[Environment]] ──► Context {
                                               a: 1 (Smi)
                                               b: → Array[1000000] ← RETAINED!
                                             }
```

To fix: `return useA; b = null;` would NOT work (code after return doesn't execute). Instead:
```javascript
function shared() {
    let a = 1;
    {
        let b = new Array(1000000);
        // use b here
    }
    // b is out of scope — NOT in the closure
    function useA() { return a; }
    return useA;
}
```
