# 📌 3.1 — Primitive Types
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Primitive types are the simplest, most basic data types in JavaScript. They are **immutable** (cannot be changed after creation) and are compared **by value** (not by reference). There are exactly 7 primitive types: `string`, `number`, `bigint`, `boolean`, `undefined`, `null`, and `symbol`.

**Technical Definition:** A primitive value is data that is not an object and has no methods of its own. Primitives are stored directly in the variable's memory location (stack). When you call methods on primitives (like `"hello".toUpperCase()`), JavaScript temporarily wraps them in an object wrapper (autoboxing), calls the method, and discards the wrapper.

**One-line mental model:**
> *Primitives are raw values — they don't have properties, they're compared by their actual value, and they can't be mutated.*

**Where does this live?**
- Small integers → Smi (directly in the tagged pointer, no heap allocation)
- Doubles/large numbers → HeapNumber (heap-allocated 64-bit float)
- Short strings → may be inlined; longer strings → heap
- `true`, `false`, `undefined`, `null` → V8 singleton constants (odd-ball objects on heap, but treated as primitives)
- Symbols → heap-allocated (unique identity)
- BigInts → heap-allocated (arbitrary precision)

## ② DRILL DOWN

### 🔹 A. string

```javascript
// String literals:
const s1 = "hello";        // Double quotes
const s2 = 'hello';        // Single quotes (identical)
const s3 = `hello`;        // Template literal (backticks)
const s4 = `Hello, ${s1}`; // Template with expression: "Hello, hello"

// String properties:
console.log(s1.length); // 5
console.log(s1[0]);     // "h" (bracket access)
console.log(s1.charAt(0)); // "h"

// Strings are IMMUTABLE:
let str = "hello";
str[0] = "H";       // Silent fail — does nothing!
console.log(str);    // "hello" (unchanged)
str = "Hello";       // This works — you're reassigning the variable, not mutating the string

// String methods return NEW strings:
const upper = "hello".toUpperCase(); // "HELLO" (new string)
const sliced = "hello".slice(1, 3);  // "el" (new string)
```

**V8 String Internals:**
```
V8 has multiple string representations:
┌─────────────────────────────┐
│ SeqOneByteString             │ ← ASCII/Latin1 (1 byte per char)
│ SeqTwoByteString             │ ← Unicode (2 bytes per char, UTF-16)
│ ConsString                   │ ← Concatenation: left + right (lazy)
│ SlicedString                 │ ← Substring: parent + offset + length
│ ExternalString               │ ← Points to external (C++) memory
│ InternalizedString           │ ← Unique, used for property names
│ ThinString                   │ ← Pointer to internalized version
└─────────────────────────────┘
```

```javascript
// ConsString optimization:
const a = "Hello";
const b = " World";
const c = a + b; // V8 creates a ConsString: { left: a, right: b }
                  // The actual concatenation is DEFERRED until c is read
                  // This makes repeated concatenation O(1) instead of O(n)

// Internalized strings:
const obj = { name: "John" }; // "name" is internalized → singleton in string table
const key = "name";           // If "name" already exists, V8 reuses the same pointer
obj[key];                      // Property lookup uses pointer comparison (fast!)
```

**String edge cases:**
```javascript
// UTF-16 encoding issues:
const emoji = "😀";
console.log(emoji.length);     // 2! (emoji is a surrogate pair: two UTF-16 code units)
console.log([...emoji].length); // 1 (spread uses Symbol.iterator — counts code points)

// String comparison:
"a" < "b"   // true (lexicographic, by code point)
"10" < "9"  // true! ("1" < "9" in first character comparison)
"café" === "café" // might be false! (é can be one codepoint or two: e + combining accent)

// Template literals:
const multiline = `
    Line 1
    Line 2
`; // Includes the newlines and leading spaces!

// Tagged template literals:
function tag(strings, ...values) {
    console.log(strings); // ["Hello ", "!"]
    console.log(values);  // ["World"]
}
tag`Hello ${"World"}!`;
```

### 🔹 B. number

JavaScript has ONE number type: 64-bit IEEE 754 double-precision floating point.

```javascript
// All of these are the same type:
const int = 42;
const float = 3.14;
const negative = -7;
const scientific = 1.5e10;  // 15000000000
const hex = 0xFF;            // 255
const octal = 0o77;          // 63
const binary = 0b1010;       // 10

// Special number values:
console.log(Infinity);       // Infinity
console.log(-Infinity);      // -Infinity
console.log(NaN);            // NaN (Not a Number)
console.log(Number.MAX_SAFE_INTEGER); // 9007199254740991 (2^53 - 1)
console.log(Number.MIN_SAFE_INTEGER); // -9007199254740991
console.log(Number.EPSILON);          // 2.220446049250313e-16
```

**The floating-point trap:**
```javascript
console.log(0.1 + 0.2);           // 0.30000000000000004 (NOT 0.3!)
console.log(0.1 + 0.2 === 0.3);   // false!

// Why? 0.1 and 0.2 cannot be exactly represented in binary floating point
// 0.1 in binary: 0.0001100110011001100... (repeating)

// Fix 1: Use epsilon comparison:
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON; // true

// Fix 2: Use integers (cents instead of dollars):
const priceInCents = 1099; // $10.99 as integer
```

**NaN quirks:**
```javascript
console.log(NaN === NaN);    // false! NaN is not equal to itself
console.log(NaN !== NaN);    // true — the only value where this is true
console.log(typeof NaN);     // "number" — NaN IS a number type
console.log(isNaN("hello")); // true (coerces to number first)
console.log(Number.isNaN("hello")); // false (strict check, no coercion)

// How to check for NaN:
const x = NaN;
Number.isNaN(x);  // true ← Use this
x !== x;          // true ← Works but obscure
```

**V8 number representation:**
```
Smi (Small Integer):
- Range: -2^30 to 2^30 - 1 (on 64-bit with pointer compression)
- Stored directly in the tagged pointer (no heap allocation)
- Used for: loop counters, array indices, most integer arithmetic

HeapNumber:
- IEEE 754 double (64-bit float)
- Allocated on the heap
- Used for: decimals, large integers, NaN, Infinity
```

### 🔹 C. bigint

Arbitrary-precision integers, introduced in ES2020.

```javascript
const big = 9007199254740992n; // Note the 'n' suffix
const big2 = BigInt("9007199254740992");
const big3 = BigInt(42); // 42n

// BigInt arithmetic:
console.log(big + 1n);    // 9007199254740993n (exact!)
console.log(2n ** 100n);  // 1267650600228229401496703205376n

// Cannot mix BigInt and Number:
console.log(1n + 1);     // TypeError! Cannot mix BigInt and other types
console.log(1n + BigInt(1)); // 2n ✅

// Comparison works across types:
console.log(1n === 1);   // false (different types)
console.log(1n == 1);    // true (loose equality with coercion)
console.log(2n > 1);     // true

// No decimals:
console.log(5n / 2n);    // 2n (truncated, not 2.5)
```

### 🔹 D. boolean

```javascript
const t = true;
const f = false;

// Boolean conversion (truthy/falsy):
// Falsy values (exactly 8):
Boolean(false);     // false
Boolean(0);         // false
Boolean(-0);        // false
Boolean(0n);        // false
Boolean("");        // false
Boolean(null);      // false
Boolean(undefined); // false
Boolean(NaN);       // false

// EVERYTHING else is truthy:
Boolean("0");       // true (non-empty string!)
Boolean([]);        // true (empty array!)
Boolean({});        // true (empty object!)
Boolean("false");   // true (non-empty string!)
Boolean(-1);        // true (non-zero number!)
```

### 🔹 E. undefined

```javascript
// undefined means "declared but not assigned" or "missing value"
let x;
console.log(x); // undefined

// Function with no return:
function f() {}
console.log(f()); // undefined

// Missing parameter:
function greet(name) { console.log(name); }
greet(); // undefined

// Missing property:
const obj = { a: 1 };
console.log(obj.b); // undefined (property doesn't exist)

// typeof undefined:
console.log(typeof undefined); // "undefined"
console.log(typeof undeclaredVar); // "undefined" (no error!)

// undefined is a global property (not a keyword in sloppy mode!):
// In sloppy mode, you can (but NEVER should) reassign it:
// undefined = 42; // Works in old browsers! Don't ever do this.
// In strict mode: TypeError

// void operator always returns undefined:
console.log(void 0);     // undefined (reliable way to get undefined)
console.log(void "abc"); // undefined
```

### 🔹 F. null

```javascript
// null means "intentionally empty" or "no value"
let x = null;

// typeof null — THE famous JavaScript bug:
console.log(typeof null); // "object" ← BUG from 1995, never fixed for backward compat

// Proper null check:
x === null;                      // true ✅
x == null;                       // true (also matches undefined)
x == undefined;                  // true (null == undefined is true)
x === undefined;                 // false (strict: null !== undefined)

// null vs undefined:
// undefined = "nothing was assigned"
// null = "I explicitly set this to nothing"
let a;         // undefined — not set yet
let b = null;  // null — intentionally empty

// In JSON:
JSON.stringify({ a: undefined, b: null }); // '{"b":null}' — undefined is omitted!
```

### 🔹 G. symbol

```javascript
// Symbols are unique, immutable identifiers
const sym1 = Symbol();
const sym2 = Symbol();
console.log(sym1 === sym2); // false — ALWAYS unique

// Description (for debugging):
const sym3 = Symbol("id");
console.log(sym3.toString()); // "Symbol(id)"
console.log(sym3.description); // "id"

// Symbols as object keys (private-ish properties):
const SECRET = Symbol("secret");
const obj = {
    name: "public",
    [SECRET]: "hidden"
};
console.log(obj.name);     // "public"
console.log(obj[SECRET]);  // "hidden"
console.log(Object.keys(obj)); // ["name"] — Symbol keys are not enumerated!

// Global symbol registry:
const s1 = Symbol.for("app.id"); // Create or retrieve from global registry
const s2 = Symbol.for("app.id"); // Retrieved from registry
console.log(s1 === s2); // true! (same global symbol)
console.log(Symbol.keyFor(s1)); // "app.id"

// Well-known symbols:
Symbol.iterator;     // Defines iteration behavior (for...of)
Symbol.toPrimitive;  // Defines type conversion behavior
Symbol.hasInstance;   // Defines instanceof behavior
Symbol.species;      // Defines constructor for derived objects
```

## ③ WHY IT EXISTS

**Why 7 primitives instead of just objects?**

1. **Performance:** Primitives (especially Smi integers) are stored directly on the stack — no heap allocation, no GC overhead. If everything were an object, simple arithmetic would require heap allocation.

2. **Simplicity:** `42 + 3` should be fast and straightforward. Object-wrapping every number would make basic operations slow and complex.

3. **Value semantics:** When you copy a primitive, you get a truly independent copy. This prevents accidental mutation — `let b = a; b = 5;` doesn't change `a`.

**Historical evolution:**
- ES1 (1997): 5 primitives — `string`, `number`, `boolean`, `null`, `undefined`
- ES6 (2015): Added `symbol` — for unique property keys and metaprogramming
- ES2020: Added `bigint` — for arbitrary-precision integer arithmetic

## ④ INTERNAL WORKING

### V8 Tagged Values

V8 uses tagged values to represent all JavaScript values in a uniform 64-bit format:

```
Tagged Value (64-bit with pointer compression):
┌──────────────────────────────────┐
│ Smi: value (31 bits) | tag: 0   │ ← No heap allocation
├──────────────────────────────────┤
│ Heap pointer (aligned) | tag: 1 │ ← Points to heap object
└──────────────────────────────────┘

Smi examples:
  42 → 0x000000540 (42 shifted left, tag = 0)
  -1 → 0xFFFFFFF0 (two's complement, tag = 0)

HeapNumber (for 3.14):
  Tagged pointer → HeapNumber object on heap
  HeapNumber { map: heap_number_map, value: IEEE754(3.14) }
```

### Autoboxing (Primitive Wrappers)

When you call a method on a primitive, V8 temporarily wraps it:

```javascript
"hello".toUpperCase();
// Internally:
// 1. V8 sees .toUpperCase() on a string primitive
// 2. V8 creates a temporary String wrapper object
// 3. Calls String.prototype.toUpperCase on the wrapper
// 4. Returns the result ("HELLO")
// 5. The wrapper is discarded (never actually allocated in optimized code)

// In V8's optimized path, this wrapper is NEVER actually created.
// TurboFan directly calls the built-in string function.
```

### typeof Implementation

```javascript
typeof 42;        // "number"  → V8 checks tag bit: 0 = Smi → "number"
typeof 3.14;      // "number"  → V8 checks tag bit: 1 → dereference → is HeapNumber? → "number"
typeof "hello";   // "string"  → V8 checks instance type of heap object → is String? → "string"
typeof true;      // "boolean" → V8 checks if value is the true_value oddball → "boolean"
typeof undefined; // "undefined" → V8 checks if value is the undefined oddball → "undefined"
typeof null;      // "object"  → V8 checks tag: 1, instance type: Null → but returns "object" (spec bug)
typeof Symbol();  // "symbol"  → V8 checks instance type → Symbol → "symbol"
typeof 42n;       // "bigint"  → V8 checks instance type → BigInt → "bigint"
```

## ⑤ ADVANTAGES

1. **Speed:** Smi integers are as fast as native C integers — no heap allocation, no GC, direct arithmetic.
2. **Simplicity:** `===` on primitives compares values, not references. `42 === 42` is always true.
3. **Immutability:** You can't corrupt a primitive's value. `"hello"` will always be `"hello"`.
4. **Memory efficiency:** Smis use zero extra memory. Strings are shared via internalization. Booleans are singletons.
5. **Predictability:** No aliasing problems. `let b = a; b = 5;` never affects `a`.

---

## ⑥ DISADVANTAGES / TRAPS

1. **`typeof null === "object"`** — a 1995 bug that can never be fixed.
2. **Floating-point precision:** `0.1 + 0.2 !== 0.3` surprises everyone.
3. **String immutability confusion:** `str[0] = 'X'` silently fails.
4. **NaN !== NaN:** The only value that is not equal to itself.
5. **Autoboxing creates confusion:** `"hello".x = 5; console.log("hello".x); // undefined`
6. **BigInt/Number mixing:** `1n + 1` throws TypeError.

```javascript
// Currency: Use integers (cents) to avoid floating-point:
const priceInCents = 1099;
const display = `$${(priceInCents / 100).toFixed(2)}`; // "$10.99"
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: Name all 7 primitive types.** **A:** string, number, bigint, boolean, undefined, null, symbol.

**Q2: What does `typeof null` return?** **A:** `"object"` — a bug from the first JS implementation, kept for backward compatibility.

**Q3: Is `NaN === NaN` true?** **A:** No. NaN is the only JavaScript value not equal to itself. Use `Number.isNaN()` to check.

**Q4: What are falsy values?** **A:** `false`, `0`, `-0`, `0n`, `""`, `null`, `undefined`, `NaN` — exactly 8 values.

**Q5: Are strings mutable?** **A:** No. Strings are immutable. `str[0] = 'X'` silently fails. Methods like `.toUpperCase()` return new strings.

### 🟡 Intermediate (5)

**Q6: Why does `0.1 + 0.2 !== 0.3`?** **A:** JavaScript uses IEEE 754 double-precision floats. `0.1` and `0.2` can't be exactly represented in binary, so their sum has a tiny rounding error.

**Q7: What is the difference between `null` and `undefined`?** **A:** `undefined` means a variable exists but has no assigned value. `null` means "intentionally empty." `undefined` is the default for uninitialized variables; `null` is explicitly set by developers.

**Q8: What is a Symbol and why would you use it?** **A:** A Symbol is a unique, immutable identifier. Use it for non-enumerable property keys (avoiding naming collisions), metaprogramming (well-known symbols like Symbol.iterator), and library/framework-internal properties.

**Q9: How does V8 store the number `42` vs `3.14`?** **A:** `42` is stored as a Smi (Small Integer) — directly in the tagged pointer, no heap allocation. `3.14` is stored as a HeapNumber — a heap object containing a 64-bit IEEE 754 double.

**Q10: What's the difference between `==` and `===` for primitives?** **A:** `===` checks type AND value (no coercion). `==` performs type coercion first. Example: `1 == "1"` is `true` (coercion), `1 === "1"` is `false`.

### 🔴 Advanced (5)

**Q11: What is autoboxing and how does V8 optimize it?** **A:** Autoboxing wraps primitives in object wrappers to call methods (`"hi".length`). In V8's optimized code (TurboFan), the wrapper is never actually allocated — V8 directly calls the built-in function on the raw value, avoiding the overhead.

**Q12: Why is `typeof` not enough for null checking?** **A:** Because `typeof null === "object"`. A proper null check requires `value === null` or `value == null` (also catches undefined).

**Q13: What is string internalization in V8?** **A:** V8 maintains a string table of unique strings. Strings used as property names are "internalized" — deduplicated so the same string is only stored once. This enables property lookup via pointer comparison instead of string comparison.

**Q14: What is the maximum safe integer and why?** **A:** `Number.MAX_SAFE_INTEGER` is `2^53 - 1` (9007199254740991). IEEE 754 doubles have a 52-bit mantissa (plus 1 implied bit). Beyond this, integers lose precision: `2^53 + 1 === 2^53` evaluates to `true`.

**Q15: How does `Symbol.toPrimitive` work?** **A:** It's a well-known Symbol that defines how an object converts to a primitive. When used in type coercion, JS calls `obj[Symbol.toPrimitive](hint)` where hint is "number", "string", or "default".

### 🧠 "Explain What Happens Internally" (2)

**Q16: What happens in V8 when you do `"hello".toUpperCase()`?**

**A:** 1. V8 sees a property access (`.toUpperCase`) on a string primitive. 2. In unoptimized code: V8 checks the primitive type → String → looks up `String.prototype.toUpperCase`. 3. In optimized code: TurboFan recognizes the pattern and directly emits machine code that calls the built-in `StringToUpperCase` function on the raw character data — no wrapper object is ever allocated. 4. V8 allocates a new string on the heap with the uppercase characters. 5. Returns the new string. The original `"hello"` is unchanged (immutable).

**Q17: What happens when you compare two strings with `===`?**

**A:** 1. V8 checks if both values are strings (type check). 2. If both are internalized strings (property names), V8 does a **pointer comparison** — O(1). 3. If one or both are not internalized, V8 first compares lengths (fast reject if different). 4. If lengths match, V8 compares character-by-character. 5. For OneByteStrings: memcmp of the byte arrays. For TwoByteStrings: memcmp of 2-byte arrays. 6. ConsStrings are flattened before comparison. This is why string comparison can be O(1) for short/internalized strings but O(n) for long non-internalized strings.
