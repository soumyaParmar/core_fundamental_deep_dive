# 📌 3.2 — Reference Types
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Reference types are complex data structures that hold collections of values and/or functionality. Unlike primitives (which store values directly), reference types store a **pointer/reference** to an object on the heap. When you assign a reference type to another variable, you copy the pointer — both variables point to the same object.

**Technical Definition:** Reference types are objects — instances of built-in or user-defined constructors allocated on the V8 heap. They include Object, Array, Function, Date, RegExp, Map, Set, WeakMap, WeakSet, and any user-defined class instances. They are mutable, compared by reference identity (not value), and garbage collected when unreachable.

**One-line mental model:**
> *Reference types = objects on the heap. Variables hold pointers to them, not the data itself. Copy the variable = copy the pointer, not the object.*

## ② DRILL DOWN

### 🔹 Object

The fundamental reference type. Everything else (Array, Function, Map, etc.) inherits from Object.

```javascript
// Object creation:
const obj1 = {};                          // Object literal (most common)
const obj2 = new Object();                // Constructor (rarely used)
const obj3 = Object.create(null);         // No prototype (truly empty)
const obj4 = Object.create(Object.prototype); // Same as {}

// Property access:
const user = { name: "Alice", "last-name": "Smith", 42: "answer" };
user.name;           // "Alice" (dot notation)
user["last-name"];   // "Smith" (bracket notation — needed for special chars)
user[42];            // "answer" (bracket notation — numeric keys)

// Computed properties:
const key = "email";
const obj = { [key]: "alice@example.com" }; // { email: "alice@example.com" }

// Shorthand:
const name = "Alice";
const age = 30;
const person = { name, age }; // { name: "Alice", age: 30 }

// Methods:
const calc = {
    add(a, b) { return a + b; },        // Method shorthand
    multiply: function(a, b) { return a * b; }, // Traditional
    divide: (a, b) => a / b,            // Arrow (different `this`!)
};

// Property iteration:
Object.keys(obj);    // Own enumerable string keys
Object.values(obj);  // Own enumerable values
Object.entries(obj); // Own enumerable [key, value] pairs
for (const key in obj) {} // Own + inherited enumerable keys (use hasOwnProperty!)
```

### 🔹 Array

Ordered, indexed collection. In V8, arrays are specialized objects.

```javascript
const arr = [1, 2, 3];
const arr2 = new Array(5);      // [empty × 5] (sparse!)
const arr3 = Array.from({length: 3}, (_, i) => i); // [0, 1, 2]
const arr4 = Array.of(1, 2, 3); // [1, 2, 3]

// Arrays are objects:
typeof []; // "object"
Array.isArray([]); // true (reliable check)

// Arrays can hold mixed types:
const mixed = [1, "two", true, null, {x: 1}, [2, 3]];

// Sparse vs dense:
const sparse = [1, , , 4]; // Holes! sparse[1] === undefined but slot is empty
const dense = [1, 2, 3, 4]; // No holes — V8 uses fast elements

// Array length:
const a = [1, 2, 3];
a.length = 1; // Truncates: a is now [1]
a.length = 5; // Extends: a is now [1, empty × 4]
```

### 🔹 Function

Functions are first-class objects in JavaScript — they can be assigned to variables, passed as arguments, returned from functions, and have properties.

```javascript
function greet(name) { return `Hello, ${name}`; } // Declaration
const greet2 = function(name) { return `Hi, ${name}`; }; // Expression
const greet3 = (name) => `Hey, ${name}`; // Arrow

// Functions are objects:
typeof greet; // "function" (special typeof result)
greet instanceof Object; // true
greet.name;   // "greet" (function name property)
greet.length; // 1 (number of expected parameters)

// Functions have prototypes:
function Foo() {}
Foo.prototype; // { constructor: Foo }

// Functions can have properties:
function counter() { counter.count++; return counter.count; }
counter.count = 0;
counter(); // 1
counter(); // 2
```

### 🔹 Date

```javascript
const now = new Date();                    // Current date/time
const specific = new Date(2024, 0, 15);   // Jan 15, 2024 (month is 0-indexed!)
const fromString = new Date("2024-01-15"); // ISO 8601
const fromMs = new Date(1705276800000);    // Milliseconds since epoch

now.getFullYear();  // 2024
now.getMonth();     // 0-11 (NOT 1-12!)
now.getDate();      // 1-31 (day of month)
now.getDay();       // 0-6 (day of week, 0 = Sunday)
now.getTime();      // Milliseconds since epoch
now.toISOString();  // "2024-01-15T00:00:00.000Z"

// Date is mutable (unlike most modern libraries):
now.setFullYear(2025); // Mutates the date object!
```

### 🔹 RegExp

```javascript
const re1 = /pattern/flags;        // Literal
const re2 = new RegExp("pattern", "flags"); // Constructor

// Common methods:
/hello/.test("hello world"); // true
"hello world".match(/(\w+)/g); // ["hello", "world"]
"hello".replace(/l/g, "L"); // "heLLo"

// Flags: g (global), i (case-insensitive), m (multiline), s (dotAll), u (unicode), y (sticky)

// Named groups:
const match = "2024-01-15".match(/(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/);
match.groups.year; // "2024"
```

### 🔹 Map

```javascript
const map = new Map();
map.set("key", "value");
map.set(42, "number key");
map.set({}, "object key"); // Objects can be keys!

map.get("key");  // "value"
map.has("key");  // true
map.size;        // 3
map.delete("key");

// Map vs Object:
// Map: any type as key, maintains insertion order, has .size, no prototype pollution
// Object: string/symbol keys only, no guaranteed order (though modern engines do maintain it)

// Iteration:
for (const [key, value] of map) { console.log(key, value); }
map.forEach((value, key) => console.log(key, value));
```

### 🔹 Set

```javascript
const set = new Set([1, 2, 3, 2, 1]); // {1, 2, 3} — duplicates removed!
set.add(4);
set.has(3);    // true
set.size;      // 4
set.delete(2);

// Unique array:
const unique = [...new Set([1, 2, 2, 3, 3, 3])]; // [1, 2, 3]

// Set uses SameValueZero comparison (like ===, but NaN === NaN):
const s = new Set();
s.add(NaN);
s.add(NaN);
s.size; // 1 — NaN is treated as equal to NaN in Sets!
```

### 🔹 WeakMap & WeakSet

```javascript
// WeakMap: keys MUST be objects, keys are held weakly (GC can collect them)
const wm = new WeakMap();
let obj = { data: "important" };
wm.set(obj, "metadata");
wm.get(obj); // "metadata"
obj = null;  // The object can now be GC'd (WeakMap doesn't prevent it!)
// After GC: the entry is automatically removed from the WeakMap

// WeakSet: values MUST be objects, held weakly
const ws = new WeakSet();
let el = document.querySelector('.item');
ws.add(el);
ws.has(el); // true
el = null;  // The DOM node can be GC'd

// Neither is iterable — you can't list entries:
// wm.keys() → doesn't exist
// ws.forEach() → doesn't exist
// This is by design — you can't enumerate weak references
```

## ③ WHY IT EXISTS

Reference types exist because primitives can't represent complex data. You need:
- **Collections** (Arrays, Maps, Sets) — to group related data
- **Structured data** (Objects) — to represent entities with multiple properties
- **Behavior** (Functions) — to encapsulate reusable logic
- **Weak references** (WeakMap, WeakSet) — to associate metadata without preventing GC

**Why references instead of copies?** Copying large objects (e.g., a 1MB array) on every assignment would be extremely slow. References make assignment O(1) — just copy a pointer.

## ④ INTERNAL WORKING

### V8 Object Layout

```
JSObject (on heap):
┌─────────────────────────┐
│ Map (hidden class ptr)   │ → Describes shape/layout
│ Properties (ptr)         │ → Named properties (overflow or dictionary)
│ Elements (ptr)           │ → Indexed properties (for arrays)
│ [In-object prop 0]      │ → Fast property slot
│ [In-object prop 1]      │ → Fast property slot
│ ...                      │
└─────────────────────────┘
```

**V8 Array Representations:**

| Elements Kind | Description | Example |
|--------------|-------------|---------|
| PACKED_SMI_ELEMENTS | Dense, all Smis | `[1, 2, 3]` |
| PACKED_DOUBLE_ELEMENTS | Dense, all doubles | `[1.1, 2.2, 3.3]` |
| PACKED_ELEMENTS | Dense, mixed types | `[1, "two", true]` |
| HOLEY_SMI_ELEMENTS | Has holes, Smis | `[1, , 3]` |
| DICTIONARY_ELEMENTS | Sparse, hash table | `a = []; a[10000] = 1` |

Transitions only go "down" (more general):
```
PACKED_SMI → PACKED_DOUBLE → PACKED_ELEMENTS
     ↓              ↓               ↓
HOLEY_SMI  → HOLEY_DOUBLE  → HOLEY_ELEMENTS → DICTIONARY
```

---

## ⑤ ADVANTAGES

1. **Flexibility:** Objects can represent any data structure — maps, lists, trees, graphs.
2. **Mutability:** You can modify objects in place, avoiding the cost of copying.
3. **Reference sharing:** Multiple parts of code can work with the same object.
4. **Prototype chain:** Objects inherit behavior from prototypes — powerful code reuse.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Reference aliasing:** `const b = a;` means both mutate the same object.
2. **Accidental mutation:** Passing objects to functions lets the function modify the original.
3. **Equality confusion:** `{} === {}` is `false` — reference comparison, not value.
4. **Memory leaks:** Objects stay alive as long as any reference exists.
5. **Array element kind degradation:** Adding a string to a `PACKED_SMI` array permanently degrades it.

---

## ⑦ PRODUCTION USAGE

```javascript
// Use Map instead of Object for dynamic keys:
const cache = new Map();
cache.set(userId, userData); // Keys can be any type, no prototype pollution

// Use Set for unique collections:
const activeUsers = new Set();
activeUsers.add(user.id);
if (activeUsers.has(user.id)) { /* already active */ }

// Use WeakMap for DOM metadata:
const elementData = new WeakMap();
elementData.set(domNode, { clickCount: 0 });
// When domNode is removed from DOM and GC'd, the metadata is automatically cleaned

// Use Object.freeze for config:
const CONFIG = Object.freeze({ api: "https://api.example.com" });
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is the difference between primitive and reference types?** **A:** Primitives store values directly, are immutable, and compared by value. Reference types store a pointer to a heap object, are mutable, and compared by reference.

**Q2: What does `typeof []` return?** **A:** `"object"`. Arrays are objects. Use `Array.isArray()` for reliable array detection.

**Q3: How do you check if something is an array?** **A:** `Array.isArray(value)`. Don't use `typeof` (returns "object") or `instanceof` (fails across iframes/realms).

**Q4: What is the difference between Map and Object?** **A:** Map allows any type as keys, maintains insertion order, has `.size`, is iterable, and has no prototype pollution risk. Object only allows string/symbol keys.

**Q5: What is a WeakMap?** **A:** A Map where keys must be objects and are held weakly — they don't prevent garbage collection. Used for associating metadata with objects without causing memory leaks.

### 🟡 Intermediate (5)

**Q6: Why is `[] === []` false?** **A:** Arrays are objects, and objects are compared by reference. Two separate array literals create two different objects on the heap with different memory addresses.

**Q7: How do V8 element kinds affect Array performance?** **A:** V8 tracks the types of elements in arrays (PACKED_SMI, PACKED_DOUBLE, PACKED_ELEMENTS, etc.). Homogeneous arrays (all same type) are stored more efficiently and accessed faster. Mixing types or creating holes degrades performance.

**Q8: What's the difference between WeakMap and Map?** **A:** WeakMap keys are held weakly (GC can collect them), keys must be objects, and WeakMap is not iterable. Map holds strong references, allows any key type, and is iterable.

**Q9: How does a Set determine uniqueness?** **A:** Sets use SameValueZero algorithm: like `===` but treats `NaN === NaN` as true. For objects, uniqueness is by reference (two different objects with same properties are both added).

**Q10: Why is Date considered a problematic API?** **A:** Months are 0-indexed (0=Jan), it's mutable (`.setFullYear()` changes the object), timezone handling is inconsistent, parsing is unreliable across browsers, and there's no duration type.

### 🔴 Advanced (5)

**Q11: How does V8 decide between fast (in-object) and slow (dictionary) properties?** **A:** Objects start in fast mode with in-object properties at fixed offsets. V8 switches to dictionary mode (hash table) when: too many property additions/deletions, `delete` is used, or too many shape transitions occur. Dictionary mode is slower but more flexible.

**Q12: What is the difference between `Object.create(null)` and `{}`?** **A:** `Object.create(null)` creates an object with no prototype (`__proto__` is `null`). It has no inherited methods like `toString`, `hasOwnProperty`, etc. `{}` inherits from `Object.prototype`. `Object.create(null)` is useful for dictionary-like objects without prototype pollution risk.

**Q13: What are V8 element kind transitions and why do they matter?** **A:** V8 categorizes array elements (SMI, DOUBLE, generic, holey). Transitions only go toward more general types — never back. Once an array becomes HOLEY_ELEMENTS, it can never go back to PACKED_SMI_ELEMENTS, even if you fill the holes. This affects property access speed permanently.

**Q14: How does WeakRef differ from WeakMap?** **A:** WeakRef holds a weak reference to a single object (you can `deref()` to get it, or get `undefined` if GC'd). WeakMap associates keys weakly with values — you can't deref individual keys. WeakRef gives direct access to the referenced object; WeakMap gives access to associated values.

**Q15: Can you use a Symbol as a Map key?** **A:** Yes. Map accepts any value as a key, including Symbols. This is one advantage over regular Objects, where keys are limited to strings and Symbols.

### 🧠 "Explain What Happens Internally" (2)

**Q16: What happens in V8 when you push an item to an array?** **A:** V8 checks if the array's backing store has capacity. If yes, it writes the value at the next index and increments length. If no capacity, V8 allocates a new, larger backing store (typically 1.5x or 2x), copies existing elements, writes the new value, and updates the array's elements pointer. If the new value's type differs from the array's element kind (e.g., pushing a string to a PACKED_SMI array), V8 transitions to a more general element kind.

**Q17: What happens when you use an object as a key in a regular Object?** **A:** Objects can only have string or Symbol keys. When you use an object as a key (`obj[{x:1}] = "val"`), JavaScript calls `.toString()` on the key object, which returns `"[object Object]"`. So ALL object keys map to the same string key! This is why Map exists — it uses the actual object reference as the key.
