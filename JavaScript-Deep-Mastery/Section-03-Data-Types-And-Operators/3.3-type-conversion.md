# 📌 3.3 — Type Conversion (Coercion)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Type conversion is how JavaScript changes a value from one type to another. **Implicit coercion** happens automatically (JavaScript decides for you), and **explicit coercion** happens when you intentionally convert (using `Number()`, `String()`, etc.). JavaScript's type coercion rules are one of the most confusing and bug-prone aspects of the language.

**Technical Definition:** Type conversion follows the ECMAScript Abstract Operations: `ToPrimitive`, `ToNumber`, `ToString`, `ToBoolean`, and `ToObject`. These are invoked by operators (+, ==, <), built-in functions, and language constructs depending on context.

**One-line mental model:**
> *JavaScript will secretly convert types to make operations work. `==` coerces; `===` doesn't. Learn the rules or use `===` everywhere.*

---

## ② DRILL DOWN

### 🔹 A. Implicit Coercion

JavaScript coerces types automatically in certain contexts:

```javascript
// String coercion (+ with a string):
"5" + 3;      // "53" (number → string)
"5" + true;   // "5true" (boolean → string)
"5" + null;   // "5null" (null → string)
"5" + undefined; // "5undefined"
"5" + {};     // "5[object Object]" (object.toString())
"5" + [];     // "5" (array.toString() → "")

// Numeric coercion (other operators):
"5" - 3;      // 2 (string → number)
"5" * 3;      // 15 (string → number)
"5" / 1;      // 5 (string → number)
"5" ** 2;     // 25 (string → number)
+true;        // 1 (boolean → number)
+false;       // 0
+null;        // 0
+undefined;   // NaN
+"";          // 0
+"hello";     // NaN
+[];          // 0 ([] → "" → 0)
+{};          // NaN ({} → "[object Object]" → NaN)

// Boolean coercion (in boolean contexts):
if ("hello") {}    // true (non-empty string)
if (0) {}          // false (zero)
!!"hello";         // true (double negation → boolean conversion)
!!0;               // false
```

### 🔹 B. Explicit Coercion

```javascript
// To Number:
Number("42");      // 42
Number("3.14");    // 3.14
Number("");        // 0
Number(" ");       // 0 (whitespace → 0)
Number("abc");     // NaN
Number(true);      // 1
Number(false);     // 0
Number(null);      // 0
Number(undefined); // NaN
Number([]);        // 0
Number([1]);       // 1
Number([1,2]);     // NaN
parseInt("42px");  // 42 (parses until non-numeric)
parseFloat("3.14em"); // 3.14

// To String:
String(42);        // "42"
String(true);      // "true"
String(null);      // "null"
String(undefined); // "undefined"
String([1,2,3]);   // "1,2,3" (Array.toString)
String({});        // "[object Object]"
(42).toString();   // "42"
(255).toString(16); // "ff" (hex)
(7).toString(2);   // "111" (binary)

// To Boolean:
Boolean(0);        // false
Boolean("");       // false
Boolean(null);     // false
Boolean(undefined);// false
Boolean(NaN);      // false
Boolean("0");      // true! (non-empty string)
Boolean([]);       // true! (object)
Boolean({});       // true! (object)
```

### 🔹 C. Truthy and Falsy

```
FALSY (exactly 8 values):
false, 0, -0, 0n, "", null, undefined, NaN

TRUTHY (everything else):
"0", "false", " ", [], {}, function(){}, 42, -1, Infinity, new Date(), new Boolean(false)
```

**Common traps:**
```javascript
// Empty array is truthy but equals false:
Boolean([]);      // true (object → truthy)
[] == false;      // true! ([] → "" → 0 → false)
// This is why == is dangerous

// "0" is truthy but equals false:
Boolean("0");     // true (non-empty string)
"0" == false;     // true! ("0" → 0 → false)
```

### 🔹 D. `==` vs `===`

**`===` (Strict Equality):** No coercion. Both type AND value must match.

```javascript
1 === 1;       // true
1 === "1";     // false (different types)
null === undefined; // false (different types)
NaN === NaN;   // false (NaN is not equal to itself)
```

**`==` (Abstract Equality):** Performs type coercion according to complex rules:

```javascript
// The algorithm (simplified):
// 1. If same type → use ===
// 2. null == undefined → true (special case)
// 3. Number == String → convert String to Number
// 4. Boolean == anything → convert Boolean to Number first
// 5. Object == Primitive → convert Object to Primitive (ToPrimitive)

1 == "1";       // true ("1" → 1)
0 == false;     // true (false → 0)
0 == "";        // true ("" → 0)
null == undefined; // true (special case)
null == 0;      // false (null only equals undefined and null)
false == "0";   // true (false → 0, "0" → 0)
"" == false;    // true ("" → 0, false → 0)
[] == false;    // true ([] → "" → 0, false → 0)
[] == ![];      // true! ([] is truthy, ![] is false, then [] → 0 == false → 0)
```

**The infamous ones:**
```javascript
"" == 0;              // true
" " == 0;             // true
"" == false;          // true
[] == false;          // true
null == false;        // false (null only == null and undefined)
undefined == false;   // false
NaN == NaN;           // false
[] == ![];            // true (this is the most confusing)
```

### 🔹 E. Object to Primitive Conversion

When an object needs to be converted to a primitive, JS calls:

1. `[Symbol.toPrimitive](hint)` if it exists (hint = "number", "string", or "default")
2. For "number" hint: `valueOf()`, then `toString()`
3. For "string" hint: `toString()`, then `valueOf()`

```javascript
const obj = {
    [Symbol.toPrimitive](hint) {
        if (hint === 'number') return 42;
        if (hint === 'string') return 'forty-two';
        return true; // default
    }
};

+obj;          // 42 (numeric hint)
`${obj}`;      // "forty-two" (string hint)
obj + "";      // "true" (default hint → string concatenation)

// Without Symbol.toPrimitive:
const obj2 = {
    valueOf() { return 100; },
    toString() { return "hundred"; }
};

+obj2;         // 100 (valueOf for numeric context)
`${obj2}`;     // "hundred" (toString for string context)
obj2 + 1;      // 101 (default → valueOf → 100 + 1)
```

### 🔹 F. valueOf and toString

```javascript
// Default valueOf returns the object itself:
({}).valueOf();    // {} (the object itself — not very useful)
[1,2].valueOf();  // [1, 2]
new Date().valueOf(); // 1705276800000 (milliseconds — useful!)

// Default toString:
({}).toString();   // "[object Object]"
[1,2].toString();  // "1,2"
(42).toString();   // "42"
true.toString();   // "true"

// Custom valueOf/toString:
class Price {
    constructor(amount) { this.amount = amount; }
    valueOf() { return this.amount; }
    toString() { return `$${this.amount.toFixed(2)}`; }
}

const p = new Price(9.99);
p + 1;       // 10.99 (valueOf → 9.99 + 1)
`${p}`;      // "$9.99" (toString)
console.log(p > 5); // true (valueOf → 9.99 > 5)
```

## ③ WHY IT EXISTS

JavaScript was designed to be forgiving — operators should "just work" even with mixed types. This was intentional for beginner-friendly web scripting in 1995. The cost: confusing coercion rules.

**`===` was added in ES3 (1999)** to give developers a way to compare without coercion. The recommendation since then: **always use `===`**.

---

## ④ INTERNAL WORKING

### V8's ToPrimitive Algorithm

When V8 needs to coerce an object to a primitive:
1. Check for `[Symbol.toPrimitive]` method → call it with the appropriate hint
2. If hint is "number": call `valueOf()`. If result is primitive, return it. Else call `toString()`. If primitive, return. Else throw `TypeError`.
3. If hint is "string": call `toString()` first, then `valueOf()`.
4. If hint is "default": same as "number" (except for `Date`, which uses "string")

### How `+` Operator Works Internally

```
V8's + operator:
1. Let lprim = ToPrimitive(left)
2. Let rprim = ToPrimitive(right)
3. If either lprim or rprim is a string:
   → ToString(lprim) + ToString(rprim) (string concatenation)
4. Else:
   → ToNumber(lprim) + ToNumber(rprim) (numeric addition)
```

This is why `"5" + 3 = "53"` (step 3: string found) but `"5" - 3 = 2` (- only does numeric).

---

## ⑤ ADVANTAGES

1. **Convenience:** `"5" * 2` working as `10` is convenient for processing form inputs.
2. **Truthiness:** `if (value)` as a quick null/undefined/empty check is idiomatic.

---

## ⑥ DISADVANTAGES / TRAPS

1. **`==` bugs:** The most common source of subtle bugs in JavaScript.
2. **`+` is overloaded:** Addition vs concatenation depends on operand types — confusing.
3. **Silent conversions:** `"" == false` being true can mask real bugs.
4. **`parseInt` traps:** `parseInt("08")` was octal in old engines. `parseInt(0.000001)` → `0` (reads as "0.000001", parses "0").

---

## ⑦ PRODUCTION USAGE

```javascript
// Always use === and !== in production:
if (value === null || value === undefined) { /* handle */ }
// Or use nullish coalescing:
const result = value ?? defaultValue; // Only null/undefined trigger default

// Explicit conversion when needed:
const port = Number(process.env.PORT) || 3000;
const isEnabled = String(config.feature) === 'true';

// Safe boolean conversion:
const isActive = Boolean(user.active); // Explicit, readable
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is type coercion?** **A:** Automatic type conversion by JavaScript when an operator or context requires a different type.

**Q2: What's the difference between `==` and `===`?** **A:** `===` checks type and value without coercion. `==` performs type coercion before comparing. Always use `===`.

**Q3: What are the falsy values?** **A:** `false`, `0`, `-0`, `0n`, `""`, `null`, `undefined`, `NaN`.

**Q4: What does `+"42"` return?** **A:** `42` (number). Unary `+` converts the string to a number.

**Q5: What does `"5" + 3` return?** **A:** `"53"`. The `+` operator with a string performs concatenation, not addition.

### 🟡 Intermediate (5)

**Q6: Why is `[] == ![]` true?** **A:** `![]` is `false` (array is truthy, negated = false). `[] == false` → `[] → "" → 0`, `false → 0`. `0 == 0` → `true`.

**Q7: What is `Symbol.toPrimitive`?** **A:** A well-known symbol that defines custom primitive conversion behavior. When an object is coerced, JS calls `obj[Symbol.toPrimitive](hint)` where hint is "number", "string", or "default".

**Q8: What does `null == undefined` return?** **A:** `true`. This is a special case in the spec — `null` and `undefined` are only loosely equal to each other and nothing else.

**Q9: Explain `Number([])` vs `Number({})`.** **A:** `Number([])` → `[]` → `""` → `0`. `Number({})` → `{}` → `"[object Object]"` → `NaN`.

**Q10: What is the difference between `Number("42")` and `parseInt("42")`?** **A:** `Number("42")` uses ToNumber (strict: entire string must be numeric). `parseInt("42px")` parses from left until non-numeric character. `Number("42px")` → `NaN`, `parseInt("42px")` → `42`.

### 🔴 Advanced (5)

**Q11: Explain the full coercion chain for `[] + {}`.** **A:** `[].toString()` → `""`. `({}).toString()` → `"[object Object]"`. `"" + "[object Object]"` → `"[object Object]"`.

**Q12: What does `{} + []` return in the console?** **A:** In the browser console, `{}` is parsed as an empty block (not an object literal). So it becomes `+[]` → `+""` → `0`. In an expression context (like `({} + [])`), it returns `"[object Object]"`.

**Q13: How does ToPrimitive work with Date objects?** **A:** Date objects use "string" as the default hint (unlike other objects which use "number"). So `date + ""` calls `toString()` first (returns human-readable date string), not `valueOf()` (which returns milliseconds).

**Q14: Explain `Number(null)` vs `Number(undefined)`.** **A:** `Number(null)` → `0` (spec defines null → 0). `Number(undefined)` → `NaN` (spec defines undefined → NaN). This inconsistency is a historical design choice.

**Q15: How does `==` handle object-to-primitive comparison?** **A:** When comparing an object to a primitive with `==`, JS calls `ToPrimitive` on the object with "default" hint. This calls `valueOf()` first (for non-Date objects). If the result is a primitive, it's used for comparison. If not, `toString()` is called.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Trace the full coercion chain for `"5" + 3 - 2`.** **A:** Step 1: `"5" + 3` → ToPrimitive on both sides. "5" is already primitive (string). 3 is primitive (number). One is string → concatenation: `"53"`. Step 2: `"53" - 2` → `-` only does numeric. ToNumber("53") → 53. `53 - 2` → `51`. Final result: `51` (number).

**Q17: What happens internally when `if ([])` evaluates?** **A:** The `if` statement calls ToBoolean on `[]`. ToBoolean rules: if the value is an object → `true`. Arrays are objects → `true`. The `if` body executes. This is why `[] == false` (coercion through ToPrimitive) but `Boolean([])` is `true` — different conversion paths!
