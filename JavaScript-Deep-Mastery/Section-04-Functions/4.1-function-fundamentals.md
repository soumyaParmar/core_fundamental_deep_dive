# 📌 4.1 — Function Fundamentals
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Functions are reusable blocks of code that can accept inputs (parameters), perform operations, and return outputs. In JavaScript, functions are **first-class citizens** — they can be assigned to variables, passed as arguments, returned from other functions, and have properties just like objects.

**Technical Definition:** A function is a callable object with an internal `[[Call]]` method. Functions have three creation forms: function declarations, function expressions, and arrow functions. Each creates a function object on the heap with an associated `[[Environment]]` (scope chain) and `[[FunctionKind]]` (Normal, Arrow, Generator, Async, etc.).

**One-line mental model:**
> *Functions are objects you can call. They're JavaScript's fundamental unit of reuse and abstraction.*

## ② DRILL DOWN

### 🔹 A. Function Declaration

```javascript
// Hoisted — available before the declaration line
function greet(name) {
    return `Hello, ${name}!`;
}

greet("Alice"); // "Hello, Alice!"

// Characteristics:
// ✅ Fully hoisted (name + body)
// ✅ Has a .name property ("greet")
// ✅ Has a .prototype property
// ✅ Can be used as constructor (new greet())
// ✅ Has its own `this` binding
// ✅ Has `arguments` object
```

### 🔹 B. Function Expression

```javascript
// NOT hoisted (follows the variable's hoisting rules)
const greet = function(name) {
    return `Hello, ${name}!`;
};

// Named function expression (NFE):
const factorial = function fact(n) {
    return n <= 1 ? 1 : n * fact(n - 1); // fact is only available INSIDE
};
console.log(factorial.name); // "fact" — useful in stack traces
// console.log(fact);        // ReferenceError — fact not in outer scope

// Characteristics:
// ❌ NOT hoisted (var → undefined, const/let → TDZ)
// ✅ Has .name property (uses variable name or explicit name)
// ✅ Has .prototype property
// ✅ Can be constructor
// ✅ Has its own `this`
// ✅ Has `arguments`
```

### 🔹 C. Arrow Function

```javascript
// Concise syntax (ES6+):
const greet = (name) => `Hello, ${name}!`;  // Implicit return
const add = (a, b) => a + b;               // Two params
const getObj = () => ({ x: 1 });            // Return object (needs parens)
const log = x => console.log(x);           // One param (parens optional)

// Block body (explicit return needed):
const process = (data) => {
    const result = transform(data);
    return result;
};

// Characteristics (CRITICAL DIFFERENCES):
// ❌ No own `this` — inherits from enclosing scope (lexical this)
// ❌ No `arguments` object
// ❌ No `.prototype` property
// ❌ Cannot be used as constructor (new arrow() → TypeError)
// ❌ Cannot be used as generator (no yield)
// ✅ Has .name property
// ✅ More concise
```

### 🔹 D. Anonymous Function

```javascript
// Functions without names — used in callbacks:
setTimeout(function() {
    console.log("anonymous");
}, 1000);

[1, 2, 3].map(function(x) { return x * 2; });

// Arrow functions are often anonymous:
[1, 2, 3].map(x => x * 2);

// V8 infers names from context:
const fn = function() {};
console.log(fn.name); // "fn" (V8 infers from assignment)

const obj = {
    method: function() {}
};
console.log(obj.method.name); // "method" (inferred)
```

### 🔹 E. Named Function Expression (NFE)

```javascript
// The name is only accessible inside the function:
const calculate = function calc(n) {
    if (n <= 1) return 1;
    return n * calc(n - 1); // Recursive call using the internal name
};

// Benefits of NFE:
// 1. Self-referencing (recursion) without relying on the outer variable
// 2. Better stack traces in debugging
// 3. The name is read-only inside the function:
const test = function inner() {
    inner = 42; // Silently fails in sloppy mode, TypeError in strict
    console.log(typeof inner); // "function" — can't reassign
};
```

### 🔹 F. Function Parameters Deep Dive

```javascript
// Default parameters (ES6):
function create(name, role = "user", active = true) {
    return { name, role, active };
}
create("Alice"); // { name: "Alice", role: "user", active: true }
create("Bob", undefined, false); // { name: "Bob", role: "user", active: false }
// undefined triggers default; null does NOT

// Rest parameters (ES6):
function sum(first, ...rest) {
    return rest.reduce((acc, val) => acc + val, first);
}
sum(1, 2, 3, 4); // 10

// Destructuring parameters:
function createUser({ name, age, role = "user" }) {
    return { name, age, role };
}
createUser({ name: "Alice", age: 30 }); // { name: "Alice", age: 30, role: "user" }

// Arguments length vs parameters length:
function test(a, b, c) {}
test.length; // 3 (number of parameters BEFORE first default/rest)

function test2(a, b = 1, c) {}
test2.length; // 1 (stops counting at first default)

function test3(...args) {}
test3.length; // 0 (rest doesn't count)
```

### 🔹 G. Return Values

```javascript
// Implicit return (arrow functions without block body):
const double = x => x * 2; // Returns x * 2

// Explicit return:
function process(x) {
    if (x < 0) return null; // Early return
    return x * 2;
}

// No return = returns undefined:
function noReturn() {
    console.log("side effect");
}
console.log(noReturn()); // undefined

// Return in constructors:
function Foo() {
    this.x = 1;
    return { y: 2 }; // Returning an object OVERRIDES the new instance
}
new Foo(); // { y: 2 } (NOT { x: 1 })

function Bar() {
    this.x = 1;
    return 42; // Returning a primitive is IGNORED by new
}
new Bar(); // { x: 1 } (42 is ignored)
```

### 🔹 H. IIFE (Immediately Invoked Function Expression)

```javascript
// Classic IIFE — creates a private scope:
(function() {
    var private = "hidden";
    console.log("IIFE executed");
})();
// private is not accessible here

// IIFE with return value:
const module = (function() {
    let count = 0;
    return {
        increment() { count++; },
        getCount() { return count; }
    };
})();

// Arrow IIFE:
const result = (() => {
    const x = computeX();
    const y = computeY();
    return x + y;
})();

// IIFE with parameters:
(function(global, doc) {
    // Use global and doc instead of window and document
    // Helpful for minification and dependency injection
})(window, document);
```

## ③ WHY IT EXISTS

Functions are the **primary unit of abstraction** in JavaScript. Before classes (ES6), everything was built with functions:
- Modules (IIFE pattern)
- "Classes" (constructor functions + prototypes)
- Encapsulation (closures)
- Async patterns (callbacks)

JavaScript was influenced by **Scheme** (a Lisp), which treats functions as first-class values. This gives JavaScript its functional programming capabilities.

## ④ INTERNAL WORKING

### V8 Function Object

```
Function Object (on heap):
┌────────────────────────────────────┐
│ Map (hidden class)                  │
│ SharedFunctionInfo → {             │
│   bytecode/code                    │ ← Compiled code
│   formal_parameter_count           │ ← .length property
│   function_name                    │ ← .name property
│   expected_nof_properties          │ ← For constructor optimization
│ }                                  │
│ Context (scope chain) ──────────── │ ← [[Environment]]
│ Feedback vector                    │ ← Type feedback for JIT
│ Prototype (for constructors)       │ ← .prototype property
└────────────────────────────────────┘
```

### Lazy Compilation

V8 lazily compiles functions — only when first called:
```javascript
function neverCalled() {
    // V8 pre-parses this (checks syntax) but does NOT generate bytecode
    // Bytecode is generated on first call
}

function calledOnce() {
    // First call: V8 compiles to bytecode (Ignition)
    // If called many times: TurboFan optimizes to machine code
}
```

## ⑤ ADVANTAGES

1. **First-class functions** enable functional programming — map, filter, reduce, callbacks, promises.
2. **Closures** provide encapsulation and state management.
3. **Arrow functions** provide concise syntax and lexical `this`.
4. **Default/rest parameters** eliminate boilerplate argument handling.

---

## ⑥ DISADVANTAGES / TRAPS

1. **`this` in regular functions depends on call site** — source of countless bugs.
2. **Arrow functions can't be constructors** — `new (() => {})` throws TypeError.
3. **`arguments` is not a real array** — needs `Array.from(arguments)`.
4. **Default parameters are evaluated at call time** — `f(x = [])` creates a new array each call (good for immutability, bad if you expect sharing).

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the ways to create a function?** **A:** Function declaration, function expression, arrow function, `new Function()` constructor (rarely used), and method shorthand in objects/classes.

**Q2: What's the difference between function declaration and expression?** **A:** Declarations are fully hoisted (usable before declaration line). Expressions follow the hoisting rules of their variable keyword (var → undefined, let/const → TDZ).

**Q3: What does a function return if there's no return statement?** **A:** `undefined`.

**Q4: What is an IIFE?** **A:** Immediately Invoked Function Expression — a function that runs immediately when defined: `(function() { ... })()`. Used to create a private scope.

**Q5: What is `function.length`?** **A:** The number of formal parameters before the first one with a default value or rest parameter.

### 🟡 Intermediate (5)

**Q6: What is a named function expression and why use it?** **A:** A function expression with an explicit name: `const f = function factorial(n) { ... }`. The name is only accessible inside the function. Benefits: recursion without relying on outer variable, better stack traces.

**Q7: Why can arrow functions not be constructors?** **A:** Arrow functions don't have a `[[Construct]]` internal method or a `.prototype` property. They're designed to be lightweight, inheriting `this` from the enclosing scope rather than creating their own.

**Q8: What happens when a constructor returns a non-object?** **A:** `new` ignores non-object return values and returns the newly created `this` object. But if the constructor returns an object, that object replaces the new instance.

**Q9: How are default parameters evaluated?** **A:** Default parameters are evaluated at call time, left to right. Each default can reference earlier parameters: `function f(a, b = a * 2) {}`.

**Q10: What is the difference between `arguments` and rest parameters?** **A:** `arguments` is array-like (not a real array), includes ALL arguments, and is not available in arrow functions. Rest parameters are real arrays, only collect "extra" arguments, and work in arrow functions.

### 🔴 Advanced (5)

**Q11: How does V8 lazily compile functions?** **A:** V8 pre-parses function bodies on initial script load (syntax check only, no bytecode). Full parsing and bytecode generation occur on first invocation. This reduces startup time for large codebases.

**Q12: What is a SharedFunctionInfo in V8?** **A:** V8's internal structure shared by all instances of the same function. It contains bytecode, parameter count, name, and other metadata. Multiple closures from the same function literal share one SharedFunctionInfo but have different Context objects.

**Q13: How does `Function.prototype.toString()` work?** **A:** In modern engines, it returns the exact source code of the function (including comments and whitespace). This was standardized in ES2019. For native/built-in functions, it returns `"function name() { [native code] }"`.

**Q14: What is the `caller` and `callee` property?** **A:** `arguments.callee` refers to the currently executing function (deprecated, forbidden in strict mode). `func.caller` refers to the function that called the current function (non-standard, forbidden in strict mode). Both prevent V8 optimization.

**Q15: Can you modify a function's `.length` property?** **A:** `.length` is non-writable but configurable. You can redefine it with `Object.defineProperty`: `Object.defineProperty(fn, 'length', { value: 5 })`. This is used by libraries that create wrapper functions and want to preserve the original arity.

### 🧠 "Explain What Happens Internally" (2)

**Q16: What happens in V8 when you call a function?** **A:** 1. V8 checks the `[[Call]]` internal method exists (TypeError if not callable). 2. Creates a new stack frame with parameter slots and local variable registers. 3. Sets up the execution context (this binding, scope chain via Context pointer). 4. If the function hasn't been compiled yet, triggers lazy compilation (parse → bytecode). 5. Executes the Ignition bytecode. 6. If the function is hot (called many times), TurboFan compiles it to optimized machine code. 7. On return, the stack frame is popped and the return value goes to the caller.

**Q17: How does V8 handle function inlining in TurboFan?** **A:** TurboFan can inline small, frequently-called functions — replacing the call site with the function's body. This eliminates call overhead and enables cross-function optimizations (constant folding, dead code elimination). V8 considers: function size (small = more likely to inline), call frequency (hot = more likely), and total inlined size budget. Inlined functions must be monomorphic (same function every time). Megamorphic call sites (calling different functions) prevent inlining.
