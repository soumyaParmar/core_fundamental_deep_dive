# 📌 4.4 — Higher-Order Functions
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** A higher-order function (HOF) is a function that either **takes one or more functions as arguments** or **returns a function** as its result. This is possible because JavaScript treats functions as first-class citizens — they can be passed around like any other value.

**Technical Definition:** A higher-order function is any function `f` where at least one parameter is a function or the return value is a function. This concept comes from functional programming and lambda calculus. In JavaScript, functions are objects, making HOFs natural and ubiquitous.

**One-line mental model:**
> *Higher-order functions operate on other functions — either by taking them as input or producing them as output.*

---

## ② DRILL DOWN

### 🔹 A. Functions as First-Class Citizens

```javascript
// 1. Assign to variables:
const greet = function(name) { return `Hello, ${name}`; };

// 2. Pass as arguments:
[1, 2, 3].map(x => x * 2); // map takes a function as argument

// 3. Return from functions:
function createGreeter(greeting) {
    return (name) => `${greeting}, ${name}!`;
}
const hello = createGreeter("Hello");
hello("Alice"); // "Hello, Alice!"

// 4. Store in data structures:
const strategies = {
    add: (a, b) => a + b,
    multiply: (a, b) => a * b,
};
strategies.add(2, 3); // 5

// 5. Have properties:
function fn() {}
fn.custom = "metadata";
```

### 🔹 B. Callback Patterns

```javascript
// Synchronous callback:
function processArray(arr, callback) {
    const results = [];
    for (const item of arr) {
        results.push(callback(item));
    }
    return results;
}
processArray([1, 2, 3], x => x * 2); // [2, 4, 6]

// Asynchronous callback (Node.js style):
function readFileAsync(path, callback) {
    fs.readFile(path, 'utf-8', (err, data) => {
        if (err) return callback(err, null); // Error-first pattern
        callback(null, data);
    });
}
readFileAsync('./config.json', (err, data) => {
    if (err) { console.error(err); return; }
    console.log(data);
});

// Event callback:
document.addEventListener('click', (event) => {
    console.log('Clicked at', event.clientX, event.clientY);
});

// Array HOFs (the most common callbacks):
const nums = [1, 2, 3, 4, 5];
nums.map(n => n * 2);           // [2, 4, 6, 8, 10]
nums.filter(n => n > 2);        // [3, 4, 5]
nums.reduce((sum, n) => sum + n, 0); // 15
nums.find(n => n > 3);          // 4
nums.some(n => n > 4);          // true
nums.every(n => n > 0);         // true
nums.forEach(n => console.log(n)); // Side effect only
nums.sort((a, b) => a - b);    // Ascending sort
nums.flatMap(n => [n, n * 2]); // [1, 2, 2, 4, 3, 6, 4, 8, 5, 10]
```

### 🔹 C. Functional Composition

```javascript
// Compose: combine multiple functions into one
// compose(f, g)(x) = f(g(x))
function compose(...fns) {
    return (x) => fns.reduceRight((acc, fn) => fn(acc), x);
}

// Pipe: left-to-right composition
// pipe(f, g)(x) = g(f(x))
function pipe(...fns) {
    return (x) => fns.reduce((acc, fn) => fn(acc), x);
}

const double = x => x * 2;
const addOne = x => x + 1;
const square = x => x ** 2;

const transform = pipe(double, addOne, square);
transform(3); // square(addOne(double(3))) = square(addOne(6)) = square(7) = 49

// Real-world compose:
const processUser = pipe(
    normalizeEmail,
    validateAge,
    hashPassword,
    saveToDatabase
);
```

### 🔹 D. Currying

```javascript
// Currying: converting a function that takes multiple arguments
// into a sequence of functions each taking one argument

// Manual curry:
function multiply(a) {
    return function(b) {
        return a * b;
    };
}
multiply(3)(4); // 12
const triple = multiply(3);
triple(4); // 12
triple(5); // 15

// Generic curry implementation:
function curry(fn) {
    return function curried(...args) {
        if (args.length >= fn.length) {
            return fn.apply(this, args);
        }
        return function(...moreArgs) {
            return curried.apply(this, [...args, ...moreArgs]);
        };
    };
}

const curriedAdd = curry((a, b, c) => a + b + c);
curriedAdd(1)(2)(3);     // 6
curriedAdd(1, 2)(3);     // 6
curriedAdd(1)(2, 3);     // 6
curriedAdd(1, 2, 3);     // 6

// Practical currying:
const log = curry((level, prefix, message) => {
    console.log(`[${level}] ${prefix}: ${message}`);
});
const errorLog = log("ERROR");
const authError = errorLog("AUTH");
authError("Invalid token"); // [ERROR] AUTH: Invalid token
```

### 🔹 E. Partial Application

```javascript
// Partial application: fixing some arguments of a function
// Different from currying: partial fixes N args at once, curry fixes one at a time

function partial(fn, ...fixedArgs) {
    return (...remainingArgs) => fn(...fixedArgs, ...remainingArgs);
}

function greet(greeting, name, punctuation) {
    return `${greeting}, ${name}${punctuation}`;
}

const greetHello = partial(greet, "Hello");
greetHello("Alice", "!"); // "Hello, Alice!"

const greetHelloAlice = partial(greet, "Hello", "Alice");
greetHelloAlice("?"); // "Hello, Alice?"

// Using bind for partial application:
const add = (a, b) => a + b;
const add5 = add.bind(null, 5); // Partially apply first arg
add5(3); // 8
```

## ③ WHY IT EXISTS

HOFs exist because JavaScript was inspired by **Scheme** (a functional language). They enable:
1. **Abstraction over actions** — not just data, but behavior can be parameterized
2. **Code reuse** — `.map`, `.filter`, `.reduce` eliminate repetitive loop patterns
3. **Declarative code** — describe *what* to do, not *how* to do it
4. **Composability** — build complex behavior from simple functions

---

## ④ INTERNAL WORKING

V8 treats callback functions the same as any other function call. When you pass a callback to `.map()`:
1. V8 calls the `.map()` built-in (C++ or optimized JS)
2. For each element, V8 calls the callback function (pushes a new frame, executes, pops)
3. TurboFan can **inline** small callbacks directly into the loop body, eliminating call overhead
4. For monomorphic callbacks (same function every time), V8 generates very efficient machine code

**Performance insight:** `for` loops are slightly faster than `.map()` in unoptimized code, but TurboFan-optimized `.map()` with inline arrows is nearly identical. Use HOFs for clarity unless profiling shows a bottleneck.

---

## ⑤ ADVANTAGES

1. **DRY code** — avoid writing the same loop patterns repeatedly.
2. **Composability** — combine small functions into complex pipelines.
3. **Testability** — pure functions as callbacks are easy to unit test.
4. **Declarative** — `.filter(isActive).map(getName)` reads like English.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Callback hell** — deeply nested callbacks are unreadable (solved by Promises/async-await).
2. **Performance in hot loops** — function call overhead on each iteration (mitigated by TurboFan inlining).
3. **Debugging stack traces** — anonymous callbacks can be harder to trace.
4. **`this` loss in callbacks** — passing methods as callbacks loses `this`.

## ⑦ PRODUCTION USAGE

```javascript
// Data transformation pipeline (common in APIs):
const processUsers = (rawUsers) => 
    rawUsers
        .filter(u => u.active)
        .map(u => ({ ...u, email: u.email.toLowerCase() }))
        .sort((a, b) => a.name.localeCompare(b.name));

// Middleware composition (Express/Koa):
const withAuth = (handler) => (req, res, next) => {
    if (!req.user) return res.status(401).json({ error: "Unauthorized" });
    handler(req, res, next);
};

const withLogging = (handler) => (req, res, next) => {
    console.log(`${req.method} ${req.path}`);
    handler(req, res, next);
};

app.get('/api/data', withAuth(withLogging(actualHandler)));

// Retry with configurable strategy:
function withRetry(fn, { maxRetries = 3, delay = 1000 } = {}) {
    return async (...args) => {
        for (let attempt = 1; attempt <= maxRetries; attempt++) {
            try {
                return await fn(...args);
            } catch (err) {
                if (attempt === maxRetries) throw err;
                await new Promise(r => setTimeout(r, delay * attempt));
            }
        }
    };
}
const fetchWithRetry = withRetry(fetch, { maxRetries: 3 });
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is a higher-order function?** **A:** A function that takes another function as an argument or returns a function.

**Q2: Name three built-in higher-order functions.** **A:** `Array.prototype.map`, `Array.prototype.filter`, `Array.prototype.reduce`.

**Q3: What does "first-class functions" mean?** **A:** Functions can be assigned to variables, passed as arguments, returned from functions, and have properties — treated like any other value.

**Q4: What is a callback function?** **A:** A function passed as an argument to another function, to be called at a specific point during execution.

**Q5: What is the difference between `map` and `forEach`?** **A:** `map` returns a new array with transformed elements. `forEach` returns undefined — used for side effects only.

### 🟡 Intermediate (5)

**Q6: Implement `Array.prototype.map` from scratch.**
```javascript
Array.prototype.myMap = function(callback, thisArg) {
    const result = [];
    for (let i = 0; i < this.length; i++) {
        if (i in this) { // Handle sparse arrays
            result.push(callback.call(thisArg, this[i], i, this));
        } else {
            result.length++;
        }
    }
    return result;
};
```

**Q7: What is function composition?** **A:** Combining two or more functions to produce a new function. `compose(f, g)(x)` = `f(g(x))`.

**Q8: What is currying?** **A:** Transforming a function with multiple arguments into a sequence of functions each taking one argument.

**Q9: What is partial application?** **A:** Creating a new function by fixing some arguments of an existing function. Different from currying because partial fixes N args at once.

**Q10: How does `reduce` differ from `map` and `filter`?** **A:** `reduce` transforms an entire array into a single value (accumulation). `map` transforms each element 1:1. `filter` selects elements based on a predicate.

### 🔴 Advanced (5)

**Q11: Implement a generic `curry` function.** **A:** See Section ② Drill Down D above.

**Q12: How does V8 optimize `.map()` with inline arrow functions?** **A:** TurboFan can inline the arrow function body directly into the loop. If the callback is monomorphic (always the same function) and small enough, TurboFan eliminates the function call overhead entirely, generating machine code equivalent to a hand-written loop.

**Q13: Implement `compose` and `pipe`.** **A:** `compose = (...fns) => x => fns.reduceRight((acc, fn) => fn(acc), x)`. `pipe = (...fns) => x => fns.reduce((acc, fn) => fn(acc), x)`.

**Q14: What is a "point-free" style?** **A:** Writing functions without explicitly mentioning their arguments. Example: `const getNames = users => users.map(u => u.name)` becomes `const getNames = map(prop('name'))` using utilities.

**Q15: What is a thunk?** **A:** A function with no arguments that wraps a computation for lazy evaluation: `const thunk = () => expensiveComputation()`. Called when the value is needed, not when created.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain what happens when V8 executes `[1,2,3].map(x => x * 2)`.** **A:** 1. V8 accesses `.map` on the array via the prototype chain (Array.prototype.map). 2. The built-in `map` implementation iterates the array. 3. For each element, it calls the arrow function: pushes a call frame, passes the element as argument, executes `x * 2` (Smi multiplication), returns. 4. Results are collected into a new array (allocated on heap). 5. If this runs many times, TurboFan inlines the arrow: the function call is eliminated, and the multiplication happens directly in the loop body. 6. V8 may also recognize the output array type (PACKED_SMI_ELEMENTS) and use optimized backing storage.

**Q17: What is the difference between synchronous and asynchronous callbacks internally?** **A:** Synchronous callbacks run immediately on the call stack (e.g., `.map()`). The callback is called within the same event loop tick. Asynchronous callbacks are registered with the runtime (e.g., `setTimeout`, `addEventListener`) and placed on the callback/microtask queue. They run later, after the current call stack is empty and the event loop picks them up. The key difference: sync callbacks block; async callbacks don't.
