# 📌 5.1 — Object Fundamentals
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** An object is a collection of key-value pairs where keys are strings or symbols, and values can be any data type (including other objects or functions). They are the primary structures used to store structured data and build complex models in JavaScript.

**Technical Definition:** An object is a dynamic collection of properties containing a reference to a prototype object. It is implemented as a heap-allocated dictionary or as a structured shape in the JIT compiler (V8 Map/Hidden Class). Every property has an associated **Property Descriptor** defining its metadata (value, writable, enumerable, configurable, or getter/setter accessors).

**One-line mental model:**
> *An object is a dynamic bag of properties linked to a prototype, optimized as a static 'shape' under the hood by the engine.*

---

## ② DRILL DOWN

### 🔹 A. Creating Objects in JavaScript

There are multiple ways to construct objects, each with distinct performance and prototype implications.

```javascript
// 1. Object Literal (Most common, fastest, uses Object.prototype)
const user = { name: "Alice", age: 30 };

// 2. Constructor Function (Classic OOP pattern)
function User(name, age) {
    this.name = name;
    this.age = age;
}
const user1 = new User("Alice", 30);

// 3. Object.create (Explicit prototype assignment)
const proto = { greet() { return `Hello, I'm ${this.name}`; } };
const user2 = Object.create(proto); // prototype linked to 'proto'
user2.name = "Alice";

// 4. Object.create(null) (Pure dictionary - no prototype, no toString/hasOwnProperty)
const dict = Object.create(null);
console.log(dict.toString); // undefined

// 5. Object.assign & Spread Operator (Shallow Cloning)
const original = { a: 1, b: { c: 2 } };
const clone1 = Object.assign({}, original);
const clone2 = { ...original };
// clone1.b and clone2.b still point to original.b (shared references!)
```

### 🔹 B. Keys and Coercion

All object keys are either **Strings** or **Symbols**. If you use any other type as a key, JavaScript will silently coerce it to a string.

```javascript
const obj = {};
const key1 = {};
const key2 = {};

obj[key1] = "Object A"; // Coerces key1 to "[object Object]"
obj[key2] = "Object B"; // Overwrites because key2 also coerces to "[object Object]"

console.log(obj[key1]); // "Object B"
console.log(Object.keys(obj)); // ["[object Object]"]
```

### 🔹 C. Property Descriptors

Every object property has a hidden configuration called a **Property Descriptor**. You can view and modify descriptors using `Object.getOwnPropertyDescriptor()` and `Object.defineProperty()`.

There are two main types of descriptors: **Data Descriptors** and **Accessor Descriptors**.

```javascript
const car = {};

// 1. Defining a Data Descriptor:
Object.defineProperty(car, 'make', {
    value: 'Tesla',
    writable: false,      // Can't be reassigned
    enumerable: false,    // Won't show up in for...in or Object.keys()
    configurable: false   // Can't be deleted or changed later
});

car.make = 'Ford';        // Silently ignored (throws TypeError in strict mode)
delete car.make;          // Returns false (throws in strict mode)
console.log(car.make);    // "Tesla"

// 2. Defining an Accessor Descriptor (Getter/Setter):
Object.defineProperty(car, 'model', {
    get() {
        return this._model;
    },
    set(value) {
        this._model = value.toUpperCase();
    },
    enumerable: true,
    configurable: true
});
car.model = 'model s';
console.log(car.model);   // "MODEL S"
```

#### Descriptor Default Attributes:
* **Literal creation (`{ x: 1 }`)**: `writable`, `enumerable`, and `configurable` are all `true`.
* **`Object.defineProperty()` creation**: Unspecified attributes default to `false` or `undefined`.

### 🔹 D. Controlling Object Mutability

JavaScript provides three levels of protection to freeze or lock down objects.

| Method | Can Add Properties? | Can Delete Properties? | Can Modify Values? | Configurable Descriptor? |
| :--- | :---: | :---: | :---: | :---: |
| **Default** | ✅ Yes | ✅ Yes | ✅ Yes | ✅ Yes |
| **`Object.preventExtensions(obj)`** | ❌ No | ✅ Yes | ✅ Yes | ✅ Yes |
| **`Object.seal(obj)`** | ❌ No | ❌ No | ✅ Yes | ❌ No (`configurable: false`) |
| **`Object.freeze(obj)`** | ❌ No | ❌ No | ❌ No | ❌ No (`writable: false`, `configurable: false`) |

*Important:* These protections are **shallow**. Nested objects inside a frozen object can still be fully modified unless recursively frozen!

## ③ WHY IT EXISTS

JavaScript was built as a lightweight, dynamic scripting language for the early web.
1. **Dynamic Extension**: Rather than forcing developers to pre-compile class structures, JavaScript allows objects to act as highly dynamic associative arrays (dictionaries). You can attach or remove properties and methods at runtime.
2. **Simplified Execution**: By relying on prototype chains and property descriptors, JavaScript unified functional programming (methods are just properties storing functions) and object-oriented paradigms within a single, lightweight primitive.

## ④ INTERNAL WORKING: V8 SHAPES (HIDDEN CLASSES)

### V8 Hidden Classes (Maps/Shapes)

In C++ or Java, objects have static structures defined by classes; offsets of fields are calculated at compilation. JavaScript's dynamic objects would require a slow hash-table lookup for every single property access. To solve this, V8 invented **Hidden Classes** (also called **Maps** or **Shapes**).

```
When you create: const obj = { x: 1, y: 2 }

V8 internally builds a Transition Chain of Hidden Classes (Maps):

 ┌─────────────────┐       Add property 'x'       ┌─────────────────┐       Add property 'y'       ┌─────────────────┐
 │     Map C0      │ ───────────────────────────► │     Map C1      │ ───────────────────────────► │     Map C2      │
 ├─────────────────┤                              ├─────────────────┤                              ├─────────────────┤
 │ Properties:     │                              │ Properties:     │                              │ Properties:     │
 │ (Empty)         │                              │ x @ offset 0    │                              │ x @ offset 0    │
 └─────────────────┘                              └─────────────────┘                              │ y @ offset 1    │
                                                                                                   └─────────────────┘
```

1. **Instantiation**: `const o1 = {}` starts with Map `C0`.
2. **First Transition**: `o1.x = 1` changes `o1`'s map pointer to Map `C1`, which contains property `x` at offset `0`.
3. **Second Transition**: `o1.y = 2` changes `o1`'s map pointer to Map `C2`, which contains property `x` at offset `0` and property `y` at offset `1`.

#### The Danger of Shape Mismatch:
```javascript
const objA = {};
objA.x = 10;
objA.y = 20; // Transition: C0 -> C1 -> C2

const objB = {};
objB.y = 20;
objB.x = 10; // Transition: C0 -> C3 -> C4
```
Even though `objA` and `objB` have identical properties, they have **different shapes** (Maps C2 vs C4). This prevents JIT compilers from optimizing methods that process these objects together.

### Fast Properties vs. Slow Properties (Dictionary Mode)

V8 classifies object property storage into three structures depending on property counts and creation patterns:

```
┌────────────────────────────────────────────────────────┐
│                   JSObject (on heap)                   │
├────────────────────────────────────────────────────────┤
│ • Map Pointer (Points to Shape)                        │
│ • Prototype Pointer                                    │
│ • Properties Backing Store (Array / Dictionary) ────────┼──► [ Out-of-Object Properties ]
│ • Elements Backing Store (Array for numeric keys) ─────┼──► [ Number-indexed elements ]
│ • In-Object Fields (Pre-allocated slots on the instance)├──► [ Inline properties for fast access ]
└────────────────────────────────────────────────────────┘
```

1. **In-Object Properties (Fastest)**:
   - Properties stored directly within the object structure on the heap.
   - Access requires zero indirections; read directly from the object's memory offset.
   - V8 allocates a fixed number of in-object slots based on initial constructor/literal sizes.

2. **Normal/Out-of-Object Properties (Fast)**:
   - When in-object slots are exhausted, extra properties are pushed to an external backing store array.
   - Access requires one indirection (lookup array, then fetch value).

3. **Slow Properties (Dictionary Mode)**:
   - If you delete properties frequently or have too many dynamic properties, V8 converts the object to a **Self-contained Dictionary (hash table)**.
   - Hidden classes are completely discarded.
   - Property access becomes significantly slower but handles arbitrary, sparse shapes gracefully.

---

## ⑤ ADVANTAGES

1. **Extremely Dynamic**: Modify structure, attach dynamic methods, or intercept properties at runtime.
2. **Engine Optimization**: JIT hidden classes allow JavaScript objects to execute as fast as static C++ structs when coded systematically.
3. **Descriptor Safety**: Property descriptors provide fine-grained API security, preventing consumers from overriding or deleting critical internal state.

---

## ⑥ DISADVANTAGES / TRAPS

1. **GC Pressure**: In-object shapes mutate easily. Creating thousands of temporary mismatched object layouts generates thousands of hidden classes, bloating V8 memory.
2. **Implicit Conversions**: Non-string keys coerced into `"[object Object]"` lead to silent bugs and key overwrite hazards.
3. **No True Deep Cloning**: Native copy mechanisms (`{...obj}`, `Object.assign`) perform shallow clones. Modifying nested properties on clones inadvertently corrupts the original instance.

## ⑦ PRODUCTION USAGE

### 🛠️ Safe Deep Freeze Utility

Standard `Object.freeze()` is shallow. In production configurations, use a recursive freezing utility:

```javascript
function deepFreeze(object) {
    // Retrieve property names
    const propNames = Reflect.ownKeys(object);

    // Freeze properties before freezing self
    for (const name of propNames) {
        const value = object[name];
        if (value && typeof value === 'object') {
            deepFreeze(value);
        }
    }

    return Object.freeze(object);
}

// Usage:
const APP_CONFIG = deepFreeze({
    api: {
        host: "api.production.com",
        timeout: 5000
    },
    features: {
        useNewBeta: true
    }
});

// Any attempt to mutate nested configs will throw errors in strict mode
APP_CONFIG.api.host = "malicious.com"; // Error!
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the primary ways to create an object in JavaScript?**  
**A:** Object literal (`{}`), Constructor functions (`new Fn()`), `Object.create()`, class instantiation (`new Class()`), and `Object.assign` or spread extensions.

**Q2: What is the difference between `Object.seal()` and `Object.freeze()`?**  
**A:** `Object.seal` prevents adding or deleting properties but allows modifying existing property values. `Object.freeze` prevents adding, deleting, and modifying values completely.

**Q3: What happens when you use a non-string or non-symbol value as an object key?**  
**A:** JavaScript coerces it to its string representation (e.g., numbers become `"1"`, objects become `"[object Object]"`).

**Q4: How do you check if a property exists directly on an object rather than its prototype chain?**  
**A:** Use `Object.hasOwn(obj, 'prop')` (modern standard) or `obj.hasOwnProperty('prop')` (legacy).

**Q5: What is a property descriptor?**  
**A:** An internal configuration object for a property containing attributes like `value`, `writable`, `enumerable`, and `configurable`.

---

### 🟡 Intermediate (5)

**Q6: What is a "pure dictionary" in JavaScript, and how do you create one?**  
**A:** An object with absolutely no prototype links (bypassing `Object.prototype`). Created using `Object.create(null)`. It lacks standard utility methods like `.toString()`, `.hasOwnProperty()`, and is immune to prototype pollution.

**Q7: Explain the default descriptor values of a property created via literal assignment vs. `Object.defineProperty()`.**  
**A:** Properties created via literals are fully mutable: `writable`, `enumerable`, and `configurable` default to `true`. Properties created via `Object.defineProperty` default to restrictive states: omitted descriptor properties evaluate to `false` (non-writable, non-enumerable, non-configurable).

**Q8: Why does `const obj = { x: 1 }; Object.freeze(obj); obj.x = 2;` silently fail in sloppy mode but throw in strict mode?**  
**A:** In sloppy mode, assigning to a non-writable property is quietly ignored to maintain compatibility. In strict mode (`"use strict"`), mutating frozen properties violates engine execution safety, throwing a `TypeError`.

**Q9: How do you create a read-only property that cannot be deleted or reconfigured?**  
**A:** Use `Object.defineProperty` with `writable: false` and `configurable: false`:
```javascript
Object.defineProperty(obj, 'id', { value: 100, writable: false, configurable: false });
```

**Q10: What is the difference between data descriptors and accessor descriptors?**  
**A:** A data descriptor maps a key to a concrete value (with `value` and `writable`). An accessor descriptor associates a property with getter/setter functions (`get` and `set`). A property cannot have both types of descriptors concurrently.

### 🔴 Advanced (5)

**Q11: Explain V8 Hidden Classes (Maps) and how object property order affects performance.**  
**A:** V8 matches dynamic objects to static structural definitions called Hidden Classes (Maps) using transition tracks. If two objects receive identical properties in a different order (e.g., `{a: 1, b: 2}` vs `{b: 2, a: 1}`), their transition paths branch, creating different hidden classes. This mismatches type-feedback loops at call sites, disabling JIT compilation optimization and forcing slower runtime resolution checks.

**Q12: Walk through V8's "Fast Properties" vs "Slow Properties" internal transitions.**  
**A:** 
1. **In-Object**: Pre-allocated slots direct on the heap object instance itself (fastest).
2. **Normal (Out-of-Object)**: If properties exceed in-object limits, they transfer to a backing store array, accessed via index offsets defined in the Map (one indirection).
3. **Slow (Dictionary Mode)**: If properties are dynamically deleted, or the object has a massive, highly sparse footprint, V8 discards the Map and converts the property store to a standard hash table, bypassing JIT shapes entirely.

**Q13: How does prototype pollution exploit vulnerable object implementations, and how do you protect against it?**  
**A:** Prototype pollution occurs when user-controlled inputs (like JSON query structures) are recursively merged into objects without input validation. A malicious payload containing keys like `__proto__` can inject attributes directly into `Object.prototype`, affecting all objects in the runtime.
*Protection:* 1. Use `Object.create(null)` for input stores. 2. Freeze the prototype: `Object.freeze(Object.prototype)`. 3. Use Maps instead of objects. 4. Validate input keys against a blacklist (`__proto__`, `constructor`, `prototype`).

**Q14: Explain why `Reflect.ownKeys()` differs from `Object.keys()` and `Object.getOwnPropertyNames()`.**  
**A:** 
* `Object.keys(obj)`: Returns only *enumerable, string-keyed* properties owned by the object.
* `Object.getOwnPropertyNames(obj)`: Returns all *string-keyed* properties owned by the object, including non-enumerable ones.
* `Reflect.ownKeys(obj)`: Returns all owned keys, including both *enumerable and non-enumerable string and symbol keys*.

**Q15: What is "Scalar Replacement of Aggregates" in V8 escape analysis?**  
**A:** A JIT compiler optimization pass where TurboFan analyzes a local object variable. If the object never escapes the function scope (not returned, not passed to other functions), TurboFan avoids allocating the object on the heap entirely. Instead, it extracts the object's property fields and binds them directly to local CPU registers, treating them as individual primitive scalar variables for maximum speed.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain exactly what occurs in V8 when you execute `delete obj.prop` on a fast property.**  
**A:** Executing `delete` on a property is an "optimization killer" for fast-mode structures. Under normal conditions, V8 manages object shapes using forward transitions. Removing a property in the middle of a shape chain breaks the transition tree. Rather than building expensive "reverse transition maps," V8 immediately degrades the target object's backing storage to **Slow Dictionary Mode**. The hidden class map pointer is reassigned to a dynamic dictionary shape, and future property lookups bypass fast JIT paths, executing via slow hash-table lookups.

**Q17: Explain the physical layout of a `JSObject` on the V8 heap, detailing the header, Map pointer, properties backing store, and elements store.**  
**A:** On the heap, a `JSObject` is stored as a contiguous memory block divided into distinct regions:
1. **Header (Map/Hidden Class Pointer)**: The first 4-8 bytes store a pointer to the Map defining the object's shape, descriptors, and transitions.
2. **Properties Pointer**: Points to the out-of-object backing store array containing named properties that did not fit in-object.
3. **Elements Pointer**: Points to the elements backing store. V8 separates integer-indexed properties (array indices) from named properties to perform optimized array operations.
4. **In-Object Field Slots**: Contiguous memory slots reserved for fast property access. Values are read directly at constant byte offsets relative to the start address of the `JSObject` pointer.
