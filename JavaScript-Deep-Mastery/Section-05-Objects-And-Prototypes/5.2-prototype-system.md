# 📌 5.2 — Prototype System
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** In JavaScript, object inheritance is not based on copying blueprints (like class-based languages). Instead, objects are directly linked to other objects in a chain. If you try to read a property or call a method that does not exist on a given object, JavaScript automatically looks up the "parent" object in the chain to find it. This parent link is called a **prototype**.

**Technical Definition:** Every JavaScript object contains an internal, implicit slot called `[[Prototype]]` (accessible via `Object.getPrototypeOf()` or the legacy `__proto__` accessor). If a property lookup (`[[Get]]`) fails on the target object, the engine recursively traverses the linked `[[Prototype]]` objects until it finds the property or reaches `null`. This mechanism is called **Delegation-based Prototype Inheritance**.

**One-line mental model:**
> *Objects in JavaScript don't copy properties; they delegate lookups to their prototype link until they hit null.*

---

## ② DRILL DOWN

### 🔹 A. `[[Prototype]]` vs. `prototype`

This is the absolute biggest point of confusion in JavaScript. They are completely different concepts!

```
                                  ┌───────────────────────────┐
                      ┌──────────►│  Function.prototype       │
                      │           ├───────────────────────────┤
                      │           │  .constructor ────────────┼──┐
                      │           │  .toString()              │  │
                      │           └───────────────────────────┘  │
                      │                         ▲                │
     [[Prototype]]    │                         │ [[Prototype]]  │
 (Internal Proto Link)│                         │                │
                      │           ┌─────────────┴─────────────┐  │
                      │           │   Object.prototype        │◄─┘
                      │           ├───────────────────────────┤
                      │           │   .hasOwnProperty()       │
                      │           │   .toString()             │
                      │           └───────────────────────────┘
                      │                         ▲
                      │                         │ [[Prototype]]
                      │                         │
 ┌────────────────────┴─────┐     ┌─────────────┴─────────────┐
 │   function User() {}     │     │   const u1 = new User()   │
 ├──────────────────────────┤     ├───────────────────────────┤
 │  • prototype ───────────┼────►│  • name: "Alice"          │
 └──────────────────────────┘     └───────────────────────────┘
```

1. **`[[Prototype]]` (Internal Slot)**:
   - Present on **every object**.
   - Direct link to the object's parent in the delegation chain.
   - Accessed via `Object.getPrototypeOf(obj)`.

2. **`prototype` (Ordinary Property)**:
   - Present **only on functions** (specifically constructible functions; arrow functions and bound functions do not have it).
   - It is just an ordinary property that points to an object.
   - It acts as the **blueprint object** that will be assigned as the `[[Prototype]]` of any new instances created when the function is called with the `new` keyword.

```javascript
function User(name) { this.name = name; }
const u1 = new User("Alice");

// u1's internal prototype points to User's prototype property:
console.log(Object.getPrototypeOf(u1) === User.prototype); // true

// User's own internal prototype points to Function's prototype property:
console.log(Object.getPrototypeOf(User) === Function.prototype); // true
```

### 🔹 B. The Prototype Lookup Chain

When you read a property `obj.foo`, the engine's internal `[[Get]]` method performs the following steps:
1. Checks if `foo` exists as a **direct property** of `obj` (Own property). If yes, returns it.
2. If not, calls `[[Get]]` recursively on `obj`'s internal `[[Prototype]]`.
3. Follows the chain upward (`obj.__proto__.__proto__`) until it either finds `foo` or hits `null`.
4. If it hits `null` without finding the property, it returns `undefined`.

```javascript
const grandparent = { hair: "curly" };
const parent = Object.create(grandparent);
parent.height = "tall";

const child = Object.create(parent);
child.age = 10;

console.log(child.age);    // 10 (Found on child)
console.log(child.height); // "tall" (Delegated to parent)
console.log(child.hair);   // "curly" (Delegated to grandparent)
console.log(child.eyes);   // undefined (Reached null prototype at top of chain)
```

---

### 🔹 C. Property Shadowing

**Shadowing** occurs when you write a property to an object that already exists on its prototype chain. The child object gets a direct, local copy, "shadowing" (hiding) the prototype's property.

However, writing behaves differently depending on property descriptors in the prototype chain:

```javascript
const proto = { value: 42 };
const obj = Object.create(proto);

// Scenario 1: Normal shadowing
obj.value = 100;
console.log(obj.value);   // 100 (Shadowed!)
console.log(proto.value); // 42 (Prototype untouched)

// Scenario 2: Prototype property is non-writable
Object.defineProperty(proto, 'readOnlyVal', { value: 999, writable: false });
const obj2 = Object.create(proto);
obj2.readOnlyVal = 10;    // Silently fails in sloppy mode, throws in strict mode!
console.log(obj2.readOnlyVal); // 999 (NOT shadowed!)

// Scenario 3: Prototype property has a setter
Object.defineProperty(proto, 'setVal', {
    set(v) { this._internal = v; }
});
const obj3 = Object.create(proto);
obj3.setVal = 55;         // Triggers the setter on proto, but 'this' points to obj3!
console.log(obj3._internal); // 55 (Stored on obj3)
```

### 🔹 D. Inspecting and Mutating Prototypes

JavaScript provides APIs to interact with prototype chains, though some should be avoided in production.

#### 1. `Object.getPrototypeOf(obj)` (Safe)
Retrieves the target's `[[Prototype]]` link. Always prefer this over the legacy `__proto__`.

#### 2. `Object.create(proto)` (Safe)
Creates a new empty object with its `[[Prototype]]` linked directly to the `proto` argument.

#### 3. `Object.setPrototypeOf(obj, newProto)` (❌ STRICTLY AVOID)
Dynamically rewrites an existing object's prototype.
> [!WARNING]
> Mutating prototypes using `Object.setPrototypeOf` or by writing directly to `__proto__` is a severe performance killer in all JavaScript engines. It ruins V8 Hidden Class/Map optimizations for the modified object and all its child objects downstream.

#### 4. The `__proto__` property (Legacy)
An accessor property (getter/setter) on `Object.prototype` that exposes the internal `[[Prototype]]` slot. It was standardized in ES6 only for browser compatibility and is deprecated.

---

### 🔹 E. Primitive Autoboxing

Primitives (strings, numbers, booleans) do not have prototypes and cannot store properties. Yet, we can call methods on them:
```javascript
const str = "hello";
console.log(str.toUpperCase()); // "HELLO"
```

#### How it works:
1. When you access a property on a primitive, JavaScript performs **autoboxing**.
2. It temporarily wraps the primitive in its object equivalent (e.g., `new String(str)`).
3. It performs the prototype lookup on the wrapper object (`String.prototype.toUpperCase`).
4. Once the operation finishes, the wrapper object is immediately discarded and garbage collected.

---

## ③ WHY IT EXISTS

1. **Massive Memory Savings**: Imagine an application with 10,000 instances of a `User` object. If each instance had its own copy of a massive `calculateSalary()` function, it would consume megabytes of RAM. Linking all instances to a single `User.prototype` means all 10,000 instances share the exact same function in memory.
2. **Behavior Delegation**: Prototype delegation allows building powerful inheritance trees that can adapt at runtime. Unlike compiled static class definitions, behaviors can be attached to prototypes dynamically, instantly updating all active instances.

---

## ④ INTERNAL WORKING: V8 PROTOTYPE OPTIMIZATIONS

### Prototype Chain Lookup V8 Internals

Without optimization, traversing a long prototype chain would require crawling heap references on every property lookup, tanking execution speeds. V8 solves this with **Inline Caching (IC)** and **Prototype Maps**.

```
    [ Object Instance ]
            │ (Map: C1)
            ▼
    [ Prototype Object ] ───► [ Hidden Class Map ] ───► Contains "Prototype User" list
            │ (Map: C2)
            ▼
    [ Global Object Prototype ] (e.g., Object.prototype)
```

1. **Prototype Maps**:
   - V8 keeps track of whether a hidden class represents a prototype object.
   - If an object is used as a prototype, its Map is marked as "immutable-by-delegation" and tracks all of its "users" (objects referencing it).

2. **Prototype Users Registration**:
   - When V8 compiles optimized code that reads `child.hair` (defined on `grandparent`), TurboFan doesn't generate lookup loop bytecode.
   - Instead, it embeds a constant offset pointer directly to the `grandparent` property address in the compiled machine code.
   - If a developer modifies `grandparent` (e.g., `grandparent.hair = "straight"`), V8 triggers a invalidation signal, invalidating the optimized JIT code for all connected child user structures instantly.

---

## ⑤ ADVANTAGES

1. **Memory Efficiency**: Shared method definitions reside on a single prototype reference rather than multiplying across instances.
2. **Dynamic Extension**: Adding a method to a prototype instantly updates all existing and future instances at runtime.
3. **Flexible Composition**: Avoids rigid multi-level hierarchical classes; objects can delegate to any arbitrary object, allowing runtime mixins and traits.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Performance Penalties (SetPrototypeOf)**: Modifying prototype links on-the-fly forces V8 to discard hidden class shapes and optimized machine code for the target and all downstream instances.
2. **Prototype Pollution Vulnerabilities**: Shared prototype tables can be polluted by recursive merges, exposing application environments to global cross-site scripting (XSS) or system level compromises.
3. **No Direct Properties Control**: Misunderstanding shadowed assignments leads to unexpected state changes when prototype values are reference types (e.g., arrays or nested objects).

---

## ⑦ PRODUCTION USAGE

### 🛠️ Safe Performance-Friendly Mixin Pattern

Instead of dynamically rewriting prototype chains using `Object.setPrototypeOf()`, use object compositions or establish linkages at object creation:

```javascript
// Dynamic functional features
const canFly = {
    fly() { return `${this.name} takes off!`; }
};

const canSwim = {
    swim() { return `${this.name} dives in!`; }
};

// GOOD: Composition at creation time (Safe, JIT-optimized)
function createSuperHero(name) {
    const instance = { name };
    // Merges mixins safely into the instance
    return Object.assign(instance, canFly, canSwim);
}

const hero = createSuperHero("Superman");
console.log(hero.fly());  // "Superman takes off!"
console.log(hero.swim()); // "Superman dives in!"
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is a prototype in JavaScript?**  
**A:** An object linked to another object, providing a delegation backup target for properties or methods that do not exist directly on the child object.

**Q2: What is the difference between `__proto__` and `.prototype`?**  
**A:** `__proto__` is a legacy accessor property exposing the internal `[[Prototype]]` link of an object instance. `.prototype` is an ordinary property unique to constructible functions that defines what prototype new instances created by that function will receive.

**Q3: Where does the prototype chain end?**  
**A:** The prototype chain ends at `Object.prototype.__proto__`, which is `null`.

**Q4: How do you create an object that does not inherit from `Object.prototype`?**  
**A:** Create it using `Object.create(null)`.

**Q5: What happens when you read a property that exists on both the object and its prototype?**  
**A:** The property on the object is returned immediately, "shadowing" the property on the prototype.

---

### 🟡 Intermediate (5)

**Q6: What is Autoboxing, and how does it relate to prototypes?**  
**A:** Autoboxing is the engine mechanism that temporarily wraps a primitive value (like a string) in an object wrapper (like `String()`) when a property is accessed. This allows primitive types to invoke methods stored on their respective wrapper prototypes (e.g., `String.prototype`) before the wrapper is garbage collected.

**Q7: Explain the prototype write lookup rule. Why does `obj.prop = 5` sometimes fail to shadow `prop` if `prop` exists on the prototype chain?**  
**A:** When setting `obj.prop = 5`, if `prop` is not on `obj` but exists on its prototype:
1. If the prototype's `prop` is `writable: true`, shadowing occurs, creating `prop` directly on `obj`.
2. If the prototype's `prop` is `writable: false`, the operation silently fails (or throws in strict mode); shadowing is blocked.
3. If the prototype's `prop` is an accessor descriptor with a setter, the setter is invoked instead of shadowing.

**Q8: Why is using `Object.setPrototypeOf()` or `obj.__proto__ = newProto` considered a severe anti-pattern?**  
**A:** Mutating prototype links breaks all existing Inline Cache (IC) assumptions and V8 hidden class maps down the entire inheritance chain. V8 must completely discard optimized machine code for these structures and perform slow, dynamic lookups.

**Q9: What is the difference between `Object.create(proto)` and assigning properties using `Object.assign()`?**  
**A:** `Object.create(proto)` creates a new empty object and links its internal `[[Prototype]]` slot to `proto` (creating a child-parent delegation relation). `Object.assign()` copies properties from source objects directly onto a target object (cloning values, no prototype linkage created).

**Q10: What does `Object.prototype.constructor` point to, and can it be changed?**  
**A:** It points back to the function that created the prototype object. It is writable and configurable, meaning it can be changed or deleted, which is why it cannot be fully trusted in production code to verify an instance's type.

---

### 🔴 Advanced (5)

**Q11: Explain V8's internal mechanism for optimizing prototype lookups. How does JIT compilation bypass prototype crawling?**  
**A:** V8 tracks objects used as prototypes. When TurboFan compiles an optimized property lookup (e.g., `child.method` which resides on `Object.prototype`), it registers the child's hidden class in a "Prototype Users" list on the prototype's Map. It then hardcodes the direct memory address of `method` inside the optimized machine code. If a prototype property is modified, V8 scans the Users list and instantly invalidates all optimized code pathways using that prototype, falling back to interpretation.

**Q12: Detail what occurs when you execute `new Foo()`. Step through the prototype assignment phase and the internal ES6 constructor logic.**  
**A:** 
1. A new empty object `{}` is allocated on the heap.
2. The engine retrieves the `prototype` property of the constructor function `Foo`. If `Foo.prototype` is not an object (e.g., `null`), it falls back to `Object.prototype`.
3. The new object's internal `[[Prototype]]` link is bound to that prototype object.
4. The constructor function `Foo` is executed, binding its `this` context to the newly created object.
5. If the constructor returns an Object type, that object is returned. Otherwise, the newly created object is returned.

**Q13: Explain why arrow functions do not possess a `.prototype` property and cannot be invoked as constructors.**  
**A:** Under the ECMAScript spec, arrow functions do not implement the internal `[[Construct]]` method or have a `.prototype` property. They are designed to be lightweight, lexical-scope-inheriting functions. Attempting to call an arrow function with `new` throws a `TypeError` because the engine cannot perform the object-prototype linkage or allocate a `this` frame.

**Q14: Explain the security implications of Prototype Pollution in Node.js applications and write a defense strategy.**  
**A:** Prototype pollution occurs when unsanitized input is merged into an object structure recursively. If an attacker inputs `{"__proto__": {"isAdmin": true}}`, the merge logic can write `isAdmin` directly onto `Object.prototype`. Now, every object in the runtime inherits `isAdmin: true`.
*Defense:* 
1. Use `Object.create(null)` for parsing user inputs.
2. Freeze global prototypes: `Object.freeze(Object.prototype)`.
3. Use `Map` objects instead of plain objects for dynamic dictionaries.
4. Run input keys through a rigorous blacklist/whitelist filter.

**Q15: Explain how `instanceof` works internally by stepping through the prototype matching algorithm.**  
**A:** The `instanceof` operator evaluates `obj instanceof Constructor` using the internal `[Symbol.hasInstance]` method:
1. The engine checks if `Constructor` is an object and is callable (throws `TypeError` if not).
2. It fetches `Constructor.prototype`.
3. It extracts `Object.getPrototypeOf(obj)` (the implicit prototype parent).
4. It runs a loop: if the extracted prototype is `null`, it returns `false`. If the prototype matches `Constructor.prototype`, it returns `true`. Otherwise, it advances to `Object.getPrototypeOf(proto)` and repeats.

---

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain how the V8 engine handles property lookups on an object when the prototype chain is extremely long (e.g., 20 levels deep), and how Megamorphic caches operate under this scenario.**  
**A:** If a prototype chain is extremely deep, a standard lookup traversal is highly expensive. To prevent this, V8 uses **Inline Caching (IC)** at every property lookup instruction site in the bytecode.
1. **Monomorphic IC**: If the target object shape remains uniform, V8 caches the memory offset of the property directly, even if it's 20 levels deep, requiring zero traversals after the first lookup.
2. **Polymorphic/Megamorphic IC**: If the call site encounters many different object shapes, the cache degrades. Under megamorphic conditions, V8 falls back to a **global lookup table (Global Lookup Cache)** rather than crawling the heap. The GLC caches the tuple `(Object Map, Prototype Chain Layout, Property Name)` mapping to the direct offset, preventing costly chain crawling.

**Q17: Explain the exact sequence of memory operations and hidden class transitions that occur when you dynamically modify `Object.prototype` (e.g., `Object.prototype.myCustomMethod = function() {}`) during runtime execution of a large codebase.**  
**A:** When `Object.prototype.myCustomMethod` is assigned:
1. V8 identifies that `Object.prototype`'s hidden class Map is marked as a **Prototype Map**.
2. It immediately walks the **Prototype Users** registration list. This list links every active hidden class Map in the entire runtime that descends from `Object.prototype` (which is almost all objects).
3. The engine invalidates the prototype caches for all registered Maps.
4. All JIT-optimized functions that perform property lookups or method execution on any object are immediately marked as stale and compiled-code de-optimized.
5. Next time those functions run, they execute a bailout back to Ignition interpreter bytecode to recalculate shapes, causing a massive, temporary global CPU spike across the runtime.
