# 📌 5.5 — Inheritance
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Inheritance allows you to create a new class (subclass) based on an existing class (base class), inheriting all of its properties and methods. In JavaScript, inheritance links both the prototype of the instances *and* the prototype of the classes themselves, allowing static methods to be inherited as well.

**Technical Definition:** In ES6 classes, inheritance is declared via the `extends` keyword. This establishes a dual prototype chain: an **Instance Chain** (where `SubClass.prototype.__proto__` points to `BaseClass.prototype`) and a **Class/Constructor Chain** (where `SubClass.__proto__` points to `BaseClass`). Subclass constructors must call `super()` before referencing `this`, as the actual object allocation is delegated to the base class constructor using `new.target`.

**One-line mental model:**
> *Class inheritance builds a dual-prototype link, where subclass instances delegate to base prototypes, and subclass constructors delegate to base constructors.*

---

## ② DRILL DOWN

### 🔹 A. The Dual Prototype Chain of Inheritance

Unlike simple constructor prototypes, ES6 class inheritance links **two separate prototype chains**:

```
INSTANCES PROTOTYPE CHAIN (Delegation of instance methods):
 u1 (Instance of Admin) ──[[Prototype]]──► Admin.prototype ──[[Prototype]]──► User.prototype ──[[Prototype]]──► Object.prototype

CLASS CONSTRUCTORS PROTOTYPE CHAIN (Delegation of static methods):
 Admin (Constructor Function) ──[[Prototype]]──► User (Constructor Function) ──[[Prototype]]──► Function.prototype
```

```javascript
class User {
    constructor(name) { this.name = name; }
    greet() { return `Hello, I'm ${this.name}`; }
    static getRole() { return "Guest"; }
}

class Admin extends User {
    constructor(name, permissions) {
        // Must call super() before accessing 'this'!
        super(name); 
        this.permissions = permissions;
    }
}

const admin = new Admin("Alice", ["read", "write"]);

// 1. Instance method inheritance:
console.log(admin.greet()); // "Hello, I'm Alice" (Delegated to User.prototype)

// 2. Static method inheritance:
console.log(Admin.getRole()); // "Guest" (Delegated directly to User class!)

// 3. Confirming Dual Chains:
console.log(Object.getPrototypeOf(Admin.prototype) === User.prototype); // true
console.log(Object.getPrototypeOf(Admin) === User); // true (Static inheritance!)
```

---

### 🔹 B. The Golden Rule of Derived Constructors: `super()`

If a class extends another class, it is classified as a **derived constructor**. Derived constructors have a strict execution rule: you **must** call `super()` before accessing `this`.

```javascript
class Base {}
class Derived extends Base {
    constructor() {
        // console.log(this); // ReferenceError: Must call super constructor in derived class before accessing 'this'
        super();
        console.log(this); // Works now!
    }
}
```

#### Why does this happen?
In legacy ES5 inheritance, the subclass allocated the object (`this = {}`) and passed it to the parent constructor to modify. In ES6 class inheritance, the **parent class allocates the object** using the `new.target` of the subclass. Therefore, `this` does not exist until the parent constructor (`super()`) allocates and returns it.

---

### 🔹 C. Overriding Methods and Static Inheritance

Subclasses can override methods of their parent class. The parent method can still be called using the `super` keyword.

```javascript
class Worker {
    work() { return "Working hard."; }
}

class Manager extends Worker {
    work() {
        const baseWork = super.work(); // Delegates to parent method
        return `${baseWork} And managing people.`;
    }
}

const manager = new Manager();
console.log(manager.work()); // "Working hard. And managing people."
```

---

### 🔹 D. Subclassing Built-In Constructors

Before ES6, subclassing native constructors like `Array`, `Map`, or `Error` was impossible. Under legacy patterns, calling `Array.apply(this)` could not allocate the internal slots required for arrays (like `.length` tracking).

ES6 classes natively support subclassing built-ins:

```javascript
class CustomArray extends Array {
    first() { return this[0]; }
    last() { return this[this.length - 1]; }
}

const arr = new CustomArray(1, 2, 3);
console.log(arr instanceof CustomArray); // true
console.log(arr instanceof Array);       // true
console.log(arr.first()); // 1
console.log(arr.last());  // 3
```

---

## ③ WHY IT EXISTS

1. **Static Inheritance Support**: Legacy ES5 constructor functions lacked a clean way to inherit static properties and methods. Developers had to manually copy functions or set proto-links.
2. **Standardized Native Subclassing**: Prior to ES6 classes, there was no way to extend native array classes, error models, or promises because you could not execute parent constructors with subclass-specific allocator contexts (`new.target`).
3. **Structured Polymorphism**: Classes establish a clean, standard language for overriding methods and organizing logical trees without massive boilerplate.

## ④ INTERNAL WORKING: ALLOCATION & STATIC INHERITANCE

### The "Base Constructor Allocates" Model

Under the hood, when `new SubClass()` is evaluated, the allocation phase works differently for derived classes:

```
new SubClass()
      │
      ▼
SubClass Constructor Activated (Lexical Flag: [[ConstructorKind]] = "derived")
      │
      ├─► Does NOT allocate an object for 'this'
      │
      ▼
   super() Invoked ────────────────────────────────────────┐
                                                           ▼
                                            BaseClass Constructor Activated
                                            ([[ConstructorKind]] = "base")
                                                           │
                                                           ▼
                                            Allocates brand new JSObject {}
                                            Links prototype to new.target.prototype 
                                            (which points to SubClass.prototype!)
                                                           │
                                                           ▼
                                            Returns the object up the chain
                                                           │
                                                           ▼
                                            Subclass receives returned object 
                                            and binds it to 'this'
```

This elegant design ensures that if the base class is a built-in (e.g., `Array`), the engine's low-level array allocation code is executed, creating all required internal array properties (like special index lookup caches) while linking it to the subclass prototype.

---

## ⑤ ADVANTAGES

1. **Dual Prototype Delegation**: Clean, out-of-the-box static method inheritance makes utility systems highly intuitive.
2. **Native Extension**: Extends standard built-ins (`Array`, `Error`, `RegExp`) securely without corrupting runtime environments.
3. **Reflective Super Binding**: Super delegation (`super.method()`) automatically resolves binding contexts (`this`) to the calling subclass instance, preventing binding loss.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Violating super() rules**: Accessing `this` before calling `super()` leads to immediate runtime failures (`ReferenceError`).
2. **Deep Inheritance Bloat**: Building deep vertical inheritance hierarchies makes code brittle and tightly coupled, violating the modern production best practice: "Favor composition over inheritance."
3. **Static Method Loss on Instances**: Developers often try to invoke static inherited methods directly on instance objects (e.g., `instance.staticMethod()`), throwing `TypeError` because instances are not part of the class constructor's prototype chain.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Custom Error Subclass with Stack Preservation

Creating a custom error class is one of the most common production use cases for class inheritance. It allows preserving correct names and stack traces for monitoring systems:

```javascript
class DatabaseError extends Error {
    constructor(message, query, code) {
        super(message); // Sets the message property on the base Error instance

        // 1. Explicitly restore prototype chain (needed for some compilation targets)
        Object.setPrototypeOf(this, new.target.prototype);

        // 2. Set custom debugging metadata
        this.name = this.constructor.name; // "DatabaseError"
        this.query = query;
        this.code = code;

        // 3. Capture clean stack trace, excluding the constructor call itself
        if (Error.captureStackTrace) {
            Error.captureStackTrace(this, this.constructor);
        }
    }
}

try {
    throw new DatabaseError("Connection timed out", "SELECT * FROM users", "ERR_DB_TIMEOUT");
} catch (err) {
    console.error(err.name);    // "DatabaseError"
    console.error(err.code);    // "ERR_DB_TIMEOUT"
    console.error(err.stack);   // Clean, relevant stack trace
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: How do you make a class inherit from another class in JavaScript?**  
**A:** Use the `extends` keyword (e.g., `class Sub extends Base {}`).

**Q2: What is the purpose of the `super()` call inside a constructor?**  
**A:** It executes the parent constructor, allocating the instance and initializing parent properties.

**Q3: What happens if you access `this` before calling `super()` in a derived constructor?**  
**A:** The engine throws a `ReferenceError`.

**Q4: Can static methods be inherited in JavaScript?**  
**A:** Yes. In JavaScript classes, the subclass constructor's prototype is linked to the parent class constructor, allowing static methods to be inherited.

**Q5: How can a subclass invoke a method defined on the parent class that it has overridden?**  
**A:** By using the `super` keyword (e.g., `super.methodName()`).

---

### 🟡 Intermediate (5)

**Q6: Why was it impossible to properly subclass built-in constructors (like `Array` or `Error`) in ES5?**  
**A:** In ES5, the subclass instance was created first, and then the parent constructor was called with the subclass `this` (e.g., `Array.call(this)`). Native constructors like `Array` ignored the passed `this` context and returned a brand-new array, making it impossible for the subclass to inherit internal array structures (like `.length` tracking).

**Q7: Explain the dual prototype chain created by the `extends` keyword.**  
**A:** 
1. **Instance Chain**: Links class prototype objects (`SubClass.prototype.__proto__ === BaseClass.prototype`), enabling subclass instances to inherit parent methods.
2. **Class Chain**: Links class constructors directly (`SubClass.__proto__ === BaseClass`), enabling the subclass to inherit static methods.

**Q8: Explain the outcome of this code: `class A {} class B extends A { constructor() {} } new B();`**  
**A:** The code will throw a `ReferenceError` during instantiation. In a derived class (class B), if you define a custom `constructor()`, you **must** call `super()` inside it. Leaving out `super()` prevents the instance from being allocated, so `this` is never bound and cannot be returned.

**Q9: If a subclass does not define a constructor, how does the default constructor handle inheritance?**  
**A:** The JS engine automatically injects a default constructor that passes all arguments to the parent constructor:
```javascript
constructor(...args) {
    super(...args);
}
```

**Q10: Can a class extend an ordinary constructor function?**  
**A:** Yes. Since classes are built on prototypes, the `extends` keyword can target any constructible function that has a `.prototype` property (e.g., `class MyComponent extends LegacyWidget {}`).

---

### 🔴 Advanced (5)

**Q11: Explain V8's internal `[[ConstructorKind]]` flag and how it changes object allocation during a `new` call.**  
**A:** 
* When a class is declared, V8 assigns it a `[[ConstructorKind]]` flag: `"base"` for standard classes or `"derived"` for classes using `extends`.
* When calling `new BaseClass()`, V8 immediately allocates the heap memory block for `{}`.
* When calling `new DerivedClass()`, V8 sees the `"derived"` flag and skips memory allocation.
* Only when the derived constructor executes `super()` does the engine delegate the allocation step to the base constructor. The base constructor allocates the memory and sets the new object's prototype to `new.target.prototype` (the subclass prototype).

**Q12: How does the `super` keyword resolve its scope? Explain how it binds the correct `this` dynamically.**  
**A:** The `super` keyword is **statically bound** when a method is defined, using an internal slot called `[[HomeObject]]` on the method's function.
* `[[HomeObject]]` points to the prototype object containing the method.
* When calling `super.method()`, the engine retrieves `Object.getPrototypeOf([[HomeObject]])` to find the parent prototype, then invokes the parent method with `this` dynamically set to the calling subclass instance (using the internal `[[GetSuperValue]]` operation).

**Q13: Why is `Object.setPrototypeOf(SubClass.prototype, BaseClass.prototype)` insufficient to fully replicate ES6 class inheritance?**  
**A:** It only sets up the **Instance Chain**, allowing instances to delegate methods. It misses:
1. **Static Inheritance**: It does not link the constructors (`Object.setPrototypeOf(SubClass, BaseClass)`), so static methods cannot be inherited.
2. **Native Subclassing**: It does not configure the internal `[[ConstructorKind]]: "derived"` allocation chain, making it impossible to extend built-ins.

**Q14: Explain the performance implications of deep class inheritance chains in V8, specifically focusing on polymorphic inline caches (PICs).**  
**A:** A deep inheritance chain means a property search might crawl multiple prototype layers. If a hot function receives objects from various depths or shapes in the chain, it triggers a polymorphic or megamorphic state.
V8's Inline Cache (IC) must perform costly checks across all registered prototype maps in the chain. Deep chains dramatically increase the chances of IC invalidation and deoptimization, causing the engine to fall back to slow global lookup caches.

**Q15: How does `new.target` act as an instantiation hook during `super()` calls in multi-tiered inheritance (e.g., A ➔ B ➔ C)?**  
**A:** When calling `new C()`, `new.target` points to `C` at the start.
As `C` calls `super()` (delegating to `B`), and `B` calls `super()` (delegating to `A`), `new.target` remains bound to `C` throughout the chain.
When class `A` finally allocates the object, it links its prototype to `new.target.prototype` (`C.prototype`). This ensures that even though class `A` allocated the memory, the object is correctly linked to the prototype of the subclass `C` that was actually instantiated.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact sequence of memory operations and prototype modifications that occur under the hood when a transpiler like Babel converts ES6 class inheritance to ES5-compatible code.**  
**A:** 
1. **Inheritance Setup**: Babel generates a helper function (usually `_inherits(subClass, superClass)`) which executes:
   - `subClass.prototype = Object.create(superClass.prototype, { constructor: { value: subClass, writable: true, configurable: true } })` (Instance Chain setup).
   - `Object.setPrototypeOf(subClass, superClass)` or copies properties manually if `setPrototypeOf` is missing (Static Chain setup).
2. **Constructor Wrapper**: Babel wraps the subclass constructor.
3. **Super Allocation Hack**: Since ES5 engines lack `new.target` and `"derived"` allocation flags, Babel converts `super()` calls into:
   - `superClass.apply(this, arguments)` to invoke the parent constructor with the subclass `this` context.
   - For built-ins, Babel uses a fallback hack: it allocates a real parent object (`new Array()`), and then dynamically changes its prototype using `__proto__` to point to the subclass prototype, which incurs a substantial performance penalty in all engines.

**Q17: Explain the exact memory layouts, hidden class shapes, and JIT compilation steps that occur in V8 when a subclass overrides a parent field (e.g., `class Parent { x = 1 }` and `class Child extends Parent { x = 2 }`).**  
**A:** 
1. **Allocation**: `new Child()` triggers allocation via `super()`, pre-sizing the instance based on both parent and child fields.
2. **Parent Initialization**: The parent field initializer runs first, writing `x = 1`. This creates Map `C1` (stores `x` at offset 0).
3. **Child Initialization**: The child field initializer runs next, writing `x = 2`.
4. **V8 Optimization**: Since `x` is already present at offset 0 on Map `C1`, V8 does not allocate a new property slot or transition to a new hidden class. It simply overwrites the value at offset 0 with `2`.
5. **JIT compilation**: Since the hidden class Map transitions remain stable (`C0 ➔ C1`), TurboFan compiles a highly optimized monomorphic access path to offset 0, ignoring the fact that the field was overwritten during initialization.
