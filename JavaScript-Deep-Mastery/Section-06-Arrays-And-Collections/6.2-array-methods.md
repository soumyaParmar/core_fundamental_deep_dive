# 📌 6.2 — Array Methods
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Array methods are built-in functions provided by JavaScript to manipulate, search, filter, sort, and transform arrays. These methods are categorized into **Mutators** (which change the original array in-place) and **Accessors/Iterators** (which leave the original array untouched and return new arrays or values).

**Technical Definition:** Array methods are high-performance functions defined on `Array.prototype`. Mutator methods (e.g., `push`, `pop`, `shift`, `unshift`, `splice`) modify the array's underlying heap-allocated backing store in-place. Iterator methods (e.g., `map`, `filter`, `reduce`) execute user-provided callback functions via higher-order loops, often highly optimized by modern engines through inline callback parsing and loop unrolling.

**One-line mental model:**
> *Array methods are highly optimized, built-in loops and backing-store modifiers designed to make collection handling expressive and fast.*

---

## ② DRILL DOWN

### 🔹 A. Mutator Methods (In-Place Modifications)

Mutators modify the array they are called on. They are highly efficient but can lead to side effects.

```javascript
const list = [2, 3];

// 1. push / pop (Add/remove from end - O(1) amortized)
list.push(4); // [2, 3, 4]
list.pop();    // [2, 3]

// 2. unshift / shift (Add/remove from start - O(N) because elements must be shifted in memory!)
list.unshift(1); // [1, 2, 3]
list.shift();     // [2, 3]

// 3. splice (Insert, replace, or remove at arbitrary indices - O(N))
// Syntax: splice(start, deleteCount, ...itemsToAdd)
list.splice(1, 1, "a", "b"); // Removes 1 item at index 1, adds "a" and "b"
console.log(list); // [2, "a", "b"]

// 4. reverse / sort (In-place ordering)
list.reverse(); // ["b", "a", 2]
```

---

### 🔹 B. Accessor and Iterator Methods (Pure / Non-Mutating)

These methods do not modify the original array; instead, they return a new representation.

```javascript
const numbers = [1, 2, 3, 4];

// 1. slice (Creates a shallow copy of a portion of an array - O(N))
const sub = numbers.slice(1, 3); // [2, 3] (index 1 to 3 exclusive)

// 2. concat (Combines multiple arrays into a new one)
const combined = numbers.concat([5, 6]); // [1, 2, 3, 4, 5, 6]

// 3. filter (Creates a new array with all elements passing the test)
const evens = numbers.filter(x => x % 2 === 0); // [2, 4]

// 4. map (Transforms each element)
const doubled = numbers.map(x => x * 2); // [2, 4, 6, 8]

// 5. reduce (Accumulates the array into a single value)
// Syntax: reduce(callback(accumulator, currentValue, index, array), initialValue)
const sum = numbers.reduce((acc, val) => acc + val, 0); // 10
```

---

### 🔹 C. The `sort()` Method Gotchas and Custom Comparators

By default, calling `sort()` without arguments converts elements to **strings** and sorts them lexicographically (Unicode point order). This leads to notorious bugs with numbers:

```javascript
const unsorted = [10, 2, 5, 1];
unsorted.sort();
console.log(unsorted); // [1, 10, 2, 5] ("10" comes before "2" alphabetically!)

// The Correct Way (Custom Numeric Comparator):
unsorted.sort((a, b) => a - b); // Ascending sort
console.log(unsorted); // [1, 2, 5, 10]

// Descending sort:
unsorted.sort((a, b) => b - a); // [10, 5, 2, 1]
```

### 🔹 D. Performance & Big-O Complexity of Common Methods

Understanding method complexity is crucial for writing scalable code:

| Method | Time Complexity | Space Complexity | Notes |
| :--- | :---: | :---: | :--- |
| **`push` / `pop`** | $O(1)$ | $O(1)$ | Fast. Only interacts with the end of the array backing store. |
| **`shift` / `unshift`** | $O(N)$ | $O(1)$ | Slow. Forces the engine to shift all remaining elements in memory by 1 slot. |
| **`slice`** | $O(N)$ | $O(N)$ | Shallow copies the backing store references. |
| **`splice`** | $O(N)$ | $O(N)$ | In-place shifts elements after the target index. |
| **`sort`** | $O(N \log N)$ | $O(N)$ | Uses Timsort (stable sorting algorithm). |
| **`map` / `filter`** | $O(N)$ | $O(N)$ | Allocates a brand-new array. |
| **`forEach` / `reduce`**| $O(N)$ | $O(1)$ | Modifies accumulator or executes side effects. No new array allocated. |

## ③ WHY IT EXISTS

Array methods exist to transition JavaScript from imperative loops (e.g., `for (let i = 0; i < len; i++)`) to functional, declarative programming.
1. **Expressive Code**: Expressing intent (`filter`, `map`) is significantly easier to read and maintain than writing manual index trackers, push buffers, and conditional statement blocks.
2. **Safety**: Manual index tracking frequently introduces "off-by-one" boundary bugs. High-level methods completely abstract iteration boundaries away.
3. **Engine-Optimized**: V8 writes built-in prototype methods in highly optimized C++ or JIT assembly, executing loops much faster than ordinary user-written JavaScript `for` loops.

---

## ④ INTERNAL WORKING: TIMSORT & INLINING

### V8's Sorting Engine: Timsort

Prior to ES2019, `Array.prototype.sort()` was unstable (equal elements could change their relative order) because V8 used QuickSort for large arrays and InsertionSort for small ones.
Today, V8 uses **Timsort** (adapted from Python):
* **Stable Sort**: Guarantees equal elements retain their original relative order.
* **Hybrid Algorithm**: It works by finding subsets of data that are already sorted (called "runs") and uses InsertionSort to complete small runs, and then merges them using a highly optimized MergeSort.

---

### JIT Callback Inlining and Loop Unrolling

When you run:
```javascript
const double = arr.map(x => x * 2);
```
Under normal conditions, calling a JavaScript callback function `x => x * 2` once per element would incur heavy call-stack and activation-record execution overhead.
To optimize this, **TurboFan perform JIT Callback Inlining**:
1. If the callback is small and the loop is hot, the compiler replaces the callback invocation step with the direct inner expression `x * 2` inside the compiled machine-code loop.
2. It performs **Loop Unrolling** — executing multiple index operations per CPU clock cycle, completely bypassing JavaScript call frame overhead and executing at raw C++ speeds.

---

## ⑤ ADVANTAGES

1. **Declarative Clarity**: Code describes *what* to do with data rather than *how* to crawl memory registers.
2. **Built-in Performance**: Standard methods run on native C++ backing loops, highly optimized by the JIT compiler.
3. **Chaining**: Methods can be seamlessly chained (`arr.filter().map().reduce()`) to build expressive data transform pipelines.

---

## ⑥ DISADVANTAGES / TRAPS

1. **In-Place Side Effects (Mutator Bug)**: Modifying arrays using `sort()`, `reverse()`, or `splice()` directly alters shared references, creating hard-to-track state bugs.
2. **Intermediate Allocation Bloat**: Chaining methods (e.g., `.filter().map().slice()`) forces the engine to allocate a brand-new intermediate array at every single step, generating heavy Garbage Collection (GC) load.
3. **Monomorphism Breakage**: If an array method callback accepts or returns values of varying shapes, it forces the array elements store to degrade to generic `HOLEY_ELEMENTS`, slowing down JIT execution.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Optimization: Single-Pass Pipelines

To process large datasets without creating slow, intermediate memory-allocated arrays, consolidate multiple chained operations into a single-pass `reduce` or a single `for` loop:

```javascript
const largeDataset = Array.from({ length: 100000 }, (_, i) => ({ id: i, value: i * 2 }));

// BAD: Chaining creates 2 intermediate arrays on the heap (Bloats GC)
const badResult = largeDataset
    .filter(item => item.id % 2 === 0)
    .map(item => item.value)
    .slice(0, 100);

// GOOD: Single-pass iteration using a simple loop or reduce (Zero intermediate arrays allocated!)
const goodResult = [];
for (let i = 0; i < largeDataset.length; i++) {
    const item = largeDataset[i];
    if (item.id % 2 === 0) {
        goodResult.push(item.value);
        if (goodResult.length === 100) break; // Early termination support!
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is the difference between a mutator method and an accessor method?**  
**A:** A mutator method modifies the original array in-place (e.g., `push`, `splice`). An accessor method does not change the original array, returning a new representation or value instead (e.g., `slice`, `concat`).

**Q2: What does `Array.prototype.sort()` do by default if you don't pass a comparator function?**  
**A:** It converts all elements into strings and sorts them in ascending lexicographical order based on UTF-16 Unicode point values.

**Q3: How do `slice()` and `splice()` differ?**  
**A:** 
* `slice(start, end)` is an accessor. It returns a shallow copy of a sub-segment of the array without modifying the original.
* `splice(start, deleteCount, ...items)` is a mutator. It removes or replaces existing elements in-place and returns the deleted elements.

**Q4: What is the time complexity of the `shift()` and `unshift()` methods?**  
**A:** $O(N)$. Because adding or removing an element at the beginning of the array forces the engine to shift every remaining element in memory to the adjacent index slot.

**Q5: Which array method is best suited for accumulating all values into a single output?**  
**A:** `Array.prototype.reduce()`.

---

### 🟡 Intermediate (5)

**Q6: Explain the stability of `Array.prototype.sort()` and why it matters in production.**  
**A:** A sorting algorithm is **stable** if it guarantees that elements with equal sorting keys retain their original relative order. Prior to ES2019, V8's sorting was unstable for larger arrays. Today, ES2019 standardizes stable sorting (using Timsort). This is critical when sorting objects by multiple criteria (e.g., sorting users by first name, then by last name; a stable sort ensures the first-name order isn't scrambled during the second sort).

**Q7: Explain the difference between `forEach()` and `map()`.**  
**A:** 
* `forEach()` executes a callback for each element to perform side effects. It always returns `undefined` and does not allocate a new array.
* `map()` transforms each element using a callback and returns a brand-new array containing the returned values, leaving the original array intact.

**Q8: What is intermediate allocation bloat when chaining array methods, and how do you optimize it?**  
**A:** Intermediate allocation bloat occurs when you chain methods like `.filter().map().reverse()`. Each method allocation instantiates a new array on the heap. For large datasets, this wastes memory and triggers frequent garbage collection (GC) pauses.
*Optimization:* Combine operations into a single-pass `reduce` or use a standard `for` loop to build the final output in a single run.

**Q9: Why does `[1, 2, 3].reduce((acc, val) => acc + val)` throw a TypeError if called on an empty array `[]` without an initial value?**  
**A:** Under the ECMAScript specification, if no initial value is passed to `reduce()`, the engine assigns index `0` as the accumulator and begins iteration at index `1`. If the array is empty, there is no index `0` element to use as the accumulator, and no initial value is defined. Having no value to return, the engine throws a `TypeError: Reduce of empty array with no initial value`.

**Q10: What is the difference between `some()` and `every()`, and do they support short-circuiting?**  
**A:** 
* `some()` returns `true` if *at least one* element passes the callback test. It short-circuits (stops iteration immediately) as soon as it finds a matching element.
* `every()` returns `true` only if *all* elements pass the callback test. It short-circuits as soon as it encounters a single failing element.

---

### 🔴 Advanced (5)

**Q11: Explain V8's JIT loop-unrolling and callback inlining optimizations for `Array.prototype.map`.**  
**A:** When V8 encounters a hot loop invoking `map(callback)`, the JIT compiler (TurboFan) analyzes the callback. If the callback is monomorphic and small, TurboFan **inlines** it, inserting the compiled callback body directly into the loop. It then performs **loop unrolling**, duplicating the unrolled loop instructions to process multiple index slots (e.g., 4 at a time) per CPU clock cycle. This eliminates function call frame allocation and register save/restore steps, matching native C++ loop speeds.

**Q12: Why is the time complexity of `push()` written as $O(1)$ amortized instead of a strict $O(1)$?**  
**A:** Standard `push()` operations write directly to the next pre-allocated empty slot in the array's contiguous memory block, which is $O(1)$. However, when the array capacity limit is reached, V8 must execute a **reallocation**. It allocates a new, larger memory block (growing by ~50%), copies all existing elements over, and updates pointers. Since this copy takes $O(N)$ time but occurs rarely, the average cost per `push` remains $O(1)$, which is mathematically termed **amortized constant time**.

**Q13: Explain how `flat()` and `flatMap()` resolve nested arrays internally. What are their time and space complexities?**  
**A:** 
* `flat(depth)` recursively traverses the array up to the specified depth, allocating elements directly into a new flattened array.
* `flatMap(callback)` is equivalent to calling `.map(callback)` followed by `.flat(1)`. However, `flatMap` is highly optimized: it flattens elements directly during the mapping pass, skipping the allocation of the intermediate array.
* *Complexity:* Both operate in $O(N + M)$ time (where $N$ is input elements, $M$ is nested elements) and $O(N + M)$ space due to allocating the flat output array.

**Q14: Explain why `arr.sort()` on an array of objects can sometimes cause JIT de-optimizations in V8 if the comparator function returns variable types.**  
**A:** V8's JIT compiler optimizes the comparator function `(a, b) => a.val - b.val` assuming the inputs are monomorphic objects and `val` is always a number. If the objects have varying hidden classes (shapes), or `val` occasionally returns `undefined`, a string, or a boolean, the type-feedback loop registers a shape change or type mismatch. This pollutes the Inline Cache (IC), forcing TurboFan to de-optimize the compiled comparator code, falling back to the slow interpreter mid-sort.

**Q15: How do array methods handle holes in sparse arrays? Detail the differences between `forEach`, `filter`, `find`, and spread syntax `[...arr]`.**  
**A:** 
* `forEach` and `filter` skip holes completely. They only execute the callback on indices that physically exist as keys on the array object.
* `find` and `findIndex` **do not** skip holes. They treat holes as index slots containing the value `undefined`.
* Spread syntax `[...arr]` uses the Iterator Protocol, walking every index from `0` to `length - 1` sequentially and yielding `undefined` for holes, converting them into concrete elements in the new array.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact memory operations, Hidden Class transitions, and GC impacts that occur in V8 when you call `arr.shift()` inside a hot loop on a large packed array.**  
**A:** Executing `.shift()` in a loop is a severe performance anti-pattern.
1. **Backing Store Shift**: To remove index `0`, V8's runtime must shift every single remaining pointer in the contiguous backing store array to the left by 1 index slot (e.g., `arr[1] ➔ arr[0]`, `arr[2] ➔ arr[1]`). This requires a memory-copy operation (`memmove`) of size $O(N)$ on the heap.
2. **Length Update**: V8 updates the magic `.length` property descriptor.
3. **GC Pressure**: If the loop runs $N$ times, the overall time complexity is $O(N^2)$. The constant shifting and copying do not allocate new objects directly, but they block execution threads and generate high volume of CPU cache misses as the engine constantly invalidates memory cache lines.

**Q17: Explain the step-by-step bytecode execution and on-stack frames layout when an optimized JIT compiled `map` loop triggers a bailout (de-optimization) because the callback function returns a non-monomorphic object shape half-way through the array.**  
**A:** 
1. **Speculative Execution**: TurboFan compiles the `map` loop assuming the callback returns a uniform object shape (Map `C1`). The loop runs at machine-code level, allocating pre-sized objects.
2. **Type Violation**: At index 5000, the callback returns a new object shape (Map `C2`).
3. **De-opt Detection**: The compiled machine code executes a shape check (Inline Cache check). The check fails.
4. **Bailout Trigger**: The CPU jumps to a deoptimization exit stub in V8's runtime.
5. **Frame Reconstruction**: V8 pauses the loop. It reads the CPU register states and reconstructs the unoptimized interpreter execution frames for the active `map` loop and the callback function on the call stack, restoring variables to their bytecode register slots.
6. **Reversion**: V8 discards the TurboFan machine code. It resumes the loop at index 5000 inside the Ignition interpreter using slow, dynamic bytecode lookups, while marking the map JIT profile as polymorphic.
