# 📌 6.4 — Maps & Sets
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Maps and Sets are advanced collection types introduced in ES6. A **Map** is a collection of key-value pairs where, unlike plain objects, the keys can be any data type (including objects or functions) and preserve their insertion order. A **Set** is a collection of unique values, meaning it automatically filters out duplicate entries. **WeakMap** and **WeakSet** are special versions that hold only weak references to their keys, allowing garbage collection to clean up memory automatically when the keys are no longer used elsewhere.

**Technical Definition:** Maps and Sets are built-in hash-table structures defined in the ECMAScript standard that provide sub-linear $O(1)$ lookup complexities. Unlike plain objects, they do not suffer from prototype pollution hazards, do not coerce keys into strings, and preserve insertion order. WeakMaps and WeakSets are implemented using **Ephemeron** mappings, ensuring their keys do not prevent garbage collection of otherwise unreferenced heap objects.

**One-line mental model:**
> *Maps are prototype-free, insertion-ordered key-value dictionaries; Sets are unique lists; Weak counterparts prevent memory leaks.*

---

## ② DRILL DOWN

### 🔹 A. Map vs. Plain Object

Plain objects have been used as maps for decades, but they have major limitations:

```javascript
// 1. Keys can be ANY type in a Map:
const map = new Map();
const keyObj = { id: 1 };
const keyFunc = () => {};

map.set(keyObj, "Object Value");
map.set(keyFunc, "Function Value");

console.log(map.get(keyObj));  // "Object Value"
console.log(map.get(keyFunc)); // "Function Value"

// 2. Objects coerce all keys to strings:
const obj = {};
obj[keyObj] = "Object Value"; // obj["[object Object]"] = ...

// 3. Getting Size:
console.log(map.size); // 2
console.log(Object.keys(obj).length); // 1 (Slow O(N) operation)

// 4. Prototype Pollution Safety:
const dangerousObj = JSON.parse('{"__proto__": {"polluted": true}}');
// Plain objects can be polluted; Maps are completely immune!
```

---

### 🔹 B. Set vs. Array

A `Set` is a collection of unique values, which makes it ideal for deduplication:

```javascript
const set = new Set([1, 2, 2, 3]);
console.log(set); // Set(3) { 1, 2, 3 } (Duplicates removed!)

// 1. O(1) existence checks (Unbelievably fast compared to Array.includes!)
console.log(set.has(2)); // true (Sub-linear time complexity)

// 2. Converting Array to Set and back (Instant Deduplication):
const numbers = [1, 2, 2, 3, 4, 4];
const unique = [...new Set(numbers)]; // [1, 2, 3, 4]
```

---

### 🔹 C. WeakMap and WeakSet (Preventing Memory Leaks)

A standard `Map` holds a **strong reference** to its keys. If a key is an object, that object will **never be garbage collected** as long as the Map is in memory, even if no other variable references it. A `WeakMap` resolves this by holding a **weak reference** to its keys.

```javascript
// 1. Map Memory Leak Scenario:
let user = { name: "Alice" };
const strongMap = new Map();
strongMap.set(user, "Metadata");

user = null; // Drop strong reference
// The object { name: "Alice" } is STILL stored on the heap because strongMap keeps it alive!

// 2. WeakMap Memory Solution:
let user2 = { name: "Bob" };
const weakMap = new WeakMap();
weakMap.set(user2, "Metadata");

user2 = null; // Drop strong reference
// The object { name: "Bob" } is now completely eligible for Garbage Collection!
// The engine will automatically erase the mapping key/value inside weakMap.
```

#### Strict WeakMap Rules:
1. Keys **must be Objects or Symbols** (primitives cannot be garbage collected and are rejected).
2. It is **non-iterable** (since GC runs asynchronously, the size of a WeakMap could fluctuate unpredictably, so the spec blocks iteration).
3. It has no `.size` property and no `.clear()` method.

---

### 🔹 D. Iterating Over Maps and Sets

Maps and Sets preserve their **insertion order** during iteration:

```javascript
const myMap = new Map([
    ["a", 1],
    ["b", 2]
]);

// 1. Iterating Entries:
for (const [key, val] of myMap.entries()) {
    console.log(key, val); // Logs: "a 1", "b 2"
}

// 2. Iterating Keys or Values:
for (const key of myMap.keys()) { console.log(key); }
for (const val of myMap.values()) { console.log(val); }
```

---

## ③ WHY IT EXISTS

1. **Primitive Coercion Bypass**: Objects coerce keys into strings. Prior to Maps, storing metadata on complex objects required injecting properties directly into them, polluting their namespace. Maps allow clean object-to-object association.
2. **$O(1)$ Complexity at Scale**: Arrays require $O(N)$ linear scans (`includes`, `indexOf`) to search. Sets and Maps use hash-tables to execute lookups in $O(1)$ sub-linear time, which is critical for large datasets.
3. **Automated Memory Cleanup**: In large single-page applications (SPAs), caching metadata or DOM node states using standard objects/maps is the number one cause of long-term memory leaks. WeakMaps automate memory management.

---

## ④ INTERNAL WORKING: ORDERED HASH TABLES

### V8's Deterministic Ordered Hash Table

Standard hash tables scatter elements across memory based on hash codes, completely losing the order of insertions.
To enforce the ECMAScript requirement that Maps and Sets must preserve insertion order, V8 uses a **Deterministic Ordered Hash Table** structure.

```
V8 ORDERED HASH TABLE LAYOUT:

  HASH BUCKETS ARRAY            DATA STORES ARRAY (Contiguous)
 ┌──────────────────┐          ┌──────────────────────────────────────┐
 │ Bucket 0 ────────┼─────────►│ Entry 0: Key: "a", Val: 1, Next: null│ ◄── First Inserted
 ├──────────────────┤          ├──────────────────────────────────────┤
 │ Bucket 1 ────────┼─┐        │ Entry 1: Key: "b", Val: 2, Next: 2   │
 ├──────────────────┤ │        ├──────────────────────────────────────┤
 │ Bucket 2 ────────┼─┼───────►│ Entry 2: Key: "c", Val: 3, Next: null│ ◄── Last Inserted
 └──────────────────┘ │        └──────────────────────────────────────┘
                      └───────► (Linked list collision chain resolves here)
```

1. **Double Arrays**: The structure contains two primary backing arrays: a **Hash Buckets Array** and a contiguous **Data Stores Array**.
2. **Sequential Inserts**: When a new key-value pair is added, V8 pushes it sequentially to the end of the **Data Stores Array**. This guarantees that iterating the Map is simply a linear sweep of this data array, ensuring perfect insertion order preservation!
3. **Bucket Indexing**: The **Hash Buckets Array** stores pointers to indices in the **Data Stores Array** to perform fast $O(1)$ hash lookups.

---

### WeakMaps and Ephemerons

WeakMaps are not implemented by having the Garbage Collector poll the Map. They use **Ephemeron Mappings**:
* An ephemeron is a key-value pair where the garbage collectability of the value is tied directly to the reachability of the key.
* If a key object `K` is only reachable via the WeakMap, both `K` and its associated value `V` are swept from memory simultaneously.
* This requires the V8 garbage collector (specifically the **Mark-Sweep** phase) to actively process WeakMap links as a specialized phase, clearing key-value slots during the sweep step.

---

## ⑤ ADVANTAGES

1. **Universal Keys**: Attach data directly to objects or functions without polluting their properties.
2. **Sub-linear Performance**: $O(1)$ lookup time complexity is maintained even as the collection size scales to millions of items.
3. **No Namespace Pollution**: Completely immune to prototype pollution security exploits.

---

## ⑥ DISADVANTAGES / TRAPS

1. **No Native JSON Serialization**: Maps and Sets cannot be converted to JSON using `JSON.stringify()` directly (returns `{}`), requiring custom serialization wrappers.
2. **GC Latency on WeakMaps**: You cannot know when or if an item inside a WeakMap has been garbage collected because GC runs asynchronously in the background.
3. **Object Hash Collisions**: Using objects with volatile properties as Map keys can occasionally cause JIT performance hits if the engine's internal hash generator triggers collisions.

---

## ⑦ PRODUCTION USAGE

### 🛠️ High-Performance DOM Metadata Cache (WeakMap)

In modern frontend libraries, WeakMaps are used to attach metadata or event states to DOM elements without causing memory leaks when those DOM elements are removed from the page:

```javascript
// A metadata store for DOM elements
const domMetadataStore = new WeakMap();

function trackElementInteraction(element) {
    let metadata = domMetadataStore.get(element);
    
    if (!metadata) {
        metadata = { clickCount: 0, lastAccessed: Date.now() };
        domMetadataStore.set(element, metadata);
    }
    
    metadata.clickCount++;
    metadata.lastAccessed = Date.now();
    console.log(`Element clicked ${metadata.clickCount} times.`);
}

// Usage:
const button = document.createElement("button");
trackElementInteraction(button); // Logs: "Element clicked 1 times."

// When button is deleted and reference dropped:
button.remove();
// The button and its associated metadata in the WeakMap are automatically cleared from heap memory!
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the main differences between a Map and a plain object?**  
**A:** Maps accept keys of any data type (objects, functions, etc.), guarantee insertion order during iteration, have a built-in `.size` property, and are safe from prototype pollution. Plain objects coerce keys to strings, do not guarantee insertion order, require $O(N)$ calculations to count keys, and can be polluted.

**Q2: What is the primary feature of a Set?**  
**A:** A Set stores only unique values, automatically filtering out duplicate entries.

**Q3: How do you check if an item exists inside a Set, and what is its time complexity?**  
**A:** Use `set.has(item)`. Its time complexity is sub-linear $O(1)$ on average.

**Q4: Can you use primitive values (like numbers or strings) as keys in a WeakMap?**  
**A:** No. WeakMaps require keys to be Objects or Symbols because primitives cannot be garbage collected.

**Q5: How do you clear all elements from a Map?**  
**A:** Use `map.clear()`.

---

### 🟡 Intermediate (5)

**Q6: Explain why WeakMaps and WeakSets are not iterable. What is the design reason behind this?**  
**A:** Garbage Collection (GC) in JavaScript engines is non-deterministic and runs asynchronously in the background. If iteration were allowed on a WeakMap, the size and keys of the Map would change randomly depending on whether the GC had executed, leading to unpredictable, non-deterministic program behavior.

**Q7: Write a one-liner utility to remove duplicates from an array, and explain its efficiency.**  
**A:** 
```javascript
const unique = [...new Set(array)];
```
*Efficiency:* This creates a Set from the array ($O(N)$ time) which filters out duplicates, and spreads it back to a new array ($O(N)$ time). Total time complexity is $O(N)$, which is highly efficient compared to nested array $O(N^2)$ solutions using `filter` or `indexOf`.

**Q8: Explain the garbage collection behavior of a standard `Map` vs. a `WeakMap` when a key object's reference is set to `null` outside the collection.**  
**A:** 
* In a standard `Map`, the map retains a **strong reference** to the key object. Setting the external reference to `null` does not allow GC to free the object; it remains trapped in memory.
* In a `WeakMap`, the map holds a **weak reference** to the key object. Setting the external reference to `null` leaves the object with zero strong references, making it instantly eligible for GC, which will reclaim its heap memory and clear its entry in the WeakMap.

**Q9: How does the equality comparison work for Map keys? Does it use `==` or `===`?**  
**A:** It uses the **SameValueZero** algorithm. It is highly similar to strict equality `===`, but with two key differences:
1. `NaN` is considered equal to `NaN` (so you can use `NaN` as a key in a Map).
2. `-0` is considered equal to `+0` (standard `SameValueZero` treats them identically).

**Q10: Why would you use a Map instead of an Object for a high-performance, frequently updated lookup cache?**  
**A:** 
1. **Dynamic Size**: Maps have a `.size` property ($O(1)$ lookup), whereas objects require counting keys ($O(N)$ time).
2. **Frequency Optimization**: Maps are optimized for frequent addition and removal of key-value pairs.
3. **No Keys Pollution**: Objects inherit prototype keys (like `toString` or `valueOf`), which can cause collision bugs. Maps only search keys explicitly inserted.

---

### 🔴 Advanced (5)

**Q11: Explain V8's "Deterministic Ordered Hash Table" and how it preserves insertion order while maintaining $O(1)$ lookups.**  
**A:** V8 maps elements using two backing store arrays:
1. **Data Stores Array**: A contiguous array where key-value pairs are pushed sequentially upon insertion. Iterating the Map is a simple linear crawl of this array, guaranteeing perfect insertion order.
2. **Hash Buckets Array**: An array storing hash offset points. When looking up a key, V8 hashes the key, finds the bucket index, reads the pointer index pointing to the Data Stores Array, and fetches the value. This dual array architecture maintains $O(1)$ speeds without scattering sequence order.

**Q12: Detail the concept of "Ephemerons" and how V8's Mark-Sweep Garbage Collector processes WeakMap entries.**  
**A:** An ephemeron is a key-value mapping where the value is reachable only if the key is reachable. During the **Mark-Sweep** GC phase, the garbage collector maintains a list of discovered WeakMaps.
* It first marks all strongly reachable objects.
* If a key `K` in a WeakMap is marked as reachable, the GC marks its associated value `V` as reachable.
* If, at the end of the marking phase, key `K` is **unmarked** (not reachable elsewhere), both `K` and `V` are swept from memory, and their slot inside the WeakMap backing store is cleared to `undefined`.

**Q13: Why does JSON serialization (`JSON.stringify(map)`) return `{}` for Map objects, and how do you implement a custom serialization replacer?**  
**A:** `JSON.stringify` only enumerates an object's enumerable, own string-keyed properties. Maps store their entries in internal engine backing slots, which are invisible to standard property enumeration.
*Custom Replacer:*
```javascript
function mapReplacer(key, value) {
    if (value instanceof Map) {
        return { __type: "Map", data: Array.from(value.entries()) };
    }
    return value;
}
JSON.stringify(myMap, mapReplacer);
```

**Q14: Explain the differences in memory allocation and performance when adding 1,000,000 items to a Map vs. dynamically writing 1,000,000 properties to a plain Object in V8.**  
**A:** 
* **Plain Object**: Writing 1,000,000 dynamic keys triggers massive Hidden Class Map transitions. V8 quickly exhausts in-object slots, shifts to out-of-object arrays, and eventually degrades the object to **Slow Dictionary Mode**. The overhead of creating thousands of transition shapes and constantly resizing heap arrays creates massive memory bloat and GC stress.
* **Map**: V8 instantiates the Deterministic Ordered Hash Table backing store immediately. It grows the Data Stores Array exponentially without creating any Hidden Class transitions or shapes. The Map handles 1,000,000 items with vastly lower memory footprints and stable, constant lookup times.

**Q15: How can you implement a private state container for a class using a WeakMap, and what are its structural advantages over using traditional closures?**  
**A:** 
```javascript
const privateState = new WeakMap();

class SecureCounter {
    constructor() {
        privateState.set(this, { count: 0 });
    }
    increment() {
        const state = privateState.get(this);
        state.count++;
    }
    get count() { return privateState.get(this).count; }
}
```
*Structural Advantages:* Unlike traditional closures (which duplicate method allocations inside the constructor for every instance), the WeakMap pattern keeps methods on the class prototype (`SecureCounter.prototype`), saving massive memory. The private state remains 100% secure from external access, and when the instance is garbage collected, the state entry in the WeakMap is automatically swept, preventing memory leaks.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact sequence of V8 engine operations, heap reallocations, and hash collisions that occur when a poorly-designed key object constantly changes its internal properties after being inserted as a key in a Map.**  
**A:** 
1. **Hashing Phase**: When `map.set(obj, val)` is executed, V8 generates a unique hash code for the object. This hash is stored in the object's header (as a private symbol) so that the hash remains constant even if the object's properties change.
2. **Object Mutation**: The developer mutates the object properties (e.g., `obj.name = "changed"`).
3. **Lookup Phase**: When calling `map.get(obj)`, V8 reads the hash code from the object's header.
4. **Collision and Retrieval**: Even though the object properties changed, its header hash remains identical. V8 maps the hash to the correct bucket in the Hash Buckets Array and locates the Data Stores entry.
5. **JIT Impact**: While lookups still succeed, mutating the object constantly changes its Hidden Class shape. When JIT optimized code attempts to resolve `map.get(obj)`, the polymorphic nature of the passing key's shape invalidates the Inline Caches (ICs) inside the lookup helper, forcing V8 to drop out of optimized machine-code and run the lookup via slow interpreter calls.

**Q17: Explain the step-by-step memory lifecycle and GC tracing phases when a WeakMap's key object becomes unreachable in a Web worker, detailing how Ephemeron markers prevent infinite GC loops during mark phases.**  
**A:** 
1. **Reference Drop**: The main thread drops the final strong reference to object `K`.
2. **GC Mark Phase Begins**: The GC sweeps the heap from roots (global, stack frames).
3. **Ephemeron Registration**: The GC encounters a `WeakMap` object. It does not mark the keys as reachable. Instead, it registers the WeakMap in an internal **Ephemeron Queue**.
4. **Iterative Marking**: The GC continues marking other objects.
5. **Ephemeron Verification**: The GC sweeps the Ephemeron Queue:
   - If a key `K` was marked as reachable from a strong root, its value `V` is marked as reachable.
   - If key `K` is not marked, V8 leaves the pair unmarked.
6. **Infinite Loop Prevention**: If key `K` points to value `V`, and `V` contains a strong reference to another key `K2`, standard transitive marking could loop. V8 prevents this by running an **incremental, fixpoint iteration algorithm**. It continues sweeping the queue until a pass occurs where no new keys transition from unmarked to marked.
7. **Sweep Phase**: Once stable, all unmarked keys (including `K`) and their associated values `V` are swept from memory, and the WeakMap backing store elements are cleared, freeing heap sectors cleanly.
