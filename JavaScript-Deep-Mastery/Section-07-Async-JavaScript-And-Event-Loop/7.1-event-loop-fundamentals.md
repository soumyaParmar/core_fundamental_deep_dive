# 📌 7.1 — Event Loop Fundamentals
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** JavaScript is single-threaded, meaning it can only execute one line of code at a time on its main thread. To handle time-consuming operations (like fetching data, loading files, or setting timers) without freezing the entire webpage, JavaScript delegates these tasks to the browser or system background. When a background task finishes, its result is sent to a queue. The **Event Loop** is a continuously running monitor that watches the main call stack and these queues. The moment the stack becomes empty, it pushes the waiting background results back onto the stack to run.

**Technical Definition:** The Event Loop is a coordination mechanism built into JavaScript runtimes (browsers or Node.js) that manages the execution of multiple code chunks, executing them sequentially via a single Call Stack. It operates by coordinating the **Call Stack**, the **Task Queue (Macrotasks)**, and the **Microtask Queue**. The runtime guarantees that all microtasks are drained to completion before the next macrotask is executed or rendering frames are drawn.

**One-line mental model:**
> *The Event Loop is a high-speed monitor: when the Call Stack is empty, it drains all microtasks, updates the UI, runs one macrotask, and repeats.*

## ② DRILL DOWN

### 🔹 A. JavaScript's Concurrency Model

JavaScript is single-threaded because it has only **one Call Stack**. However, the runtime environment (browser or Node.js) provides extra APIs and threads in the background, enabling asynchronous concurrency.

```
┌───────────────────────────────────────────────┐
│              BROWSER RUNTIME (Or Node)       │
│                                               │
│  ┌──────────────────┐   ┌──────────────────┐  │
│  │   JS ENGINE      │   │   WEB APIs       │  │
│  │  ┌────────────┐  │   │  • DOM/Events    │  │
│  │  │ Call Stack │  │   │  • Fetch/XHR     │  │
│  │  └────────────┘  │   │  • setTimeout    │  │
│  │  ┌────────────┐  │   └────────┬─────────┘  │
│  │  │ Heap       │  │            │            │
│  │  └────────────┘  │            │ (Callback  │
│  └────────┬─────────┘            │  delegated)│
│           ▲                      ▼            │
│           │             ┌──────────────────┐  │
│      EVENT LOOP ────────┤ QUEUES           │  │
│   (Checks if Stack      │ • Microtasks     │  │
│    is empty, pushes)    │ • Macrotasks     │  │
│                         └──────────────────┘  │
└───────────────────────────────────────────────┘
```

---

### 🔹 B. Macrotasks (Task Queue) vs. Microtask Queue

When an asynchronous callback is ready to run, the runtime pushes it into one of two primary queues. These queues have strictly different execution priorities.

#### 1. The Microtask Queue (Highest Priority)
Microtasks are executed **immediately after the current execution context finishes**, before any rendering passes or other macrotasks are processed. The queue is fully drained to completion in one go.
* **Sources**:
  * `Promise` resolution/rejection handlers (`.then()`, `.catch()`, `.finally()`)
  * `queueMicrotask()`
  * `MutationObserver` (Browser-only DOM tracking)
  * `process.nextTick()` (Node.js specific — technically a specialized "pre-microtask" queue)

#### 2. The Macrotask Queue (Task Queue - Lower Priority)
Macrotasks are executed one-by-one. The Event Loop pulls **only one macrotask** from the queue, executes it, drains any newly generated microtasks, updates the UI rendering, and then retrieves the next macrotask.
* **Sources**:
  * `setTimeout()`, `setInterval()`
  * `setImmediate()` (Node.js only)
  * User input events (clicks, typing)
  * Network I/O responses (`fetch` complete events)
  * MessagePort, `postMessage()`, `requestAnimationFrame` (specialized task)

---

### 🔹 C. The Event Loop Execution Cycle

The Event Loop executes a highly strict, deterministic cycle:

```
                  ┌────────────────────────┐
                  │ 1. Execute Call Stack  │
                  └───────────┬────────────┘
                              │ (Drained to empty)
                              ▼
                  ┌────────────────────────┐
                  │ 2. Drain ALL           │ ◄─── (Loops until Microtask Queue
                  │    Microtasks          │      is completely empty!)
                  └───────────┬────────────┘
                              │
                              ▼
                  ┌────────────────────────┐
                  │ 3. UI Render Phase?    │ ───► (Runs at ~60-120fps; checks
                  │    (Browsers only)     │      if repaint is needed)
                  └───────────┬────────────┘
                              │
                              ▼
                  ┌────────────────────────┐
                  │ 4. Run ONE Macrotask   │
                  └───────────┬────────────┘
                              │
                              └───────────────── Loop repeats
```

---

### 🔹 D. Chronological Execution Trace

Consider this classic interview snippet:

```javascript
console.log("1. Start");

setTimeout(() => {
    console.log("2. Timeout");
}, 0);

Promise.resolve().then(() => {
    console.log("3. Promise 1");
}).then(() => {
    console.log("4. Promise 2");
});

queueMicrotask(() => {
    console.log("5. queueMicrotask");
});

console.log("6. End");
```

#### Step-by-Step Execution:
1. **Call Stack Exec**:
   * Logs `"1. Start"`.
   * Encounter `setTimeout`. Background timer registers. Once done (instantly), its callback is placed in the **Macrotask Queue**.
   * Encounter `Promise.resolve()`. The first `.then()` callback is placed in the **Microtask Queue**.
   * Encounter `queueMicrotask`. Its callback is placed in the **Microtask Queue**.
   * Logs `"6. End"`.
   * Current execution context finishes; **Call Stack is empty**.
2. **Drain Microtask Queue**:
   * Pulls first microtask: Logs `"3. Promise 1"`. This returns a resolved promise, which appends the second `.then()` callback (`Promise 2`) to the end of the Microtask Queue!
   * Pulls next microtask: Logs `"5. queueMicrotask"`.
   * Pulls newly appended microtask: Logs `"4. Promise 2"`.
   * **Microtask Queue is now completely empty**.
3. **Rendering Phase**: Checks if browser window repainting is needed.
4. **Macrotask Phase**:
   * Pulls the single pending macrotask: Logs `"2. Timeout"`.
5. **Output**:
   ```
   1. Start
   6. End
   3. Promise 1
   5. queueMicrotask
   4. Promise 2
   2. Timeout
   ```

## ③ WHY IT EXISTS

1. **Non-Blocking I/O**: Operating systems perform I/O (like network lookups) slowly. If JavaScript blocked threads waiting for these requests, the browser would freeze. The Event Loop allows delegating I/O to background system kernels, keeping the UI perfectly responsive.
2. **Concurrency Without Multi-threading Complexity**: True multi-threaded programming requires dealing with complex race conditions, deadlocks, and shared-memory state synchronization. The single-threaded Event Loop solves concurrency cleanly, avoiding all thread locks.
3. **High Network Throughput**: Runtimes like Node.js can handle tens of thousands of concurrent database connections on a single CPU thread, because thread resource limits are bypassed via asynchronous event dispatching.

## ④ INTERNAL WORKING: BROWSER SPEC VS. LIBUV

### The Browser Specification (HTML5)

In modern browsers, the Event Loop is governed by the **HTML5 Specification**:
* **Multiple Task Sources**: Browsers manage multiple task queues (e.g., a "User Interaction Queue" for clicks, and a "Timer Queue" for `setTimeout`). The spec allows the browser to prioritize user interactions over timers to keep the page feeling responsive.
* **Microtask Checkpoint**: The engine executes a "Perform a microtask checkpoint" whenever the Call Stack becomes empty. If a microtask itself schedules more microtasks, the loop will continue executing them indefinitely, blocking the macrotask queue and freezing the page.

---

### Node.js: The Libuv Event Loop Phases

While browsers use the HTML5 spec, Node.js uses **Libuv** (a C++ library) to manage its Event Loop. The Libuv loop contains **six distinct execution phases** that run sequentially in a loop:

```
 ┌────────────────────────────────────────────────────────┐
 │                      LIBUV EVENT LOOP                  │
 ├────────────────────────────────────────────────────────┤
 │   ┌───────────────────────────────────────────────┐    │
 └──►│ 1. TIMERS (setTimeout, setInterval callbacks) │    │
     └──────────────────────┬────────────────────────┘    │
                            ▼                             │
     ┌───────────────────────────────────────────────┐    │
     │ 2. PENDING CALLBACKS (Deferred TCP errors)    │    │
     └──────────────────────┬────────────────────────┘    │
                            ▼                             │
     ┌───────────────────────────────────────────────┐    │
     │ 3. IDLE, PREPARE (Internal engine operations) │    │
     └──────────────────────┬────────────────────────┘    │
                            ▼                             │
     ┌───────────────────────────────────────────────┐    │
     │ 4. POLL (Retrieve I/O events, execute files)  │    │
     └──────────────────────┬────────────────────────┘    │
                            ▼                             │
     ┌───────────────────────────────────────────────┐    │
     │ 5. CHECK (setImmediate callbacks execute)      │    │
     └──────────────────────┬────────────────────────┘    │
                            ▼                             │
     ┌───────────────────────────────────────────────┐    │
     │ 6. CLOSE CALLBACKS (socket.on('close'))       │────┘
     └───────────────────────────────────────────────┘
```

* **Microtask Execution in Node**: Node.js drains `process.nextTick` and `Promise` microtasks **between every single phase** of the Libuv event loop, prioritizing nextTick before standard promise microtasks.

## ⑤ ADVANTAGES

1. **Perfect UI Responsiveness**: Delegation keeps the main thread free, preventing webpage freeze during long API requests.
2. **Simplified State Management**: A single thread means zero race conditions on memory variables.
3. **Lightweight Concurrency**: Bypasses the heavy CPU and RAM memory overhead of spinning up system-level OS threads.

---

## ⑥ DISADVANTAGES / TRAPS

1. **CPU Blockage (Starvation)**: Running a heavy, synchronous calculation (e.g., sorting 1,000,000 items) blocks the Call Stack, preventing the Event Loop from executing microtasks, timers, or paint repaints, freezing the browser.
2. **Microtask Infinite Loop (Starvation of Macrotasks)**: Scheduling microtasks recursively inside a microtask callback will block the macrotask queue forever:
   ```javascript
   function starve() {
       Promise.resolve().then(starve); // Recursive microtask loop
   }
   // The page will freeze instantly! Timer and click handlers will never run.
   ```
3. **Unpredictable Timer Accuracy**: Timers are not high-precision guarantees. They only define the *minimum* delay before the callback is queued. If the stack or microtask queue is busy, the timer will execute much later than specified.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Breaking Up CPU-Intensive Tasks

If you must run a heavy, long-running calculation on the main thread, do not run it synchronously. Use recursive asynchronous slicing (with `setTimeout` or `setImmediate`) to yield control back to the Event Loop periodically, keeping the UI responsive:

```javascript
// BAD: Synchronous loop blocks the main thread for seconds (UI freezes)
function processHugeDataSync(data) {
    const results = [];
    for (const item of data) {
        results.push(heavyCalculate(item));
    }
    return results;
}

// GOOD: Process in batches, yielding control to the Event Loop periodically
function processHugeDataAsync(data, index = 0, results = []) {
    const batchSize = 1000;
    const end = Math.min(index + batchSize, data.length);

    // 1. Process a small chunk synchronously
    for (let i = index; i < end; i++) {
        results.push(heavyCalculate(data[i]));
    }

    // 2. If elements remain, schedule the next batch as a MACROTASK
    if (end < data.length) {
        setTimeout(() => {
            processHugeDataAsync(data, end, results);
        }, 0); // Yields stack control back to Event Loop so clicks/renders can run!
    } else {
        console.log("Batch processing complete!", results);
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: Is JavaScript multi-threaded or single-threaded?**  
**A:** JavaScript is single-threaded; it has only one Call Stack and executes one instruction at a time on its main thread. However, the runtime host environment is multi-threaded, executing timers or network queries in the background.

**Q2: What is the main difference between Macrotasks and Microtasks?**  
**A:** Microtasks (Promises, `queueMicrotask`) have higher priority and are fully drained to completion as soon as the Call Stack becomes empty. Macrotasks (timers, UI events) are executed one-by-one; the loop runs only one macrotask before checking for microtasks and repainting.

**Q3: What does a `setTimeout(fn, 0)` call do?**  
**A:** It registers a timer with a 0ms delay. When the delay expires, the callback is queued at the end of the **Macrotask Queue**, executing only after the current stack and all pending microtasks are drained.

**Q4: Name three sources of Microtasks.**  
**A:** `Promise.then/catch/finally` handlers, `queueMicrotask()`, and `MutationObserver` callbacks.

**Q5: What happens to the browser UI if the Call Stack is running a heavy, infinite synchronous loop?**  
**A:** The UI freezes completely. The browser cannot repaint, process clicks, or run timers because the Call Stack is never emptied, preventing the Event Loop from cycling.

---

### 🟡 Intermediate (5)

**Q6: Explain the "Microtask Starvation" trap and write a code snippet that triggers it.**  
**A:** Microtask Starvation occurs when microtasks are recursively added to the queue during a microtask checkpoint. Since the Event Loop does not stop until the Microtask Queue is completely empty, it is trapped in an infinite loop, starving macrotasks and repaints.
```javascript
const starve = () => Promise.resolve().then(starve);
starve(); // Freezes page instantly
```

**Q7: Why are timers (like `setTimeout`) not guaranteed to execute at their exact specified millisecond?**  
**A:** A timer only guarantees the *minimum* time before its callback is pushed into the Macrotask Queue. If the main Call Stack is currently executing a long function, or the Microtask Queue is crowded with callbacks, the timer callback will wait in the queue until the engine is free to run it.

**Q8: Explain the difference between `setTimeout(fn, 0)` in a browser and `setImmediate(fn)` in Node.js.**  
**A:** 
* `setTimeout(fn, 0)` queues a callback in the Timer phase.
* `setImmediate(fn)` is Node.js specific and is designed to execute immediately after the I/O polling phase finishes (in the Check phase). Under Libuv architecture, `setImmediate` is typically processed faster than a `setTimeout(fn, 0)` when scheduled inside I/O callbacks.

**Q9: How does `requestAnimationFrame` (rAF) fit into the browser Event Loop cycle?**  
**A:** `requestAnimationFrame` callbacks run during the **Rendering Phase** of the browser event loop cycle. They execute just before the browser calculates CSS styles, layouts, and performs physical screen repainting, typically running at 60Hz or matching the monitor's refresh rate.

**Q10: Explain the chronological order of execution for `process.nextTick` vs. standard Promise microtasks in Node.js.**  
**A:** In Node.js, `process.nextTick` callbacks are stored in a specialized queue separate from the Microtask Queue. When the Call Stack empties, Node.js **fully drains the nextTick queue first**, and only then drains the Promise microtask queue. Thus, `process.nextTick` always executes before standard Promise microtasks.

### 🔴 Advanced (5)

**Q11: Walk through the V8 engine and system-level thread activities that occur when you execute a `fetch('https://api.com')` request in a browser.**  
**A:** 
1. **Call Stack**: The engine executes `fetch()`, allocates a Promise, and passes the network request to the Browser runtime's Network Web API.
2. **Web API**: The Browser spawns a background thread from its OS network thread pool to manage the TCP handshake and transmit HTTP data. The JS Call Stack continues executing.
3. **Execution Pause**: The main JS thread goes idle.
4. **I/O Complete**: When the server returns data, the network thread reads the buffer, closes the socket, and pushes a "network fetch resolved" task to the **Macrotask Queue**.
5. **Event Loop Action**: The Event Loop detects the Call Stack is empty, pulls the fetch callback macrotask, executes it, resolves the promise, and pushes the `.then()` callbacks into the **Microtask Queue** to run immediately.

**Q12: Explain the step-by-step phases of the Libuv Event Loop in Node.js, and describe where microtasks are drained relative to these phases.**  
**A:** The Libuv Event Loop runs through six phases:
1. **Timers**: Executes expired `setTimeout` and `setInterval` callbacks.
2. **Pending Callbacks**: Executes deferred I/O callbacks (like TCP system errors).
3. **Idle, Prepare**: Used internally by the engine.
4. **Poll**: Retrieves new I/O events, reads buffers, blocks if empty.
5. **Check**: Executes `setImmediate` callbacks.
6. **Close Callbacks**: Executes socket cleanups (e.g., `socket.on('close')`).
*Microtasks Drainage:* Unlike browsers, which run microtasks after the Call Stack empties, Node.js drains `process.nextTick` and `Promise` microtasks **between every single phase** of the Libuv loop.

**Q13: How can you write a custom wrapper for `MutationObserver` that leverages its microtask queuing properties to build a microtask deferral helper?**  
**A:** 
```javascript
const queueMicrotaskFallback = (callback) => {
    if (typeof queueMicrotask === 'function') {
        queueMicrotask(callback);
        return;
    }
    // MutationObserver fallback
    let count = 0;
    const textNode = document.createTextNode(String(count));
    const observer = new MutationObserver(() => {
        callback();
    });
    observer.observe(textNode, { characterData: true });
    
    // Trigger mutation synchronously, which queues the observer callback as a microtask
    textNode.data = String(++count);
};
```

**Q14: Explain the differences in event loop behavior when processing event listener callbacks during the capture/bubble phase vs. when events are dispatched programmatically via `.dispatchEvent()`.**  
**A:** 
* **User Triggered (Capture/Bubble)**: The browser triggers listeners sequentially. When a bubble listener completes, the Call Stack empties. The engine **drains the Microtask Queue immediately** before triggering the next listener in the bubble chain.
* **Programmatically Dispatched (`.dispatchEvent`)**: The JS call to `element.dispatchEvent()` is active on the Call Stack. The engine triggers all listeners **synchronously** within that call frame. The Call Stack is not empty between listeners; therefore, the Microtask Queue is **not** drained until the entire `dispatchEvent()` execution block finishes.

**Q15: Explain how the browser schedules repaints during a heavy DOM mutation loop, and why writing `document.body.appendChild` in a loop doesn't cause visible flashing.**  
**A:** Writing DOM modifications is a synchronous operation that modifies the in-memory DOM tree representation. However, the browser does not repaint the screen immediately.
Repaints are scheduled at standard intervals (~16.6ms for 60fps) inside the Event Loop's **Rendering Phase**.
The Event Loop executes the main stack and drains microtasks. Since the loop is busy running the synchronous mutations loop, the stack is never empty. The browser's rendering phase is completely blocked.
Only when the script completes, the Call Stack empties, the loop advances to the Rendering Phase, and the browser performs a single, batch calculate layout and paint, preventing visible flashing.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact physical CPU activity, stack frame states, and thread switches that occur when a Node.js process encounters thread-pool blocking during cryptography or file systems tasks (e.g., `crypto.pbkdf2Sync` vs. `crypto.pbkdf2`).**  
**A:** 
1. **Synchronous Call (`pbkdf2Sync`)**: The JS thread executes the stack frame. Since it is synchronous, the V8 engine runs the intensive hashing loop directly on the **main JS execution thread**. The CPU core runs at 100%, and the Call Stack is blocked. The Libuv event loop is completely frozen, and no incoming network sockets can be processed, causing severe thread starvation.
2. **Asynchronous Call (`pbkdf2`)**: 
   - The Call Stack executes `pbkdf2`, and delegates the task to Libuv.
   - Libuv reads its thread pool configuration (default: 4 threads).
   - It assigns the cryptography work to one of the **Worker Threads** in the pool.
   - **Thread Switch**: The OS switches execution context of the cryptography calculation to a separate CPU core matching the worker thread.
   - The main JS execution thread remains at 0% block, immediately returning to cycle the Event Loop.
   - Once the background worker completes, it issues an interrupt signal, and Libuv pushes a completion callback into the Check or Poll phase of the main thread Event Loop.

**Q17: Explain the exact mechanics and execution timings inside browser engines (like WebKit or Blink) when a microtask checkpoint takes place. How are exceptions caught inside a microtask handled, and does a microtask crash halt the rest of the microtask queue?**  
**A:** 
1. **Checkpoint Activation**: When the call stack reaches a depth of 0, the engine's execution coordinator initiates the "Perform a microtask checkpoint" operation.
2. **Queue Crawl**: The engine iterates through the Microtask Queue. Unlike the Macrotask Queue, which runs tasks in batches, the checkpoint loop continuously shifts and processes tasks.
3. **Execution**: The microtask's `JSFunction` bytecode is loaded into the interpreter and executed.
4. **Exception Handling**: If a microtask throws an unhandled exception (e.g., an unhandled promise rejection or a raw error inside `queueMicrotask`):
   - The engine catches the exception at the boundary of the microtask wrapper.
   - It fires the global `unhandledrejection` or `error` events asynchronously.
   - **No Queue Halt**: Crucially, the exception **does not halt** the execution of the remaining items in the queue. The engine catches the error, discards the crashed frame, shifts the next microtask in the queue, and continues processing until the queue is completely drained.
