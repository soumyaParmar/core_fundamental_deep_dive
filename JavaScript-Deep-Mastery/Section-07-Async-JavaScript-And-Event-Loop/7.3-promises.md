# 📌 7.3 — Promises
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** A Promise is an object that acts as a placeholder for a value that is not yet available but will be resolved in the future (like an I/O result or an API response). It is a "trust contract": once created, the Promise guarantees it will eventually transition into either a successful resolved state (with the data) or a failed rejected state (with the error). Once a Promise resolves, its state is locked forever and cannot be changed.

**Technical Definition:** A Promise is a state-machine object conforming to the **Promises/A+ Specification**. It exists in one of three mutually exclusive states: `pending`, `fulfilled`, or `rejected`. Promises resolve via an internal **Promise Resolution Procedure**, guaranteeing that all fulfillment or rejection reactions (`.then`, `.catch`) are executed strictly as asynchronous Microtasks once the Call Stack is empty.

**One-line mental model:**
> *A Promise is a secure state-machine container representing a future value, executing its handlers strictly via the Microtask Queue.*

---

## ② DRILL DOWN

### 🔹 A. The Promise State Machine

A Promise is an object with two primary internal slots: `[[PromiseState]]` and `[[PromiseResult]]`.

```
                  ┌─────────────────────────────────┐
                  │       PENDING (Initial)         │
                  └──────────────┬───┬──────────────┘
                                 │   │
                Resolve Callback │   │ Reject Callback
                                 ▼   ▼
        ┌─────────────────────────┐ ┌─────────────────────────┐
        │  FULFILLED (Resolved)   │ │   REJECTED (Failed)     │
        │  [[PromiseState]]:      │ │   [[PromiseState]]:     │
        │    "fulfilled"          │ │     "rejected"          │
        │  [[PromiseResult]]:     │ │   [[PromiseResult]]:    │
        │     data value          │ │      error object       │
        └─────────────────────────┘ └─────────────────────────┘
```

#### Immutability of Resolved State:
Once a Promise transitions from `pending` to `fulfilled` or `rejected`, **it is locked**. Any subsequent calls to `resolve()` or `reject()` are silently ignored:

```javascript
const p = new Promise((resolve, reject) => {
    resolve("First success"); // State locked to fulfilled
    resolve("Second success"); // Ignored!
    reject(new Error("Fail")); // Ignored!
});
```

---

### 🔹 B. Chaining and Promise Consumers

The `.then()`, `.catch()`, and `.finally()` methods do not attach handlers to the original promise; they **return a brand-new Promise**, enabling seamless, linear chaining:

```javascript
// A linear async chain (No nesting!)
fetchUser(1)
    .then(user => fetchUserOrders(user.id))       // Returns new Promise
    .then(orders => fetchOrderDetails(orders[0])) // Returns new Promise
    .then(details => {
        console.log("Order loaded:", details);
    })
    .catch(err => {
        console.error("Failed somewhere in the chain:", err);
    });
```

#### Resolution Chaining Rules:
* If a handler returns a **primitive or plain object**, the new promise resolves with that value.
* If a handler returns another **Promise**, the new promise adopts the state and result of that returned promise (dynamic nesting resolution).
* If a handler throws an error, the new promise is automatically **rejected** with that error.

---

### 🔹 C. Promise Combinators (Static Methods)

JavaScript provides four static combinator methods to coordinate concurrent promises:

| Method | Resolution Condition | Rejection Condition | Primary Use Case |
| :--- | :--- | :--- | :--- |
| **`Promise.all([...])`** | Resolves when **all** promises fulfill. Returns array of results. | Rejects immediately when **any** single promise rejects. | Parallel execution where all parts are required (All-or-nothing). |
| **`Promise.allSettled([...])`** | Resolves when **all** promises settle (either fulfill or reject). Never rejects. | Never rejects. | Auditing concurrent APIs where you need all results regardless of failure. |
| **`Promise.race([...])`** | Settles (either fulfills or rejects) as soon as the **first** promise settles. | Same as first settler. | Implementing request timeout caps. |
| **`Promise.any([...])`** | Resolves when the **first** promise fulfills. | Rejects only when **all** promises reject (returns AggregateError). | Querying multiple redundant mirror APIs; first success wins. |

```javascript
// Promise.all example
Promise.all([fetch('/users'), fetch('/posts')])
    .then(([res1, res2]) => { /* Both complete successfully */ })
    .catch(err => { /* Triggers if either fails */ });
```

---

### 🔹 D. Error Propagation and Uncaught Rejections

Errors propagate down the promise chain automatically until they encounter a `.catch()` block. If no handler catches the rejection, it triggers an **Uncaught Promise Rejection** event:

```javascript
// Node.js processes crash on uncaught rejections by default:
process.on('unhandledRejection', (reason, promise) => {
    console.error('Unhandled Rejection at:', promise, 'reason:', reason);
    // Perform cleanup and exit safely
});
```

---

## ③ WHY IT EXISTS

1. **Escaping Callback Hell**: Promises transition nested, indented pyramids into clean, linear chains (`.then().then()`), restoring top-to-bottom readability.
2. **Solving Inversion of Control**: With callbacks, you surrender control to third-party code. With Promises, you control when handlers run. The Promise guarantees it will only resolve once, lock its state, and trigger standard microtasks, eliminating dual-calling hazards.
3. **Unified Future Value (Monad)**: Promises treat asynchronous data as a tangible, first-class JavaScript value (an object wrapper) that can be passed around, returned, and composed before the data is physically loaded.

## ④ INTERNAL WORKING: V8 PROMISE STRUCTURE

### V8 Heap Layout of a Promise

In V8, a Promise is represented by a `JSPromise` heap object containing:

```
┌────────────────────────────────────────────────────────┐
│                   JSPromise Heap Object                │
├────────────────────────────────────────────────────────┤
│ • Map (Hidden Class Map pointer)                       │
│ • State: 0 (Pending), 1 (Fulfilled), 2 (Rejected)      │
│ • Result (Contains resolve value or reject error)      │
│ • Reactions List (Linked list of fulfillment callbacks)├──► [ PromiseReaction ] ──► [ PromiseReaction ]
└────────────────────────────────────────────────────────┘
```

1. **Reactions List**: A linked list of callbacks registered via `.then()` or `.catch()`.
2. **Fulfillment Phase**:
   - When `resolve(data)` is executed, the engine updates the `State` to `1` and writes `data` to `Result`.
   - V8 walks the **Reactions List**, wraps each callback in a **PromiseReactionJob**, and pushes it directly onto the **Microtask Queue**.
   - Because it goes to the Microtask Queue, these handlers run immediately when the current Call Stack empties, bypassing the slower Macrotask timer queues entirely.

---

## ⑤ ADVANTAGES

1. **Immutable States**: Guarantees values are secure; once resolved, data cannot be altered or double-fired.
2. **Microtask Priority**: Execution is prioritized, bypassing the latency of timer-based macro tasks.
3. **Linear Composition**: Built-in prototypes allow easy, readable pipeline structures and concurrent combinator operations.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Allocation Overhead**: Every promise and `.then()` call allocates a wrapper object on the heap, increasing garbage collection load compared to raw callbacks.
2. **Un-cancelable**: Once created, a standard Promise cannot be canceled. It will run its task to completion even if the user navigates away from the page.
3. **Silent Failure**: Forgetting to attach a `.catch()` block swallows errors, hiding critical system bugs in production.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Writing a Custom `Promise.all` Combinator

Implementing `Promise.all` is a classic senior engineering test, showing deep understanding of promise aggregation, state tracking, and asynchronous concurrency:

```javascript
function customPromiseAll(promises) {
    return new Promise((resolve, reject) => {
        // Handle non-arrays or empty arrays
        if (!Array.isArray(promises)) {
            return reject(new TypeError("Argument must be an array"));
        }
        if (promises.length === 0) {
            return resolve([]);
        }

        const results = new Array(promises.length);
        let completedCount = 0;

        promises.forEach((promise, index) => {
            // Coerce non-promise values to resolved promises
            Promise.resolve(promise)
                .then(val => {
                    results[index] = val; // Store at exact index
                    completedCount++;

                    // Once all fulfill, resolve the final promise
                    if (completedCount === promises.length) {
                        resolve(results);
                    }
                })
                .catch(err => {
                    // Reject immediately if ANY single promise fails
                    reject(err);
                });
        });
    });
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the three states of a Promise?**  
**A:** `pending` (initial state), `fulfilled` (operation completed successfully), and `rejected` (operation failed).

**Q2: What is the primary difference between how Promises and callbacks execute?**  
**A:** Promises execute their resolution handlers strictly as high-priority **Microtasks**. Asynchronous callbacks are typically executed as lower-priority **Macrotasks** (e.g., through timers or I/O events).

**Q3: Can a resolved Promise transition back to the pending state or change its value?**  
**A:** No. Promise state transitions are one-way and immutable. Once resolved, the state is locked forever.

**Q4: What does `.then()` return?**  
**A:** It returns a brand-new Promise. This allows chaining multiple asynchronous operations.

**Q5: What happens if an error is thrown inside a Promise constructor block?**  
**A:** The engine catches the exception and automatically transitions the Promise to the `rejected` state with the thrown error as its result.

---

### 🟡 Intermediate (5)

**Q6: Explain the difference between `Promise.all()` and `Promise.allSettled()`.**  
**A:** 
* `Promise.all()` is all-or-nothing: it resolves only when *all* input promises fulfill, but rejects immediately if *any* single promise fails.
* `Promise.allSettled()` never rejects. It waits for *all* input promises to either fulfill or reject, returning an array of objects describing the outcome and result of each promise.

**Q7: Explain the outcome of this code: `Promise.resolve(1).then(x => { throw x; }).catch(err => err * 2).then(y => console.log(y))`**  
**A:** 
1. `Promise.resolve(1)` creates a fulfilled promise with value `1`.
2. The `.then()` throws `1`, making the chain promise transition to the `rejected` state with value `1`.
3. The `.catch()` catches `1`, evaluates `1 * 2`, and returns `2` (since the catch callback returns normally, it resolves the promise with `2`).
4. The final `.then()` receives `2` and logs `"2"`.

**Q8: What is an uncaught promise rejection, and why is it dangerous in production?**  
**A:** An uncaught promise rejection occurs when a promise is rejected but has no `.catch()` block or rejection handler attached in its chain. It is dangerous because it can leave applications in an unhandled, corrupted state, and in modern Node.js environments, it crashes the entire process by default.

**Q9: Explain the difference between `Promise.race()` and `Promise.any()`.**  
**A:** 
* `Promise.race()` settles (resolves or rejects) as soon as the *first* input promise settles.
* `Promise.any()` resolves as soon as the *first* input promise *fulfills* (succeeds). It only rejects if *all* input promises reject.

**Q10: Why does `new Promise((res) => res(Promise.resolve(42)))` resolve to `42` instead of resolving to the nested Promise object?**  
**A:** By specification, the Promise constructor executes the **Promise Resolution Procedure**. If the value passed to `resolve()` is a promise or a thenable, the engine flattens it: it hooks into the nested promise's `.then()` and delays resolution until the inner promise settles, resolving the outer promise with the raw unwrapped value (`42`).

### 🔴 Advanced (5)

**Q11: Walk through the exact V8 microtask queuing sequence of a promise chain consisting of three nested `.then()` handlers.**  
**A:** 
1. **Fulfillment**: The initial promise resolves, changing its state to `fulfilled` and queuing the first `.then()` callback in the Microtask Queue.
2. **First Cycle**: The Event Loop drains the Microtask Queue, executes the first callback, and retrieves its return value. It uses this value to resolve the first chained promise, which immediately appends the second `.then()` callback to the end of the Microtask Queue.
3. **Second Cycle**: The Event Loop processes the second callback, resolving the second chained promise, which appends the third `.then()` callback to the Microtask Queue.
4. **Third Cycle**: The Event Loop executes the third callback, resolving the final promise.
*Crucial Detail:* Handlers are not queued all at once; each `.then()` callback is queued only when the preceding promise in the chain actually settles.

**Q12: Detail the heap layout of a V8 `JSPromise` object, explaining the role of "Reactions".**  
**A:** A V8 `JSPromise` contains a Map pointer, a State slot (0=pending, 1=fulfilled, 2=rejected), and a Result slot. Crucially, it has a **Reactions** pointer.
* When the promise is pending and `.then(cb)` is called, the cb reference is wrapped in a `PromiseReaction` heap structure.
* If multiple `.then()` calls are attached to the same promise, V8 links these reaction structures in a **singly-linked list** off the Reactions slot.
* When the promise is resolved, V8 walks this linked list, extracts the callbacks, creates Microtask objects, and pushes them onto the Microtask Queue, clearing the Reactions pointer to avoid memory leaks.

**Q13: Why is a Promise chain considered a Monad? Explain the conceptual map/flatMap mapping in JavaScript Promises.**  
**A:** In functional programming, a Monad is a structure that wraps a value in a context and allows chaining operations via a flatMap.
JavaScript Promises act as monads:
* **Wrap**: The Promise constructor wraps a raw value inside a state-machine context.
* **flatMap**: The `.then()` method acts as both `map` and `flatMap`. If the callback returns a raw value, `.then` maps it. If the callback returns a Promise (a wrapped value), `.then` automatically flattens (resolves) it, behaving as a monad's `flatMap` to keep the chain flat.

**Q14: Explain the performance and allocation trade-offs of using native Promises in a high-frequency event listener (e.g., tracking mouse movement) vs. using raw callbacks.**  
**A:** Using Promises in high-frequency listeners (firing 100+ times per second) is a major anti-pattern.
* **Heap Pressure**: Every Promise creation allocates a `JSPromise` object, `PromiseReaction` slots, and closure contexts on the heap. This rapidly fills the V8 Scavenger (New Space) memory.
* **GC Stress**: The constant allocation triggers frequent GC Scavenger cycles, causing visible micro-stutters in UI rendering.
* **Callback Efficiency**: Raw callbacks do not allocate objects per invocation; they reuse a single static function pointer, running significantly faster with zero GC footprint.

**Q15: Write a function `timeoutPromise(promise, ms)` that rejects with a "TimeoutError" if the input promise does not settle within `ms` milliseconds, using optimal combinator patterns.**  
**A:** 
```javascript
function timeoutPromise(promise, ms) {
    const timeout = new Promise((_, reject) => {
        const id = setTimeout(() => {
            clearTimeout(id);
            reject(new Error("TimeoutError: Operation timed out"));
        }, ms);
    });
    return Promise.race([promise, timeout]);
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact execution timings, queue states, and engine phases that occur when a microtask scheduled by a Promise resolution attempts to block screen repainting.**  
**A:** 
1. **Loop Cycle**: A promise resolves, queuing its `.then()` callback in the Microtask Queue.
2. **Stack Empty**: The Call Stack becomes empty.
3. **Checkpoint**: V8 enters a microtask checkpoint, running the promise callback.
4. **Recursive Blocking**: The callback resolves another promise, which appends a new microtask. The checkpoint loop continues executing.
5. **Rendering Block**: The browser's Event Loop specification dictates that the **Rendering Phase** (CSS layouts, paints) can only run *after* the microtask checkpoint finishes.
6. **Freeze**: Because the microtask queue is continually refilled, the checkpoint never exits. The Event Loop is stuck in step 2 (microtask drainage), and the CPU core runs at 100% capacity. The rendering phase is completely starved, freezing the UI.

**Q17: Explain the Promise Resolution Procedure step-by-step as outlined in the Promises/A+ specification, specifically focusing on how the engine prevents infinite recursive cycles when resolving a promise with a thenable that references itself.**  
**A:** 
1. **Self-Reference Check**: When evaluating `resolve(promise, x)`, the engine first checks if the promise `promise` and the resolved value `x` refer to the exact same object.
2. **TypeError Trigger**: If `promise === x`, the procedure immediately halts, rejecting the promise with a `TypeError` to prevent an infinite loop.
3. **Thenable Identification**: If `x` is an object or function, the engine attempts to read `x.then` inside a `try/catch` block.
4. **Accessor Guard**: If reading `x.then` throws an exception, the promise is rejected with that exception.
5. **Callback Binding**: If `then` is a callable function, the engine executes it with `x` as the `this` context, passing a `resolvePromise` and a `rejectPromise` callback.
6. **Recursion Resolution**: If `resolvePromise` is called with a nested thenable, the procedure runs recursively. To prevent infinite cycles with cyclic thenables, the engine uses a flag to ensure the first callback to execute locks the state, ignoring any subsequent calls.
