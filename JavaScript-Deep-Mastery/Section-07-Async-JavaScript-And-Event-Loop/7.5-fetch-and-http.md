# 📌 7.5 — Fetch & HTTP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** The Fetch API is a built-in browser tool used to send network requests (HTTP/HTTPS) to servers and retrieve data (like JSON or files). Unlike old browser tools, Fetch is fully Promise-based, allowing you to easily handle request success and failure. Crucially, Fetch streams data: when the server starts sending a large file, Fetch lets you access and process the data in chunks as it arrives, rather than forcing you to wait for the entire download to finish.

**Technical Definition:** The Fetch API is an ES6 promise-based native window specification that replaces the legacy `XMLHttpRequest`. It exposes three core primitives: `Request`, `Response`, and `Headers`. Fetch leverages **WHATWG Readable Streams** under the hood; the response body is a stream that locks upon reading, meaning its buffer can only be consumed once via methods like `.json()` or `.text()`. Fetch only rejects on physical network failures or CORS security blocks, resolving HTTP error codes (like 404 or 500) as successful resolutions.

**One-line mental model:**
> *Fetch is a native Promise-based HTTP client built on readable body streams, rejecting only on true network failures.*

---

## ② DRILL DOWN

### 🔹 A. Syntax and the HTTP Status Catch

The most important gotcha with `fetch()` is that **it does not reject on HTTP error statuses** (like 404 or 500). The promise only rejects if the request physically fails to reach the network (e.g., DNS error, offline, or CORS violation).

```javascript
// ❌ BAD: Expecting catch to intercept HTTP 500 Server Errors
fetch('https://api.com/users/999')
    .then(res => res.json())
    .catch(err => console.error("Caught error:", err)); // Will NOT run on 404/500!

// ✅ GOOD: Manually checking response.ok or response.status
fetch('https://api.com/users/999')
    .then(res => {
        if (!res.ok) { // res.ok is true if status is 200-299
            throw new Error(`HTTP Error Status: ${res.status}`);
        }
        return res.json();
    })
    .then(data => console.log(data))
    .catch(err => console.error("Caught error:", err)); // Intercepts both HTTP and network errors!
```

---

### 🔹 B. Body Reading Formats and the Single-Read Lock

When a server responds, the body is not loaded into memory as a completed string. It is a **ReadableStream**. When you call a body parsing method (like `.json()`, `.text()`, or `.blob()`), the stream is locked and read to completion.

```javascript
fetch('https://api.com/data')
    .then(async (res) => {
        const text = await res.text(); // Stream locked and read!
        
        // Attempting to read again:
        // const json = await res.json(); // TypeError: Failed to execute 'json' on 'Response': body stream already read!
    });
```

#### How to read a response multiple times:
If you need to inspect the response body multiple times (e.g., logging raw text and then parsing JSON), you must **clone** the response before reading:

```javascript
fetch('https://api.com/data')
    .then(async (res) => {
        const clonedRes = res.clone(); // Clones the stream
        const text = await res.text(); // Consumes first stream
        const json = await clonedRes.json(); // Consumes cloned stream safely!
    });
```

---

### 🔹 C. Aborting Requests via `AbortController`

Modern JavaScript supports canceling active network requests using `AbortController` and `AbortSignal`.

```javascript
const controller = new AbortController();
const { signal } = controller;

// 1. Pass the signal option to fetch
fetch('https://api.com/huge-file', { signal })
    .then(res => res.json())
    .catch(err => {
        if (err.name === 'AbortError') {
            console.log("Fetch request successfully canceled!");
        } else {
            console.error("Fetch failed:", err);
        }
    });

// 2. Cancel the request (e.g., if user clicks "Cancel" or navigates away)
controller.abort();
```

## ③ WHY IT EXISTS

1. **Escaping XMLHttpRequests (XHR)**: Legacy XHR was incredibly verbose, required complex state tracking (`readyState`, `onreadystatechange`), and did not support modern Promise structures out-of-the-box.
2. **Stream-based Architectures**: Loading large multi-megabyte payloads fully into memory before processing causes extreme browser lag. Fetch introduces WASWG Streams, allowing data to be processed chunk-by-chunk.
3. **Decoupled Primitives**: By exposing `Request` and `Response` as first-class object classes, developers can construct, clone, and pass around mock server configurations easily (which is crucial for Service Workers).

---

## ④ INTERNAL WORKING: STREAMS & BROWSER NETWORKING Threads

### ReadableStream and the Backpressure Mechanism

Under the hood, when you call `fetch()`, the browser networking layer initiates a connection. As packets arrive over TCP, they are written to a low-level browser C++ buffer.

```
 [ Server TCP Packets ] ──► [ Browser C++ Ring Buffer ] ──► [ WHATWG ReadableStream ] ──► [ JS Heap (json/text) ]
                                                                     │
                                                      (Flow control / Backpressure)
```

1. **Stream Locking**: The `body` property on the `Response` object is a `ReadableStream`. Reading it locks the stream to a reader, preventing other scripts from stealing data packets.
2. **Backpressure**: If the JavaScript thread is busy running heavy calculations, it pauses reading from the `ReadableStream`. The browser detects this and slows down reading from the low-level TCP socket, signaling the OS to use **TCP flow control (backpressure)** to tell the server to slow down transmission, preventing browser memory bloat.

---

### Browser Request Dispatching

The JavaScript thread does not perform network operations. Runtimes delegate all fetch operations to the browser's **Network Service**:
* When `fetch()` is executed, the engine serializes the `Request` object and sends it via IPC (Inter-Process Communication) to the browser's Network Service process.
* The Network Service manages DNS resolution, establishes TCP/TLS connections, manages HTTP/2 or HTTP/3 multiplexing, and pipes the resulting data stream back to the rendering process.

---

## ⑤ ADVANTAGES

1. **Native Promises**: Seamless integration with async/await pipelines and standard promise catch patterns.
2. **First-Class Primitives**: Decoupled `Request`, `Response`, and `Headers` classes simplify API mocking and Service Worker routing.
3. **Memory Efficient**: WHATWG streaming bodies prevent heavy RAM consumption when reading massive file payloads.

---

## ⑥ DISADVANTAGES / TRAPS

1. **HTTP Status Blindness**: Does not reject on status codes like 404 or 500, requiring developers to write defensive status check boilerplate.
2. **No Timeout Parameter**: Unlike Axios or legacy XHR, `fetch` lacks a built-in `timeout` option, forcing developers to manually link `AbortController` to a `setTimeout` timer.
3. **Double Read Crash**: Attempting to read a response body twice without explicitly cloning throws a terminal `TypeError`, crashing active loops.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Production-Grade Robust HTTP Client

To use Fetch safely in production, you should wrap it in a custom client that handles JSON parsing, adds automatic timeout limits, and verifies HTTP status codes:

```javascript
async function secureFetch(url, options = {}) {
    const { timeout = 5000, ...fetchOptions } = options;

    // 1. Create AbortController to implement timeout
    const controller = new AbortController();
    const id = setTimeout(() => controller.abort(), timeout);

    try {
        const response = await fetch(url, {
            ...fetchOptions,
            signal: controller.signal
        });

        clearTimeout(id); // Clear timeout on success

        // 2. Guard against HTTP error statuses
        if (!response.ok) {
            throw new Error(`HTTP_${response.status}: ${response.statusText}`);
        }

        // 3. Auto-parse JSON
        return await response.json();

    } catch (err) {
        clearTimeout(id);
        if (err.name === 'AbortError') {
            throw new Error(`Request timeout: Operation exceeded ${timeout}ms`);
        }
        throw err;
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: Does `fetch()` reject when the server responds with an HTTP status code `500`?**  
**A:** No. `fetch()` only rejects on network failures, DNS errors, or CORS violations. HTTP status codes like `500` or `404` are resolved as successful promises.

**Q2: What is the `response.ok` property?**  
**A:** A boolean property that returns `true` if the HTTP status code of the response is in the range `200` to `299`.

**Q3: Why can you only read a response body (e.g., `res.json()`) once?**  
**A:** The response body is represented as a raw **ReadableStream**. When you read it, the stream is locked and fully consumed (clearing the buffer). Attempting to read it again throws a `TypeError`.

**Q4: How do you cancel an active `fetch` request?**  
**A:** Instantiate an `AbortController`, pass its `AbortSignal` to the fetch options, and call `controller.abort()`.

**Q5: What are the three core class primitives exposed by the Fetch API?**  
**A:** `Request`, `Response`, and `Headers`.

---

### 🟡 Intermediate (5)

**Q6: Explain the purpose of the `response.clone()` method and when you must use it.**  
**A:** `response.clone()` creates a duplicate copy of the `Response` object, including a separate, unread clone of the body stream. You must use it when you need to read the response body multiple times (e.g., reading it once as text for logging, and a second time as JSON for processing).

**Q7: Explain the difference between sending credentials with a request using the `credentials` options: `omit`, `same-origin`, and `include`.**  
**A:** 
* `omit`: Never sends cookies or authorization headers with the request.
* `same-origin`: Sends credentials only if the destination URL matches the origin of the calling script (default in modern browsers).
* `include`: Always sends credentials (cookies and headers), even on cross-origin requests.

**Q8: Why does `fetch()` reject on cross-origin requests that violate CORS rules, even if the server returns a `200 OK`?**  
**A:** CORS (Cross-Origin Resource Sharing) is a browser-enforced security specification. The Network Service sends the request, and the server processes it and returns data. However, the browser inspects the response headers. If it detects that the `Access-Control-Allow-Origin` header is missing or mismatched, it blocks the JavaScript thread from accessing the response and rejects the promise with a `TypeError`.

**Q9: How do you send a POST request with JSON payload using Fetch? Write a quick snippet.**  
**A:** 
```javascript
fetch(url, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data)
});
```

**Q10: What is the difference between `response.blob()` and `response.arrayBuffer()`?**  
**A:** 
* `response.blob()` parses the response stream into a high-level **Blob (Binary Large Object)**, representing file-like raw data (used for images, files).
* `response.arrayBuffer()` parses the stream into a low-level, fixed-length binary data **ArrayBuffer**, allowing direct manipulation of the binary bytes using TypedArrays or DataViews.

### 🔴 Advanced (5)

**Q11: Walk through the low-level browser process architectures and IPC communications that occur when you trigger a `fetch()` request.**  
**A:** 
1. The JS engine executes `fetch()`, instantiates a `Request` object on the heap, and passes it to the renderer process.
2. The renderer uses **IPC (Inter-Process Communication)** to serialize and pass the request metadata to the browser's centralized **Network Service Process**.
3. The Network Service assigns a background C++ network thread to perform DNS lookup, establish a TCP socket, and negotiate TLS handshakes.
4. When the response arrives, the Network Service parses HTTP headers, validates security permissions, and pipes the raw data packets back to the renderer process via an IPC data pipe, feeding the `ReadableStream` backing buffer.

**Q12: Detail the concept of "Backpressure" in WHATWG ReadableStreams and explain how V8 handles slow consumption of network streams.**  
**A:** Backpressure is a flow-control mechanism that prevents a slow reader from being overwhelmed by a fast writer.
If the JS thread reads the `ReadableStream` slower than packets arrive, the stream's internal queue fills up.
Once the queue reaches its **High Water Mark** limit, the stream signals the browser's C++ network socket reader.
The browser pauses reading from the OS TCP socket. The OS kernel's TCP stack stops sending ACKs (acknowledgements) to the server. The sending server detects the full window size and halts transmission, pausing packet flow until the JS thread resumes reading, protecting RAM from overflowing.

**Q13: How does the browser enforce CORS preflight checks? Walk through the conditions that trigger an `OPTIONS` preflight request and its effect on Promise resolution.**  
**A:** A preflight check is triggered if a cross-origin request is not considered "simple".
* *Conditions triggering preflight:* HTTP methods other than `GET`, `HEAD`, or `POST`; or custom headers (like `Authorization` or `Content-Type` other than `application/x-www-form-urlencoded`, `multipart/form-data`, or `text/plain`).
* *Preflight Execution:* The browser sends a synchronous `OPTIONS` request with `Access-Control-Request-Method` and `Access-Control-Request-Headers`.
* *Promise Effect:* The `fetch()` Promise remains pending. Only if the server returns status `200` with valid CORS headers does the browser transmit the actual request. If the preflight fails, the browser blocks the actual request and rejects the promise with a `TypeError`.

**Q14: Explain the difference in memory allocations and stream handling when reading a 100MB video response using `await response.blob()` vs. consuming it via its `response.body` reader.**  
**A:** 
* `response.blob()`: The engine reads the entire 100MB stream to completion, allocating a massive contiguous 100MB memory block on the heap or temporary browser disk storage, causing memory pressure and potential crash risks on mobile devices.
* `response.body`: Allows consuming the data in small chunks:
  ```javascript
  const reader = response.body.getReader();
  while(true) {
      const { value, done } = await reader.read(); // Reads small chunks (~64KB) sequentially
      if (done) break;
      processChunk(value); // Memory footprint remains strictly capped at chunk size (~64KB)
  }
  ```

**Q15: How would you construct a custom mock response class that implements the `Response` primitive to test your fetch handlers offline?**  
**A:** 
```javascript
const mockData = { id: 1, name: "Alice" };
const mockResponse = new Response(JSON.stringify(mockData), {
    status: 200,
    headers: { 'Content-Type': 'application/json' }
});

// Verifying compatibility
console.log(mockResponse.ok); // true
console.log(await mockResponse.json()); // { id: 1, name: "Alice" }
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact memory layouts and garbage collection behavior that occur when a response object containing a 50MB file is cloned (`res.clone()`), and how the V8 heap manages duplicate data streams.**  
**A:** 
1. **Piping Allocation**: When `res.clone()` is called, the browser allocates a **TeeReadableStream** C++ structure.
2. **Buffer Splitting**: Instead of copying the 50MB binary buffer in memory (which would waste 50MB of RAM), the low-level network C++ service maintains a **single shared packet buffer**.
3. **Pointers Mapping**: Both `res` and `clonedRes` point to separate pointer cursors referencing the same underlying shared C++ ring buffer.
4. **Consumption**: As packet chunks are read, they are pushed to both readers.
5. **GC Cleanup**: If the developer reads and drops `res`, its pointer cursor is garbage collected. However, the shared C++ buffer **is not freed** because the clone `clonedRes` still references it. Only when both cloned stream readers have fully consumed or canceled their streams does V8 signal the network service to free the underlying 50MB C++ backing store, protecting against memory bloat.

**Q17: Explain the exact sequence of low-level browser C++ socket events, inter-process communications, and JavaScript microtask queues that occur when a `fetch` request experiences a TCP reset (RST) packet from a remote server mid-download.**  
**A:** 
1. **TCP Reset**: The remote server sends a TCP packet with the `RST` flag set.
2. **Kernel Event**: The OS network kernel receives the packet and triggers a socket error on the TCP connection.
3. **Network Service Alert**: The browser's Network Service process detects the socket crash.
4. **IPC Error Transmission**: The Network Service serializes a "Connection Reset" error and sends it via IPC to the Renderer process.
5. **Stream Break**: The C++ backing reader for the `ReadableStream` receives the error. It sets the stream's internal state to "errored" and frees the C++ pipe buffer.
6. **Task Queuing**: The renderer pushes an "Errored Stream Complete" task to the **Macrotask Queue**.
7. **Execution & Microtask Trigger**: When the Event Loop pulls the task:
   - The active `read()` or `json()` promise on the Call Stack is rejected with a `TypeError: Failed to fetch`.
   - The rejection reaction callback is instantly pushed to the **Microtask Queue** to handle the error immediately within the same loop tick.
