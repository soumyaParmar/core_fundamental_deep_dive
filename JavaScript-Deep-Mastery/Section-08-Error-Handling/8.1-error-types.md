# 📌 8.1 — Error Types
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** In JavaScript, when something goes wrong during the execution of your code, the runtime engine creates and throws a specialized object called an **Error**. These objects contain critical details about the problem, such as a readable description (message), the specific category of the error (name), and a list of the exact lines of code that were executing when the error occurred (stack trace).

**Technical Definition:** An error in JavaScript is a heap-allocated instance of the base `Error` class or one of its specialized derived sub-constructors. Errors carry a standardized interface consisting of `message` and `name` properties, along with an engine-specific `stack` property. Built-in error categories represent distinct runtime exception failures evaluated during AST parsing or virtual machine execution.

**One-line mental model:**
> *Errors are heap-allocated diagnostic records wrapping descriptive metadata and a snapshot of the active Call Stack.*

---

## ② DRILL DOWN

### 🔹 A. Anatomy of the Standard `Error` Object

Every standard Error object contains three primary properties:

```javascript
const err = new Error("Something went wrong!");

console.log(err.name);    // "Error" (The name of the error constructor)
console.log(err.message); // "Something went wrong!" (The descriptive text)
console.log(err.stack);   // String containing the trace of active stack frames:
                          // Error: Something went wrong!
                          //     at <anonymous>:1:13
```

*Important:* The `stack` property is non-standard but universally implemented by all modern engines (V8, JavaScriptCore, SpiderMonkey).

---

### 🔹 B. Built-in Error Types

JavaScript provides seven built-in error constructors to classify different categories of runtime failures:

```
                            ┌────────────────────────┐
                            │      base Error        │
                            └───────────┬────────────┘
                                        │
      ┌──────────────┬──────────────┬───┴──┬──────────────┬──────────────┐
      ▼              ▼              ▼      ▼              ▼              ▼
SyntaxError   ReferenceError    TypeError RangeError   URIError   AggregateError
```

#### 1. `SyntaxError`
Thrown when the JavaScript parser encounters code that violates the grammatical rules of the language during the compilation phase:
```javascript
// const x = ; // SyntaxError: Unexpected token ';'
// JSON.parse("invalid-json"); // SyntaxError: Unexpected token i in JSON
```

#### 2. `ReferenceError`
Thrown when attempting to read or write a variable that has not been declared, or is currently in its Temporal Dead Zone (TDZ):
```javascript
console.log(nonExistentVar); // ReferenceError: nonExistentVar is not defined

const a = 10;
// let a = 20; // ReferenceError: Cannot access 'a' before initialization
```

#### 3. `TypeError`
Thrown when an operand or argument passed to a function does not match the expected type, or when attempting to modify a read-only property:
```javascript
const num = 42;
// num.toUpperCase(); // TypeError: num.toUpperCase is not a function

const obj = {};
Object.defineProperty(obj, 'x', { value: 1, writable: false });
// obj.x = 2; // TypeError in strict mode
```

#### 4. `RangeError`
Thrown when a numeric value or parameter is outside its allowed range or boundaries:
```javascript
// 1. Array size out of bounds:
// const arr = new Array(-1); // RangeError: Invalid array length

// 2. Call Stack overflow:
function recurse() { recurse(); }
// recurse(); // RangeError: Maximum call stack size exceeded
```

#### 5. `URIError`
Thrown when the global URI handling functions (`encodeURI`, `decodeURI`, `encodeURIComponent`, `decodeURIComponent`) are passed malformed parameters:
```javascript
// decodeURIComponent('%'); // URIError: URI malformed
```

#### 6. `AggregateError` (ES2021)
Introduced in ES2021 to wrap multiple errors inside a single object (primarily returned by `Promise.any()` when all input promises fail):
```javascript
const err = new AggregateError([
    new Error("Failed DB A"),
    new Error("Failed DB B")
], "Multi-connection failure");

console.log(err.errors); // [Error: Failed DB A, Error: Failed DB B]
```

---

## ③ WHY IT EXISTS

1. **Diagnostic Safety**: Without specific error types, a program that fails would halt immediately without leaving any debugging records. Standardized errors pinpoint the exact line, context, and structural failure.
2. **Defensive Programming**: Categorizing failures (e.g., separating typing mistakes `TypeError` from variable lookups `ReferenceError`) allows developers to write intelligent `catch` blocks that handle specific recovery actions while allowing unknown bugs to bubble up for auditing.

---

## ④ INTERNAL WORKING: CALL STACK CAPTURE & JIT EXCEPTION STUBS

### `Error.captureStackTrace` V8 Internals

In V8, creating an `Error` object does not immediately construct the massive stack trace string.
1. **Lightweight Allocation**: The constructor allocates the error object and records a pointer to V8's current **Call Stack frame array**. This is highly efficient and saves memory.
2. **Lazy Evaluation**: The `stack` property is defined as an accessor descriptor (getter). Only when you explicitly read `error.stack` does V8 parse the recorded frame pointers, resolve them to file names and line numbers, and construct the final stack trace string.
3. **`Error.captureStackTrace(obj, constructorOpt)`**: A V8-specific API that appends a `.stack` property to any custom object, while allowing you to pass a constructor function (`constructorOpt`) to exclude all internal helper function frames above that constructor from the stack trace output.

---

### How Exceptions Affect JIT Optimizations

Historically, writing `try/catch` blocks inside a function turned off all TurboFan optimizations (marking the function as "de-optimized").
* **Modern V8 Handling**: Today, V8 can optimize functions containing `try/catch` blocks.
* **Bailout Penalty**: However, actually **throwing an exception** is still highly expensive. When an error is thrown, the engine must halt execution, discard JIT registers, execute a **bailout** to the interpreter, and search backward through the active stack frames for the nearest exception handler. Writing control flow logic that relies on constant throwing and catching creates severe CPU execution overhead.

---

## ⑤ ADVANTAGES

1. **Clear Classifications**: Seven built-in constructors allow fast structural identification of bugs.
2. **Call Stack Snapshots**: Universal `stack` accessors provide exact line-by-line tracing for logs.
3. **Lazy Trace Strings**: V8's lazy parsing prevents memory overhead unless the stack trace is explicitly printed.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Non-Standard Stack Properties**: The `.stack` field is not officially part of the ECMAScript standard; while widely supported, its formatting varies between V8 (Chrome/Node) and SpiderMonkey (Firefox).
2. **Silent Swallowing**: Using a generic catch-all `catch(e) {}` blocks diagnostic reporting, hiding critical coding typos (like `ReferenceError` spelling mistakes).
3. **Performance Costs**: Throwing errors in hot loops destroys JIT register caches and forces costly frame search rollbacks.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Production-Ready Exception Auditor

Use structural checks (`instanceof`) to parse exception types dynamically, safely separating predictable validation failures from terminal syntax or reference bugs:

```javascript
function processPayload(jsonString) {
    try {
        const user = JSON.parse(jsonString); // Can throw SyntaxError
        
        if (typeof user.id !== 'number') {
            throw new TypeError("User ID must be a number!"); // Validated TypeError
        }

        executeTransaction(user.id); // Can throw ReferenceError/TypeError if bug exists
    } catch (err) {
        if (err instanceof SyntaxError) {
            console.error("[MONITOR] Malformed JSON payload received:", err.message);
            // Handle recovery: return fallback data
        } else if (err instanceof TypeError) {
            console.error("[MONITOR] Validation constraint violated:", err.message);
        } else {
            // CRITICAL: ReferenceError or SyntaxError in our execution code
            // Alert operations team immediately and crash or rethrow!
            console.error("[CRITICAL SYSTEM BUG]:", err.stack);
            throw err; 
        }
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the three standard properties found on a JavaScript `Error` object?**  
**A:** `name` (the category name), `message` (the error description), and `stack` (the snapshot string trace of active stack frames).

**Q2: Which error type is thrown if you try to call a variable that is not a function (e.g., `const x = 5; x();`)?**  
**A:** `TypeError`.

**Q3: What does `ReferenceError` represent?**  
**A:** It represents a failure when attempting to read or write a variable that does not exist in the active scope or is currently trapped in the Temporal Dead Zone (TDZ).

**Q4: Can you throw values other than `Error` objects in JavaScript?**  
**A:** Yes. You can throw any JavaScript value, including primitives (e.g., `throw "Crash!"` or `throw 404`). However, throwing primitives is highly discouraged because they lack a `.stack` property, making debugging difficult.

**Q5: What is `AggregateError` and when is it primarily encountered?**  
**A:** A built-in error class that groups multiple independent errors into a single wrapper. It is primarily returned by `Promise.any()` when every input promise fails.

---

### 🟡 Intermediate (5)

**Q6: Explain the difference between a `SyntaxError` and a `ReferenceError` regarding compile-time vs. runtime execution.**  
**A:** 
* `SyntaxError` is generally detected during the **Parsing/Compilation Phase** before V8 executes any code. If a file has a syntax error, the engine halts immediately without running a single line.
* `ReferenceError` is a **Runtime Exception**. It occurs during code execution when V8 evaluates a statement and attempts to resolve an identifier that is not declared in the environment records.

**Q7: Explain the outcome of this code: `try { decodeURIComponent('%'); } catch(e) { console.log(e instanceof Error); }`**  
**A:** It logs `true`. `decodeURIComponent('%')` is malformed and throws a `URIError`. Since `URIError` inherits from `Error` (`URIError.prototype.__proto__ === Error.prototype`), `e instanceof Error` evaluates to `true`.

**Q8: Explain what V8's `Error.captureStackTrace` method does and write a brief usage example.**  
**A:** `Error.captureStackTrace(targetObject, constructorOpt)` is a V8-specific method that creates a `.stack` property on `targetObject`. The optional `constructorOpt` parameter allows passing a function reference; V8 will exclude that function and all parent frames above it from the generated stack trace, providing a cleaner log.
```javascript
class MyCustomError {
    constructor(message) {
        this.message = message;
        Error.captureStackTrace(this, MyCustomError); // Hides constructor call frame
    }
}
```

**Q9: Why does V8 capture stack traces lazily, and how does this optimize memory?**  
**A:** Resolving stack traces requires mapping raw CPU instruction pointers to source file names, class names, and line numbers, which is a slow and memory-intensive operation. To optimize, V8 only records a lightweight list of raw frame pointers when an error is instantiated. It only resolves these pointers and constructs the actual stack trace string if and when the developer explicitly reads the `.stack` property.

**Q10: What is the temporal dead zone (TDZ) reference error? Write a snippet that triggers it.**  
**A:** The TDZ ReferenceError occurs when attempting to read a variable declared with `let` or `const` before its declaration line is parsed.
```javascript
console.log(x); // ReferenceError: Cannot access 'x' before initialization
let x = 10;
```

### 🔴 Advanced (5)

**Q11: Detail what occurs inside the V8 virtual machine when an exception is thrown. How does it search the Call Stack for catch handlers?**  
**A:** When `throw` executes, V8 halts the current execution path.
1. It allocates the exception value.
2. It executes a **Stack Rollback (unwinding)**. V8 searches the current execution context frame for an active exception handler (try/catch block).
3. If not found in the current frame, it pops the frame off the Call Stack, rolls back register states, and inspects the parent frame.
4. This continues upward through all active frames.
5. If it reaches the root frame without finding a catch block, the exception becomes uncaught, triggering the host environment's crash handler (or unhandled rejection events).

**Q12: Why are throwing exceptions in hot loops considered a severe performance anti-pattern? Detail JIT compilation de-optimizations.**  
**A:** JIT compilers optimize code assuming a smooth, predictable execution path. When an exception is thrown:
* TurboFan must execute a **bailout**, discarding optimized CPU registers and reverting execution back to the slow Ignition interpreter.
* The interpreter must perform costly stack unwinding, scanning heap frame markers to find a handler.
* Throwing errors in loops constantly destroys compiled CPU instruction cache lines, forcing the JIT compiler to repeatedly optimize and de-optimize the hot code path.

**Q13: How can you write a custom exception parser that intercepts the non-standard `.stack` trace string and extracts the exact filename, function name, and line number of the crash?**  
**A:** 
```javascript
function parseErrorStack(error) {
    if (!error.stack) return null;
    
    // Split stack string into lines
    const lines = error.stack.split('\n');
    // Grab the first active code frame (line 1 is the error message, line 2 is the top frame)
    const topFrame = lines[1];
    
    // Parse: "at FunctionName (path/to/file.js:12:34)"
    const match = topFrame.match(/at\s+(.+?)\s+\((.+?):(\d+):(\d+)\)/) 
               || topFrame.match(/at\s+(.+?):(\d+):(\d+)/); // Fallback for anonymous
               
    if (!match) return null;
    
    return {
        functionName: match[1],
        filePath: match[2],
        lineNumber: parseInt(match[3], 10),
        columnNumber: parseInt(match[4], 10)
    };
}
```

**Q14: Explain the differences in stack trace capture between: `new Error()` vs. `Error.captureStackTrace` in custom class structures.**  
**A:** 
* `new Error()`: Automatically captures the entire Call Stack. However, the top stack frame will show the constructor initialization line itself (e.g., `at new CustomError (file.js:5:9)`), cluttering the logs with internal instantiation frames.
* `Error.captureStackTrace(this, CustomConstructor)`: Attaches the stack to `this` but explicitly hides all stack frames from the top frame down to `CustomConstructor`. The top frame in the stack trace will be the line *where the custom error was instantiated*, which is much cleaner for debugging.

**Q15: Explain how `AggregateError` represents its nested errors internally, and how to recursively print the full, multi-layered stack trace in production systems.**  
**A:** `AggregateError` has an `.errors` property which is a standard Array containing the sub-error objects.
*Recursive Printer:*
```javascript
function printDeepStack(error, depth = 0) {
    const indent = "  ".repeat(depth);
    console.error(`${indent}Error: [${error.name}] ${error.message}`);
    if (error.stack) {
        console.error(`${indent}Stack: ${error.stack.split('\n')[1]}`); // Top frame only
    }
    
    if (error instanceof AggregateError) {
        console.error(`${indent}Contains ${error.errors.length} sub-errors:`);
        error.errors.forEach(subErr => printDeepStack(subErr, depth + 1));
    }
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact memory leaks and GC tracking paths that can occur if you continuously attach references to active Error objects to a long-lived global singleton, detailing how V8 stack frame references capture outer lexical closures.**  
**A:** 
1. **Error Creation**: An `Error` is created inside a closure referencing large local variables.
2. **Global Reference**: The `Error` object is pushed into a global errors log array (GC Root).
3. **Lexical Capture**: Because the `Error`'s stack trace stores references to V8 Call Stack frame records, these frame records actively capture the **Scope/Context** active when the error occurred.
4. **GC Tracing**: The Mark-Sweep GC traces from the global array to the `Error` object, and follows its internal stack frame pointers.
5. **Closure Traversal**: These frame pointers keep the entire lexical environment record strongly reachable. If the error occurred inside a function that allocated large arrays or DOM references, V8 **cannot garbage collect those parent variables** because the global Error's stack frame pointers keep them alive, causing a massive memory leak.

**Q17: Explain the exact sequence of low-level V8 engine operations, bytecode compilation steps, and CPU instructions when a `TypeError` is resolved synchronously. What happens to the CPU's Instruction Pointer (IP) and Frame Pointer (FP) during this resolution?**  
**A:** 
1. **TypeError Condition**: The interpreter processes bytecode `CallMethod` on a primitive number. V8's type checker detects a type mismatch.
2. **Error Factory**: V8 jumps to its internal C++ runtime function `Runtime_ThrowTypeError`.
3. **Allocation**: V8 allocates a `TypeError` object on the heap and captures the raw stack frame addresses.
4. **Instruction Pointer Redirect**: The CPU's Instruction Pointer (IP) is instantly redirected to V8's **Exception Thrower Stub** in machine code.
5. **Stack Unwinding**: The exception stub reads the active Frame Pointer (FP) to locate the current stack frame. It scans the frame's metadata table for a `Catch Handler` instruction address.
6. **Frame Popping**: Finding no local handler, the stub rolls back the FP to the parent frame's base address, discarding all local variable registers, and updates the IP to scan the parent.
7. **Catch Jump**: Once it finds a frame with a catch block, V8 sets the FP to that frame, redirects the IP to the catch block's entry bytecode address, writes the error object to the catch parameter register, and resumes normal execution.
