# 📌 8.3 — Custom Errors
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** While JavaScript has built-in errors (like `TypeError` or `SyntaxError`), real-world production applications need domain-specific errors. For example, if a user enters an incorrect password, you want to throw an `AuthenticationError` with an associated HTTP status code `401`. Creating a **Custom Error** means extending the built-in `Error` class to build specialized exception objects that carry unique properties, metadata, and error codes.

**Technical Definition:** A Custom Error is a user-defined class that extends the standard `Error` class. It maintains prototype chain integrity, overrides the `.name` property to match the class name, sets custom attributes (like HTTP status or database codes), and leverages `Error.captureStackTrace` to ensure the generated stack trace omits the error's internal constructor frames, starting exactly at the line where the exception was thrown.

**One-line mental model:**
> *Custom Errors are subclassed Error objects carrying domain-specific metadata and clean stack traces for production auditing.*

---

## ② DRILL DOWN

### 🔹 A. Creating a Custom Error Subclass

Creating a robust custom error requires executing several key operations inside the constructor:

```javascript
class DatabaseError extends Error {
    constructor(message, query, code) {
        // 1. Call parent constructor to initialize 'message'
        super(message);

        // 2. Explicitly restore prototype chain (Crucial for ES5 compilation targets!)
        Object.setPrototypeOf(this, new.target.prototype);

        // 3. Configure the name property correctly
        this.name = this.constructor.name; // Evaluates to "DatabaseError"

        // 4. Attach domain-specific metadata
        this.query = query;
        this.code = code;

        // 5. Capture clean, relevant stack trace (V8 specific)
        if (Error.captureStackTrace) {
            Error.captureStackTrace(this, this.constructor);
        }
    }
}
```

---

### 🔹 B. The Prototype Chain and `instanceof` Integrity

If you do not set up the class prototype correctly, inheritance checks using `instanceof` will fail, especially when transpiling to older environments (ES5):

```javascript
const dbErr = new DatabaseError("Query failed", "SELECT * FROM...", "ERR_QUERY");

console.log(dbErr instanceof DatabaseError); // true (Direct instance)
console.log(dbErr instanceof Error);         // true (Inherits from base)
console.log(dbErr.name);                     // "DatabaseError"
```

#### Why we write `Object.setPrototypeOf(this, new.target.prototype)`:
When compiling down to ES5, the parent constructor (`Error`) does not support subclassing natively. It ignores the subclass prototype and returns a plain error object. Explicitly restoring the prototype chain guarantees that `instanceof` operations evaluate correctly under all environments and bundlers.

---

### 🔹 C. A Structured Production Error Hierarchy

In a large enterprise system, you should not create scattered individual errors. Instead, design a clean, structured **Error Hierarchy** branching from a base application error:

```
                            ┌────────────────────────┐
                            │       Base Error       │
                            └───────────┬────────────┘
                                        │
                            ┌───────────┴────────────┐
                            │    ApplicationError    │ (Base for your app)
                            └───────────┬────────────┘
                                        │
      ┌────────────────────────┬────────┴───────────────┬────────────────────────┐
      ▼                        ▼                        ▼                        ▼
ValidationError          AuthenticationError      DatabaseError            NetworkTimeoutError
```

```javascript
// Base Application Error containing HTTP metadata
class ApplicationError extends Error {
    constructor(message, statusCode = 500) {
        super(message);
        Object.setPrototypeOf(this, new.target.prototype);
        this.name = this.constructor.name;
        this.statusCode = statusCode;
        if (Error.captureStackTrace) {
            Error.captureStackTrace(this, this.constructor);
        }
    }
}

// Subclasses inherit from ApplicationError:
class ValidationError extends ApplicationError {
    constructor(message, fields = []) {
        super(message, 400); // 400 Bad Request
        this.fields = fields;
    }
}

class AuthenticationError extends ApplicationError {
    constructor(message) {
        super(message, 401); // 401 Unauthorized
    }
}
```

---

## ③ WHY IT EXISTS

1. **Domain-Specific Logic**: Standard JavaScript errors cannot represent application concepts. Throwing a `ValidationError` with custom fields allows an Express error-handling middleware to automatically convert the error into a clean JSON API response.
2. **Telemetry and Auditing**: Custom errors allow monitoring systems (like Sentry or Datadog) to group and route alerts based on specific error classes and custom metadata (e.g., routing `DatabaseError` to DBAs, and `AuthenticationError` to security teams).

---

## ④ INTERNAL WORKING: V8 STACK RESOLUTION & PROTOTYPES

### How `Error.captureStackTrace` Operates

When you execute `Error.captureStackTrace(this, this.constructor)`, V8 processes the stack collection dynamically:

```
 CALL STACK
 ┌──────────────────────────────────────┐
 │ Error.captureStackTrace()            │
 ├──────────────────────────────────────┤
 │ new DatabaseError()                  │ ◄── Omitted from stack trace!
 ├──────────────────────────────────────┤
 │ saveUserRecord()                     │ ◄── Top frame shown in stack!
 ├──────────────────────────────────────┤
 │ handleRequest()                      │
 └──────────────────────────────────────┘
```

1. **Stack Scan**: V8 reads the current Frame Pointer (FP) to map the active Call Stack.
2. **Omission Check**: V8 reads the second argument (`this.constructor` which points to `DatabaseError`).
3. **Truncation**: V8 crawls down the Call Stack, locating the frame for `DatabaseError`. It completely **trims and discards** that frame and any frames above it, ensuring the generated `.stack` trace starts directly at the line where `DatabaseError` was instantiated (`saveUserRecord`), keeping the log relevant and clean.

---

## ⑤ ADVANTAGES

1. **Rich Metadata**: Pack custom telemetry (validation fields, HTTP codes, query parameters) directly inside the exception object.
2. **Clean Call Stacks**: Omit internal constructor frames to keep traces focused on active application files.
3. **Structured Filtering**: Construct robust, scalable API middleware by catching and sorting errors using clean hierarchies.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Transpilation Prototype Loss**: Compiling classes to ES5 breaks prototype chains if the constructor fails to restore them, causing `instanceof` checks to return `false`.
2. **Class Name Minification Trap**: Relying on `this.name = this.constructor.name` can fail if you minify your code (e.g., using Webpack/Terser), as class names are minified to single characters (e.g., class `ValidationError` becomes `class e`), corrupting logs.
3. **Extra Allocations**: Multi-layered custom hierarchies increase the number of heap allocations per exception, adding minor garbage collection overhead.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Express.js Global Error Middleware

Use custom error hierarchies to build a highly scalable, automated Express.js error handling middleware that parses exceptions and formats clean, standardized API responses:

```javascript
// Express Error Middleware
function apiErrorHandler(err, req, res, next) {
    // 1. Check if error is part of our custom application hierarchy
    if (err instanceof ApplicationError) {
        return res.status(err.statusCode).json({
            status: "error",
            name: err.name,
            message: err.message,
            ...(err.fields && { validationErrors: err.fields }) // Conditionally append fields
        });
    }

    // 2. If it is an unhandled system error (e.g., ReferenceError or standard crash)
    console.error("[CRITICAL SYSTEM FAULT]:", err.stack);
    
    return res.status(500).json({
        status: "fatal",
        message: "An internal server error occurred."
    });
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: How do you create a custom error class in JavaScript?**  
**A:** By using ES6 class syntax to extend the built-in `Error` class (e.g., `class MyError extends Error {}`).

**Q2: What is the purpose of calling `super(message)` in the constructor of a custom error?**  
**A:** It calls the base `Error` class constructor, which correctly initializes the `.message` property and configures standard internal error descriptors on the instance.

**Q3: Why should you manually assign `this.name = this.constructor.name` in a custom error constructor?**  
**A:** By default, a subclassed error inherits `.name` from `Error.prototype` (evaluating to `"Error"`). Manually assigning it ensures the `.name` property reflects the actual custom class name (e.g., `"ValidationError"`).

**Q4: How can you attach custom data to a custom error?**  
**A:** By assigning custom fields directly to the `this` context inside the constructor (e.g., `this.statusCode = 404;`).

**Q5: What standard operator is used to identify a specific custom error class inside a `catch` block?**  
**A:** The `instanceof` operator (e.g., `if (err instanceof ValidationError)`).

---

### 🟡 Intermediate (5)

**Q6: Why is `Object.setPrototypeOf(this, new.target.prototype)` often added to custom error constructors?**  
**A:** When compiling ES6 classes down to ES5, parent constructors like `Error` do not natively support ES6-style subclassing. They create a plain Error object instead of linking the child prototype. Adding `Object.setPrototypeOf` manually restores the prototype chain on `this` so that `instanceof` checks evaluate correctly.

**Q7: Explain the minification trap associated with using `this.name = this.constructor.name` in production environments, and how to resolve it.**  
**A:** Production build tools (like Terser/Webpack) minify JavaScript class names to single characters to save space (e.g., class `AuthenticationError` becomes `class t`). As a result, `this.constructor.name` evaluates to `"t"` instead of `"AuthenticationError"`.
*Resolution:* Manually hardcode the string name inside the constructor (e.g., `this.name = "AuthenticationError"`) instead of dynamically reading `constructor.name`.

**Q8: Explain the difference in stack traces between `new Error()` and a custom error class that implements `Error.captureStackTrace(this, this.constructor)`.**  
**A:** 
* `new Error()` stack trace starts inside the error instantiation line itself, showing the internal constructor calls.
* A custom error using `Error.captureStackTrace(this, this.constructor)` trims the top frames, omitting the constructor execution frame itself. The stack trace starts exactly at the line *where the custom error class was instantiated*, making the trace cleaner and more relevant.

**Q9: Can you extend built-in sub-errors like `TypeError` or `SyntaxError` to create a custom error? Write a quick snippet.**  
**A:** Yes, you can extend any built-in error class.
```javascript
class InvalidParameterError extends TypeError {
    constructor(message) {
        super(message);
        this.name = "InvalidParameterError";
    }
}
```

**Q10: Why does `instanceof` evaluate to `true` for `customError instanceof Error`? Explain the prototype link.**  
**A:** The `extends` keyword links the prototype of the subclass prototype object to the base class prototype object (`CustomError.prototype.__proto__ === Error.prototype`). The `instanceof` operator crawls this `__proto__` chain; finding `Error.prototype` in the path, it returns `true`.

---

### 🔴 Advanced (5)

**Q11: Detail what V8 does during a `new CustomError()` invocation when compiling to ES6. Walk through the `new.target` propagation and memory allocation.**  
**A:** 
1. **Allocation**: V8 allocates the base heap instance. Since `CustomError` is a derived class, allocation is delegated to the base `Error` constructor.
2. **`new.target` Preservation**: Throughout the chain, `new.target` remains bound to `CustomError`.
3. **Prototype Link**: V8 sets the instance's `[[Prototype]]` link directly to `new.target.prototype` (`CustomError.prototype`).
4. **Execution**: The base `Error` constructor runs first via `super()`, configuring message descriptors and internal stack pointers.
5. **Initialization**: The derived `CustomError` constructor runs next, writing custom properties (like `statusCode`) directly to the pre-sized in-object memory slots.

**Q12: How would you design a custom error class in Node.js that securely serializes its properties (including the non-enumerable `.stack` property) to JSON format for shipping to logging aggregates?**  
**A:** Standard `JSON.stringify(error)` outputs `{}` because `message`, `name`, and `stack` are non-enumerable. To resolve this, override `.toJSON()`:
```javascript
class LoggableError extends Error {
    constructor(message, metadata = {}) {
        super(message);
        this.name = "LoggableError";
        this.metadata = metadata;
    }

    toJSON() {
        return {
            name: this.name,
            message: this.message,
            stack: this.stack, // Explicitly expose non-enumerable stack
            metadata: this.metadata,
            timestamp: new Date().toISOString()
        };
    }
}
```

**Q13: Explain why setting `.prototype = Object.create(Error.prototype)` inside a legacy ES5 custom error constructor pattern breaks the `error.stack` property generation, and how to preserve it.**  
**A:** In legacy ES5, `Object.create(Error.prototype)` correctly links the prototype chain but does **not** execute the internal initialization code unique to the `Error` constructor.
Standard ES5 `Error()` called as a function (without `new`) ignores `this` and returns a new Error.
*Preservation:* You must manually invoke `Error.captureStackTrace` or assign the stack manually inside the constructor:
```javascript
function LegacyError(message) {
    this.message = message;
    Error.captureStackTrace(this, LegacyError); // Resolves stack generation!
}
LegacyError.prototype = Object.create(Error.prototype);
LegacyError.prototype.constructor = LegacyError;
```

**Q14: Explain the differences in memory overhead and garbage collection (GC) tracing between throwing 10,000 standard `Error` objects vs. 10,000 lightweight custom object dictionaries (e.g., `throw { message: "Failed" }`).**  
**A:** 
* **Standard Errors**: Extremely high overhead. Each `Error` triggers V8's stack walker to record a chain of Call Stack frame pointers, allocating memory for frames metadata. If `.stack` is read, V8 allocates a massive trace string on the heap, creating high garbage collection (GC) copy/sweep stress.
* **Custom Object Dictionaries**: Minimal overhead. Allocates a simple key-value plain object on the heap with zero stack crawling or frame recording.
* *Trade-off:* While dictionaries are fast, they lack debugging stacks, making them useless for tracing structural bugs.

**Q15: How would you build a multi-layered custom error handler in a microservice environment that maps incoming HTTP error payloads (from external APIs) back into concrete local Custom Error instances?**  
**A:** 
```javascript
class MicroserviceError extends Error {
    constructor(message, serviceName, httpStatus) {
        super(message);
        this.name = "MicroserviceError";
        this.serviceName = serviceName;
        this.httpStatus = httpStatus;
    }

    static fromResponsePayload(payload, serviceName) {
        const message = payload.error?.message || "Unknown API error";
        const status = payload.status || 500;
        
        return new MicroserviceError(message, serviceName, status);
    }
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact memory layouts and hidden class transitions that V8 processes when you instantiate a deep three-level custom error hierarchy (A ➔ B ➔ C), detailing how in-object field size expectations propagate from the root.**  
**A:** 
1. **SFI Parsing**: V8 parses the class chain `C extends B extends A extends Error`. It reads static fields and constructor writes across all three classes.
2. **Pre-Sizing Estimation**: V8 estimates the total number of property writes (e.g., `this.a`, `this.b`, `this.c`) and sets `expected_nof_properties` in C's `SharedFunctionInfo` to 3.
3. **Allocation**: `new C()` triggers C's derived constructor. Control propagates via `super()` through `B` to `A` and finally `Error`.
4. **Layout Setup**: V8 allocates a `JSObject` with 3 pre-allocated in-object fields. The Map is set to `C0` (empty template of C).
5. **State Transitions**:
   - `A`'s constructor writes `this.a = 1`. V8 transitions map `C0` ➔ `C1`.
   - `B`'s constructor writes `this.b = 2`. V8 transitions map `C1` ➔ `C2`.
   - `C`'s constructor writes `this.c = 3`. V8 transitions map `C2` ➔ `C3`.
6. **Unified Map**: All future instances of `C` transition smoothly through this identical, pre-sized map pipeline, maximizing monomorphic JIT speed.

**Q17: Explain how a minified production bundle causes V8's Inline Cache (IC) lookups to degrade when using a polymorphic `instanceof` check against multiple custom error subclasses compiled down to ES5.**  
**A:** 
1. **ES5 Transpilation**: Babel transpiles classes to standard prototype functions. The dual constructor chain (`SubClass.__proto__ === BaseClass`) is lost on older browsers.
2. **Minification**: Class constructors are minified (e.g., `ValidationError` becomes `class a`, `AuthError` becomes `class b`).
3. **PIC Invalidation**: In the Express middleware, you run `if (err instanceof ValidationError)`. Under ES5 transpilation, the minified prototype links are evaluated dynamically.
4. **IC Degradation**: If `err` changes frequently between minified class `a`, `b`, and plain `Error`, the `instanceof` call site becomes **polymorphic** or **megamorphic**.
5. **Bailout**: V8's JIT compiler cannot hardcode the prototype matching offset. It invalidates the Inline Cache, discards the optimized machine-code stubs, and performs slow, dynamic prototype crawls across the minified heap objects, degrading CPU efficiency.
