# 📌 9.1 — CommonJS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** CommonJS is the legacy module system introduced to allow JavaScript to work on servers (specifically in Node.js). Before modules, all JavaScript files shared one single global scope, making it easy for variables in one file to accidentally overwrite variables in another. CommonJS solves this by wrapping each file in its own private sandbox. To share code between files, you explicitly use `module.exports` to export values, and `require()` to import them synchronously into other files.

**Technical Definition:** CommonJS is a synchronous, runtime-evaluated module specification. Each file is treated as a private module instance. Upon execution, the runtime wraps the file's code inside a C++ function wrapper, passing five local variables: `exports`, `require`, `module`, `__filename`, and `__dirname`. The module is executed synchronously, and its evaluated `module.exports` object is cached in `require.cache` for all subsequent imports.

**One-line mental model:**
> *CommonJS is a synchronous, runtime-evaluated module system that sandboxes files via function wrappers and caches exports in a global registry.*

---

## ② DRILL DOWN

### 🔹 A. `require()` and `module.exports` vs. `exports`

A major source of confusion for developers is the difference between `module.exports` and the `exports` shortcut.

```javascript
// 1. How the runtime initializes them:
// exports is simply a reference pointer pointing to the same object as module.exports!
// exports = module.exports = {};

// ✅ GOOD: Attaching properties to exports
exports.greet = () => "Hello";
exports.farewell = () => "Goodbye";

// ❌ BAD: Reassigning exports directly (The Shortcut Trap!)
exports = {
    greet: () => "Hello"
};
// Reassignment breaks the reference link! module.exports remains an empty object {},
// and other files requiring this module will receive absolutely nothing!

// ✅ GOOD: Always reassign module.exports if you want to export a single class, function, or primitive
module.exports = class User {};
```

---

### 🔹 B. Dynamic & Synchronous Loading

Unlike modern ES Modules, CommonJS loads modules **synchronously** and **dynamically** at runtime. You can call `require()` anywhere in your code, including inside loops or conditional `if` blocks:

```javascript
const environment = process.env.NODE_ENV;

if (environment === 'production') {
    const logger = require('./production-logger'); // Loaded dynamically only in production!
    logger.log("App started");
} else {
    const logger = require('./dev-logger');
    logger.log("Dev started");
}
```

---

### 🔹 C. The Module Cache (`require.cache`)

When you call `require('./module.js')` for the first time, Node.js loads, parses, and executes the code inside that file. The resulting `module.exports` object is cached in a global registry: `require.cache`.
Subsequent calls to `require('./module.js')` **do not re-execute the code**; they instantly return the cached object:

```javascript
// counter.js
let count = 0;
count++;
module.exports = count;

// main.js
const c1 = require('./counter'); // Executes code: count becomes 1
const c2 = require('./counter'); // Returns cached value: does NOT re-execute!

console.log(c1); // 1
console.log(c2); // 1 (Exactly the same cached instance)
```

#### Clearing the Cache for Hot Reloading:
If you need to force a module to re-execute (e.g., during live hot-reloading in dev servers), you must manually delete its entry in the cache:

```javascript
delete require.cache[require.resolve('./counter.js')];
const c3 = require('./counter'); // Code re-executes! count becomes 2
```

---

### 🔹 D. CommonJS Circular Dependencies

If two modules require each other (e.g., `A` requires `B`, and `B` requires `A`), it is called a circular dependency. CommonJS handles this by returning a **partially finished** exports object to prevent infinite loops:

```javascript
// a.js
exports.done = false;
const b = require('./b.js');
console.log('in a, b.done =', b.done);
exports.done = true;

// b.js
exports.done = false;
const a = require('./a.js'); // Circular link!
console.log('in b, a.done =', a.done); // Returns partially finished object: a.done is still false!
exports.done = true;

// Output on execution:
// in b, a.done = false
// in a, b.done = true
```

## ③ WHY IT EXISTS

Before Node.js was created in 2009, JavaScript was strictly a front-end browser language.
1. **Namespace Preservation**: Browsers ran scripts by dumping them into the shared `window` global scope. On servers managing files, databases, and secrets, sharing a global namespace would cause disastrous variable name collision bugs.
2. **Synchronous Disk Operations**: In local server environments, importing a module means reading it from a hard drive. Disk reads are fast enough that performing them synchronously via blocking I/O was simple and practical, eliminating the need for complex async wrappers.

## ④ INTERNAL WORKING: THE MODULE WRAPPER FUNCTION

### The Node.js Module Wrapper

CommonJS sandboxing is not accomplished by virtual machines or browser sandboxes.
Before V8 executes your file, Node.js wraps your code inside an anonymous **C++ C-bindings function wrapper**:

```javascript
// 1. The Wrapper Template:
(function(exports, require, module, __filename, __dirname) {
    // ──────── YOUR ACTUAL FILE CODE STARTS HERE ────────
    const data = 42;
    module.exports = data;
    // ──────── YOUR ACTUAL FILE CODE ENDS HERE ────────
});
```

### The Step-by-Step Compile and Run Lifecycle:
1. **File Read**: Node reads the file content from disk as a raw string.
2. **Wrapper Injection**: Node wraps the code inside the function template string.
3. **V8 Compilation**: Node passes the wrapper string to V8's JIT parser via `vm.runInThisContext()`, compiling it into a callable `JSFunction`.
4. **Execution**: Node invokes the compiled function, passing the five local variable parameters:
   - `exports`: A pointer to `module.exports` (initially `{}`).
   - `require`: The local resolver function tied to this module's directory path.
   - `module`: The active `Module` class heap instance.
   - `__filename` & `__dirname`: Absolute strings calculated from the active file path.
5. **Return**: The function finishes, and Node fetches `module.exports` to return it to the caller.

---

## ⑤ ADVANTAGES

1. **Dynamic Versatility**: Load modules on-the-fly inside functions or loops.
2. **Simple Architecture**: Synchronous disk reading removes the complexity of promises or async wrapping.
3. **Unified Registry**: Highly efficient `require.cache` prevents duplicate execution and conserves RAM.

---

## ⑥ DISADVANTAGES / TRAPS

1. **The Exports Reassignment Trap**: Reassigning `exports` directly breaks the pointer connection, returning empty objects and causing hard-to-debug import failures.
2. **Blocked Browser Portability**: Browsers do not support synchronous local disk reading. Running CommonJS code in browsers requires complex bundlers (like Webpack) to compile wrappers, slowing down build pipelines.
3. **Circular Reference Fragility**: Returns partially initialized objects during circular dependency loops, creating silent undefined reference bugs.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Module Resolver Utility

Use `require.resolve()` in production services to verify the existence of modules or resolve their absolute physical path without executing their code (useful for locating assets or checking plugins):

```javascript
function loadPluginSafely(pluginName) {
    try {
        // 1. Resolve absolute path of module without executing it
        const absolutePath = require.resolve(pluginName);
        console.log(`Plugin located at: ${absolutePath}`);
        
        // 2. Load the plugin now
        return require(pluginName);
    } catch (err) {
        console.warn(`Plugin [${pluginName}] could not be found. Using fallback plugin.`);
        return require('./fallback-plugin');
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the two core keywords used to export and import code in CommonJS?**  
**A:** `module.exports` (to export values) and `require()` (to import modules).

**Q2: Are CommonJS modules loaded synchronously or asynchronously?**  
**A:** Synchronously. The `require()` call blocks the execution thread until the target file is read from disk, compiled, and executed.

**Q3: What is the difference between `exports` and `module.exports`?**  
**A:** `module.exports` is the actual object returned to the caller by `require()`. `exports` is simply a local shortcut variable pointing to the same memory reference as `module.exports`.

**Q4: What happens if you require the exact same module 5 times in different files?**  
**A:** The module code is executed exactly **once** (upon the first `require`). The resulting exports object is stored in the global `require.cache` registry. The subsequent 4 `require` calls instantly return the cached object, bypassing re-execution.

**Q5: What are `__filename` and `__dirname` inside a CommonJS module?**  
**A:** They are local string variables injected by the Node wrapper: `__filename` holds the absolute path to the active module file, and `__dirname` holds the absolute path to the directory containing the file.

---

### 🟡 Intermediate (5)

**Q6: Explain what happens if you write `exports = class User {}` in a CommonJS module, and why this is considered a critical bug.**  
**A:** This is a critical bug because it breaks the pointer reference. Initially, `exports` points to the same object as `module.exports` (e.g., `exports = module.exports = {}`). Writing `exports = ...` reassigns the local `exports` variable to point to a new class reference, but `module.exports` remains unchanged pointing to the original empty object `{}`. Since `require()` only returns `module.exports`, the class is never exported.

**Q7: Explain the concept of circular dependencies in CommonJS. How does the engine prevent infinite execution loops?**  
**A:** When module `A` requires `B`, and `B` requires `A`, they form a circular dependency.
To prevent infinite loops, Node returns a **partially completed exports object** from the cache.
During execution, when `B` requires `A`, Node halts execution of `A`, stores `A`'s current incomplete export states in the cache, redirects control to `B` using `A`'s partial exports, finishes `B`, and then resumes `A`.

**Q8: How do you completely clear the cache for a specific module so that the next `require()` is forced to re-execute the file?**  
**A:** Delete the target file's absolute path entry from the global `require.cache` registry:
```javascript
const modulePath = require.resolve('./my-module.js');
delete require.cache[modulePath];
```

**Q9: Explain what `require.resolve()` does and how it differs from a standard `require()` call.**  
**A:** 
* `require()` resolves the path, loads the file, compiles and executes it, and returns the exported object.
* `require.resolve()` runs only the resolution algorithm: it searches directories to locate the file, checks security permissions, and returns its absolute path **without ever loading or executing the code**.

**Q10: Why can you call `require()` dynamically inside a function body or `if` block, but you cannot do the same with standard ES6 `import` statements?**  
**A:** CommonJS `require()` is a standard synchronous function executed sequentially at runtime, allowing dynamic execution. ES6 `import` statements are part of the static syntax of ES Modules; they are parsed and resolved during the compilation phase **before any code runs**, so they are strictly banned inside conditional runtime blocks.

### 🔴 Advanced (5)

**Q11: Write out the exact wrapper function Node.js uses to wrap CommonJS code, and explain the V8 compilation steps.**  
**A:** 
```javascript
(function(exports, require, module, __filename, __dirname) {
    // File code inserted here
});
```
*Compilation Steps:* 
1. **Read**: Node's C++ layer reads the file as a raw text string.
2. **Wrap**: Node concatenates the wrapper header and footer strings around the code.
3. **Compile**: Node passes the wrapper string to V8's JIT compiler via the `vm` context API. V8 parses the string into an AST, compiles the code to Ignition bytecode, and wraps it in a callable `JSFunction`.
4. **Execution**: Node executes this function, injecting the five arguments from its active execution environment.

**Q12: Detail the Node.js Module Resolution Algorithm. How does the engine resolve a require call to: `require('lodash')`?**  
**A:** When `require('lodash')` is called (non-path import):
1. **Core Verification**: Node checks if `lodash` is a native core module (like `fs`).
2. **Search Directory**: If not, Node checks the current directory for a folder named `node_modules`. If found, it searches for `node_modules/lodash`.
3. **Folder Resolution**:
   - Checks `node_modules/lodash/package.json` for a `"main"` field pointing to the entry file.
   - If missing, checks for `node_modules/lodash/index.js` or `node_modules/lodash/index.json`.
4. **Parent Crawl**: If not found in the current folder, Node moves to the parent directory and scans `../node_modules/lodash`.
5. **System Root**: This directory crawl repeats upward until it reaches the root directory. If still not resolved, it checks global node directories, throwing a `MODULE_NOT_FOUND` error if completely missing.

**Q13: How does V8 optimize subsequent `require()` calls? What are the memory costs of keeping thousands of module entries inside `require.cache`?**  
**A:** V8 optimizes requires using the `require.cache` hash table.
* **JIT Speed**: Since subsequent calls bypass compilation and disk reads, the `require` lookup evaluates in near-instant $O(1)$ time, returning the cached reference directly.
* **Memory Costs**: Every entry in `require.cache` retains a strong reference to the exported objects and their closed-over V8 Heap Context scopes. In massive enterprise applications, keeping thousands of unused modules strongly referenced in the cache blocks V8 from garbage collecting their scopes, contributing to long-term memory bloat.

**Q14: Explain why CommonJS circular dependencies are highly fragile compared to ES Modules. How does the binding mechanism differ?**  
**A:** 
* **CommonJS**: Copies values or reference pointers by value when executing `require()`. During circular dependencies, a module gets a copy of a **partially initialized object**. If it attempts to read a property that has not been initialized yet, it receives `undefined`, leading to runtime crashes.
* **ES Modules**: Use **Live Read-Only Bindings**. Instead of copying values, the compiler maps imports as direct reference pointers to the exported module scope variables. The bindings resolve before execution, making circular links highly robust.

**Q15: How would you build a custom require hook in Node.js (without using external libraries) that intercepts all `.json` imports and automatically decrypts their contents before returning the parsed object?**  
**A:** CommonJS exposes the `require.extensions` API (legacy but functional) to intercept extensions:
```javascript
const fs = require('fs');

require.extensions['.json'] = function(module, filename) {
    const rawContent = fs.readFileSync(filename, 'utf8');
    const decryptedContent = decryptString(rawContent); // Custom decryption
    
    module.exports = JSON.parse(decryptedContent); // Bind to module exports
};
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact C++ memory allocations, file-system blockages, and V8 Heap context creations that occur when Node.js runs `require('./large-file.js')` inside a highly active HTTP server handling 5,000 requests per second.**  
**A:** Running `require` dynamically inside a hot HTTP request handler is a production disaster.
1. **Thread Blocking**: `require` is strictly synchronous. V8 halts the main execution thread and calls C++ file APIs (`fs.readFileSync`), causing the entire process to block.
2. **Context Starvation**: Because the main thread is blocked waiting for disk I/O, the Event Loop is frozen, and 5,000 incoming network requests wait in the OS TCP socket buffer.
3. **C++ Allocation**: Node's C++ layer reads the file, allocating binary memory blocks.
4. **Compile Phase**: V8 parses the string, generating an AST and compile bytecode. The compiled `JSFunction` and a brand-new heap Context object are allocated, capturing variables.
5. **Memory Churn**: If this is done per request, thousands of new compiled functions and context scopes are allocated, causing massive heap fragmentation.
6. **GC Thrashing**: The high allocation rate triggers severe GC Scavenger cycles, freezing the thread further, culminating in a memory heap collapse.

**Q17: Explain the internal V8 heap references that are created when you clear `require.cache` but some files still retain a strong reference to the previously exported module object. How does this result in a V8 "Detached Module" memory leak?**  
**A:** 
1. **Cache Clearing**: You delete a module from the cache: `delete require.cache[require.resolve('./module.js')]`.
2. **New Import**: Future imports compile and execute a new instance, allocating new objects.
3. **The Trap**: However, an active global module (`main.js`) still stores a local reference to the **original** exports object created before the cache was cleared (e.g., `const oldInstance = require('./module.js')`).
4. **Detached State**: The old exports object is "detached" from `require.cache`.
5. **GC Tracing**: During GC, the marking algorithm traces from the active `main.js` root to `oldInstance`.
6. **Leak**: Because `oldInstance` remains reachable, its entire closure context, compiled code metadata, and all connected variables are kept alive on the heap. Since they are detached from the cache, they cannot be updated or cleared, creating a permanent **Detached Module Memory Leak**.
