# 📌 10.2 — Events
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** An event is a notification from the browser that something has happened, such as a user clicking a button, scrolling the page, or a file finishing its download. Events travel through the HTML tree in three phases: traveling down to the clicked item (Capture Phase), hitting the item (Target Phase), and traveling all the way back up to the top of the tree (Bubble Phase). This allows you to handle actions directly on the element or intercept them at a higher parent level (Event Delegation).

**Technical Definition:** Events are standardized object structures conforming to the **W3C DOM Event Specification**. Event propagation is a two-way tree traversal consisting of the **Capture Phase** (root to target) and the **Bubble Phase** (target to root). Runtimes manage event handlers asynchronously; however, the dispatching of listeners along the propagation path occurs synchronously within the same Event Loop tick once a task is dequeued.

**One-line mental model:**
> *Events are tree-traveling signals that flow down during capture and rise up during bubble, notifying registered listener functions.*

---

## ② DRILL DOWN

### 🔹 A. The Three Phases of Event Flow

When an event occurs (e.g., a user clicks a `<span>` nested inside a `<div>`), the event travels along a strict, three-phase path:

```
  1. CAPTURE PHASE (Root ──► Parent ──► Target)
  ┌────────────────────────────────────────────────────────┐
  │  document ──► <html> ──► <body> ──► <div> ──► [span]    │
  └───────────────────────────────────────────────────┬────┘
                                                      │ 2. TARGET PHASE
  3. BUBBLE PHASE  (Target ──► Parent ──► Root)       ▼
  ┌────────────────────────────────────────────────────────┐
  │  document ◄── <html> ◄── <body> ◄── <div> ◄── [span]    │
  └────────────────────────────────────────────────────────┘
```

1. **Capture Phase**: The event travels down from the global window and document objects, through all parent elements, to the target.
2. **Target Phase**: The event reaches the target element that triggered the action.
3. **Bubble Phase**: The event rises back up through all parent elements to the root. By default, standard listeners (`addEventListener`) only fire during the **Bubble Phase**.

---

### 🔹 B. Advanced Listener Options

The standard `addEventListener(type, listener, options)` method accepts a highly configureable `options` object:

```javascript
element.addEventListener('click', callback, {
    capture: true,  // Force listener to execute during the CAPTURE phase instead of bubble
    once: true,     // Automatically remove the listener after it fires once!
    passive: true,  // Promise never to call preventDefault() (Maximizes scroll performance!)
    signal: abortSignal // Allows removing the listener dynamically via an AbortController!
});
```

---

### 🔹 C. Event Control: preventDefault vs. stopPropagation

It is critical to understand the boundaries of event control methods:

```javascript
// 1. preventDefault()
// Cancels the browser's default behavior (e.g., stops links from navigating or forms from submitting)
// Does NOT stop the event from bubbling up the tree!
link.addEventListener('click', (e) => {
    e.preventDefault();
});

// 2. stopPropagation()
// Stops the event from traveling further up (bubbling) or down (capturing) the DOM tree.
// However, if the active element has other listeners attached, they WILL still execute!
btn.addEventListener('click', (e) => {
    e.stopPropagation();
});

// 3. stopImmediatePropagation()
// Stops the event from bubbling AND immediately kills all other listeners attached
// to the same element for this event type!
btn.addEventListener('click', (e) => {
    e.stopImmediatePropagation();
    console.log("I run!");
});
btn.addEventListener('click', (e) => {
    console.log("I will NOT run!"); // Killed by stopImmediatePropagation!
});
```

---

### 🔹 D. Event Delegation (Memory Optimization)

If you have a list containing 10,000 items, attaching `addEventListener` to every item would allocate 10,000 function closures in V8 heap memory, slowing down the page.
Instead, leverage **Event Delegation**: attach **one single listener** to the parent element, and intercept the event as it bubbles up!

```javascript
// A single listener on the parent <ul> container:
listContainer.addEventListener('click', (event) => {
    // Check if the actual clicked target is a list item:
    if (event.target.tagName === 'LI') {
        console.log("List item clicked:", event.target.textContent);
    }
});
```

---

### 🔹 E. Custom Events

JavaScript allows creating and dispatching custom event payloads to build decoupled APIs:

```javascript
// 1. Create a custom event with metadata inside 'detail'
const cartEvent = new CustomEvent('cartUpdate', {
    detail: { itemId: 99, quantity: 2 },
    bubbles: true, // Allow the event to bubble up the DOM!
    cancelable: true
});

// 2. Listen for the custom event
document.addEventListener('cartUpdate', (e) => {
    console.log(`Cart updated: Item ${e.detail.itemId}`);
});

// 3. Dispatch the event from any element
button.dispatchEvent(cartEvent);
```

---

## ③ WHY IT EXISTS

1. **Decoupled User-Action Pipeline**: User inputs (clicks, keyboard, touch) occur asynchronously. Events decouple these inputs from the application's layout render loop, acting as a clean bridge that routes hardware input interrupts directly into JS execution scopes.
2. **Unified Interactive Standard**: Standardizing event flows across bubble/capture stages ensures developers have a uniform pipeline to intercept, cancel, or proxy user events dynamically across any browser architecture.

---

## ④ INTERNAL WORKING: THE PASSIVE SCROLL OPTIMIZATION

### The `passive: true` Performance Secret

When you scroll a page or swipe a mobile screen, the browser must fire the `touchstart` or `touchmove` event.
* **The Problem**: By default, the browser engine does not know if your code will call `event.preventDefault()` to stop the scroll.
* **The Delay**: To be safe, the browser **halts page scrolling** and waits for the JavaScript main thread to execute the callback. If the JS thread is busy or takes 50ms, the user experiences visible **scroll stutter (jank)**.
* **The Optimization**: By setting `passive: true`, you tell the browser: *"I promise I will never call `preventDefault()`."* The browser instantly performs the visual scroll on the background GPU thread **without waiting for JavaScript**, delivering buttery-smooth 120fps scrolling.

```
  [ Touch Input ] ──► [ GPU Compositor Thread ] ──► Instantly Scroll Screen (120fps)
                              │
                      (Pipes event in background)
                              ▼
                      [ JS Main Thread ] ──► Runs Touch Listener Callback
```

---

## ⑤ ADVANTAGES

1. **Memory Conservation**: Event delegation prevents allocating thousands of temporary closures.
2. **Smooth Scrolling**: Passive options bypass main thread JS blocks to deliver fluid visual animations.
3. **Decoupled Architecture**: Custom events allow building clean, modular UI components that communicate safely without tight coupling.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Silent delegation bugs**: If elements inside the list item contain nested children (like `<i>` icons), `event.target` will point to the `<i>` instead of the `<li>`, breaking simple element tag checks.
2. **StopPropagation Overuse**: Aggressive use of `stopPropagation` blocks global analytical trackers (like Google Analytics) that monitor clicks by listening at the `document` level.
3. **Un-removed Listeners**: Failing to remove active window resize or scroll listeners when components are unmounted leads to permanent memory leaks.

---

## ⑦ PRODUCTION USAGE

### 🛠️ High-Performance Safe Event Delegator

Use a robust delegation utility that handles deep nested child elements correctly (using `.closest()`) to prevent common delegation bugs:

```javascript
function delegateEvent(parentElement, selector, eventType, callback) {
    parentElement.addEventListener(eventType, (event) => {
        // 1. Find the closest matching ancestor element, up to the delegated parent boundary
        const targetElement = event.target.closest(selector);
        
        if (targetElement && parentElement.contains(targetElement)) {
            // 2. Execute callback, ensuring 'this' context binds to the target element
            callback.call(targetElement, event);
        }
    });
}

// Usage:
// Correctly intercepts clicks even if the user clicks a nested <i> inside the <button class="action-btn">
delegateEvent(document.body, '.action-btn', 'click', function(e) {
    console.log("Button clicked:", this.id);
});
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the three phases of event propagation in the DOM?**  
**A:** The **Capture Phase** (event travels down to the target), the **Target Phase** (event reaches the clicked element), and the **Bubble Phase** (event rises back up to the root).

**Q2: What is the difference between `event.preventDefault()` and `event.stopPropagation()`?**  
**A:** 
* `preventDefault()` cancels the browser's default behavior for the event (e.g., stopping link navigation) but does not stop propagation.
* `stopPropagation()` stops the event from traveling further up or down the DOM tree, but does not cancel default behaviors.

**Q3: What is Event Delegation?**  
**A:** A performance optimization pattern where a single event listener is attached to a parent element rather than attaching individual listeners to multiple child elements, leveraging event bubbling to intercept events.

**Q4: How do you create and trigger a custom event in JavaScript?**  
**A:** Create it using `new CustomEvent('name', { detail: payload })` and trigger it using `element.dispatchEvent(event)`.

**Q5: Will an event listener attached via `addEventListener('click', cb)` execute during the capture phase by default?**  
**A:** No. By default, listeners are registered to execute during the **Bubble Phase**. To run during capture, you must set `capture: true` in the options.

---

### 🟡 Intermediate (5)

**Q6: Explain what `event.stopImmediatePropagation()` does and how it differs from `event.stopPropagation()`.**  
**A:** 
* `stopPropagation()` stops the event from traveling to other elements up or down the DOM tree.
* `stopImmediatePropagation()` stops tree propagation **and immediately blocks** any other event listeners of the same type attached to the *active* element from running.

**Q7: Explain why `passive: true` is highly critical for scroll performance on mobile browsers.**  
**A:** Mobile scrolling is processed on a high-speed background GPU compositor thread. By default, if a touch listener exists, the GPU must pause scrolling to wait for JavaScript's main thread to run the event and confirm if `preventDefault()` was called. Setting `passive: true` guarantees JS won't prevent scrolling, allowing the GPU to scroll immediately, preventing visual stutter.

**Q8: Explain the difference between `event.target` and `event.currentTarget` inside an event listener.**  
**A:** 
* `event.target` points to the **actual, deep element** that triggered the event (where the click physically occurred).
* `event.currentTarget` points to the **element that is currently executing the event listener** (the element where the listener was attached).

**Q9: How can you dynamically remove an event listener inside its own callback function without having a named reference to the function?**  
**A:** Use the `{ once: true }` option, or reference `arguments.callee` (legacy sloppy mode), or leverage an `AbortSignal`:
```javascript
const controller = new AbortController();
element.addEventListener('click', (e) => {
    console.log("Fired");
    controller.abort(); // Dynamically removes the listener!
}, { signal: controller.signal });
```

**Q10: Why does custom event delegation fail if you check `event.target.tagName === 'LI'` when the `<li>` contains a nested `<i>` icon? How do you resolve this?**  
**A:** If the user clicks directly on the icon, `event.target` points to the `<i>` element. Its `.tagName` is `"I"`, so the check `=== 'LI'` fails.
*Resolution:* Use `event.target.closest('li')` to search upward for the matching list item parent.

---

### 🔴 Advanced (5)

**Q11: Detail the browser C++ rendering pipeline and thread-context swaps that occur when a user swipes a mobile screen containing an active, non-passive `touchmove` listener.**  
**A:** 
1. **Input**: Touch hardware registers a swipe.
2. **GPU Compositor**: The GPU thread captures the swipe. It inspects its compositor map and detects a non-passive touch region.
3. **Thread Switch**: The GPU thread blocks. It sends an IPC signal to the main browser thread.
4. **JS Queue**: The main thread pushes a `touchmove` task to the Macrotask Queue.
5. **Execution**: The Event Loop pulls the task, runs V8, and executes the callback.
6. **Verification**: Blink checks if `preventDefault()` was called.
7. **Thread Switch Back**: Blink passes the scroll authorization back to the GPU thread via IPC.
8. **Render**: The GPU finally scrolls the screen. If the JS thread took 100ms, the screen remains frozen for 100ms, creating severe visual jank.

**Q12: Explain the Event listener leak inside V8. How does keeping a reference to a parent DOM node inside a callback closure prevent the reclaiming of an unmounted child DOM node?**  
**A:** 
If you attach a listener `window.addEventListener('resize', () => { console.log(parentEl.id); })`, the anonymous callback is registered globally on the window.
Because the callback references `parentEl` (forming a closure), V8 allocates a heap context holding `parentEl`.
Even if you remove `parentEl` from the DOM, V8's GC tracing starts at the global `window` object, traverses the listener list, finds the callback, follows its context pointer, and marks `parentEl` as reachable. `parentEl` remains trapped in memory, creating a permanent memory leak until the listener is explicitly removed.

**Q13: Explain how browser engines dispatch events programmatically via `.dispatchEvent()` vs. user actions. Why do microtask checkpoints differ?**  
**A:** 
* **User Triggered**: The event callback executes, and the Call Stack empties. The engine **immediately runs a microtask checkpoint** before dispatching the next bubbling listener.
* **Programmatic (`.dispatchEvent`)**: The JS call to `dispatchEvent()` is active on the Call Stack. The engine evaluates all bubble listeners **synchronously** inside that single call frame. The stack is not empty between bubble listeners; therefore, the Microtask Queue **is not drained** until the entire programmatic `dispatchEvent()` script block finishes.

**Q14: Explain why `event.stopPropagation()` fails to prevent other listeners attached to the *same* element from running. How does V8 store listeners list?**  
**A:** V8/Blink stores event listeners in a contiguous C++ array linked to the DOM node.
When an event fires, Blink iterates through this array synchronously.
Calling `stopPropagation()` sets a boolean flag on the C++ `Event` object to block transitions to other DOM nodes.
However, it does not halt the C++ loop iterating over the active element's array. To stop that array loop, you must call `stopImmediatePropagation()`, which immediately sets a "halt loop" flag in the C++ iterator.

**Q15: How would you implement a custom event bus in pure JavaScript that fully replicates the DOM's Capture/Bubble propagation phases for a non-DOM hierarchical JSON data tree?**  
**A:** 
```javascript
class TreeNode {
    constructor(name, parent = null) {
        this.name = name;
        this.parent = parent;
        this.listeners = new Map();
    }
    addEventListener(type, cb) {
        if (!this.listeners.has(type)) this.listeners.set(type, []);
        this.listeners.get(type).push(cb);
    }
    dispatchEvent(event) {
        event.target = this;
        // 1. Build Propagation Path (Root to Target)
        const path = [];
        let curr = this;
        while (curr) {
            path.unshift(curr);
            curr = curr.parent;
        }
        // 2. Capture Phase
        event.phase = "capture";
        for (const node of path) {
            node._fire(event);
            if (event._stopped) return;
        }
        // 3. Bubble Phase
        event.phase = "bubble";
        path.reverse();
        for (const node of path) {
            node._fire(event);
            if (event._stopped) return;
        }
    }
    _fire(event) {
        event.currentTarget = this;
        const list = this.listeners.get(event.type) || [];
        for (const cb of list) {
            cb(event);
        }
    }
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact C++ memory allocations, array index lookups, and V8 engine states that occur when a mouse click event is physical triggered on a coordinate corresponding to a button element nested deep inside the DOM.**  
**A:** 
1. **Interrupt**: The OS registers a hardware mouse click and passes coordinates to the browser.
2. **Hit Testing**: Blink's rendering pipeline walks its C++ layout tree, performing a **Hit Test** calculation to determine which specific element lies at those coordinate parameters (resolves to `blink::HTMLButtonElement`).
3. **Path Construction**: Blink builds a C++ vector list of all ancestor nodes up to the root to define the propagation path.
4. **Capture Phase**: Blink crawls the vector index backward (Root ──► button), executing any V8 listener stubs registered for capture.
5. **Target & Bubble**: Blink hits the target, and then crawls the vector forward, invoking V8 stubs for bubbling.
6. **V8 Frame Allocation**: For each matching C++ listener, Blink enters the V8 environment, allocates a physical stack frame for the JS callback, writes the `JS DOM Event` object to V8, and executes the bytecode.

**Q17: Explain the exact sequence of V8 heap tracing and garbage collection operations when an active DOM element containing thousands of click listeners is detached from the DOM tree but is captured inside a global React/Vue component closure scope.**  
**A:** 
1. **Detachment**: The parent framework removes the DOM node from the visible page.
2. **Context Retention**: The DOM wrapper remains referenced inside a global component state closure.
3. **GC Marking Crawl**: The GC sweeps the heap from global roots:
   - Locates the active framework component.
   - Crawls its state scope and finds the JS DOM wrapper.
   - Marks the wrapper as reachable.
4. **Blink Node Keep-Alive**: The wrapper keeps the C++ DOM node alive in Blink memory.
5. **Listener Retention**: Because the C++ node is alive, the attached C++ event listeners array remains active.
6. **Callback Leak**: The C++ listeners array holds strong root references to the thousands of JS callback functions on the V8 heap.
7. **GC Trap**: The GC is forced to mark all these callback functions and any variables captured inside their closures as reachable. The entire tree, components, and listeners remain permanently trapped on the heap, creating a massive, cascading memory leak due to the circular JS-C++ referencing.
