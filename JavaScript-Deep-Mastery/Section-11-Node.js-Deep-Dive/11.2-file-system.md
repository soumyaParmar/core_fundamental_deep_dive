# 📌 11.2 — File System
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** The File System (`fs`) module in Node.js allows your code to read, write, create, delete, and monitor files and folders on your computer. Because reading files from a hard drive is incredibly slow compared to executing memory calculations, Node.js provides three different ways to interact with files: **Synchronous** (blocks the whole app until finished), **Asynchronous Callbacks** (delegates to background C++ threads), and **Streams** (reads massive files chunk-by-chunk in a continuous pipeline to prevent your server from running out of memory).

**Technical Definition:** The Node.js `fs` module exposes low-level POSIX file system system calls. It integrates with **Libuv's ThreadPool** to execute blocking file operations (like `open`, `read`, `write`, `close`) asynchronously on background C++ worker threads. It implements **Node.js Streams**, facilitating high-throughput, chunk-based binary flow control equipped with **Backpressure** management to prevent memory buffer overflows.

**One-line mental model:**
> *The fs module provides synchronous, callback, and streaming wrappers over operating system file system calls, managed asynchronously via Libuv worker threads.*

---

## ② DRILL DOWN

### 🔹 A. Three Ways to Read Files

Understanding when to use each API style is critical for production scalability:

```javascript
const fs = require('fs');

// 1. SYNCHRONOUS: Blocks the main thread! (Use ONLY during app startup for loading configs)
try {
    const data = fs.readFileSync('config.json', 'utf8');
} catch (err) { console.error(err); }

// 2. ASYNCHRONOUS CALLBACKS: Non-blocking, delegates to Libuv ThreadPool
fs.readFile('data.txt', 'utf8', (err, data) => {
    if (err) return console.error(err);
    console.log(data);
});

// 3. PROMISE-BASED (ES6+): Modern, clean async/await pipeline
const fsPromises = require('fs').promises;
async function loadData() {
    try {
        const data = await fsPromises.readFile('data.txt', 'utf8');
    } catch (err) { console.error(err); }
}
```

---

### 🔹 B. File Descriptors

A **File Descriptor** is a unique, non-negative integer assigned by the operating system to represent an open file session. It acts as a direct reference pointer:

```javascript
// Open a file to get its unique File Descriptor (fd)
fs.open('logs.txt', 'r', (err, fd) => {
    if (err) throw err;
    console.log(`File Descriptor allocated: ${fd}`); // e.g., 3

    // Read a specific chunk of bytes using the file descriptor
    const buffer = Buffer.alloc(100);
    fs.read(fd, buffer, 0, 100, 0, (err, bytesRead) => {
        console.log(`Read ${bytesRead} bytes safely.`);
        
        // ALWAYS close the file descriptor when finished to prevent OS memory leaks!
        fs.close(fd, () => console.log("File closed safely."));
    });
});
```

---

### 🔹 C. Node.js Streams and the Memory Advantage

Attempting to read a massive 5GB video file using `fs.readFile` will load the **entire 5GB** into V8's heap memory at once, instantly triggering a terminal `RangeError: File size is too large` or crashing the server due to Out of Memory (OOM) boundaries.
**Streams** solve this by reading and writing files in small, manageable chunks (typically 64KB) sequentially:

```javascript
// Reading a massive file chunk-by-chunk with zero memory bloat
const readStream = fs.createReadStream('large-video.mp4');
const writeStream = fs.createWriteStream('copy-video.mp4');

// Pipe automatically manages the transfer of data packets
readStream.pipe(writeStream);

readStream.on('data', (chunk) => {
    console.log(`Received chunk of size: ${chunk.length} bytes`); // e.g., 65536 bytes
});

readStream.on('end', () => console.log("Streaming complete!"));
```

---

### 🔹 D. Stream Backpressure Management

If the read stream is fast (loading data from a high-speed SSD) and the write stream is slow (writing data over a slow network connection), the temporary buffer memory will overflow. This is resolved by **Backpressure**:

```
 [ Read Stream ] ──► (Fast 10MB/s) ──► [ Buffer Array ] ──► (Slow 1MB/s) ──► [ Write Stream ]
                                            │
                                  High Water Mark reached!
                                            │
                                            ▼
                                   readStream.pause() 
                                   (Signals Libuv to halt disk reads)
```

#### How `.pipe()` handles backpressure automatically under the hood:
1. `writeStream.write(chunk)` returns `false` when its internal buffer exceeds its limit (**`highWaterMark`**, default: 16KB).
2. The pipe logic detects `false` and instantly executes `readStream.pause()`, halting file reads.
3. Once the write stream drains its buffer to disk, it fires the `'drain'` event.
4. The pipe logic catches `'drain'` and executes `readStream.resume()`, restarting the data flow.

---

### 🔹 E. Monitoring Files: `fs.watch` vs. `fs.watchFile`

Monitoring file changes is a common task (e.g., in hot-reload servers):

```javascript
// 1. fs.watch (Highly Efficient - Native OS Events!)
// Uses native OS kernel hooks: inotify (Linux), FSEvents (macOS), ReadDirectoryChangesW (Windows)
const watcher = fs.watch('config.json', (eventType, filename) => {
    console.log(`File updated via native OS event: ${filename}`);
});

// 2. fs.watchFile (Inefficient - Polling stat checks!)
// Periodically wakes up the CPU to query file statistics (blocking disk I/O)
fs.watchFile('config.json', { interval: 5007 }, (curr, prev) => {
    console.log("File stat check completed.");
});
```

## ③ WHY IT EXISTS

1. **Server-Side File Operations**: Unlike front-end JS, which is sandboxed in browsers to prevent local file reads, server-side frameworks must interact with disk databases, parse logs, write images, and read configuration files.
2. **Infinite Data Processing**: Provides high-performance stream pipelines to allow server nodes to parse massive gigabyte-scale datasets cleanly on standard, low-spec cloud instances.

## ④ INTERNAL WORKING: LIBUV THREADPOOL DELEGATION

### How `fs` Calls Delegate to Libuv in C++

JavaScript does not have native APIs to read files.
When you run `fs.readFile('file.txt', cb)`, the execution flows through these steps:

```
  [ JS Main Thread ] ──► fs.readFile() ──► [ C++ Binding Layer ] ──► [ uv_fs_read ]
                                                                           │
                                                                           ▼
  [ JS Callback ] ◄── Dequeues Callback ◄── [ Poll Phase ] ◄── Pushes Task onto ThreadPool
```

1. **JS Call**: The JS thread executes `fs.readFile` and passes the absolute file path and callback function to Node's C++ binding layer.
2. **Handle Allocation**: C++ allocates a `uv_fs_t` struct request and delegates the execution to Libuv.
3. **ThreadPool Push**: Libuv places the `uv_fs_t` request onto the **ThreadPool** queue.
4. **Worker Thread Execution**: An idle C++ worker thread picks up the request. It executes the blocking OS system call (`open`, `read`) directly on the OS filesystem.
5. **Event Loop Completion**: Once the read finishes, the worker thread posts an interrupt signal. The Event Loop Poll phase catches the completed event and executes the original JS callback.

---

## ⑤ ADVANTAGES

1. **Memory Safety**: Stream APIs allow processing massive files with a constant, small RAM footprint.
2. **Asynchronous Scaling**: Offloading file I/O to background C++ threads prevents freezing the main JS execution loop.
3. **Native OS Watchers**: Highly efficient file monitoring using native system kernel events.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Synchronous Core Blockage**: Forgetting and using `fs.readFileSync` inside API request routes stalls the entire server, freezing all concurrent network requests.
2. **File Descriptor Leak**: Opening files via `fs.open` without calling `fs.close` leaves File Descriptors open, eventually exhausting the operating system limit (`EMFILE: too many open files`).
3. **Static Stream Boundaries**: Streams are one-directional. You cannot rewind or seek specific byte coordinates without closing and reopening the stream.

---

## ⑦ PRODUCTION USAGE

### 🛠️ High-Performance Stream-Based Log Parser

Use stream pipelines to parse massive multi-gigabyte log files line-by-line, filtering out specific error levels with zero memory overhead:

```javascript
const fs = require('fs');
const readline = require('readline');

async function parseLargeLogFile(inputPath, outputPath) {
    // 1. Establish the high-throughput read and write streams
    const readStream = fs.createReadStream(inputPath);
    const writeStream = fs.createWriteStream(outputPath);

    // 2. Wrap read stream in readline interface to parse line-by-line
    const rl = readline.createInterface({
        input: readStream,
        crlfDelay: Infinity
    });

    for await (const line of rl) {
        // 3. Filter lines containing "ERROR"
        if (line.includes('ERROR')) {
            const hasBufferCapacity = writeStream.write(line + '\n');
            
            // 4. Manual backpressure management: wait for drain if buffer is full!
            if (!hasBufferCapacity) {
                await new Promise(resolve => writeStream.once('drain', resolve));
            }
        }
    }

    writeStream.end();
    console.log("Log filtering complete!");
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is the primary difference between `fs.readFile` and `fs.readFileSync`?**  
**A:** 
* `fs.readFile` is asynchronous; it offloads disk operations to background C++ threads and executes a callback, keeping the main JS thread free.
* `fs.readFileSync` is synchronous and blocking; it halts all JavaScript execution on the main thread until the file is fully read.

**Q2: What is a File Descriptor in Node.js?**  
**A:** An integer assigned by the operating system to represent an open file session, acting as a pointer reference for read/write APIs.

**Q3: Why should you avoid using `fs.readFile` to read a massive 5GB file?**  
**A:** Because `fs.readFile` loads the entire file into V8's heap memory at once, which will instantly exceed V8's memory limits and crash the process with an Out of Memory (OOM) or RangeError exception.

**Q4: What is the purpose of `fs.createReadStream`?**  
**A:** It creates a readable stream that reads files in small, continuous chunks (default: 64KB) sequentially, keeping memory consumption low.

**Q5: How do you listen for incoming data packets in a readable stream?**  
**A:** By listening to the `'data'` event on the stream instance (e.g., `stream.on('data', callback)`).

---

### 🟡 Intermediate (5)

**Q6: What is "Backpressure" in Node.js streams, and why is it critical for memory safety?**  
**A:** Backpressure occurs when the data reading speed exceeds the data writing speed. Without backpressure controls, the temporary buffer array would grow infinitely, bloating RAM and crashing the server. Backpressure halts the read stream dynamically when the write buffer is full, resuming only when the write buffer drains.

**Q7: Explain the difference between `fs.watch()` and `fs.watchFile()`. Which is preferred in production?**  
**A:** 
* `fs.watch()` uses **native OS kernel events** (`inotify` in Linux, `FSEvents` in macOS) to monitor changes. It is highly efficient and consumes near-zero CPU resources.
* `fs.watchFile()` periodically wakes up the CPU to **poll the file statistics** (`fs.stat`). It is highly inefficient and blocks disk I/O.
* *Production choice:* Always prefer `fs.watch()`.

**Q8: Explain what occurs to the operating system resources if you open multiple files via `fs.open()` but forget to call `fs.close()`.**  
**A:** The operating system maintains a hard limit on the number of concurrent open files per process. If you forget to close files, the file descriptors remain active. Once the threshold is exceeded, subsequent file operations fail instantly, throwing the terminal error `EMFILE: too many open files`.

**Q9: Explain how `fs.promises.readFile` manages asynchronous execution compared to standard callback-based `fs.readFile`.**  
**A:** Both offload the file read to Libuv's ThreadPool. However:
* Callback-based `fs.readFile` executes a plain JS callback in the event loop Poll phase.
* `fs.promises.readFile` returns a native JS `Promise` object. When the C++ read completes, the engine resolves the promise, pushing its `.then` or `await` handlers to the high-priority **Microtask Queue** to execute immediately.

**Q10: Why does `fs.createReadStream('file.txt', { highWaterMark: 1024 })` differ from standard stream creation?**  
**A:** It overrides the default chunk size. Standard readable streams read files in **64KB** chunks. Setting `highWaterMark: 1024` restricts the chunk size to exactly **1KB** (1024 bytes) per read loop cycle.

---

### 🔴 Advanced (5)

**Q11: Walk through the exact C++ memory allocations and Libuv ThreadPool thread context switches that occur when an asynchronous file read completes.**  
**A:** 
1. **Completion**: The Libuv worker thread finishes reading the file chunk into an external C++ buffer.
2. **Interrupt**: The worker thread posts a message to Libuv's event loop handler.
3. **Context Switch**: The operating system schedules a thread context switch back to Node's main JS execution thread.
4. **V8 Transition**: During the Poll phase, the loop reads the completed message.
5. **ArrayBuffer Alloc**: Node wraps the raw C++ buffer address in a V8 `v8::ArrayBuffer` object.
6. **Stack Push**: V8 allocates a stack frame and executes the JS callback, passing the buffer.
7. **GC Sweep**: When the JS buffer is dropped, V8's GC sweeps the wrapper, while C++ calls `free()` on the external memory address.

**Q12: Detail the exact event sequence and state changes that occur inside the `.pipe()` method when backpressure is triggered and subsequently cleared.**  
**A:** 
1. `readStream` emits `'data'` chunks, writing them to `writeStream`.
2. `writeStream.write(chunk)` buffer exceeds the configured `highWaterMark` limit, returning `false`.
3. The `.pipe()` listener intercepts the `false` return and immediately calls `readStream.pause()`.
4. `readStream` stops emitting `'data'` events. The C++ reader halts disk reads.
5. The `writeStream` continues writing its buffer to disk asynchronously.
6. Once the write buffer is empty, `writeStream` emits the **`'drain'` event**.
7. The `.pipe()` listener catches the `'drain'` event and calls `readStream.resume()`, restarting the data pipeline.

**Q13: How can you write a custom Readable Stream in Node.js from scratch? Write an example and explain the `_read()` interface.**  
**A:** Extend the base `Readable` class and implement the internal `_read(size)` method:
```javascript
const { Readable } = require('stream');

class CounterStream extends Readable {
    constructor(max = 5) {
        super();
        this.max = max;
        this.current = 0;
    }
    // The internal method called by the stream consumer
    _read(size) {
        if (this.current < this.max) {
            this.push(String(++this.current)); // Push data to buffer
        } else {
            this.push(null); // Signal EOF (End of File)
        }
    }
}
```

**Q14: Explain the differences in low-level OS file systems APIs Node uses under the hood between `fs.watch` in Linux vs. macOS vs. Windows.**  
**A:** Libuv abstracts different OS file systems event engines:
* **Linux**: Uses **`inotify`** API. It allows registering watches on file descriptors and reads event buffers via system calls.
* **macOS**: Uses **`FSEvents`** API. It registers watch points inside the macOS kernel file system events daemon.
* **Windows**: Uses **`ReadDirectoryChangesW`** asynchronous system call, monitoring directory handles using completion ports (IOCP).

**Q15: Explain how a memory leak can occur in a stream pipeline if you manually pipe streams without handling the `'error'` events, and how `stream.pipeline` resolves this in production.**  
**A:** 
* *Manual Pipe:* If you write `readStream.pipe(writeStream)`, and an error occurs inside `readStream` (e.g., file permissions crash), the stream triggers an `'error'` event. If un-handled, the pipe fails to close the streams, leaving file descriptors open and memory buffers allocated on the heap, causing a memory leak.
* *Solution:* Use `stream.pipeline` which automatically destroys all connected streams and frees all resources if any single stream in the pipe throws an error:
```javascript
const { pipeline } = require('stream');
pipeline(readStream, writeStream, (err) => {
    if (err) console.error("Pipeline failed securely, resources cleaned up:", err);
});
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact sequence of V8 heap allocations, C++ binding tables, and OS-level file locking events that occur when a Node.js process attempts to write to a log file concurrently using two separate `fs.createWriteStream` instances.**  
**A:** 
1. **Instantiation**: Node allocates two separate C++ `WriteStream` structures and opens the target file.
2. **File Descriptor Allocation**: The OS allocates two unique file descriptors (`fd: 3` and `fd: 4`) pointing to the same file.
3. **Concurrent Write**: Both streams call `.write()`.
4. **V8 Scope**: V8 allocates JS wrappers and passes buffers to C++ bindings.
5. **C++ Locking**: Libuv worker threads execute `write(3, buf1)` and `write(4, buf2)`.
6. **OS Kernel Lock**: The operating system kernel manages the file's **inode write lock**.
7. **Write Race**: If both threads write concurrently:
   - One thread acquires the inode write lock first and writes its chunk.
   - The second thread waits. Once the lock is released, it writes its chunk.
   - **Data Corruption**: Because the streams have separate file descriptor pointers, they write to the file without coordinating offsets, leading to **interleaved data corruption** where log lines are split and scrambled mid-write.

**Q17: Explain the step-by-step C++ V8 engine states during a "Buffer Copy-on-Write (COW)" optimization when reading a large file using `fs.readFile` with a slice operation. What happens inside the memory allocation table when you call `buf.slice(0, 100)`?**  
**A:** 
1. **Original Buffer**: `fs.readFile` allocates a C++ backing store buffer of size 10MB, exposed as V8 `v8::ArrayBuffer`.
2. **Slice Invocation**: The script executes `const slice = buf.slice(0, 100)`.
3. **Reference Sharing**: Inside V8/Node bindings, the engine **does not allocate new C++ memory** or copy the 100 bytes of binary data.
4. **Pointer Offset Mapping**: V8 allocates a new, lightweight `v8::TypedArray` or `v8::DataView` wrapper object on the heap.
5. **Memory Mapping**: This new wrapper's properties are configured:
   - Sets its internal memory pointer to the **exact same memory address** of the original 10MB buffer.
   - Sets its byte offset parameter to `0`.
   - Sets its byte length parameter to `100`.
6. **CoW Mechanics**: Both the original buffer and the slice reference the identical C++ memory sector. Modifying bytes inside `slice` instantly mutates the original `buf`, preventing slow memory allocation copying at the cost of shared mutation side-effects.
