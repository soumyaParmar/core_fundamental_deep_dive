# 📌 12.2 — Memory Leaks
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** A memory leak occurs when your code allocates space on the computer's memory (heap) to store an object, but then forgets to clean it up when it is no longer needed. Because the garbage collector (the browser's automatic memory cleaner) only deletes objects that are completely unreachable, keeping even a single, forgotten reference to an object in a global variable or active event listener traps that object—and potentially a massive tree of connected data—in memory forever, eventually slowing down the page and crashing your app.

**Technical Definition:** A memory leak in JavaScript is the retention of heap-allocated objects that are no longer required for active program execution, but remain strongly connected to a **Garbage Collection (GC) Root** (global variables, active call stacks, DOM trees) in the engine's reachability graph. This prevents the Mark-Sweep collector from sweeping the memory slots, leading to continuous heap expansion and ultimate **Out of Memory (OOM)** process termination.

**One-line mental model:**
> *Memory leaks are caused by dead heap objects remaining strongly connected to active GC Roots, trapping them in memory forever.*

---

## ② DRILL DOWN

### 🔹 A. How V8's Garbage Collector Works

JavaScript uses the **Mark-and-Sweep** algorithm to manage memory automatically:

```
  [ GC ROOT ] (Globals, active Stack Frames)
       │
       ▼ (Follow references)
  [ Reachable Object ] ──► [ Reachable Object ] (Marked as ALIVE)
  
  [ Unreachable Object ] ──► [ Unreachable Object ] (Swiped & DELETED!)
```

1. **Mark Phase**: The GC starts at the **GC Roots** (the global `window` or `process` object, and all active variables in the current Call Stack) and recursively traverses all references. Every object it reaches is marked as "alive."
2. **Sweep Phase**: The GC sweeps the entire heap. Any object that was **not marked as alive** (unreachable) is swept, and its memory is freed.

---

### 🔹 B. Common Memory Leak Patterns

To write stable production applications, you must master the five primary memory leak patterns:

#### 1. Accidental Global Variables
If you omit `let`, `const`, or `var` in sloppy mode, JavaScript automatically attaches the variable to the global `window` or `global` object:
```javascript
function leak() {
    leakedData = new Array(1000000); // Accidental Global!
    // this.leakedData = ...; // Same leak if function is called without 'new'
}
leak(); 
// The array remains in memory forever because 'window.leakedData' is a permanent GC Root!
```

#### 2. Uncleared Timers (`setInterval` / `setTimeout`)
Timers that run continuously keep all variables captured in their closures alive on the heap:
```javascript
function startTracker() {
    const hugeData = new Array(1000000);
    
    // The interval callback captures 'hugeData' in its closure!
    const id = setInterval(() => {
        console.log("Still running...");
    }, 1000);
    
    // Even if we exit the function, hugeData CANNOT be collected as long as the interval runs!
    // Solution: Always call clearInterval(id) when finished!
}
```

#### 3. Detached DOM Nodes
A **Detached DOM Node** occurs when a DOM element is removed from the visible webpage (`remove()`), but a JavaScript variable still retains a reference to it:
```javascript
let button = document.createElement('button');
document.body.appendChild(button);

// Remove from page:
button.remove();

// ⚠️ THE LEAK:
// The physical button node is deleted from the screen, but the variable 'button' still references it.
// The button and ALL its nested children remain allocated in memory!
// Solution: Set button = null;
```

#### 4. Out of Scope Closures (The Meteor Memory Leak)
In V8, closures defined in the same parent scope **share the same internal Context object**. This can cause unexpected memory leaks:
```javascript
let leakInstance = null;

function replaceThing() {
    const originalThing = leakInstance;
    
    // ⚠️ THE LEAK CLOSURE:
    // This unused closure captures 'originalThing'
    const unused = () => {
        if (originalThing) console.log("hi");
    };
    
    leakInstance = {
        longStr: new Array(1000000).join('*'),
        // This closure is active. Because it shares the SAME context as 'unused',
        // it indirectly traps 'originalThing' in memory!
        someMethod() {} 
    };
}
setInterval(replaceThing, 1000); // Memory expands infinitely until crash!
```

#### 5. Forgotten Event Listeners
Attaching events to global objects (like `window` or `document`) and forgetting to remove them when local components are destroyed:
```javascript
class ResizableComponent {
    constructor() {
        this.handler = () => this.resize();
        window.addEventListener('resize', this.handler);
    }
    destroy() {
        // ⚠️ THE LEAK:
        // If we destroy the component but forget this line:
        // window.removeEventListener('resize', this.handler);
        // The global window object keeps the handler (and the entire component instance) alive!
    }
}
```

## ③ WHY IT EXISTS

1. **Abstraction Trade-off**: JavaScript abstracts manual memory management (like C++ `malloc` and `free`) away, replacing it with automatic garbage collection to make development faster.
2. **The Reachability Principle**: The GC cannot understand developer *intent*. It only knows if an object is *reachable*. If a developer leaves a reference active by mistake, the GC must assume the object is still alive, leading to leaks.

---

## ④ INTERNAL WORKING: GENERATIONAL SPECS & WRITE BARRIERS

### V8 Generational Garbage Collection Spaces

To optimize execution speeds, V8 divides heap memory into distinct spaces based on object lifespans (the **Generational Hypothesis**: most objects die young):

```
 ┌────────────────────────────────────────────────────────┐
 │                    V8 HEAP MEMORY                      │
 ├────────────────────────────┬───────────────────────────┤
 │ NEW SPACE (Young Gen)      │ OLD SPACE (Old Gen)       │
 │ - Semi-Space Copying       │ - Mark-Sweep-Compact      │
 │ - Size: ~1MB to 64MB       │ - Size: Gigabytes         │
 │ - Very frequent, fast (<2ms)│ - Rare, slow (20ms-100ms+)│
 └────────────────────────────┴───────────────────────────┘
```

1. **New Space (Young Generation)**:
   - Reserved for short-lived, newly allocated objects.
   - Divided into two equal halves: **From-Space** and **To-Space**.
   - During a **Scavenger** pass, V8 copies alive objects from From-Space to To-Space, clearing all dead ones instantly.
   - Objects that survive multiple Scavenger passes are promoted to **Old Space**.

2. **Old Space (Old Generation)**:
   - Stores long-lived, promoted objects.
   - Managed via the slow **Mark-Sweep-Compact** algorithm.

---

### The Write Barrier

If the Scavenger GC only scans the New Space heap, what happens if an object in Old Space points to a newly allocated object in New Space?
* To prevent scanning the entire massive Old Space (which would take 50ms), V8 uses a **Write Barrier**.
* Every time you write an object pointer (`oldObj.newRef = newObj`), V8 runs a low-level C++ hook that checks if you linked an Old-to-New reference.
* If yes, V8 registers the address in a specialized table called the **Remembered Set**.
* During New Space GC, the Scavenger only scans the New Space and this small Remembered Set, maintaining instant execution speeds.

---

## ⑤ ADVANTAGES

1. **Automatic Safety**: Developers are spared from writing manual memory release code, preventing fatal memory corruption crashes.
2. **Generational Optimizations**: Splitting the heap ensures short-lived allocations are reclaimed in microseconds.
3. **Lazy Memory Compacting**: Minimizes page fragmentation, keeping memory access sequential.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Hidden Context Traps**: Shared parent scopes in closures trap variables silently, causing severe memory leaks.
2. **Stop-the-World Latency**: Major Mark-Sweep passes pause the main JS thread, creating visible visual stutter in real-time games or sockets.
3. **No Automatic Destructors**: Lacks explicit class destroy lifecycle hooks, forcing manual removal of listeners and pointers.

---

## ⑦ PRODUCTION USAGE

### 🛠️ Memory-Safe Weak Cache Utility (WeakRef)

To store heavy calculated data without preventing garbage collection, leverage **`WeakRef`** and **`FinalizationRegistry`** (ES2021) to build a self-cleaning memory cache:

```javascript
class WeakMemoryCache {
    constructor() {
        this.cache = new Map();
        
        // 1. Register a cleanup hook called when V8 sweeps cached objects
        this.registry = new FinalizationRegistry((key) => {
            console.log(`[GC CLEANUP]: Swept expired key: ${key}`);
            this.cache.delete(key); // Remove dead weak reference from Map
        });
    }

    set(key, objectPayload) {
        // 2. Wrap object in a Weak Reference (does NOT prevent GC!)
        this.cache.set(key, new WeakRef(objectPayload));
        
        // 3. Register object for GC monitoring
        this.registry.register(objectPayload, key);
    }

    get(key) {
        const ref = this.cache.get(key);
        if (!ref) return null;

        const obj = ref.deref(); // Attempt to read the object
        if (!obj) {
            // Object was garbage collected!
            this.cache.delete(key);
            return null;
        }
        return obj;
    }
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is a Memory Leak in JavaScript?**  
**A:** An architectural bug where heap-allocated objects are no longer required by the application but remain strongly referenced, preventing the Garbage Collector from reclaiming their memory.

**Q2: What is the primary Garbage Collection algorithm used by V8?**  
**A:** The **Mark-and-Sweep** algorithm.

**Q3: What are "GC Roots" in JavaScript?**  
**A:** The base entry points of the memory reachability graph, including the global object (`window` or `global`), active variables in the Call Stack, and active DOM trees.

**Q4: What is a "Detached DOM Node"?**  
**A:** A DOM element that has been physically removed from the webpage's visible DOM tree (`remove()`), but remains referenced inside a JavaScript variable, preventing its memory from being freed.

**Q5: Why does leaving an interval timer (`setInterval`) active cause memory leaks?**  
**A:** Because the active interval callback remains registered in the event system, and its closure context retains strong references to all outer variables used by the callback, preventing their garbage collection.

---

### 🟡 Intermediate (5)

**Q6: Explain how the Mark-and-Sweep algorithm operates to identify dead memory.**  
**A:** 
1. **Mark Phase**: The GC starts at the global and active stack GC Roots, crawls the reference tree recursively, and marks every reachable object as "alive."
2. **Sweep Phase**: The GC sweeps the entire heap. Any object that was not marked as alive during the marking phase is identified as dead, and its memory space is reclaimed.

**Q7: Explain the V8 generational memory spaces: "New Space" vs. "Old Space".**  
**A:** 
* **New Space (Young Generation)**: Stores short-lived, newly allocated objects. It uses a fast semi-space copying collector (Scavenger) that runs very frequently.
* **Old Space (Old Generation)**: Stores long-lived objects promoted from New Space. It uses the slower Mark-Sweep-Compact algorithm.

**Q8: Explain what a "Write Barrier" is and why it is critical for New Space Scavenger GC performance.**  
**A:** A Write Barrier is a compiler hook that executes whenever an object pointer is updated. If an object in Old Space is modified to point to a new object in New Space, the Write Barrier registers the address in the **Remembered Set**. This allows the Scavenger GC to verify cross-generational references quickly by scanning only the New Space and the Remembered Set, bypassing the massive Old Space, keeping pauses under 2ms.

**Q9: What is the "Meteor Memory Leak" pattern in closures? How does sharing contexts cause leaks?**  
**A:** In V8, closures declared in the same parent function scope share a **single, unified parent Context object**.
If one unused closure captures a large variable (e.g., `originalThing`), and another active closure is exported (e.g., `someMethod`), the active closure's reference to the shared context **keeps the unused large variable alive on the heap**, creating a silent memory leak.

**Q10: Explain the purpose of `FinalizationRegistry` introduced in ES2021.**  
**A:** A native API that allows developers to register a callback function that is automatically executed when a target object is garbage collected, facilitating cleanups of associated external resources.

---

### 🔴 Advanced (5)

**Q11: Walk through the exact C++ V8 engine states, mark-sweep sweeps, and pointers compacting operations that occur during a Major Garbage Collection pass.**  
**A:** 
1. **GC Trigger**: Old Space allocation reaches the watermark threshold.
2. **Thread Pause**: V8 pauses the JS main thread (Stop-the-World).
3. **Tri-Color Marking**: V8 spawns concurrent threads running a **Tri-color marking algorithm**:
   - **White**: Unvisited objects (candidates for sweep).
   - **Grey**: Visited, but children not scanned yet.
   - **Black**: Visited and all children scanned (alive).
4. **Sweeping**: Unmarked (White) objects are added to Free Lists.
5. **Compacting**: To prevent page fragmentation, V8 moves all Black objects to a contiguous memory block on the page, and walks the entire heap to **update every single reference pointer** to the new memory addresses, and resumes execution.

**Q12: Detail the memory graph structure inside a V8 heap snapshot when a closure creates a memory leak. How do you trace the "Retaining Path" in Chrome DevTools?**  
**A:** In a Chrome DevTools heap snapshot:
* Select the suspected object and inspect the **Retainers** panel at the bottom.
* The panel displays the **Retaining Path**: a chain of reference pointers walking backward from the object to the GC Root.
* For a closure leak, you will see the object is referenced by `context` (the V8 Heap Context). The context is referenced by a `JSFunction` (the closure callback). The function is referenced by an array or global listener, mapping the exact leak path.

**Q13: How can you write a custom memory leak detector that leverages `FinalizationRegistry` and `WeakRef` to programmatically verify if a specific UI Component was successfully garbage collected after being unmounted?**  
**A:** 
```javascript
const unmountRegistry = new FinalizationRegistry((id) => {
    console.log(`[MEM WATCHER]: Success! Component [${id}] was garbage collected.`);
});

class ComponentTracker {
    static track(componentInstance, uniqueId) {
        // Register instance for GC observation
        unmountRegistry.register(componentInstance, uniqueId);
        
        // Keep a weak reference to check state manually
        this.weakRefs = this.weakRefs || new Map();
        this.weakRefs.set(uniqueId, new WeakRef(componentInstance));
    }
    
    static audit() {
        for (const [id, ref] of this.weakRefs.entries()) {
            if (ref.deref() !== undefined) {
                console.warn(`[WARNING]: Component [${id}] is STILL alive on the heap! Possible Memory Leak.`);
            }
        }
    }
}
```

**Q14: Explain the differences in memory allocation and GC copying overhead between: `new Array(1000000)` storing primitives vs. storing objects.**  
**A:** 
* **Primitives (Numbers/Booleans)**: Contiguous allocation of raw, scalar values in a single memory block.
  - *GC Overhead*: Extremely low. The GC treats it as a single chunk. It does not need to traverse references, and copying it during Scavenger passes is a simple fast memory block copy (`memcpy`).
* **Objects**: An array of **pointers** referencing separate object memory blocks scattered across the heap.
  - *GC Overhead*: Extremely high. The GC must recursively traverse and mark all 1,000,000 individual object heap allocations. During copying, it must copy 1,000,000 individual blocks and update all pointers, creating major CPU bottlenecks.

**Q15: Explain how a memory leak can occur when using a `WeakMap` if you use an object as a key, but the *value* associated with that key retains a strong reference back to the key.**  
**A:** This creates a **Cyclic Reference Memory Leak**.
```javascript
const map = new WeakMap();
let keyObj = { name: "Key" };
const valObj = { owner: keyObj }; // Strong reference back to keyObj!

map.set(keyObj, valObj);
keyObj = null; // Drop external reference
```
*Why it leaks:* Although `WeakMap` holds a weak reference to the key `keyObj`, the value `valObj` contains a **strong reference** pointing directly to `keyObj`. Because `valObj` is strongly kept alive inside the WeakMap, it keeps `keyObj` strongly reachable. V8's GC marks both as reachable, and the ephemeron sweeping phase is blocked, permanently leaking both objects in memory.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact sequence of V8 heap space allocations, C++ memory operations, and GC sweeps when a JavaScript script runs `window.myLeak = new Array(1000000)` inside a browser. How does the memory expand physically on the host computer?**  
**A:** 
1. **Allocation**: V8 evaluates the array creation. Since it is a massive array of 1,000,000 slots, V8 allocates a contiguous memory backing store on the heap (size: ~8MB).
2. **Promotion**: Since it is allocated on `window` (a global GC Root), the array is immediately promoted to **Old Space** memory to prevent New Space saturation.
3. **RAM Expansion**: V8's memory allocator requests virtual memory pages from the operating system kernel (using `mmap` or `VirtualAlloc`). The OS assigns physical RAM sectors to Node's heap space limit.
4. **Mark Phase**: During GC, the marking thread traces from `window` to `myLeak`, marking the 8MB backing store as alive (Black).
5. **Compaction Block**: Because it is strongly referenced, the sweep phase bypasses it. The memory remains occupied.
6. **Saturation**: If the script does this repeatedly, V8's heap size continuously expands, eventually hitting the hardcoded **heap size limit** (typically 1.4GB - 4GB). Once saturated, V8 halts execution, throws an `Out of Memory` exception, and crashes the process.

**Q17: Explain the step-by-step C++ V8 engine states, memory address swaps, and CPU registry states during a "Generational Promotion (Tenuring)" event when a long-lived object is migrated from New Space to Old Space.**  
**A:** 
1. **Survival Check**: During a New Space Scavenger pass, the GC scans an object `obj` in From-Space. It detects that `obj` has survived **two GC cycles (age threshold exceeded)**.
2. **Old Space Allocation**: The GC pauses and requests a free memory block of matching size from the **Old Space Free Lists**.
3. **Copying (Tenuring)**: The GC executes a fast memory copy (`memcpy`), copying `obj`'s raw binary payload from the New Space From-Space directly to the newly allocated address in Old Space.
4. **Forwarding Pointer**: Inside the old New Space From-Space memory address of `obj`, the GC writes a special **Forwarding Pointer** pointing directly to the new Old Space address.
5. **Reference Updating**: The GC walks the remaining heap. Any other object referencing `obj`'s old New Space address is updated to point to the new Old Space address using the Forwarding Pointer.
6. **Space Evacuation**: Once all references are updated, the New Space From-Space is swept, freeing the old memory slot, completing a generational promotion.
