# 📌 14.1 — Creational Patterns
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Creational patterns are design strategies that handle how objects are created in your code. Instead of instantiating objects directly using `new ClassName()` all over your codebase—which makes your code rigid and hard to test—you use creational patterns to abstract and control the instantiation process. This allows you to guarantee that only one instance of a class ever exists (Singleton), dynamically build complex objects step-by-step (Builder), or create objects cleanly through a central manufacturing system (Factory).

**Technical Definition:** Creational design patterns abstract the instantiation process, decoupling a system from how its objects are created, composed, and represented. In JavaScript, these patterns leverage the dynamic prototype model, dynamic closures, and structural descriptors to enforce lifecycle controls: **Singleton** (restricting allocation to a single instance), **Factory** (polymorphic class factories), **Prototype** (zero-copy object duplication via `Object.create`), and **Builder** (method-chaining step assembly).

**One-line mental model:**
> *Creational patterns abstract and control object instantiation to make systems decoupled, easily testable, and memory-efficient.*

## ② DRILL DOWN

### 🔹 A. The Singleton Pattern (Guaranteed Single Instance)

A **Singleton** ensures a class has only one instance and provides a global access point to it. In JavaScript, you can implement a highly robust, secure Singleton using closures or private modules:

```javascript
// DatabaseRegistry (Singleton ES6 Module)
const _instance = Symbol("instance");

class DatabaseRegistry {
    constructor() {
        // Prevent direct instantiation (must use getInstance()!)
        const target = new.target;
        if (target !== DatabaseRegistry && target !== undefined) {
            throw new Error("Cannot instantiate DatabaseRegistry directly. Use getInstance().");
        }
        this.connections = new Map();
    }

    static getInstance() {
        // Check if global singleton exists, if not, allocate it
        if (!this[_instance]) {
            this[_instance] = new DatabaseRegistry();
            Object.freeze(this[_instance]); // Freeze to prevent runtime overrides!
        }
        return this[_instance];
    }
    
    register(name, connection) {
        this.connections.set(name, connection);
    }
}

// Export the singleton instance
module.exports = DatabaseRegistry.getInstance();
```

---

### 🔹 B. The Factory Pattern (Abstracting Construction)

A **Factory** abstracts the instantiation details, allowing subclasses or dynamic logic to determine exactly which object class to build:

```javascript
class RegularUser { constructor(name) { this.name = name; this.role = "user"; } }
class AdminUser { constructor(name) { this.name = name; this.role = "admin"; } }

// The Factory:
class UserFactory {
    static createUser(type, name) {
        switch (type.toLowerCase()) {
            case 'admin':
                return new AdminUser(name);
            case 'user':
            default:
                return new RegularUser(name);
        }
    }
}

const admin = UserFactory.createUser('admin', 'Alice');
console.log(admin.role); // "admin"
```

---

### 🔹 C. The Prototype Pattern (Zero-Copy Cloning)

Instead of running a heavy constructor function to allocate properties, the **Prototype Pattern** clones an existing "blueprint" object directly using `Object.create()`, establishing a direct prototype delegation link which is highly memory-efficient:

```javascript
const carPrototype = {
    wheels: 4,
    drive() {
        return `Driving on ${this.wheels} wheels.`;
    }
};

// Clone the prototype directly:
const myCar = Object.create(carPrototype);
myCar.color = "red"; // Customize properties on the clone

console.log(myCar.drive()); // "Driving on 4 wheels." (Delegates to carPrototype)
```

---

### 🔹 D. The Builder Pattern (Step-by-Step Assembly)

The **Builder Pattern** separates the construction of a complex object from its representation. It uses method chaining to build objects step-by-step:

```javascript
class Request {
    constructor() {
        this.url = "";
        this.method = "GET";
        this.headers = {};
    }
}

class RequestBuilder {
    constructor() {
        this.request = new Request();
    }

    setUrl(url) {
        this.request.url = url;
        return this; // Return 'this' to enable method chaining!
    }

    setMethod(method) {
        this.request.method = method.toUpperCase();
        return this;
    }

    setHeader(key, value) {
        this.request.headers[key] = value;
        return this;
    }

    build() {
        return this.request; // Return the final constructed object
    }
}

// Usage (Highly readable method chaining!):
const apiRequest = new RequestBuilder()
    .setUrl("https://api.com/users")
    .setMethod("POST")
    .setHeader("Content-Type", "application/json")
    .build();
```

## ③ WHY IT EXISTS

1. **Decoupled Architecture**: Direct class coupling (`new Class()`) spreads class naming throughout your application files. If you rename a class or change its constructor arguments later, you must edit dozens of files. Creational patterns centralize this, keeping code maintainable.
2. **Memory and Allocation Optimization**: Patterns like Prototype or Singleton prevent redundant heap memory allocations, maximizing V8 engine speed during startup loops.

---

## ④ INTERNAL WORKING: V8 SHAPE OPTIMIZATIONS IN FACTORIES

### Hidden Class Stabilization in Factory Functions

When you use a **Factory Pattern** or **Builder Pattern** that returns object literals:
```javascript
// BAD: Dynamic property factory (causes shape branching in V8!)
function createPoint(x, y, label = null) {
    const point = { x, y };
    if (label) {
        point.label = label; // Transitions to different Hidden Class!
    }
    return point;
}
```
* **The Problem**: Objects returned by the factory have **different hidden classes (Maps)** depending on whether the optional `label` is provided. When other functions process these points, the polymorphic shapes invalidate JIT Inline Caches, slowing down execution.
* **The Solution**: Ensure the factory returns a completely uniform, pre-initialized structure (or instantiates a class constructor) to guarantee that all output objects share a **single, monomorphic Hidden Class Map**.

---

## ⑤ ADVANTAGES

1. **High Maintainability**: Centralizing instantiation logic makes refactoring constructors or class names simple.
2. **Memory Efficiency**: Prototype cloning and Singleton pooling bypass heavy runtime CPU allocations.
3. **Decoupled Testability**: Easily inject mock objects or factory stubs during unit tests, bypassing database or API connections.

---

## ⑥ DISADVANTAGES / TRAPS

1. **The Global State Trap (Singleton)**: Singletons act as global variables. If you store volatile mutable states inside a Singleton, it becomes highly difficult to track which part of your application modified the state, leading to hard-to-trace bugs.
2. **Increased Boilerplate**: Builders and Abstract Factories introduce multiple auxiliary classes, slightly increasing code complexity for small applications.
3. **Constructor Coupling in Factories**: If a factory is poorly written, it remains tightly coupled to class constructors internally, violating the Dependency Inversion Principle.

## ⑦ PRODUCTION USAGE

### 🛠️ Production-Ready Secure Microservice Factory

Use the Factory and Builder patterns to build a highly robust, secure API microservice registry that constructs authenticated client instances dynamically based on config:

```javascript
class ApiService {
    constructor(builder) {
        this.url = builder.url;
        this.timeout = builder.timeout;
        this.token = builder.token;
    }
}

// 1. The Builder
class ApiServiceBuilder {
    setUrl(url) { this.url = url; return this; }
    setTimeout(ms) { this.timeout = ms; return this; }
    setToken(token) { this.token = token; return this; }
    build() { return new ApiService(this); }
}

// 2. The Factory (Decouples client creation from calling files)
class ServiceFactory {
    static getClient(serviceType, token) {
        const builder = new ApiServiceBuilder()
            .setTimeout(5000)
            .setToken(token);

        switch (serviceType.toLowerCase()) {
            case 'payment':
                return builder.setUrl("https://api.payment.com").build();
            case 'auth':
                return builder.setUrl("https://api.auth.com").build();
            default:
                throw new Error(`Unsupported service type: ${serviceType}`);
        }
    }
}

// Usage inside routes:
const paymentClient = ServiceFactory.getClient('payment', 'JWT_TOKEN_123');
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is a Creational Design Pattern?**  
**A:** A design pattern that abstracts and controls the object instantiation process, decoupling the system from how its objects are created.

**Q2: What is the Singleton Pattern?**  
**A:** A pattern that guarantees a class has only one instance in memory and provides a single, global access point to it.

**Q3: What is the Factory Pattern?**  
**A:** A pattern that abstracts constructor calls, providing a central method or class to dynamically manufacture and return objects without exposing the instantiation logic to the caller.

**Q4: How does the Prototype Pattern duplicate objects in JavaScript?**  
**A:** By using `Object.create(prototype)`, which creates a new empty object and links its internal `[[Prototype]]` slot to the prototype object, avoiding running constructor code.

**Q5: What is the Builder Pattern and when should you use it?**  
**A:** A pattern that constructs a complex object step-by-step using method chaining. You should use it when an object requires a high number of optional parameters or a multi-stage assembly process.

---

### 🟡 Intermediate (5)

**Q6: Why is `Object.freeze()` highly critical when implementing a Singleton in JavaScript?**  
**A:** JavaScript objects are mutable by default. If you do not freeze the Singleton instance (e.g., `Object.freeze(instance)`), any external file can dynamically delete, overwrite, or mutate the Singleton's properties or methods (e.g., `dbSingleton.query = null`), breaking the entire application.

**Q7: Explain the concept of "Method Chaining" inside the Builder Pattern. Why do builder methods return `this`?**  
**A:** Method chaining allows calling multiple methods sequentially in a single statement (e.g., `.setUrl().setMethod()`). To enable this, each setter method inside the Builder class must explicitly return **`this`** (the active builder instance), passing the object context along to the next method in the chain.

**Q8: Explain the performance difference in V8 when creating an object via `Object.create(proto)` vs. `new Constructor()`.**  
**A:** 
* `new Constructor()` runs V8's constructor pipeline, parses arguments, and allocates in-object properties. If the constructor has complex logic, it consumes substantial CPU cycles.
* `Object.create(proto)` bypasses the constructor completely. It allocates a plain `JSObject` and writes the `[[Prototype]]` pointer directly, executing near-instantly with zero initialization CPU cost.

**Q9: How do you prevent developers from executing a `new` call directly on a class that is intended to be a Singleton?**  
**A:** Use **`new.target`** inside the constructor. If `new.target` points to the class itself, throw a hard exception, forcing developers to use the static `getInstance()` method instead.

**Q10: What is the difference between a Simple Factory and an Abstract Factory?**  
**A:** 
* A **Simple Factory** returns instances of different classes depending on the input string passed to a single method.
* An **Abstract Factory** is a factory that returns *other factories* (a factory of factories), providing a higher level of abstraction to group families of related factories.

---

### 🔴 Advanced (5)

**Q11: Detail what V8 does to Hidden Class Maps when a Factory function returns object literals with dynamically assigned optional keys. Why does this trigger Polymorphic Inline Cache (PIC) invalidation?**  
**A:** 
1. **Shape Branching**: If a factory returns `{ x, y }` under one branch and `{ x, y, label }` under another, V8 must allocate two different hidden classes (`MapA` and `MapB`).
2. **IC Invalidation**: When downstream optimized functions process these objects (e.g., `total += point.x`), the loop's Inline Cache (IC) encounters a mix of `MapA` and `MapB` shapes.
3. **Polymorphic State**: The IC is forced to transition to a polymorphic or megamorphic state, compiling branch comparison instructions that slow down property read speeds by 10x-50x.
*Resolution:* Always pre-initialize optional keys to `null` or `undefined` inside the factory to maintain a single, monomorphic hidden class Map.

**Q12: Detail the memory layout and pointer mechanics of a Singleton instance stored as a static property on a class vs. a Singleton scoped inside a private module closure.**  
**A:** 
* **Static Class Property**: The Singleton is stored as a property on the constructor function object (`Class.instance`). The reference is globally visible and accessible via the class namespace.
* **Module Closure Scope**: The Singleton is stored in a local variable scoped inside the private module closure (e.g., `let instance = null`). It is completely hidden and protected from external runtime introspection, accessible solely through the exported module API, providing superior security.

**Q13: How can you write a custom Prototype-based cloning engine in JS that supports deep copying of nested objects while preserving correct constructor references and prototype chains?**  
**A:** 
```javascript
function deepClonePrototype(obj, hash = new WeakMap()) {
    if (obj === null || typeof obj !== 'object') return obj;
    if (hash.has(obj)) return hash.get(obj); // Handle cyclic references!

    // 1. Create a new object preserving the original prototype chain
    const clone = Object.create(Object.getPrototypeOf(obj));
    hash.set(obj, clone);

    // 2. Deep copy properties
    for (const key of Reflect.ownKeys(obj)) {
        const desc = Object.getOwnPropertyDescriptor(obj, key);
        if (desc) {
            // Recursively clone values if they are objects
            desc.value = deepClonePrototype(desc.value, hash);
            Object.defineProperty(clone, key, desc);
        }
    }
    return clone;
}
```

**Q14: Explain the differences in V8 heap allocations and GC pressure when launching an application that instantiates 10,000 objects during startup using the Builder Pattern vs. using a raw Object Literal Factory.**  
**A:** 
* **Builder Pattern**: Extremely high startup overhead. Instantiating a builder (`new Builder()`) and chaining methods allocates the temporary `Builder` object, multiple method closure frames, and finally the target object, creating high GC Scavenger churn during startup.
* **Object Literal Factory**: Minimal overhead. The factory directly allocates the target object literal `{}` inside a single execution frame, bypassing all temporary builder wrapper allocations and keeping memory footprint flat.

**Q15: Explain how a Singleton pattern can cause severe memory leaks in multi-tenant SaaS server environments (like Node.js handling multiple user requests), and how to isolate states safely.**  
**A:** In a multi-tenant Node.js server, a Singleton (e.g., a shared `UserPreferencesStore`) is loaded once and persists globally across **all incoming HTTP requests**.
If request A mutates properties on the Singleton (e.g., `store.tenantId = 101`), and request B immediately follows, request B receives the mutated Singleton state, leaking tenant data across requests.
*Resolution:* Do not store request-specific states inside Singletons; leverage **Asynchronous Local Storage (`AsyncLocalStorage`)** to isolate tenant contexts safely across execution flows.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact sequence of V8 heap allocations, hidden class transitions, and GC sweeps that occur when a developer uses a Factory function inside an infinite loop to create objects containing dynamically assigned unique keys.**  
**A:** 
1. **Dynamic Writing**: Inside the loop, the factory runs: `const obj = {}; obj[uniqueId] = true;`.
2. **Hidden Class Explosion**: Because the key name `uniqueId` changes on every iteration, V8 is forced to compile a **brand-new Hidden Class Map** for every single object created.
3. **Memory Saturation**: The V8 metadata heap (which stores Hidden Classes and transition descriptors) rapidly expands, bloating RAM.
4. **GC Failure**: Standard Mark-Sweep GC sweeps the dynamic objects when their references are dropped. However, the internal V8 **Hidden Class transition descriptors and maps** are stored in a centralized compiler cache.
5. **Metadata Leak**: The compiler cache does not sweep these maps quickly, leading to a severe V8 Metadata Memory Leak that eventually triggers an OOM crash.

**Q17: Explain the step-by-step C++ V8 engine states and pointer allocations that occur when `Object.create(proto)` is executed, detailing how the engine links the `[[Prototype]]` slot without executing any JavaScript constructor stubs.**  
**A:** 
1. **C++ Invocation**: V8 executes `Object.create(proto)`. It jumps to the C++ runtime function `Runtime_ObjectCreate`.
2. **Allocation**: V8 allocates a fresh, blank `JSObject` structure on the heap.
3. **Prototype Link**: The engine reads the 64-bit memory address pointer of the C++ `proto` object.
4. **Slot Writing**: V8 writes this `proto` pointer directly into the newly allocated object's internal **`[[Prototype]]` slot (the `__proto__` reference header)**.
5. **Constructor Bypass**: The C++ code exits immediately and returns the object. Because it bypassed the standard `[[Construct]]` bytecode dispatcher, V8 skips all JS constructor compilation, context setups, parameter evaluations, and instruction jumps, completing a zero-overhead object linkage.
