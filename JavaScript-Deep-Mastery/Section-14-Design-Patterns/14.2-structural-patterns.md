# 📌 14.2 — Structural Patterns
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Structural patterns are design blueprints that focus on how classes and objects can be combined, connected, or wrapped to build larger, more powerful structures. Instead of modifying original objects directly—which can break existing features—structural patterns let you combine objects cleanly. For example, you can wrap a slow class to speed it up or secure it (Proxy), create a simple interface to hide a complex system of classes (Facade), translate between two incompatible interfaces (Adapter), or dynamically add new features to an object on the fly (Decorator).

**Technical Definition:** Structural design patterns explain how to assemble objects and classes into larger structures while keeping these structures flexible and efficient. In JavaScript, these patterns exploit prototype inheritance, dynamic key assignment, and meta-programming hooks (`Proxy` and `Reflect`) to modify runtime object topology. They include: **Decorator** (dynamic composition-based behavior extension), **Facade** (simplified unified interface over complex subsystems), **Adapter** (interface translation matching target requirements), and **Proxy** (surrogate access control, virtualization, and interception).

**One-line mental model:**
> *Structural patterns assemble and wrap objects to compose new interfaces and behaviors without breaking existing code.*

## ② DRILL DOWN

### 🔹 A. The Decorator Pattern (Dynamic Behavior Extension)

The **Decorator Pattern** attaches new responsibilities to an object dynamically. In standard object-oriented programming, this is an alternative to subclassing. In JavaScript, since objects are highly dynamic, we can decorate objects at runtime by wrapping them, or use experimental ES decorators to decorate classes and methods.

Here is a pure composition-based implementation of the Decorator pattern:

```javascript
// Base Component
class SimpleCoffee {
    getCost() {
        return 5;
    }
    getDescription() {
        return "Simple coffee";
    }
}

// Decorator 1: Milk
class MilkDecorator {
    constructor(coffee) {
        this.coffee = coffee;
    }
    getCost() {
        return this.coffee.getCost() + 1.5;
    }
    getDescription() {
        return this.coffee.getDescription() + ", milk";
    }
}

// Decorator 2: Sugar
class SugarDecorator {
    constructor(coffee) {
        this.coffee = coffee;
    }
    getCost() {
        return this.coffee.getCost() + 0.5;
    }
    getDescription() {
        return this.coffee.getDescription() + ", sugar";
    }
}

// Usage (Composing behaviors dynamically!):
let myCoffee = new SimpleCoffee();
myCoffee = new MilkDecorator(myCoffee);
myCoffee = new SugarDecorator(myCoffee);

console.log(myCoffee.getDescription()); // "Simple coffee, milk, sugar"
console.log(myCoffee.getCost());        // 7
```

### 🔹 B. The Facade Pattern (Simplified Interface)

A **Facade** provides a simplified, high-level interface to a complex, chaotic body of code, such as a large library, sub-modules, or Web APIs. It hides the messy details behind a clean, easy-to-use API.

```javascript
// Subsystem A: Video Converter
class VideoConverter {
    compress(file) { return `Compressed ${file}`; }
}

// Subsystem B: Audio Synthesizer
class AudioSynthesizer {
    sync(video) { return `Synced audio with ${video}`; }
}

// Subsystem C: Cloud Uploader
class CloudUploader {
    upload(media) { return `Uploaded ${media} to Cloud`; }
}

// The Facade: Hides the complex orchestration from the client!
class MediaTranscoderFacade {
    constructor() {
        this.converter = new VideoConverter();
        this.synthesizer = new AudioSynthesizer();
        this.uploader = new CloudUploader();
    }

    publishVideo(rawFile) {
        const compressed = this.converter.compress(rawFile);
        const synced = this.synthesizer.sync(compressed);
        const url = this.uploader.upload(synced);
        return url;
    }
}

// Usage (Single method call!):
const mediaFacade = new MediaTranscoderFacade();
console.log(mediaFacade.publishVideo("raw_footage.mov"));
// Output: "Uploaded Synced audio with Compressed raw_footage.mov to Cloud"
```

### 🔹 C. The Adapter Pattern (Interface Translator)

The **Adapter Pattern** allows objects with incompatible interfaces to collaborate. It acts as a translator, wrapping an incompatible service/API and transforming its inputs/outputs to match the client's expected interface.

```javascript
// Old/Internal System expects: calculateTotal()
class OldCalculator {
    calculateTotal(price, taxRate) {
        return price + (price * taxRate);
    }
}

// New Third-Party System offers: getBill(amountDetails)
class NewVendorBillingSystem {
    getBill(amountDetails) {
        // Expects: { base: number, taxPercent: number }
        return amountDetails.base * (1 + amountDetails.taxPercent / 100);
    }
}

// Adapter: Translates the new vendor API into the old billing signature
class BillingAdapter {
    constructor(newSystem) {
        this.newSystem = newSystem;
    }

    calculateTotal(price, taxRate) {
        // taxRate is represented as a decimal (e.g. 0.15 for 15%), convert to percent
        return this.newSystem.getBill({
            base: price,
            taxPercent: taxRate * 100
        });
    }
}

// Usage:
const newSystem = new NewVendorBillingSystem();
const adapter = new BillingAdapter(newSystem);

// Client calls calculateTotal() seamlessly through the adapter
const finalPrice = adapter.calculateTotal(100, 0.15);
console.log(finalPrice); // 115
```

### 🔹 D. The Proxy Pattern (Surrogate Control)

A **Proxy** provides a placeholder or surrogate object to control access to the original target object. In JavaScript, this can be done elegantly using the ES6 `Proxy` class, allowing you to intercept and customize fundamental operations (like property lookup, assignment, enumeration, or function invocation).

```javascript
// Target Object
const realUser = {
    username: "john_doe",
    role: "user"
};

// Proxy handler intercepting reads and writes
const accessControlHandler = {
    get(target, prop, receiver) {
        if (prop === "role") {
            // Mask role for standard accessors
            return "hidden_role";
        }
        return Reflect.get(target, prop, receiver);
    },
    set(target, prop, value, receiver) {
        if (prop === "role") {
            throw new Error("Security Violation: Cannot manually modify user roles.");
        }
        return Reflect.set(target, prop, value, receiver);
    }
};

const secureUserProxy = new Proxy(realUser, accessControlHandler);

console.log(secureUserProxy.username); // "john_doe"
console.log(secureUserProxy.role);     // "hidden_role" (Intercepted!)

try {
    secureUserProxy.role = "admin";    // Throws Security Violation!
} catch (e) {
    console.error(e.message);
}
```

## ③ WHY IT EXISTS

1. **Interface Harmonization & Code Migration**: When integrating legacy libraries or third-party packages, interfaces rarely align. Adapters allow developers to switch out backend services without rewriting core domain code.
2. **Adherence to Open/Closed Principle**: Instead of polluting core classes with non-core details (like security logging, performance tracking, or caching), Decorators and Proxies intercept calls cleanly, keeping classes highly cohesive.
3. **Complexity Reduction**: Subsystems tend to grow complex over time. A Facade shields external developers from internal changes, reducing cognitive load and simplifying integration.

## ④ INTERNAL WORKING: V8 INLINE CACHING AND DELEGATION IN WRAPPERS / PROXIES

### ⚠️ The Cost of Indirection: Inline Cache (IC) Deoptimization

When wrapping objects inside **Decorators**, **Adapters**, or **Proxies**, understanding the underlying V8 compiler behavior is crucial.

#### 1. Decorator/Adapter Object Delegation Overhead
If a Decorator wraps a class:
```javascript
class DecoratedClass {
    constructor(base) { this.base = base; }
    action() { return this.base.action(); }
}
```
* **Memory Cost:** Every decorator adds a layer to the heap. If you wrap an object 5 times, a property read requires traversing 5 reference pointers.
* **Inline Cache (IC) Misses:** In V8, when a function repeatedly reads a property on a specific Hidden Class shape, V8 builds a fast-path JIT transition called an Inline Cache. If a function receives wrapped objects with varying decorator configurations (some wrapped in `MilkDecorator`, others in `SugarDecorator`, and others raw), the hidden shapes mismatch. This forces JIT compiler code into a **polymorphic** or **megamorphic** state, disabling JIT optimization and dropping method lookup speeds significantly.

#### 2. ES6 Proxy Interception Cost in JIT
An ES6 `Proxy` is not a simple JavaScript wrapper—it changes how the V8 engine itself resolves memory addresses.
* **C++ Trap Dispatches:** When `Proxy` methods are called, V8 cannot execute its optimized bytecode. Instead, it must leave its JIT hot-path, check the Proxy's internal trap handler array, set up a C++ transition frame, and execute the JS trap callback.
* **Deoptimization of Inline Caches:** Because property lookups on a Proxy run dynamic JavaScript code (the `get` or `set` trap), V8 cannot assign a stable **Hidden Class Map** to a Proxy instance. Consequently, all property operations on an ES6 Proxy are permanently categorized as **megamorphic**, forcing slow dictionary lookups.
* **Benchmark Truth:** Property lookups on raw objects run in less than **0.5 nanoseconds** (due to monomorphic IC). The same property lookup executed on an ES6 `Proxy` with a `get` trap takes between **10 to 40 nanoseconds**—a **20x to 80x slowdown**.

## ⑤ ADVANTAGES

1. **Decoupled Extensions**: Decorators allow runtime features to be applied incrementally without bloating classes with unused features.
2. **Simplified Clients**: Facades shield client code from large, highly volatile third-party subsystems.
3. **Runtime Protection**: Proxies allow strict data validation, security scanning, and automated logging to be injected transparently into existing objects.
4. **Seamless Integration**: Adapters enable outdated legacy APIs to work alongside modern systems without requiring high-risk codebase rewrites.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Proxy Performance Bottlenecks**: Overusing ES6 Proxies inside tight, high-throughput loops (like rendering engines or packet parsers) can lead to severe CPU utilization spikes due to JIT cache deoptimization.
2. **The "Identity Crisis" (Proxy/Decorator)**: An adapter or proxy has a different object identity (`===`) than the original object. If your codebase relies on references (e.g. `Map` keys), checking the proxy against the raw object will fail.
3. **Highly Complex Call Stacks**: Multi-layered decorators or proxies create deep, confusing stack traces during debugging, making root-cause analysis highly difficult.
4. **Hidden Side Effects in Facades**: If a Facade obscures important configuration options, developers may struggle to fine-tune the system's performance or security attributes.

## ⑦ PRODUCTION USAGE

### 🛠️ Production-Ready Secure Microservice Logger & Resilience Adapter

This production-grade script builds an API resilience adapter and a secure logging proxy. It translates an unstable external third-party API interface into our standard internal API shape, while transparently shielding sensitive user fields (like passwords or tokens) and logging performance metrics:

```javascript
// Standard Internal Client Interface expected by our codebase:
// { requestData(endpoint): Promise<any> }

// Incompatible Third-Party Client:
class LegacyExternalClient {
    executeRawQuery(url, authHeader, bodyString) {
        // Imagine this calls an actual API
        console.log(`[HTTP POST] -> ${url}`);
        return Promise.resolve({
            status: 200,
            payload: JSON.stringify({
                userId: "usr_99",
                secretToken: "SUPER_SECRET_TOKEN",
                email: "user@domain.com"
            })
        });
    }
}

// 1. The Adapter: Standardizes signature and maps authentication
class LegacyApiAdapter {
    constructor(legacyClient, apiKey) {
        this.legacyClient = legacyClient;
        this.apiKey = apiKey;
    }

    async requestData(endpoint) {
        const url = `https://api.external.com/v1/${endpoint}`;
        const authHeader = `Bearer ${this.apiKey}`;
        const body = JSON.stringify({ timestamp: Date.now() });

        const rawResponse = await this.legacyClient.executeRawQuery(url, authHeader, body);
        
        if (rawResponse.status !== 200) {
            throw new Error(`External API Error: Status ${rawResponse.status}`);
        }
        
        return JSON.parse(rawResponse.payload);
    }
}

// 2. The Proxy: Intercepts responses, sanitizes secrets, and logs execution latency
function createSecureLoggingProxy(targetService) {
    const sensitiveKeys = new Set(["secretToken", "password", "apiKey", "auth"]);

    return new Proxy(targetService, {
        get(target, prop, receiver) {
            const value = Reflect.get(target, prop, receiver);

            if (typeof value === 'function') {
                // Return a wrapped method to intercept execution
                return async function (...args) {
                    const start = performance.now();
                    const sanitizedArgs = JSON.parse(JSON.stringify(args)); // Shallow clone for logging

                    console.log(`[PROXY CALLED] Method: "${prop}" | Args:`, sanitizedArgs);

                    try {
                        const result = await value.apply(this, args);
                        const end = performance.now();
                        const latency = (end - start).toFixed(4);

                        // Securely sanitize result object
                        const sanitizedResult = { ...result };
                        for (const key of Object.keys(sanitizedResult)) {
                            if (sensitiveKeys.has(key)) {
                                sanitizedResult[key] = "[REDACTED_SECURITY_SENSITIVE]";
                            }
                        }

                        console.log(`[PROXY SUCCESS] Method: "${prop}" | Latency: ${latency}ms | Result:`, sanitizedResult);
                        return result; // Return raw, unredacted result to the system!
                    } catch (error) {
                        const end = performance.now();
                        console.error(`[PROXY FAILURE] Method: "${prop}" | Latency: ${(end - start).toFixed(4)}ms | Error:`, error.message);
                        throw error;
                    }
                };
            }

            return value;
        }
    });
}

// --- Execution Flow ---
async function run() {
    const rawLegacyClient = new LegacyExternalClient();
    const adapter = new LegacyApiAdapter(rawLegacyClient, "API_KEY_ABCD_12345");
    
    // Secure the adapter with the Proxy
    const securedService = createSecureLoggingProxy(adapter);

    console.log("Starting secured API requests...");
    const data = await securedService.requestData("profile");
    
    console.log("\nApplication received data token safely:", data.secretToken);
}

run();
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is a Structural Design Pattern?**  
**A:** A design pattern focused on how classes and objects are composed or wrapped to construct larger, flexible, and more efficient structures.

**Q2: What is the main difference between the Adapter and the Proxy pattern?**  
**A:** 
* An **Adapter** changes/translates the original object's interface into a different interface.
* A **Proxy** implements the *exact same* interface as the target object, purely controlling, wrapping, or monitoring access to it.

**Q3: What is the purpose of the Facade Pattern?**  
**A:** To provide a simple, unified, high-level interface to a complex and messy subsystem, shielding client code from complex dependencies.

**Q4: How does the Decorator Pattern differ from Class Inheritance?**  
**A:** Class inheritance extends class behaviors statically at compile-time/definition time. The Decorator pattern dynamically wraps objects at runtime, allowing features to be added, swapped, or stacked on the fly.

**Q5: What is an ES6 Proxy?**  
**A:** A built-in JavaScript object that wraps a target object, allowing you to intercept and customize low-level engine operations (like getting, setting, deleting properties, or calling methods).

### 🟡 Intermediate (5)

**Q6: How can you implement a basic caching Proxy in JavaScript to optimize expensive calculations?**  
**A:** 
```javascript
class CachingProxy {
    constructor(heavyService) {
        this.service = heavyService;
        this.cache = new Map();
    }
    async getData(param) {
        if (this.cache.has(param)) {
            return this.cache.get(param); // Return cached output
        }
        const result = await this.service.getData(param);
        this.cache.set(param, result);
        return result;
    }
}
```

**Q7: Why does wrapping an object inside an ES6 Proxy break references like `instanceof` or object equality comparison (`===`)?**  
**A:** A Proxy is a completely separate instance in memory. If `p = new Proxy(obj, {})`, then `p === obj` evaluates to `false`. However, prototype relationships are preserved by default, meaning `p instanceof Constructor` will evaluate to `true` because the Proxy intercepts prototype-lookup operations and routes them to the target.

**Q8: Explain how you would use the Facade pattern to clean up chaotic Browser Web APIs (such as handling legacy cross-browser events or AJAX)?**  
**A:** You write an abstraction class or module (like old JQuery or Axios) that checks browser capabilities internally (e.g. standard `fetch` vs `XMLHttpRequest` vs IE's `ActiveXObject`) and exposes a single unified `request()` function, keeping the client code clean and platform-agnostic.

**Q9: Explain the trade-offs of using JavaScript class decorators vs. standard object wrapping (composition).**  
**A:** 
* **Class Decorators:** Standardized declarative syntax (`@log`), but applied statically to class definitions at loading time. They cannot be toggled easily on individual runtime instances.
* **Object Wrapping:** High runtime flexibility (dynamic application per instance), but requires boilerplate forwarders for every single method on the target interface.

**Q10: What is a "Virtual Proxy" and how does it optimize web application performance?**  
**A:** A Virtual Proxy acts as a skeleton wrapper that defers the allocation of a heavy object (like a large file loader, video processor, or database driver) until it is actually needed (lazy loading), keeping the application startup phase fast and lightweight.

### 🔴 Advanced (5)

**Q11: Detail what V8 does to JIT Inline Caching (IC) structures when properties are repeatedly accessed on objects wrapped inside deep layers of dynamic ES6 Proxies.**  
**A:** When V8 encounters an ES6 Proxy, it cannot resolve standard property offsets on a static Hidden Class. Instead, it must execute C++ runtime trap dispatches. Since the trap (`get`) is a dynamic JS closure, V8's Inline Cache (IC) transitions straight to a **megamorphic** state. If the Proxy wraps another Proxy (deep wrapping), the compiler must execute nested trap dispatches, generating deep, unoptimizable execution paths and triggering complete deoptimization of hot-path JIT loops.

**Q12: Write a robust, self-healing Proxy handler that acts as a circuit breaker, automatically redirecting requests to a backup service if the main service throws errors repeatedly.**  
**A:** 
```javascript
function createResilientService(primary, backup, threshold = 3, cooldownMs = 5000) {
    let failureCount = 0;
    let isBroken = false;
    let lastFailureTime = 0;

    return new Proxy(primary, {
        get(target, prop, receiver) {
            const activeMethod = Reflect.get(target, prop, receiver);

            if (typeof activeMethod === 'function') {
                return async function (...args) {
                    const now = Date.now();
                    if (isBroken && (now - lastFailureTime < cooldownMs)) {
                        console.warn("[Circuit Breaker: OPEN] Routing request to Backup service.");
                        return backup[prop].apply(backup, args);
                    } else if (isBroken) {
                        // Cooldown passed, attempt half-open state recovery
                        isBroken = false;
                        failureCount = 0;
                        console.log("[Circuit Breaker: HALF-OPEN] Testing primary service.");
                    }

                    try {
                        const res = await activeMethod.apply(primary, args);
                        failureCount = 0; // Reset count on success
                        return res;
                    } catch (err) {
                        failureCount++;
                        lastFailureTime = Date.now();
                        console.error(`[Circuit Breaker] Primary service failed. Total failures: ${failureCount}`);
                        
                        if (failureCount >= threshold) {
                            isBroken = true;
                            console.error("[Circuit Breaker: OPENED] Primary service offline. Switching to Backup.");
                        }
                        
                        return backup[prop].apply(backup, args); // Fallback
                    }
                };
            }
            return activeMethod;
        }
    });
}
```

**Q13: How can you implement an API Adapter that dynamically maps keys of arbitrary JSON payloads using a JSON-Schema-like mapping configuration in JS?**  
**A:** 
```javascript
class SchemaMapperAdapter {
    constructor(mappingSchema) {
        this.mappingSchema = mappingSchema; // Maps: { "newKey": "old_key_path" }
    }

    adapt(payload) {
        const adapted = {};
        for (const [newKey, oldPath] of Object.entries(this.mappingSchema)) {
            adapted[newKey] = this.resolvePath(payload, oldPath);
        }
        return adapted;
    }

    resolvePath(obj, path) {
        return path.split('.').reduce((acc, part) => (acc ? acc[part] : undefined), obj);
    }
}

// Usage:
const oldPayload = { user: { info: { email_addr: "test@domain.com" } } };

```

**Q14: Explain the differences in V8 garbage collection behavior and memory leak vectors between a standard Decorator using class wrapping references vs. a Proxy object.**  
**A:** 
* **Class Decorator (Wrapper):** Holds a hard reference to the inner object (`this.base = base`). If the wrapper is referenced elsewhere (e.g. a global array), the inner object is kept alive in the heap, causing a memory leak.
* **ES6 Proxy:** Holds a strong internal reference to both the target object and the handler object. A critical trap exists: if the handler has a reference back to the Proxy or targets, it forms a cyclic reference. While modern V8 Mark-Sweep GC handles cycles, if the handler uses global event listeners or global registry maps to store the Proxy target references, it creates a persistent leak vector, preventing both the proxy, target, and handler from ever being GC'd.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact step-by-step sequence of V8 heap allocations, Hidden Class lookups, and compiler bailouts when a program passes 5 differently decorated wrappers of a base class into a high-throughput loop that evaluates a property.**  
**A:** 
1. **Instantiation:** When the 5 differently decorated objects are constructed, V8 allocates a separate instance for each class in the Young Generation (Newspace) heap.
2. **Hidden Class Map Creation:** Since each decorator class (`MilkDecorator`, `SugarDecorator`, `VanillaDecorator`, etc.) is declared separately, V8 assigns each a distinct C++ **Hidden Class (Map)** structure containing property-to-offset descriptors.
3. **Monomorphic Access Setup:** When the loop starts, the JIT compiler generates an **Inline Cache (IC)** slot for the property access `item.getCost()`. On the first iteration, it records `MapA` (e.g., `MilkDecorator`).
4. **Polymorphic Transition:** On subsequent iterations, V8 encounters `MapB` (e.g., `SugarDecorator`) and then `MapC`. The JIT transitions the IC from **Monomorphic** to **Polymorphic**, generating comparison branches in machine code for the first few Maps.
5. **Megamorphic Bailout:** Once the number of distinct Maps exceeds the JIT polymorphic limit (typically 4 or 5 maps), V8 triggers a complete JIT compiler bailout. The IC transitions to a **Megamorphic** state.
6. **Execution Slowdown:** The JIT compiler completely deletes the optimized property-lookup assembly code and falls back to a generic C++ runtime dictionary search, checking transition hashes on every single loop iteration, resulting in a substantial drop in performance.

**Q17: Trace the low-level V8 engine mechanics, bytecode generation, and memory-barrier transitions that occur when `Proxy.revocable()` is executed, detailing how the engine invalidates the proxy's internal pointers upon revocation.**  
**A:** 
1. **C++ Invocation:** Executing `Proxy.revocable(target, handler)` dispatches to the C++ runtime function `Builtin_ProxyRevocable`.
2. **Allocation of Revocable Structure:** V8 allocates three items: a `JSProxy` object, a C++ `JSReceiver` representing the `revoke` function closure, and an internal C++ `ProxyRevocable` backing store linking both.
3. **Internal Pointer Registration:** The `JSProxy` contains two internal raw 64-bit pointers in its header: `[[ProxyTarget]]` pointing to the target object, and `[[ProxyHandler]]` pointing to the handler.
4. **Bytecode Setup:** The returned `revoke` function is compiled into a lightweight bytecode stub that references the internal `ProxyRevocable` C++ backing store.
5. **Revocation Invocation:** When the client executes `revoke()`, V8 invokes the native code which immediately grabs the backing store.
6. **Memory-Barrier Nullification:** V8 enters a critical section and writes **`NULL`** (or V8's special `the_hole` serialization sentinel) into the `JSProxy`'s internal `[[ProxyTarget]]` and `[[ProxyHandler]]` header slots.
7. **Bailout/Exception Generation:** On the very next property read, the V8 execution thread encounters `NULL` in the `[[ProxyTarget]]` slot. The engine halts processing instantly, bypasses JIT bytecode completely, and triggers a C++ exception dispatch that throws a standard JavaScript `TypeError: Cannot perform 'get' on a proxy that has been revoked.`
