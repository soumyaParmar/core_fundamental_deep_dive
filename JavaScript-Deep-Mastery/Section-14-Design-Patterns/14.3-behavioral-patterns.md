# 📌 14.3 — Behavioral Patterns
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Behavioral patterns are design strategies that handle how different objects communicate and delegate responsibilities to one another. Instead of having objects tightly coupled—where one object directly manipulates or hardcodes references to another—behavioral patterns establish clean communication pipelines. For example, you can set up a system where objects auto-update when something changes (Observer), swap out algorithms at runtime (Strategy), traverse complex collections step-by-step without exposing the underlying data structures (Iterator), or turn operations into stand-alone objects that can be queued, logged, or undone (Command).

**Technical Definition:** Behavioral design patterns are concerned with algorithms and the assignment of responsibilities between objects. They describe not just patterns of objects or classes but also the patterns of communication between them. In JavaScript, these patterns utilize lexical closures, asynchronous event loops, callback queues, and iterable protocols. They include: **Observer** (one-to-many dependency subscription model), **Strategy** (encapsulated interchangeable algorithms), **Iterator** (sequential traversal access via iterable protocols), and **Command** (encapsulated requests as parameterizable objects).

**One-line mental model:**
> *Behavioral patterns streamline communication and responsibility delegation between objects to keep systems clean, modular, and responsive.*

## ② DRILL DOWN

### 🔹 A. The Observer Pattern (Publisher-Subscriber)

The **Observer Pattern** defines a one-to-many dependency between objects, so that when one object changes state, all its dependents are notified and updated automatically. In JavaScript, this is the foundational pattern behind Event Emitters, DOM event listeners, and reactive libraries (like RxJS or Vue's reactivity system).

```javascript
class Subject {
    constructor() {
        this.observers = new Set(); // Use a Set to prevent duplicate subscribers
    }

    subscribe(observer) {
        this.observers.add(observer);
        // Return an unsubscribe function for clean cleanup!
        return () => this.unsubscribe(observer);
    }

    unsubscribe(observer) {
        this.observers.delete(observer);
    }

    notify(data) {
        for (const observer of this.observers) {
            observer.update(data);
        }
    }
}

// Concrete Observer
class LoggerObserver {
    update(data) {
        console.log(`[LOGGER] Received update:`, data);
    }
}

class DashboardObserver {
    update(data) {
        console.log(`[DASHBOARD] Rendering metrics:`, data);
    }
}

// Usage:
const monitor = new Subject();
const logger = new LoggerObserver();
const dashboard = new DashboardObserver();

const unsubLogger = monitor.subscribe(logger);
monitor.subscribe(dashboard);

monitor.notify({ temperature: 24, status: "OK" });
// Output from both logger and dashboard

unsubLogger(); // Unsubscribe the logger
console.log("--- Logger unsubscribed ---");
monitor.notify({ temperature: 29, status: "WARNING" });
// Only dashboard is notified!
```

### 🔹 B. The Strategy Pattern (Interchangeable Algorithms)

The **Strategy Pattern** defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime. It lets the algorithm vary independently from the clients that use it. In JavaScript, functions are first-class citizens, which makes the Strategy pattern incredibly simple—you don't even need wrapper classes; you can pass raw functions!

```javascript
// Concrete Strategies (Encapsulated shipping fee algorithms)
const economyShipping = (weight) => weight * 1.5;
const expressShipping = (weight) => weight * 5.0;
const overnightShipping = (weight) => weight * 12.0 + 10;

// The Context (Order Processor)
class ShippingContext {
    constructor(shippingStrategy) {
        this.strategy = shippingStrategy; // Dynamic strategy assignment
    }

    setStrategy(newStrategy) {
        this.strategy = newStrategy;
    }

    calculateCost(weight) {
        return this.strategy(weight);
    }
}

// Usage:
const orderProcessor = new ShippingContext(economyShipping);
console.log(`Economy Cost: $${orderProcessor.calculateCost(10)}`); // $15

// Swap strategies seamlessly at runtime!
orderProcessor.setStrategy(expressShipping);
console.log(`Express Cost: $${orderProcessor.calculateCost(10)}`); // $50
```

### 🔹 C. The Iterator Pattern (Sequential Traversal)

The **Iterator Pattern** provides a way to access elements of an aggregate object sequentially without exposing its underlying representation. JavaScript has native support for this via the **Iteration Protocols** (`Symbol.iterator`), allowing any custom object to work with `for...of` loops and spread operators.

```javascript
class CustomPlaylist {
    constructor() {
        this.songs = [];
    }

    addSong(title, duration) {
        this.songs.push({ title, duration });
    }

    // Implementing the standard JS Iterable protocol:
    [Symbol.iterator]() {
        let index = 0;
        const songs = this.songs;

        return {
            next() {
                if (index < songs.length) {
                    return { value: songs[index++], done: false };
                } else {
                    return { value: undefined, done: true };
                }
            }
        };
    }
}

// Usage:
const playlist = new CustomPlaylist();
playlist.addSong("Song A", "3:45");
playlist.addSong("Song B", "4:12");

// Standard loop works out-of-the-box!
for (const song of playlist) {
    console.log(`Playing: ${song.title} (${song.duration})`);
}
```

### 🔹 D. The Command Pattern (Encapsulating Requests)

The **Command Pattern** turns a request into a stand-alone object that contains all information about the request. This transformation lets you parameterize methods with different requests, delay or queue a request's execution, and support undoable operations.

```javascript
// The Receiver (Knows how to execute the actual work)
class TextEditor {
    constructor() {
        this.text = "";
    }
    write(newText) { this.text += newText; }
    erase(length) { this.text = this.text.slice(0, -length); }
}

// Concrete Commands
class InsertTextCommand {
    constructor(editor, textToInsert) {
        this.editor = editor;
        this.textToInsert = textToInsert;
    }
    execute() {
        this.editor.write(this.textToInsert);
    }
    undo() {
        this.editor.erase(this.textToInsert.length);
    }
}

// The Invoker (Manages command history for undo support)
class EditorManager {
    constructor() {
        this.history = [];
    }

    executeCommand(command) {
        command.execute();
        this.history.push(command);
    }

    undo() {
        const command = this.history.pop();
        if (command) {
            command.undo();
        }
    }
}

// Usage:
const editor = new TextEditor();
const manager = new EditorManager();

const writeHello = new InsertTextCommand(editor, "Hello ");
const writeWorld = new InsertTextCommand(editor, "World!");

manager.executeCommand(writeHello);
manager.executeCommand(writeWorld);
console.log(`Current Text: "${editor.text}"`); // "Hello World!"

manager.undo();
console.log(`After Undo: "${editor.text}"`);  // "Hello "
```

## ③ WHY IT EXISTS

1. **Decoupling Communication**: High-scale systems suffer when modules are highly coupled. The Observer pattern allows subsystems (e.g. analytical aggregators, email microservices) to listen to domain events without the primary domain logic being aware of their existence.
2. **Behavioral Flexibility**: Hardcoded `if/else` or `switch` statements for selecting algorithms pollute controllers. The Strategy pattern cleans up this nesting, moving algorithms into separate modules.
3. **Transactional Safety and Recovery**: The Command pattern makes request tracking trivial. By serializing commands, systems can recover states easily, execute batch jobs offline, or implement infinite undo/redo stacks.

## ④ INTERNAL WORKING: CLOSURE PROPAGATION AND EVENT QUEUES IN BEHAVIORAL PATTERNS

### Lexical Closures and Garbage Collection inside Observers

When implementing Observers or event listeners in JavaScript, developers frequently introduce memory leaks due to **lexical closures**.

#### 1. The Retainer Chain Trap
Consider this common observer pattern pattern:
```javascript
function attachEventMonitor(subject) {
    const heavyState = new Array(1000000).fill("MEM_DRAIN");

    subject.subscribe(() => {
        // This callback forms a closure over 'heavyState'!
        console.log(`Event triggered. State size: ${heavyState.length}`);
    });
}
```
* **V8 Heap Retained Size:** When `attachEventMonitor` finishes executing, its activation context frame would normally be garbage collected. However, the anonymous subscription callback passed to `subject.subscribe` is kept alive in the heap inside `subject`'s observer `Set`.
* **The Closure Context:** Because the callback accesses `heavyState`, the JIT compiler attaches a Lexical Environment context containing `heavyState` to the callback function object. As long as `subject` remains alive in memory, the heavy array **cannot be garbage collected**, resulting in a major memory leak.

#### 2. Microtask Queue Propagation inside Command Invokers
When commands are executed asynchronously (e.g., in queues), they interact directly with the **Event Loop**:
```javascript
class AsyncQueueInvoker {
    constructor() { this.queue = []; }
    enqueue(command) {
        this.queue.push(command);
        // Schedule queue execution on the Microtask Queue
        queueMicrotask(() => this.flush());
    }
    flush() {
        while(this.queue.length > 0) {
            const cmd = this.queue.shift();
            cmd.execute();
        }
    }
}
```
* **Bytecode Dispatch:** Calling `queueMicrotask` avoids blocking the browser rendering thread (macro-task loop). The callback to `this.flush()` is pushed directly to the V8 Microtask queue.
* **Execution Boundary:** The microtask queue is flushed *completely* before the browser performs layout and repaint. If a command schedules another command recursively, it keeps the microtask loop busy, causing **Event Loop Starvation** and freezing the UI.

## ⑤ ADVANTAGES

1. **Loose Coupling**: Publishers do not need to know the implementation details of their subscribers.
2. **Open/Closed Compliance**: New strategies can be introduced dynamically without editing the primary client file.
3. **Uniform Traversals**: Custom data structures (trees, graphs, lists) can be iterated over using standard `for...of` semantics.
4. **Action Auditability**: Commands can be serialized to JSON, queued, delayed, or tracked across networks for audit logs.

---

## ⑥ DISADVANTAGES / TRAPS

1. **The Lapsed Listener Leak**: Forgetting to unsubscribe observers leads to lingering references, keeping dead views or logic frames alive in the garbage collector.
2. **Execution Order Ambiguity**: Observers are typically notified synchronously inside a loop. If observers modify the Subject's state *during* notification, it can trigger infinite recursive cascades or inconsistent states.
3. **Complex Flow Control**: In behavioral patterns like Command, tracking cascading undos across multiple dependent objects can lead to highly fragile system states.
4. **Performance Churn**: Creating temporary Command or Iterator objects inside hot rendering frames (60fps UI loops) causes high GC Scavenger cycles.

## ⑦ PRODUCTION USAGE

### 🛠️ Production-Ready Event-Sourced Redo/Undo State Machine

This script implements an event-sourced command pipeline coupled with an Observer pattern. It records transaction steps, manages a history buffer, notifies dashboards of state changes, and allows infinite undo/redo executions.

```javascript
// 1. The Command Abstract Signature
class Command {
    execute() { throw new Error("Execute method must be implemented."); }
    undo() { throw new Error("Undo method must be implemented."); }
    toJSON() { throw new Error("Serialization required for audit."); }
}

// 2. Concrete Bank Command
class TransferFundsCommand extends Command {
    constructor(ledger, fromAcc, toAcc, amount) {
        super();
        this.ledger = ledger;
        this.fromAcc = fromAcc;
        this.toAcc = toAcc;
        this.amount = amount;
        this.transactionId = `tx_${Math.random().toString(36).substr(2, 9)}`;
    }

    execute() {
        this.ledger.debit(this.fromAcc, this.amount);
        this.ledger.credit(this.toAcc, this.amount);
    }

    undo() {
        this.ledger.credit(this.fromAcc, this.amount);
        this.ledger.debit(this.toAcc, this.amount);
    }

    toJSON() {
        return {
            id: this.transactionId,
            type: "TRANSFER",
            from: this.fromAcc,
            to: this.toAcc,
            amount: this.amount
        };
    }
}

// 3. Receiver: Ledger Store (with Reactive Observer Support)
class BankLedger {
    constructor() {
        this.balances = new Map([
            ["Alice", 1000],
            ["Bob", 500]
        ]);
        this.listeners = new Set();
    }

    subscribe(listener) {
        this.listeners.add(listener);
        return () => this.listeners.delete(listener);
    }

    debit(account, amount) {
        const current = this.balances.get(account) || 0;
        this.balances.set(account, current - amount);
        this.notifyChange(account);
    }

    credit(account, amount) {
        const current = this.balances.get(account) || 0;
        this.balances.set(account, current + amount);
        this.notifyChange(account);
    }

    notifyChange(account) {
        const balance = this.balances.get(account);
        for (const listener of this.listeners) {
            listener({ account, balance });
        }
    }
}

// 4. Invoker: Command Pipeline with History Stack & Undo/Redo Engine
class TransactionPipeline {
    constructor() {
        this.history = [];
        this.redoStack = [];
    }

    process(command) {
        command.execute();
        this.history.push(command);
        this.redoStack = []; // Clear redo stack on new action
        console.log(`[PIPELINE] Executed:`, JSON.stringify(command.toJSON()));
    }

    undoLast() {
        const command = this.history.pop();
        if (!command) return console.log("[PIPELINE] Nothing to undo.");
        
        command.undo();
        this.redoStack.push(command);
        console.log(`[PIPELINE] Undid:`, command.transactionId);
    }

    redoLast() {
        const command = this.redoStack.pop();
        if (!command) return console.log("[PIPELINE] Nothing to redo.");
        
        command.execute();
        this.history.push(command);
        console.log(`[PIPELINE] Redid:`, command.transactionId);
    }
}

// --- Execution Output ---
const ledger = new BankLedger();
const pipeline = new TransactionPipeline();

// Subscribe observers for audit reporting
ledger.subscribe(({ account, balance }) => {
    console.log(`[MONITOR] Balance Alert: ${account} now has $${balance}`);
});

const tx = new TransferFundsCommand(ledger, "Alice", "Bob", 200);

pipeline.process(tx);
// Monitor alerts should trigger automatically:
// [MONITOR] Balance Alert: Alice now has $800
// [MONITOR] Balance Alert: Bob now has $700

pipeline.undoLast();
// Alice should be restored to $1000, Bob to $500

pipeline.redoLast();
// Re-applies the $200 transfer seamlessly!
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are Behavioral Design Patterns?**  
**A:** Patterns concerned with how objects communicate, cooperate, and distribute runtime responsibilities among themselves.

**Q2: What is the primary purpose of the Observer Pattern?**  
**A:** To establish a one-to-many subscription model so that when a subject updates, all its registered observers are notified automatically.

**Q3: How does JavaScript natively implement the Iterator Pattern?**  
**A:** Through the Iteration Protocol, specifically by defining a `[Symbol.iterator]` method on an object that returns an object containing a standard `next()` method.

**Q4: What is the Command Pattern in a nutshell?**  
**A:** A pattern that wraps an action, its arguments, and its receiver context into a standalone object, enabling queued or undoable execution.

**Q5: What is the main difference between the Strategy pattern and a simple `if/else` block?**  
**A:** `if/else` blocks hardcode decision forks inside a single file/function. The Strategy pattern isolates algorithms into independent, swapable functions, satisfying the Open/Closed Principle.

### 🟡 Intermediate (5)

**Q6: Explain what a "Lapsed Listener" memory leak is in JavaScript Observers, and how you resolve it.**  
**A:** A lapsed listener leak occurs when a subscriber registers a listener callback on a publisher but fails to unsubscribe when the subscriber is destroyed. The publisher keeps a strong reference to the listener closure, preventing the garbage collector from reclaiming the listener and any objects retained in its lexical scope.  
*Resolution:* Always return and execute an unsubscribe clean-up function when discarding objects.

**Q7: How can you write a custom object in JavaScript that supports standard `for...of` iteration?**  
**A:** You assign a method to the property key **`Symbol.iterator`** that returns an iterator object with a `next()` function returning `{ value, done }`:
```javascript
const myIterable = {
    data: [10, 20],
    [Symbol.iterator]() {
        let index = 0;
        return {
            next: () => index < this.data.length 
                ? { value: this.data[index++], done: false } 
                : { done: true }
        };
    }
};
```

**Q8: Explain the difference between the Pub/Sub (Publish-Subscribe) pattern and the Observer pattern.**  
**A:** 
* In the **Observer** pattern, the subject keeps a direct list of observers and notifies them directly.
* In **Pub/Sub**, there is a broker/event channel sitting in the middle. The publisher sends messages to the broker, which routes them to subscribers without the publisher and subscriber ever knowing each other exists.

**Q9: How does the Strategy pattern leverage JavaScript's first-class functions to avoid heavy OOP boilerplate?**  
**A:** In languages like Java, every strategy requires a distinct interface and concrete class definition. Since JavaScript handles functions as first-class citizens, you can simply pass raw, anonymous arrow functions directly to the context, bypassing class and instance allocation completely.

**Q10: Explain how the Command pattern supports offline synchronization in web applications.**  
**A:** Because commands represent serialized requests (JSON schemas), when a web app goes offline, it can serialize commands and store them in `IndexedDB`. When the network reconnects, a command processor reads the queue, sends the payloads to the API, and executes them in order.

### 🔴 Advanced (5)

**Q11: Detail how V8 tracks closures formed by event observer registrations in its generational garbage collector (Newspace vs. Oldspace). Under what conditions does a leaked observer cross the allocation boundary?**  
**A:** 
1. **Initial Allocation:** When an observer listener closure is created, V8 allocates its closure context in the Young Generation (Newspace).
2. **Retention Link:** Since the Subject (usually a long-lived application module) holds a strong reference to the listener Set, the listener is flagged as "live" during garbage collector Scavenger runs.
3. **Promotion to Oldspace:** After surviving two Scavenger GC cycles, V8 promotes the listener and its entire retained lexical context (the variables captured in its scope) into the Old Generation (Oldspace) heap.
4. **Persistent Bloat:** Once promoted to Oldspace, the memory is only swept during heavy, expensive Mark-Sweep-Compact cycles. If the subscription is never cleared, the memory is locked in Oldspace permanently, leading to escalating RAM usage.

**Q12: Write a high-performance Event Emitter in JS from scratch that avoids heap allocations when notifying subscribers, explaining how to bypass array creation.**  
**A:** Standard emitters allocate arrays of handlers per event. To prevent GC churn, we can implement a linked-list observer chain stored directly on the listener objects:
```javascript
class FastEmitter {
    constructor() {
        this.events = new Map(); // Maps: eventName -> head of LinkedList node
    }

    on(event, callback) {
        const node = { callback, next: null };
        const head = this.events.get(event);
        if (!head) {
            this.events.set(event, node);
        } else {
            let curr = head;
            while (curr.next) curr = curr.next;
            curr.next = node;
        }
        // Return unsub handler
        return () => this.off(event, callback);
    }

    emit(event, data) {
        let curr = this.events.get(event);
        while (curr) {
            curr.callback(data);
            curr = curr.next; // Zero array allocation during traversal!
        }
    }

    off(event, callback) {
        let curr = this.events.get(event);
        if (!curr) return;
        if (curr.callback === callback) {
            this.events.set(event, curr.next);
            return;
        }
        while (curr.next) {
            if (curr.next.callback === callback) {
                curr.next = curr.next.next;
                return;
            }
            curr = curr.next;
        }
    }
}
```

**Q14: Explain the difference in execution order and V8 context allocation when executing 1,000 recursive commands using synchronous execution vs. scheduling them on the microtask queue via `Promise.resolve().then()`.**  
**A:** 
* **Synchronous Execution:** The JS call stack grows linearly. Every synchronous execution pushes a new execution stack frame into the V8 C++ stack. If the recursion depth exceeds limits, V8 throws a `RangeError: Maximum call stack size exceeded`.
* **Microtask Queue (Async Resolution):** Executing via `Promise.resolve().then()` returns control back to the event loop. The active stack is cleared (popped to 0). V8 allocates a microtask node containing the next call payload on the task heap. When the current frame exits, the engine immediately picks up the next microtask. This bypasses stack overflow errors entirely, but if the microtask generation is faster than execution, it can lead to heap exhaustion.

**Q15: How would you design a Strategy Pattern context that dynamically determines the optimal sorting strategy at runtime based on the array size and data density, targeting V8's internal sorting thresholds?**  
**A:**

const mergeSortStrategy = (arr) => { /* Stable Sort for large arrays */ return arr.sort(); };

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact step-by-step sequence of V8 heap allocations, scope record linkings, and GC sweeps when a subscriber function is registered to an Observer Subject inside a nested lexical closure, and then discarded without explicit unsubscription.**  
**A:** 
1. **Lexical Context Instantiation:** A function containing a nested observer listener is executed. V8 allocates a **Scope Record (Lexical Environment Context)** to hold the parent function's variables.
2. **Listener Allocation:** The observer closure itself is allocated as a `JSFunction` object. Its internal `[[Scope]]` property is linked to the newly created Scope Record.
3. **Subject Linkage:** The `JSFunction` reference is pushed into the Subject's internal `observers` collection (`Set`).
4. **Promotion and Retention:** When the outer execution finishes, its execution frame pops off the stack. However, because the Subject's `Set` retains the `JSFunction` observer, and the observer's `[[Scope]]` retains the parent scope record, the entire graph is kept alive. The generational collector (Scavenger) marks them as live and eventually promotes them to the **Old Generation (Oldspace)**.
5. **GC Verification Failure:** When the parent object is no longer referenced anywhere else in the application, the programmer expects its memory to be collected. However, during GC mark sweeps, the collector traces the root pointer of the *Subject*, finds the observer `Set`, follows the reference to the `JSFunction`, follows the `[[Scope]]` reference to the Lexical Environment, and concludes the variables are still reachable, resulting in a persistent memory leak.

**Q17: Explain the low-level V8 execution mechanics, bytecode loop transitions, and CPU instruction flows that occur when a `for await...of` loop executes on an async generator iterator.**  
**A:** 
1. **Bytecode Compilation:** The V8 JIT compiler compiles the `for await...of` loop into a series of **Suspension Points (yield/resume)** using Generator bytecodes like `LdaNamedProperty` (to lookup `Symbol.asyncIterator`), `SuspendGenerator`, and `ResumeGenerator`.
2. **Iterator Object Setup:** The loop calls `[Symbol.asyncIterator]()` and allocates a C++-backed JS Iterator object.
3. **Execution Pause:** On each loop iteration, the compiler dispatches a call to the iterator's `next()` method, returning a Promise. V8 executes a `SuspendGenerator` instruction, saving the active execution context (registers, registers file, and local variables) directly to the active generator object on the heap.
4. **C++ Event Loop Transition:** The JavaScript thread resumes other work. Meanwhile, the async promise resolves in the background. The promise's microtask handler registers a callback to resume the generator.
5. **Context Restoration:** The microtask queue executes the resumption handler. V8 invokes `ResumeGenerator`. The engine copies the saved registers, frames, and bytecode offsets back from the heap onto the physical execution stack.
6. **Value Extraction & Yield:** V8 processes the resolved `{ value, done }` tuple, assigns the `value` to the loop variable, and executes the loop body until the next suspension point is reached, yielding CPU slices efficiently.
