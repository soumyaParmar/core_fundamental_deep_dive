# 📌 18.2 — Advanced TypeScript Types (Metaprogramming)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## ① DEFINITION

**Plain English:** Advanced TypeScript is basically writing code for your types. Instead of just creating simple shapes (like "User has a name string"), you write logical operations that dynamically build and transform types based on other types. These advanced concepts include: **Generics** (creating reusable components that work with a variety of types while preserving type safety), **Mapped Types** (looping through every property of a type to change them, like making every field read-only or optional), **Conditional Types** (writing type-level `if/else` statements to dynamically choose a type), **Template Literal Types** (building new string types by combining smaller strings, e.g. mapping auto-generated CSS padding classes), and **Utility Types** (pre-built helpers like `Pick`, `Omit`, and `Record` that let you modify object shapes instantly).

**Technical Definition:** Advanced TypeScript enables type-level metaprogramming. **Generics** establish parametric polymorphism. **Mapped Types** iterate over keys using index signatures and mapping modifiers. **Conditional Types** (`T extends U ? X : Y`) enable type-level ternary evaluations, utilizing the `infer` keyword for pattern matching and type extraction. **Template Literal Types** allow type-level string manipulation, generating combinations via distributive union mapping.

**One-line mental model:**
> *Advanced TypeScript type programming translates logical conditions and key iterations into dynamic compile-time type schemas.*

## ② DRILL DOWN

### 🔹 A. Generics & Constraints

Generics allow a component to work over a variety of types rather than a single one, while maintaining full type safety. You can constrain generics using the `extends` keyword.

```typescript
// A generic API Response wrapper:
interface ApiResponse<T> {
    status: "success" | "error";
    data: T;
    timestamp: number;
}

// 1. Using constraints (K must be a valid key of T!)
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
    return obj[key];
}

const user = { id: 1, username: "alice_dev" };
const id = getProperty(user, "id"); // Valid! Type is number
// const bad = getProperty(user, "email"); // Compiler Error: Argument of type '"email"' is not assignable...
```

---

### 🔹 B. Mapped Types (Transforming Keys)

Mapped types build on index signatures, allowing you to loop through the keys of an existing type to construct a new type.

```typescript
type User = {
    id: number;
    name: string;
    email: string;
};

// Loop through keys and make every property optional and read-only!
type ImmutablePartial<T> = {
    readonly [P in keyof T]?: T[P];
};

type ReadonlyUser = ImmutablePartial<User>;
/* Resulting Type:
type ReadonlyUser = {
    readonly id?: number;
    readonly name?: string;
    readonly email?: string;
}
*/
```

---

### 🔹 C. Conditional Types & the `infer` Keyword

Conditional types select one of two possible types based on a relation expressed as a type relationship test (resembles ternary operators). The `infer` keyword allows you to extract and assign a type dynamically inside the condition.

```typescript
// 1. Simple Conditional Type
type IsString<T> = T extends string ? true : false;
type A = IsString<string>; // true
type B = IsString<number>; // false

// 2. Extracting Promise return types using 'infer'!
type UnwrapPromise<T> = T extends Promise<infer U> ? U : T;

type ResolvedData = UnwrapPromise<Promise<string[]>>; // Result Type: string[]
type NonPromise = UnwrapPromise<number>;            // Result Type: number
```

---

### 🔹 D. Template Literal Types (Type-Level String Assembly)

Template literal types build on template string literals, allowing you to generate custom string pattern sets at the type level.

```typescript
type Direction = "top" | "bottom" | "left" | "right";
type Size = "sm" | "md" | "lg";

// Dynamic generation of combinations!
type PaddingClass = `padding-${Direction}-${Size}`;

// Resulting Union:
// "padding-top-sm" | "padding-top-md" | "padding-top-lg" | "padding-bottom-sm" | ...
const elementStyle: PaddingClass = "padding-left-lg"; // Valid!
// const badStyle: PaddingClass = "padding-middle-md"; // Compiler Error!
```

---

## ③ WHY IT EXISTS

1. **Building Extreme Framework Versatility**: Libraries like React, Nest.js, and Prisma must process highly diverse, user-defined database schemas. Generics and mapped types allow framework developers to write type-safe APIs that adapt dynamically to any user schema.
2. **Preventing Code Duplication**: Without advanced types, developers must duplicate interfaces for different scenarios (e.g. creating `User`, `UpdateUserPayload`, `ReadonlyUser` separately). Advanced utility types like `Partial<T>` and `Omit<T, K>` generate these permutations automatically.
3. **Strict String Auditing (CSS/API Actions)**: In large frontend systems, template literal types prevent typos in styling configurations or API actions by strictly checking dynamic string combinations.

## ④ INTERNAL WORKING: HOW CONDITIONAL AND MAPPED TYPES ARE COMPUTED BY TS COMPILER AST

### The TypeScript Type Engine Evaluation Phases

When TypeScript compiles advanced types, it evaluates them using specific logical rules inside the Checker.

#### 1. Distributive Conditional Types
When a conditional type operates on a **union type**, it behaves distributively:
```typescript
type ToArray<T> = T extends any ? T[] : never;
type UnionArray = ToArray<string | number>;
```
* **Evaluation:** TypeScript's checker does not evaluate the union as a single block. It distributes the union, mapping the condition to each member:
  `ToArray<string> | ToArray<number>`
* **Result:** Resolves to `string[] | number[]`.
* **Bypassing Distribution:** To prevent distribution, wrap the extends operands in square brackets:
  `type ToArray<T> = [T] extends [any] ? T[] : never;` (Resolves to `(string | number)[]`).

#### 2. Mapped Type Key Re-mapping (`as` clause)
```typescript
type Getters<T> = {
    [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};
```
* **AST Evaluation:** The checker processes the mapped type.
* **Key Interception:** The `as` clause intercepts the key variable `K`. The compiler runs string transformation built-ins (like `Capitalize`) on the key's AST name, creating new string symbols for the output type descriptors, renaming the shape keys dynamically before type checking.

## ⑤ ADVANTAGES

1. **Ultra-Reusable Code**: Generics write logic once that adapts securely to arbitrary user types.
2. **Automated Schema Changes**: Updating a base interface instantly updates all derived mapped/utility types across the entire project.
3. **Typo Prevention**: Template literal types guarantee that dynamic strings (like CSS properties or routes) contain zero spelling errors.
4. **Compile-Time Validation**: Catching API payload errors at compile-time prevents production API integrations from crashing.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Deep Type Instantiation Latency**: Creating overly complex, nested conditional types can overload the TypeScript compiler's type checker thread, slowing down development server startup speeds.
2. **The `never` Distribution Trap**: Passing a `never` type to a distributive conditional type silently returns `never` instead of executing the condition, leading to confusing return behaviors:
   ```typescript
   type IsNever<T> = T extends never ? true : false;
   type Test = IsNever<never>; // Evaluates to 'never'! (Not 'true'!)
   ```
   *Resolution:* Wrap the target in square brackets: `[T] extends [never]`.
3. **Cryptic Compiler Errors**: When deep conditional generics fail to typecheck, TypeScript outputs extremely long, hard-to-read compilation error stack traces that are difficult to debug.
4. **Runtime Erasure Disconnect**: Developers often expect type validations to exist at runtime. Since advanced types are erased, you must maintain separate runtime validators (like Zod) alongside your advanced type definitions.

## ⑦ PRODUCTION USAGE

### 🛠️ Production-Ready Safe State Manager & Type-Safe Action Handler

This production-grade script builds an advanced type-safe state dispatcher. It maps state changes to specialized action string prefixes, utilizes generics and conditional extraction to automate payload checking, and enforces strict key checking:

```typescript
// 1. Target State Schema
interface AppState {
    user: { id: string; email: string };
    theme: "light" | "dark";
    status: "idle" | "busy";
}

// 2. Advanced Metaprogramming: Generate type-safe Actions from State Keys!
type ActionPayloads = {
    [K in keyof AppState as `SET_${Uppercase<string & K>}`]: AppState[K];
};

/* Generated Type-Safe ActionPayloads schema:
{
    SET_USER: { id: string; email: string };
    SET_THEME: "light" | "dark";
    SET_STATUS: "idle" | "busy";
}
*/

// 3. Conditional Type helper to extract payload matching Action
type ExtractPayload<ActionKey extends keyof ActionPayloads> = ActionPayloads[ActionKey];

// The Dispatcher Class
class TypeSafeDispatcher {
    #state: AppState;

    constructor(initialState: AppState) {
        this.#state = initialState;
    }

    /**
     * Dispatches action, enforcing strict key name and matching payload type at compile-time!
     */
    dispatch<A extends keyof ActionPayloads>(action: A, payload: ExtractPayload<A>): void {
        console.log(`[DISPATCH] Processing: ${action} | Payload:`, payload);

        // Simulated state mutation
        if (action === "SET_THEME") {
            this.#state.theme = payload as AppState["theme"];
        } else if (action === "SET_STATUS") {
            this.#state.status = payload as AppState["status"];
        }
    }

    getState(): AppState {
        return { ...this.#state };
    }
}

// --- Execution & Compilation Diagnostics ---
const dispatcher = new TypeSafeDispatcher({
    user: { id: "usr_1", email: "alice@domain.com" },
    theme: "light",
    status: "idle"
});

// Compile-time checks prevent any mismatch!
dispatcher.dispatch("SET_THEME", "dark"); // Valid!
dispatcher.dispatch("SET_STATUS", "busy"); // Valid!

// dispatcher.dispatch("SET_THEME", "invalid-theme"); 
// Compiler Error: Argument of type '"invalid-theme"' is not assignable to parameter of type '"light" | "dark"'.

// dispatcher.dispatch("SET_USER", { id: "usr_2" }); 
// Compiler Error: Property 'email' is missing in type '{ id: string; }' but required in type '{ id: string; email: string; }'.
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are Generics in TypeScript?**  
**A:** Generics are placeholders that allow classes, functions, or interfaces to work with multiple data types while preserving full compile-time type-safety and autocompletion features.

**Q2: What is the purpose of the `extends` keyword inside Generics?**  
**A:** It enforces a **type constraint** on the generic parameter, directing the compiler to reject any type argument that does not inherit from or implement the specified constraint shape.

**Q3: What does the `keyof` operator do?**  
**A:** It returns a **union type** composed of the string or numeric literal keys of a given object interface.

**Q4: Name three built-in utility types provided by TypeScript.**  
**A:** 
* `Partial<T>`: Makes all properties of type `T` optional.
* `Pick<T, K>`: Constructs a type by picking a set of keys `K` from `T`.
* `Omit<T, K>`: Constructs a type by removing a set of keys `K` from `T`.

**Q5: What is a Mapped Type?**  
**A:** A type definition that uses an index signature syntax to loop through every property key of an existing type, converting them into a new type structure based on mapping modifiers.

---

### 🟡 Intermediate (5)

**Q6: Explain what Conditional Types are in TypeScript. Give a syntax example.**  
**A:** Conditional types allow selecting one of two possible types based on a relation expressed as a type relationship test (ternary operator logic):
`T extends U ? X : Y`
If type `T` is assignable to type `U`, the type resolves to `X`; otherwise, it resolves to `Y`.

**Q7: Explain the purpose of the `infer` keyword inside conditional types.**  
**A:** The `infer` keyword is used inside the `extends` clause of a conditional type to declare a generic variable representing a type to be extracted dynamically from the examined type structure during type checking.

**Q8: Explain what a Template Literal Type is. How does it extend standard string literal types?**  
**A:** Template Literal Types allow generating new string types by combining smaller string union types inside a template string schema (e.g. ``type Padding = `pad-${Size}` ``). If a union type is passed as an operand, the compiler distributes the combinations automatically.

**Q9: Explain the difference between `Omit<T, K>` and `Exclude<T, U>`.**  
**A:** 
* `Exclude<T, U>` is a conditional helper that operates on **union types**, removing members of `U` from union `T`.
* `Omit<T, K>` operates on **object structures**, constructing a new object shape by removing a set of keys `K` from the object type `T`.

**Q10: Why does `type ReadonlyUser = Readonly<User>` prevent you from modifying properties? How does this map internally?**  
**A:** `Readonly<T>` is implemented internally as a mapped type that loops through all keys of `T` and appends the `readonly` modifier to each:
`type Readonly<T> = { readonly [P in keyof T]: T[P]; }`
Any subsequent attempt to assign a new value to these properties triggers a compile-time type-check failure.

### 🔴 Advanced (5)

**Q11: Explain the "Distributive" property of conditional types. Under what conditions is distribution disabled, and how do you bypass it?**  
**A:** Conditional types containing a bare generic type parameter (e.g. `T extends any`) distribute automatically when passed a **union type**. The checker evaluates the condition for each union member separately and joins the results as a new union.  
*Distribution is disabled* if the generic type parameter is wrapped inside a **tuple structure** on both sides of the `extends` keyword (e.g. `[T] extends [any]`), forcing the checker to evaluate the union as a single consolidated type.

**Q12: Write an advanced utility type `DeepReadonly<T>` that recursively converts all nested properties, array items, and keys of an arbitrary object into read-only types.**  
**A:** 
```typescript
export type DeepReadonly<T> = {
    readonly [K in keyof T]: T[K] extends Function
        ? T[K]
        : T[K] extends object
        ? DeepReadonly<T[K]>
        : T[K];
};
```

**Q13: Explain why passing `never` to a distributive conditional type (e.g. `type IsNever<T> = T extends never ? true : false`) evaluates to `never` instead of `false`. How do you implement a correct `IsNever<T>` check?**  
**A:** Since `never` represents an empty union type, when it is passed to a distributive conditional type, the compiler distributes the condition over the "empty" set. Since there are zero members to evaluate, the evaluation short-circuits instantly and returns `never`.  
*Resolution:* Wrap the type parameter in a tuple to prevent distribution:
`type IsNever<T> = [T] extends [never] ? true : false;`

**Q14: Explain the difference in TypeScript compilation latency between a codebase using flat, clean interfaces vs a codebase heavily utilizing nested conditional generic mappings with recursive loops.**  
**A:** 
* **Flat Interfaces:** Resolved instantly using simple dictionary lookups in the Symbol Table.
* **Recursive Conditional Generics:** Forces the type checker's C++ thread to execute deeply nested evaluation loops, allocating millions of temporary types in memory. This degrades compilation speeds, spikes CPU usage during local hot-module-reloads, and can trigger compiler stack overflows.

**Q15: Design a TypeScript utility type `GetRequired<T>` that examines an object type `T` and constructs a new type containing *only* the properties that are strictly required (omitting optional properties).**  
**A:** We use conditional keys check where we compare `{ [K in keyof T]?: T[K] }` shape to detect optionals:
```typescript
type GetRequired<T> = {
    [K in keyof T as {} extends Pick<T, K> ? never : K]: T[K];
};
```
*Explanation:* `{} extends Pick<T, K>` evaluates to `true` if property `K` is optional because picking an optional property allows an empty object as a valid assignment. We re-map those keys to `never` to strip them.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain the exact step-by-step sequence of V8 heap allocations, context transformations, and type erasure changes that occur when an application instantiates and executes a highly generic class `DataStore<T>` containing multiple generic type assertions.**  
**A:** 
1. **Compilation Phase:** The compiler reads the generic class `DataStore<T>`.
2. **Generic Substitution:** The Checker parses instances (e.g., `new DataStore<User>()`), substituting `T` with the `User` interface structure to perform static analysis.
3. **Type Erasure:** The compiler strips all `<T>`, generic type assertions (`as T`), and type constraints from the code.
4. **JS Generation:** V8 receives the raw vanilla ES Class.
5. **Engine Allocation:** When `new DataStore()` executes, V8 allocates the class instance in the Young Generation heap.
6. **Hidden Class Stabilization:** Since V8 only sees raw values (no generics), it builds standard Hidden Class Maps based on the properties initialized in the class constructor, executing at maximum monomorphic JIT speeds.

**Q17: Trace the step-by-step C++ compiler states and AST evaluations that occur when a TypeScript program compiles a template literal type mapping union combinatorics of size 1000 (e.g. `type Combo = `${Size}-${Color}-${Border}``), detailing why and how the compiler memory footprint expands.**  
**A:** 
1. **Parser Allocation:** The compiler parses the template literal type statement, generating template literal AST nodes.
2. **Distribution Phase:** The Checker detects that three union parameters (`Size` of size 10, `Color` of size 10, `Border` of size 10) are passed as operands.
3. **Combinatorial Loop:** The compiler C++ thread initiates a nested loop to calculate the Cartesian product of the three sets (`10 * 10 * 10 = 1000` combinations).
4. **String Symbol Generation:** For each combination, the compiler generates a unique internal string symbol representation and stores it in the global isolate symbol registry.
5. **Memory Bloat:** The compiler allocates a distinct `StringLiteralType` C++ object on the Node.js compiler heap for each of the 1,000 generated string combinations.
6. **Garbage Collection Trigger:** The sudden allocation of thousands of metadata objects saturates the Node.js compiler heap buffer, triggering blocking GC Scavenger runs that slow down compile speeds. If combinatorics exceed 10,000, it crashes the compiler isolates.
