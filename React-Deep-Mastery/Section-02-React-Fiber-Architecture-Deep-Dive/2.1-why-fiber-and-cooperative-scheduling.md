# 📌 2.1 — Why Fiber & Cooperative Scheduling

---

## ① DEFINITION

**Plain English:**
Imagine a single-lane bridge where cars can only cross in one continuous line. If a massive, slow-moving cargo truck starts crossing the bridge, all the faster cars (like user click events or text inputs) must wait behind it until it finishes. If the truck takes 200 milliseconds to cross, the traffic jams, representing a frozen, non-responsive browser window. This is how React's old **Stack Reconciler** worked: it would compile and update the entire UI tree in one recursive block, blocking the browser thread.
Now, imagine that same bridge, but with a traffic warden (React's **Fiber Scheduler**). When a fast sports car arrives, the warden can pause the cargo truck midway, let the sports car pass, and then resume the truck's journey.
This is **Cooperative Scheduling** and **Time Slicing**. React pauses long-running UI updates to handle user inputs instantly, keeping the app smooth.

**Technical Definition:**
**React Fiber** is a complete rewrite of React's core reconciliation algorithm introduced in React 16. It changes reconciliation from a synchronous, non-interruptible stack-based traversal to an asynchronous, pauseable frame-based traversal.
**Cooperative Scheduling** is a multitasking model where the execution environment (React) yields control back to the host environment (the browser's main thread) at regular intervals (typically 5ms) to allow layout calculation, paint, and input event handling to occur without delay.

**One-line mental model:**
> *Fiber is a virtual stack frame that can be paused, prioritized, and resumed, turning rendering into cooperative background work.*

**Where does this live?**
The Fiber representation lives in the `react-reconciler` package, and the time-slicing work scheduler lives in the `scheduler` package.

## ② DRILL DOWN

To understand why Fiber is revolutionary, we must contrast it with the Stack Reconciler.

### 🔹 The Stack Reconciler (React < 16)
The Stack Reconciler reconciled the UI using recursion. When state updated, React would recursively traverse the component tree.
- **Synchronous Execution:** Once the recursion started, it could not be paused. It was tied to the native JS call stack.
- **Main Thread Block:** If the tree was large, the recursion took 50ms to 200ms.
- **Jank:** The browser could not paint or respond to user clicks during this time. The screen appeared frozen, and typing had a noticeable delay.

```
Stack Reconciler Execution (Synchronous & Blocking):
==========================================================
[Start Reconcile] ──► Component A ──► Child B ──► Child C ... ──► [DOM Write] ──► [Finish]
(Main thread is BLOCKED for the entire duration: 150ms)
==========================================================
```

### 🔹 The Fiber Reconciler (React >= 16)
The Fiber Reconciler replaces the native recursive stack with a virtual call stack made of linked list nodes called **Fibers**.
- **Yielding (Time Slicing):** React slices work into chunks. After processing a Fiber node, it checks how much time is left in the current frame. If the frame time is exhausted (usually after 5ms), it yields control to the browser.
- **Prioritization:** It categorizes tasks into lanes (e.g., Immediate, User Blocking, Normal, Idle). A user click event can interrupt a low-priority list rendering task.

```
Fiber Reconciler Execution (Asynchronous & Yielding):
==========================================================
[Work Chunk 1] ──► Check Time (5ms) ──► Yield to Browser (Paint/Events)
                                               │
                                               ▼
[Work Chunk 2] ◄───────────────────────────────┘
  (User inputs are processed in between, preventing jank)
==========================================================
```

### 🔹 Time Slicing & Fiber Priority Categories

React categorizes tasks into lanes, representing different priorities:

| Priority Lane | Purpose / Example | Target Deadline |
|:---|:---|:---|
| **Sync Lane** | Direct text inputs, button clicks (must be immediate) | Immediate |
| **Input Continuous** | Hover effects, drag and drop operations | 16ms |
| **Default Lane** | API data fetches, page transitions | 500ms |
| **Idle Lane** | Offscreen rendering, pre-fetching analytics | No deadline |

## ③ WHY IT EXISTS

### What problem does it solve?

Browsers are single-threaded. The main thread handles JavaScript execution, HTML parsing, CSS style calculations, layout generation (reflows), and screen painting.

For a screen to look smooth to the human eye, it must refresh **60 times per second** (or more on high-refresh screens).
$$\text{Frame Time} = \frac{1000\text{ms}}{60} \approx 16.6\text{ms}$$

If any JavaScript task takes longer than **16.6ms**, the browser drops that frame, causing a visual stutter called **jank**.

```
Browser Render Loop (1 Frame = 16.6ms):
┌────────────────────────────────────────────────────────┐
│  JS Scripting ──► Style Calc ──► Layout ──► Paint      │
└────────────────────────────────────────────────────────┘
▲                                                        ▲
└─────── Must fit within this 16.6ms window ─────────────┘
```

The Stack Reconciler was a major obstacle to smooth web apps because it ran as a single JavaScript execution block. With Fiber, React broke down this execution block into smaller pieces, allowing the browser to paint and handle user interactions in between.

## ④ INTERNAL WORKING

How does the Scheduler yield control, and how does it know when to return?

### 🔹 `requestIdleCallback` vs. `MessageChannel`

The browser provides a native API called `requestIdleCallback`, which schedules a callback to run during the browser's idle periods.

#### Why React DOES NOT Use `requestIdleCallback` in Production:
1. **Low Support:** It is not supported by Safari.
2. **Unpredictable Timing:** The frequency of callbacks is determined by the browser, which can be inconsistent, sometimes running only once per second.
3. **Max Frame Ceiling:** The API has a maximum cap of 50ms for idle periods. React needs tighter control over yielding intervals (5ms).

#### React's Solution: `MessageChannel` + `requestAnimationFrame`
To control cooperative scheduling across all browsers, React built a polyfill inside its `scheduler` package using **`MessageChannel`** (a lightweight postMessage mechanism).

```
Scheduler Loop Flow:
┌───────────────────────┐
│ requestAnimationFrame │ ◄── Align work with browser's refresh rate
└──────────┬────────────┘
           │
           ▼
┌───────────────────────┐
│   Post Message to     │
│   MessageChannel      │
└──────────┬────────────┘
           │
           ▼ (Macro-task callback fires)
┌───────────────────────┐
│ Execute Fiber work loop│ ◄── Limit to 5ms chunks
└──────────┬────────────┘
           │
           ├─────► [Time remaining?] ──► Keep processing Fibers
           │
           └─────► [Time exhausted?] ──► Yield to browser, post next Message
```

### 🔹 Step-by-Step Scheduling Mechanism:
1. **Request Work:** When state changes, React requests scheduling by posting a task to the `scheduler` package.
2. **Macro-task Scheduling:** The scheduler registers a message handler on a `MessageChannel` and triggers `port.postMessage(null)`.
3. **Execution:** The message callback fires as a macro-task. Since macro-tasks run after browser paints, this ensures the event loop has a chance to execute paint cycles first.
4. **The 5ms Work Loop:** Inside the macro-task, React enters a loop:
   ```javascript
   function workLoopConcurrent() {
     // Perform work as long as there are tasks and we haven't reached the deadline
     while (workInProgress !== null && !shouldYieldToHost()) {
       performUnitOfWork(workInProgress);
     }
   }
   ```
5. **Yielding:** The function `shouldYieldToHost()` checks if the elapsed execution time has exceeded `5ms`. If it has, the loop breaks.
6. **Reschedule:** React saves its current position (`workInProgress` pointer) and schedules another macro-task via `MessageChannel` to resume execution in the next cycle.

## ⑤ ADVANTAGES

1. **Eliminates Input Lag:** Text inputs, animations, and transitions remain responsive even during heavy background rendering.
2. **Prioritized Rendering:** High-priority updates (like typing validation) render before low-priority ones (like loading a list of products below the fold).
3. **Concurrency:** React can work on multiple tasks simultaneously (e.g., rendering a tab in the background while keeping the active tab interactive).
4. **Suspense and Error Boundaries:** Fiber's ability to pause and rewind rendering is what makes `Suspense` (waiting for async resources) and Error Boundaries work.

---

## ⑥ DISADVANTAGES / TRAPS

1. **State Inconsistencies (Tearing):** Because rendering is asynchronous and can be paused, different parts of the UI could theoretically display data from different versions of the state. React prevents this using internal mechanisms like `useSyncExternalStore`.
2. **Increased Execution Overhead:** The scheduler loop, time checking, and maintaining linked lists of Fibers introduce CPU and memory overhead compared to a direct recursive loop.
3. **Side-Effect Traps:** If component functions contain side-effects (like modifying global variables or triggering API calls directly in the body), those side-effects may run multiple times because React can discard a partial render tree and start over.

### ⚠️ Side-Effect Trap: Modifying Globals inside Component Bodies

```jsx
// ❌ BAD: Side-effect inside the render function
let renderCount = 0;
function ProfileCard() {
  // If React pauses this rendering, discard it, and restart,
  // this global counter will increment incorrectly!
  renderCount++; 
  
  return <div>Rendered: {renderCount}</div>;
}

// ✅ GOOD: Keep components pure. Put side-effects in effects
function ProfileCard() {
  const [count, setCount] = React.useState(0);
  
  React.useEffect(() => {
    // Effects are guaranteed to run only after a successful commit
    document.title = `Profile Page`;
  }, []);
  
  return <div>Profile</div>;
}
```

## ⑦ PRODUCTION USAGE

### Real Production Pattern: Transitions for Non-Urgent State Updates
Use `startTransition` to mark non-urgent state updates, letting React pause them if the user interacts with the page.

```jsx
import { useState, useTransition } from 'react';

export default function SearchApp() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();

  const handleSearch = (e) => {
    const val = e.target.value;
    // 1. URGENT update: Update input field instantly
    setQuery(val);

    // 2. NON-URGENT update: Wrap heavy computations/filtering in a transition
    startTransition(() => {
      const filtered = heavySearchFilter(val); // Expensive processing
      setResults(filtered);
    });
  };

  return (
    <div>
      <input type="text" value={query} onChange={handleSearch} />
      {isPending && <p>Loading results...</p>}
      <ul>
        {results.map(item => <li key={item.id}>{item.text}</li>)}
      </ul>
    </div>
  );
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is React Fiber?**
**A:** React Fiber is a complete rewrite of React's reconciliation engine. It allows React to split rendering work into small chunks, prioritize updates, pause and resume rendering, and yield control to the browser to prevent UI lag.

**Q2: What is the main limitation of the old Stack Reconciler?**
**A:** The Stack Reconciler evaluated the component tree using synchronous recursion. Once started, it could not be paused, blocking the browser's main thread and causing UI lag (jank) during large updates.

**Q3: What is "Time Slicing"?**
**A:** Time Slicing is a technique where React divides long-running rendering tasks into small, non-blocking chunks (typically 5ms each), yielding control back to the browser in between chunks to keep the interface responsive.

**Q4: How does React schedule work if `requestIdleCallback` is not fully supported?**
**A:** React uses a custom polyfill in its `scheduler` package built with a combination of `MessageChannel` and `requestAnimationFrame`. This allows it to schedule tasks at macro-task priority across all browsers.

**Q5: Why are components required to be pure in React Fiber?**
**A:** Because Fiber can pause, discard, or restart rendering a component. If a component function contains side-effects, those side-effects could fire multiple times, leading to inconsistent application state.

---

### 🟡 Intermediate (5)

**Q6: Why did React choose `MessageChannel` over `setTimeout(fn, 0)` for scheduling work?**
**A:** `setTimeout(fn, 0)` is throttled by browsers to a minimum of 4ms after nested calls, which can cause scheduling delays. Additionally, `MessageChannel` runs as a macro-task and executes faster than nested timers, allowing React to schedule work closer to browser paint boundaries.

**Q7: Explain the difference between React Scheduler's Normal Priority and User Blocking Priority.**
**A:** 
- **User Blocking Priority:** Used for interactive actions that require immediate feedback, like keypresses or clicks. It has a short timeout (typically 250ms) before the scheduler forces execution.
- **Normal Priority:** Used for updates that don't block user interaction, like fetching data. It has a longer timeout (typically 5000ms).

**Q8: What is "Tearing" in concurrent rendering?**
**A:** Tearing occurs when a visual state inconsistency is rendered to the screen. For example, if a component reads from an external store, and rendering is paused to handle an input event that updates that store, some components in the same render cycle might read the old value while others read the new value, resulting in mismatched UI data.

**Q9: How does React's lane system replace the older priority levels (e.g., ExpirationTime)?**
**A:** The older `ExpirationTime` model assigned a timestamp to each task and scheduled tasks linearly. This model lacked the flexibility to group, filter, or merge updates. The lane system uses 32-bit integers (bitmasks) where each bit represents a "lane." This allows React to combine, prioritize, or pause groups of updates using bitwise operations.

**Q10: What does the term "Cooperative Scheduling" mean in the context of web browsers?**
**A:** It is a multitasking approach where React cooperates with the browser's main thread. Instead of running a long rendering task to completion, React works for a short interval (5ms) and voluntarily yields control back to the browser, allowing it to process pending events or paint the screen.

---

### 🔴 Advanced (5)

**Q11: Explain why React uses macro-tasks instead of micro-tasks to yield to the browser during Fiber updates.**
**A:** Micro-tasks (like `Promise.then` or `queueMicrotask`) execute immediately after the current JavaScript execution stack finishes, before the browser can perform paint, layout, or event dispatching. If React used micro-tasks to split rendering, it would block the browser from painting or handling inputs, defeating the purpose of time slicing. Macro-tasks (via `MessageChannel` or `setTimeout`) are queued after paint and layout operations, allowing the browser to refresh the screen in between updates.

**Q12: Detail the bitwise math behind React's Lane System and explain how it allows prioritizing updates.**
**A:** React uses a 32-bit binary integer to represent priorities (lanes). For example, `SyncLane = 0b0000000000000000000000000000001` (1st bit) and `DefaultLane = 0b0000000000000000000000000010000` (17th bit). Using bitwise operators, React can check if a lane is included in a set of pending lanes using `(pendingLanes & lane) !== 0`, add a lane using `pendingLanes | lane`, or clear a lane using `pendingLanes & ~lane`. This binary representation allows React to manage and group priorities quickly.

**Q13: How does cooperative scheduling handle browsers running on high-refresh-rate screens (e.g., 120Hz or 144Hz)?**
**A:** The 5ms time-slice window is hardcoded in React's scheduler as an optimal threshold. On a 60Hz screen, 16.6ms is available per frame. On a 120Hz screen, only 8.3ms is available. The 5ms yield threshold leaves enough time (~3.3ms) for the browser to run style calculations, layout updates, and paint cycles within the tighter 120Hz window.

**Q14: Explain the architectural reason why Error Boundaries could not be implemented cleanly in the Stack Reconciler.**
**A:** In the Stack Reconciler, rendering was evaluated recursively using the native JavaScript call stack. If a component threw an error midway, the call stack would unwind completely, leaving the application in an incomplete state with some DOM nodes updated and others not. Fiber uses a virtual call stack represented by a linked list. If an error is thrown, React catches it, stops processing the virtual stack, rewinds the state of that branch, and renders the closest fallback component without breaking the rest of the UI.

**Q15: How does React's concurrent scheduler resolve conflicts when a high-priority task is queued while a low-priority task is already rendering?**
**A:**
1. A low-priority task (e.g., list rendering) is executing in the concurrent work loop.
2. A user clicks a button, dispatching a high-priority state update.
3. React Scheduler receives the task, notices its priority lane is higher than the currently rendering lane, and interrupts the work loop.
4. React discards the incomplete `workInProgress` tree of the low-priority task.
5. React restarts the render cycle, processing only the high-priority update.
6. Once the high-priority work is committed to the DOM, React restarts the low-priority task using the updated state.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React Scheduler schedules, executes, and yields a task using `MessageChannel`.**
**A:**
1. State changes inside a component, and React calls `Scheduler_scheduleCallback`.
2. The Scheduler wraps the callback in a task object, assigns a priority level, calculates an expiration timeout, and inserts it into a Min-Heap task queue.
3. If no schedule loop is active, the Scheduler calls `requestHostCallback`.
4. `requestHostCallback` calls `port.postMessage(null)` on a `MessageChannel` instance.
5. The browser event loop finishes its current execution, processes any layout updates, and paints.
6. The browser event loop then pulls the message from the macro-task queue and executes the channel's `onmessage` handler (`performWorkUntilDeadline`).
7. `performWorkUntilDeadline` records the current time (`startTime = getCurrentTime()`).
8. It invokes the work loop, processing tasks from the Min-Heap.
9. Before executing each task, the scheduler calls `shouldYieldToHost()`:
    - `shouldYieldToHost()` checks: `getCurrentTime() - startTime >= 5ms`.
10. If the elapsed time is less than 5ms, it continues to the next task.
11. If the elapsed time is 5ms or more, the loop breaks.
12. If unfinished tasks remain, `performWorkUntilDeadline` calls `port.postMessage(null)` to schedule the next execution cycle, and yields control back to the browser.

---

**Q17: Explain step-by-step what happens internally in React when a user types a letter in a text input during a low-priority background render.**
**A:**
1. React is rendering a heavy list in the background at `DefaultLane` priority.
2. The user types a character in a text input.
3. The browser triggers a native input event.
4. React's event listener catches the event and schedules an update at `SyncLane` priority.
5. The scheduler registers that a new `SyncLane` task is pending.
6. During the concurrent work loop, after processing a Fiber node, the loop checks if a higher-priority task is pending:
   `hasPendingSyncWork = true` or `nextLanePriority > currentLanePriority`.
7. React pauses the work loop and saves its progress.
8. React notices that the new task's priority (`SyncLane`) is higher than the current rendering priority (`DefaultLane`).
9. React discards the partially completed `workInProgress` tree to free up memory.
10. The Scheduler immediately runs the `SyncLane` rendering task.
11. React renders the input component with the new character, diffs the trees, and commits the update to the DOM.
12. The browser paints the new character, keeping the typing experience smooth.
13. With the `SyncLane` queue empty, React schedules a new task to restart the low-priority list rendering.
