# 📌 2.2 — Fiber Reconciliation & Work Loop

---

## ① DEFINITION

**Plain English:**
Imagine building a website's layout. In the old days, you'd use a single blueprint (the Virtual DOM) and read it recursively. If you were interrupted, you'd have to start over from the beginning.
React Fiber changes this by converting the blueprint tree into a **linked list of tasks**, where each element (called a **Fiber**) is connected to its first child, its next sibling, and its parent. Think of it like a chain of post-it notes: "Do A, then do A's child B. If B is done, do B's sibling C. If C is done, go back to parent A."
Because this is represented as pointers in a linked list rather than recursive function calls, React can stop at any single post-it note, check if it needs to yield to the browser, and then resume exactly where it left off.
To make updates smooth, React uses a technique called **Double Buffering**. It maintains two trees in memory: one representing what is currently visible on the screen, and another representing the next update being worked on. Once the next update is complete, React swaps the trees instantly.

**Technical Definition:**
A **Fiber** is a plain JavaScript object representing a unit of work in React's reconciler. It is a virtual stack frame that contains information about a component, its state, its props, and its relationships to other nodes in the tree.
The **Work Loop** is the engine that drives reconciliation. It processes the Fiber tree iteratively using a Depth-First Search (DFS) traversal, updating props and state, and generating DOM mutation flags along the way.
**Double Buffering** is an optimization technique where React maintains two trees: the **Current Tree** (visible UI) and the **workInProgress Tree** (constructed during rendering).

**One-line mental model:**
> *Fibers are linked-list nodes representing virtual stack frames, reconciled iteratively through a DFS loop.*

**Where does this live?**
The Fiber structure and traversal logic live inside `react-reconciler/src/ReactFiber.js` and `ReactFiberWorkLoop.js` in the React source code.

## ② DRILL DOWN

Let's inspect the fields of a Fiber node and see how they form a linked list.

### 🔹 1. The Structure of a Fiber Node
Unlike standard Virtual DOM elements, which only have `type`, `props`, and `children`, a Fiber node contains pointers to build a doubly-linked tree, along with state queues and update flags.

Here is a simplified structure of a Fiber node from the React source code:

```javascript
const FiberNode = {
  // 1. Identification Properties
  tag: WorkTag,            // Type of work (e.g., FunctionComponent, ClassComponent, HostComponent)
  key: null | string,      // Unique identifier for reconciler diffs
  type: any,               // The function or class reference (or string for DOM nodes like 'div')

  // 2. Tree Relationship Pointers (Linked List structure)
  child: null | Fiber,     // Pointer to the first child node
  sibling: null | Fiber,   // Pointer to the next sibling node
  return: null | Fiber,    // Pointer to the parent node (returns to parent when done)

  // 3. State & Inputs Properties
  pendingProps: any,       // Props passed down that need to be applied
  memoizedProps: any,      // Props used to render the current output
  updateQueue: any,        // Queue of state updates (e.g., scheduled setStates)
  memoizedState: any,      // The internal state of the component (linked list of hooks)

  // 4. Concurrency & Effects Flags
  flags: Flags,            // Side-effect flags (e.g., Insertion, Update, Deletion)
  subtreeFlags: Flags,     // Side-effect flags from child subtrees (prevents deep traversals)
  alternate: null | Fiber, // Pointer to the matching node in the double-buffered tree
};
```

### 🔹 2. The Fiber Tree Linked List Topology
Consider the following React JSX:

```jsx
<App>
  <Header />
  <Main>
    <Sidebar />
    <Content />
  </Main>
</App>
```

In the old Stack Reconciler, this was stored as nested arrays of children. In Fiber, it is stored as a linked list:

```
           App (Fiber)
            │
            ▼ (child)
         Header (Fiber) ───► (sibling) ───► Main (Fiber)
                                             │
                                             ▼ (child)
                                          Sidebar (Fiber) ───► (sibling) ───► Content (Fiber)
```

- Each node points to its **first child** (`child`).
- A node's child points to its immediate **sibling** (`sibling`).
- Every child points back to its **parent** (`return`).

---

### 🔹 3. Double Buffering in React
To keep rendering smooth and prevent users from seeing incomplete layouts, React uses double buffering, a technique borrowed from computer graphics.

```
       [CURRENT TREE]                   [WORK IN PROGRESS TREE]
     (Reflected in DOM)                   (Built in memory)
     
        ┌─────────┐        alternate        ┌─────────┐
        │  App    │◄───────────────────────►│  App    │
        └────┬────┘                         └────┬────┘
             │                                   │
             ▼                                   ▼
        ┌─────────┐        alternate        ┌─────────┐
        │  Header │◄───────────────────────►│  Header │ (Processing updates...)
        └─────────┘                         └─────────┘
```

1. **The Current Tree:** Represents the DOM elements currently visible on the screen.
2. **The workInProgress Tree:** When an update is triggered, React traverses the current tree and constructs a parallel tree in memory, reusing existing Fiber nodes via the `alternate` pointer.
3. **Commit Phase Swap:** Once the `workInProgress` tree is fully resolved, React changes a single root pointer in `react-dom` to point to the `workInProgress` tree, updating the screen instantly. The old `current` tree becomes the new `workInProgress` tree for the next update.

## ③ WHY IT EXISTS

### What problem does it solve?

In traditional recursive architectures, when you call a function, a new frame is pushed onto the JavaScript call stack.
```javascript
function renderComponent(component) {
  // Processing logic...
  component.children.forEach(child => renderComponent(child)); // Recursive call
}
```
You cannot stop a running call stack midway; it must run until it is empty. If the component tree is large, this recursive process can block the main thread for over 100ms.

**The Fiber Solution:**
By moving the call stack from the native JavaScript engine to a heap-allocated linked list of Fiber nodes, React gained control over the execution stack.
- **Yielding:** React can pause rendering after processing any single Fiber node.
- **Priority Swap:** React can discard a partial render tree if a higher-priority update (like a keypress) arrives.
- **Parallel Work:** React can prepare offscreen tabs in memory using the `workInProgress` tree, swapping it in only when the user clicks the tab.

## ④ INTERNAL WORKING

Let's trace how React traverses the Fiber tree using Depth-First Search (DFS) and processes work.

### 🔹 Depth-First Search (DFS) Traversal Algorithm

Reconciliation uses an iterative DFS loop that avoids recursion. It uses three pointers: `child`, `sibling`, and `return`.

```
        DFS Traversal Path:
        
             1. App (beginWork)
              │       ▲
     (child)  │       │ (completeWork)
              ▼       │
           2. Header (beginWork) ──► 3. Main (beginWork)
                                         │       ▲
                                (child)  │       │ (completeWork)
                                         ▼       │
                                      4. Sidebar (beginWork) ──► 5. Content (beginWork)
```

#### Traversal Rules:
1. **Go Down First:** For the current node, execute **`beginWork`**. If the node has a `child`, make that child the next node and repeat.
2. **Go Across:** If there is no child, execute **`completeWork`** on the current node. If the node has a `sibling`, make that sibling the next node and repeat.
3. **Go Up:** If there is no child and no sibling, execute **`completeWork`** on the current node. Move up to the parent (`return` node).
   - If the parent has a sibling, make that sibling the next node.
   - If not, move up again, executing `completeWork` on the parent, until you reach the root.

---

### 🔹 The Work Loop Implementation

React runs the traversal inside a loop.

```javascript
// Synchronous Work Loop (cannot be interrupted)
function workLoopSync() {
  while (workInProgress !== null) {
    performUnitOfWork(workInProgress);
  }
}

// Concurrent Work Loop (can yield to the browser)
function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress);
  }
}

function performUnitOfWork(unitOfWork) {
  const current = unitOfWork.alternate; // The current node on screen
  
  // 1. Begin work: updates state, handles props, returns first child
  let next = beginWork(current, unitOfWork);
  
  unitOfWork.memoizedProps = unitOfWork.pendingProps;
  
  if (next === null) {
    // 2. Complete work: bubbles up side-effect flags, handles DOM node creation
    completeUnitOfWork(unitOfWork);
  } else {
    workInProgress = next;
  }
}
```

#### `beginWork` vs `completeWork`:
- **`beginWork` (Going Down):**
  - Resolves component states by processing hook queues.
  - Generates children by executing component functions (or evaluating JSX).
  - Reconciles (diffs) children and marks them with flags (e.g., `Placement`, `Update`, `Deletion`).
  - Returns the first child Fiber node, or `null` if there are no children.
- **`completeWork` (Going Up):**
  - Creates the physical DOM element (for host elements like `div` or `span`) and inserts children.
  - Prepares event listeners and props to be attached to the DOM element.
  - Bubbles up flags from the children to the parent (`subtreeFlags`), which helps the commit phase bypass clean subtrees quickly.

## ⑤ ADVANTAGES

1. **Pauseable Rendering:** React can pause long render tasks to keep the browser responsive.
2. **Double-Buffered Consistency:** The user never sees a partially updated UI; updates are committed to the screen all at once.
3. **Memory Reuse:** React reuses Fiber nodes between the `current` and `workInProgress` trees, reducing memory allocation and garbage collection overhead.
4. **Optimized Commit Traversal:** By bubbling up effect flags to `subtreeFlags`, React can bypass large, unchanged branches during the commit phase without inspecting every node.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Memory Footprint:** Each Fiber node is a large JavaScript object with dozens of properties. Maintaining two complete trees in memory can consume significant RAM in complex applications.
2. **Hidden Execution Cost:** Although cooperative scheduling prevents UI lag, the constant checks (`shouldYield`), linked-list pointer updates, and bitwise flag calculations introduce a minor CPU overhead compared to simple recursion.

### ⚠️ Trap: Triggering Cascading Renders
If you mutate state inside a component body (outside of an effect or handler), you can cause infinite render loops because `beginWork` runs the component function, which triggers another state update, causing React to start a new work loop.

```jsx
// ❌ BAD: Unconditional state modification inside the render phase
function Counter() {
  const [count, setCount] = useState(0);
  
  // Executing this inside the render body schedules another update on every render,
  // throwing React into an infinite loop!
  setCount(count + 1); 
  
  return <div>{count}</div>;
}

// ✅ GOOD: Update state inside an event handler or useEffect
function Counter() {
  const [count, setCount] = useState(0);
  
  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  );
}
```

## ⑦ PRODUCTION USAGE

### Large-Scale Optimization: Code Splitting with Suspense
Fiber's pauseable architecture allows React to stop rendering a branch if a component is not yet loaded, showing a fallback UI instead.

```jsx
import React, { Suspense, lazy } from 'react';

// The reconciler will pause rendering when it encounters this component
// until the network request for the bundle completes.
const HeavyDashboard = lazy(() => import('./HeavyDashboard'));

export default function App() {
  return (
    <div className="app">
      <header>My Dashboard</header>
      <Suspense fallback={<div className="spinner">Loading Dashboard...</div>}>
        <HeavyDashboard />
      </Suspense>
    </div>
  );
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the three main relationship pointers in a Fiber node?**
**A:**
1. `child`: Points to the first child node.
2. `sibling`: Points to the immediate sibling node.
3. `return`: Points to the parent node.

**Q2: What is Double Buffering in React?**
**A:** Double Buffering is a technique where React maintains two Fiber trees in memory: the **Current Tree** (what is visible on screen) and the **workInProgress Tree** (what is being rendered in memory). Once rendering is complete, React swaps them, updating the screen instantly.

**Q3: What is the main difference between `workLoopSync` and `workLoopConcurrent`?**
**A:** 
- `workLoopSync` runs a continuous loop until all Fiber work is finished, and cannot be interrupted.
- `workLoopConcurrent` checks if it needs to yield to the browser (`shouldYield()`) after processing each Fiber, allowing it to pause and keep the UI responsive.

**Q4: What are `flags` in a Fiber node?**
**A:** Flags (previously called `effectTag`) are binary bitmasks that record what mutations (e.g., `Placement`, `Update`, `Deletion`) need to be applied to the DOM node during the commit phase.

**Q5: What happens to a component's local state during double buffering?**
**A:** The state is stored in the Fiber's `memoizedState` property. When React clones a node from the current tree to the workInProgress tree, it copies the reference to the hook state, ensuring the component's state is preserved.

---

### 🟡 Intermediate (5)

**Q6: Explain the difference in execution between `beginWork` and `completeWork`.**
**A:** 
- `beginWork` runs during the downward traversal of the tree. It processes state updates, executes component functions, reconciles children, and returns the next child node to process.
- `completeWork` runs during the upward traversal. It constructs host DOM elements, compiles props, prepares event listeners, and bubbles up side-effect flags to the parent's `subtreeFlags`.

**Q7: How does Fiber avoid using recursive function calls during reconciliation?**
**A:** By using tree pointers (`child`, `sibling`, `return`) to build a linked list of Fiber nodes. Traversal is implemented using an iterative `while` loop that walks the list, avoiding recursion and preventing call stack overflows.

**Q8: What is the purpose of the `alternate` field in a Fiber node?**
**A:** The `alternate` field links a node in the current tree to its corresponding node in the workInProgress tree (and vice versa). This allows React to reuse nodes instead of allocating new memory for every render.

**Q9: What are `subtreeFlags` and how do they optimize the commit phase?**
**A:** `subtreeFlags` are bitmasks that aggregate all update flags from a component's children. If a parent's `subtreeFlags` is `0` (clean), the commit phase can skip that entire branch, avoiding deep traversals of unchanged nodes.

**Q10: Why does React copy `pendingProps` to `memoizedProps` at the end of `performUnitOfWork`?**
**A:** `pendingProps` represents the input arguments passed down for the current render cycle. Once `performUnitOfWork` processes the Fiber node successfully, those props are saved in `memoizedProps` to represent the node's rendered state. In future renders, React compares new props against `memoizedProps` to determine if it can skip rendering the node.

---

### 🔴 Advanced (5)

**Q11: Explain how the Fiber reconciler handles depth-first search (DFS) traversal when returning to a parent node that has no sibling.**
**A:**
1. A leaf node (no `child` and no `sibling`) completes its work, and React executes `completeWork` on it.
2. React moves up to the parent node (`return` pointer).
3. If the parent has a `sibling`, the traversal moves to that sibling.
4. If the parent has no sibling, React executes `completeWork` on the parent, and moves up again to the grandparent.
5. This upward traversal continues until React finds a node with a sibling or reaches the root node (`HostRoot`), at which point the work loop terminates.

**Q12: How does the Fiber architecture support incremental rendering and prevent UI jank?**
**A:** By breaking reconciliation into individual units of work (Fibers) and executing them in an iterative loop. Since the state is stored in the heap (on Fiber nodes) rather than the call stack, React can break the loop after any unit of work, yield the thread to let the browser process inputs and paint, and resume reconciliation in the next frame using the stored pointers.

**Q13: Why is the `alternate` pointer set to `null` on a Fiber node during its initial mount?**
**A:** When a component mounts for the first time, it only exists in the workInProgress tree and has no representation in the current tree. The `alternate` pointer remains `null` until the component is committed to the DOM, at which point the trees are swapped and the pointer is initialized.

**Q14: Explain the difference between `memoizedState` in a class component Fiber vs a functional component Fiber.**
**A:** 
- In a class component Fiber, `memoizedState` is a plain JavaScript object representing the component's state (e.g., `{ count: 0 }`).
- In a functional component Fiber, `memoizedState` is the head of a singly linked list of hook objects. Each hook object contains its own state, queue, and a pointer to the next hook in the list.

**Q15: How does React ensure that event handlers inside components capture the correct state during concurrent rendering?**
**A:** React updates references during the commit phase. When an update is committed, the current tree root pointer is swapped. Event listeners are bound to the DOM node, but React resolves the handlers by looking up the active Fiber in the current tree, ensuring event callbacks always read the committed state.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React traverses and reconciles a tree of 3 sibling components (A, B, C) where only B has a state update.**
**A:**
1. B calls `setState`, scheduling an update at the root. React initiates the work loop and sets `workInProgress = HostRoot`.
2. **BeginWork(HostRoot):** React traverses down to the root component `App`.
3. **BeginWork(App):** `App` checks if its props or state changed. They haven't, but its child `B` has pending work, so React returns `App`'s child `A`.
4. **BeginWork(A):** React checks if `A` has pending work. It does not, so it reuses the existing node and returns `A`'s sibling `B`.
5. **BeginWork(B):** React detects that B has pending updates. It runs B's component function, processing B's state update queue.
6. The component function returns B's new JSX representation. React diffs it, marks B with an `Update` flag, and returns B's first child.
7. **Processing B's children:** React runs `beginWork` on B's children. If no updates are found, it skips to `completeWork` on them.
8. Once B's children are processed, React runs `completeWork(B)`, which prepares DOM attributes and bubbles up B's update flag.
9. React moves to B's sibling `C`.
10. **BeginWork(C):** React checks if `C` has pending work. It does not, so it reuses the node and returns `null` (since C has no child).
11. **Cleanup:** React runs `completeWork(C)`, moves up to `App`, runs `completeWork(App)`, and completes the work loop.
12. React enters the commit phase and applies B's updates to the DOM.

---

**Q17: Explain step-by-step what happens internally during the double-buffering pointer swap at the end of a render cycle.**
**A:**
1. The work loop completes reconciliation on the `workInProgress` tree.
2. The reconciler exits the render phase and enters the commit phase.
3. React passes the `workInProgress` root pointer to the renderer (`react-dom`).
4. The renderer runs layout and mutation effects, updating the physical DOM nodes.
5. Once all DOM updates are committed, the renderer updates the root container's current pointer:
   `container.current = workInProgress`.
6. The old `current` tree is now disconnected from the DOM.
7. During the next state update, React calls `createWorkInProgress(current)`. Instead of allocating new objects, it takes the old disconnected tree, updates its properties with the new props and state, and sets it as the new `workInProgress` tree.
