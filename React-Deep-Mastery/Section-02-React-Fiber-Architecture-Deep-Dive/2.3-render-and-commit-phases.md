# 📌 2.3 — Render & Commit Phases

---

## ① DEFINITION

**Plain English:**
Building a stage play is divided into two distinct parts: **rehearsals** and **opening night**.
During rehearsals (the **Render Phase**), the director and actors practice the scenes in private. If the script changes, they can stop, discard their work, rewrite the scene, and practice again. No one in the audience sees these mistakes.
Opening night (the **Commit Phase**) is the live performance. The curtains rise, the stage lights turn on, and the set changes occur right in front of the audience. The performance must be fast, continuous, and cannot be interrupted midway.
React updates the UI in the same two phases:
1. **Render Phase:** React calculates the necessary UI changes in memory. This phase can be paused, interrupted, or discarded.
2. **Commit Phase:** React applies the calculated changes directly to the browser DOM. This phase must run to completion in a single, uninterrupted transaction.

**Technical Definition:**
Reconciliation is split into two phases:
- **The Render Phase:** An asynchronous, interruptible phase where React traverses the Fiber tree, runs component functions, diffs virtual representations, and builds a list of side-effects. No real DOM changes occur during this phase.
- **The Commit Phase:** A synchronous, non-interruptible phase where the renderer (`react-dom`) mutates the DOM, updates references, and schedules lifecycle methods or effect hooks.

**One-line mental model:**
> *Render calculates what changes; Commit writes them to the DOM synchronously.*

**Where does this live?**
The Render phase is managed by the Fiber work loop in `ReactFiberWorkLoop.js`, while the Commit phase is executed by `commitRoot` and its sub-methods inside `ReactFiberCommitWork.js`.

## ② DRILL DOWN

Let's compare the characteristics of the Render and Commit phases.

### 🔹 Render vs. Commit Comparison

| Attribute | Render Phase | Commit Phase |
|:---|:---|:---|
| **Execution Mode** | Asynchronous & Concurrent | Synchronous & Blocking |
| **Interruptible?** | Yes (can yield to browser or discard work) | No (must run to completion) |
| **DOM Mutations?** | None (purely in-memory operations) | Yes (inserts, updates, deletes DOM nodes) |
| **Purity** | Must be pure (no side effects) | Imperative (executes side effects, runs hooks) |
| **Key Entrypoint** | `performUnitOfWork` (`beginWork` / `completeWork`) | `commitRoot` (`commitMutationEffects` / `commitLayoutEffects`) |

---

### 🔹 The Three Sub-Phases of the Commit Phase
Once the render phase completes and all Fiber flags are collected, the reconciler calls `commitRoot`. The commit phase executes in **three sequential sub-passes** over the mutated Fibers:

```
[Commit Phase Entry: commitRoot]
               │
               ▼
┌──────────────────────────────────────────────┐
│  1. Before Mutation Phase                    │  ◄── Read DOM state (e.g. scroll positions)
│     - getSnapshotBeforeUpdate                │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  2. Mutation Phase                           │  ◄── Write to DOM (Insert/Update/Delete)
│     - DOM nodes attached/removed             │      Runs useLayoutEffect cleanups
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│  3. Layout Phase                             │  ◄── Read updated layout, run useLayoutEffect
│     - useLayoutEffect execution              │      Assigns ref.current
└──────────────────────────────────────────────┘
```

#### 1. Before Mutation Phase (`commitBeforeMutationEffects`)
React traverses the effect list to read the state of the DOM before it is changed.
- **Class Components:** Runs `getSnapshotBeforeUpdate`, allowing components to capture information (like scroll position) before the DOM changes.
- **Concurrent Mode:** Schedules passive effects (`useEffect`) to run asynchronously after the browser paint.

#### 2. Mutation Phase (`commitMutationEffects`)
React performs the actual DOM mutations.
- **DOM Insertion:** Appends new elements to their parent containers (`Placement`).
- **DOM Update:** Modifies text, style attributes, and classes (`Update`).
- **DOM Deletion:** Removes unmounted elements (`Deletion`).
- **Cleanups:** Runs cleanups for `useLayoutEffect` and unmount actions.

#### 3. Layout Phase (`commitLayoutEffects`)
React runs code that needs to execute immediately after the DOM is updated but before the browser paints the screen.
- **Hook Execution:** Runs `useLayoutEffect` callbacks synchronously.
- **Ref Assignment:** Assigns DOM node references to `ref.current`.
- **Class Components:** Runs class component lifecycles (`componentDidMount` or `componentDidUpdate`).

---

## ③ WHY IT EXISTS

### What problem does it solve?

If React mutated the DOM incrementally during the rendering process:
- **Tearing and Partial Render Bugs:** If rendering was paused midway to handle an event, the user would see a broken page with some elements updated and others not.
- **Layout Thrashing:** If component functions read and mutated the DOM in an interleaved fashion, it would trigger multiple layout reflows, degrading performance.

**The Phase Separation Solution:**
By separating reconciliation into Render and Commit phases, React ensures that:
1. **User Experience is Consistent:** Visual updates are applied to the DOM in a single synchronous transaction, ensuring the page is always rendered in a complete state.
2. **Scheduling is Flexible:** React can pause and prioritize in-memory render calculations without affecting what the user sees on screen.

## ④ INTERNAL WORKING

Let's trace how React tracks updates during the Render phase and executes them in the Commit phase.

### 🔹 Tracking DOM Mutations: The Fiber Flags

During the Render phase, when `beginWork` and `completeWork` process a Fiber node, they identify differences between the props/state and mark the node with binary flags.

```javascript
// ReactFiberFlags.js (Simplified)
export const NoFlags =         0b0000000000000;
export const Placement =       0b0000000000010; // Insert node
export const Update =          0b0000000000100; // Update attributes
export const Deletion =        0b0000000001000; // Delete node
export const Snapshot =        0b0000001000000; // Runs getSnapshotBeforeUpdate
export const Passive =         0b0000100000000; // Schedules useEffect
```

#### How flags are processed:
1. If a component's state change requires an update, its Fiber is marked with the `Update` flag:
   `fiber.flags |= Update;`
2. As `completeWork` bubbles up the tree, it aggregates these flags into `subtreeFlags`:
   `parent.subtreeFlags |= child.flags | child.subtreeFlags;`
3. In the Commit phase, React uses the `subtreeFlags` of the root node to identify and traverse only the branches that contain updates, skipping clean subtrees.

---

### 🔹 Detailed Lifecycle of Hook Execution

The timeline below shows when hooks and layout effects execute relative to DOM mutations and the browser paint cycle:

```
[Render Phase]
  └── Run Component functions (useState / useReducer state calculations)
  └── Diff Virtual Nodes
  └── Mark Fibers with Flags (Placement, Update, Deletion)
       │
       ▼
[Commit Phase: Before Mutation]
  └── Run getSnapshotBeforeUpdate (Class components)
       │
       ▼
[Commit Phase: Mutation]
  └── Run useLayoutEffect Cleanup (Synchronous)
  └── Mutate DOM (insertBefore, removeChild, textContent = ...)
       │
       ▼
[Commit Phase: Layout]
  └── Assign ref.current (DOM elements linked)
  └── Run useLayoutEffect Callbacks (Synchronous)
       │
       ▼
[Browser Reflow & Paint] (User sees the updated UI)
       │
       ▼
[Post-Paint (Macro-task callback)]
  └── Run useEffect Cleanup (Asynchronous)
  └── Run useEffect Callbacks (Asynchronous)
```

## ⑤ ADVANTAGES

1. **Consistent Visuals:** DOM updates are applied all at once, preventing the user from seeing partially rendered layouts.
2. **Safe Interruption:** React can discard or deprioritize in-memory render phases without leaving the DOM in an inconsistent state.
3. **Synchronous Layout Checks:** `useLayoutEffect` runs synchronously in the layout phase before the browser paints, allowing you to read layout dimensions and adjust elements without causing visual flickering.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Double Execution in Strict Mode:** To help developers catch side-effects in the render phase, React's Strict Mode runs component functions twice in development.
2. **Blocking Code in useLayoutEffect:** Since the Layout phase is synchronous and blocks painting, running heavy computations inside `useLayoutEffect` will delay screen updates and cause UI lag.

### ⚠️ Trap: Putting Side Effects in the Render Phase

```jsx
// ❌ BAD: Putting an API fetch or dispatch inside the render phase
function ProductPage({ id }) {
  const [product, setProduct] = useState(null);
  
  // Side-effect runs during beginWork!
  // In concurrent mode, this can fire multiple times for a single render.
  fetch(`/api/products/${id}`).then(res => res.json()).then(data => setProduct(data));
  
  return <div>{product?.name}</div>;
}

// ✅ GOOD: Wrap side effects in useEffect
function ProductPage({ id }) {
  const [product, setProduct] = useState(null);
  
  useEffect(() => {
    let active = true;
    fetch(`/api/products/${id}`)
      .then(res => res.json())
      .then(data => {
        if (active) setProduct(data);
      });
    return () => { active = false; };
  }, [id]);
  
  return <div>{product?.name}</div>;
}
```

## ⑦ PRODUCTION USAGE

### Real Production Pattern: Measuring DOM Dimensions Flicker-Free
If you need to measure a DOM element's width and adjust a tooltip's position before it renders, use `useLayoutEffect` to avoid visual flickering.

```jsx
import { useState, useLayoutEffect, useRef } from 'react';

export default function TooltipButton() {
  const [tooltipHeight, setTooltipHeight] = useState(0);
  const buttonRef = useRef(null);

  // useLayoutEffect runs in the Layout Phase (post-mutation, pre-paint).
  // We can measure the DOM and update state synchronously before the screen refreshes.
  useLayoutEffect(() => {
    if (buttonRef.current) {
      const rect = buttonRef.current.getBoundingClientRect();
      setTooltipHeight(rect.height);
    }
  }, []); // Run once on mount

  return (
    <div style={{ position: 'relative' }}>
      <button ref={buttonRef}>Hover Me</button>
      <div 
        className="tooltip"
        style={{ 
          position: 'absolute', 
          top: `-${tooltipHeight + 10}px` // Dynamic positioning
        }}
      >
        Tooltip Content
      </div>
    </div>
  );
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the two main phases of React reconciliation?**
**A:**
1. **The Render Phase:** React computes the necessary changes by diffing Virtual DOM trees in memory. This phase can be paused, restarted, or discarded.
2. **The Commit Phase:** React applies the changes to the physical browser DOM. This phase runs synchronously and cannot be interrupted.

**Q2: Does React update the DOM during the Render phase?**
**A:** No. The Render phase performs in-memory computations and tree diffing. DOM mutations are applied only during the Commit phase.

**Q3: Which hook runs synchronously before the browser paint cycle?**
**A:** `useLayoutEffect` runs synchronously in the layout sub-phase of the Commit phase, before the browser paints the screen.

**Q4: When does `useEffect` execute?**
**A:** `useEffect` executes asynchronously after the Commit phase is complete and the browser has painted the screen.

**Q5: Why does React run components twice in development mode?**
**A:** This is a feature of React's Strict Mode. It runs component functions twice to help developers detect side-effects in the render phase (which is supposed to be pure).

---

### 🟡 Intermediate (5)

**Q6: What are the three sub-phases of the Commit phase, and in what order do they execute?**
**A:**
1. **Before Mutation Phase:** Reads the state of the DOM (e.g. running `getSnapshotBeforeUpdate`).
2. **Mutation Phase:** Applies updates, insertions, and deletions to the DOM, and runs unmount cleanups.
3. **Layout Phase:** Runs `useLayoutEffect` callbacks and assigns references (`ref.current`).

**Q7: Why is it safe to pause the Render phase but unsafe to pause the Commit phase?**
**A:** The Render phase works with Virtual DOM objects in memory, so pausing does not affect the UI. The Commit phase modifies the real DOM, so pausing it midway would leave the UI in an incomplete and broken state.

**Q8: Explain the difference in ref assignment between the Mutation phase and the Layout phase.**
**A:** During the Mutation phase, React detaches old refs by setting `ref.current = null`. During the Layout phase, after the DOM has been updated, React binds new references by assigning the DOM node to `ref.current`.

**Q9: What happens to state updates triggered inside `useLayoutEffect` vs `useEffect`?**
**A:**
- Updates in `useLayoutEffect` are applied synchronously before paint, scheduling a new render cycle immediately, which blocks the paint until the update is committed.
- Updates in `useEffect` are scheduled as macro-tasks post-paint, allowing the browser to render the initial frame before starting the next update.

**Q10: What are Fiber flags and how are they used during reconciliation?**
**A:** Flags are binary bitmasks assigned to Fiber nodes during the Render phase to record necessary mutations (e.g., `Placement` or `Update`). During the Commit phase, the renderer reads these flags to perform the corresponding DOM mutations.

---

### 🔴 Advanced (5)

**Q11: Explain how React uses bitwise operations to bubble up flags and how this optimizes the Commit phase.**
**A:**
During the Render phase, as nodes complete, React aggregates update flags from child nodes into `subtreeFlags` using bitwise OR operations:
`parent.subtreeFlags |= child.flags | child.subtreeFlags;`
In the Commit phase, React checks the root node's `subtreeFlags`. If it is `NoFlags`, React skips the entire tree. If it is non-zero, React only traverses branches where `(node.subtreeFlags & MutationMask) !== 0`, avoiding inspections of clean components.

**Q12: Why are class component lifecycles like `componentWillMount` and `componentWillReceiveProps` deprecated in concurrent React?**
**A:** These lifecycles run during the Render phase. In concurrent React, the Render phase can be paused, discarded, or restarted. This means these lifecycles could run multiple times for a single update, causing duplicate side effects (like multiple API calls or state mutations). They were deprecated in favor of hooks or lifecycles that run during the Commit phase.

**Q13: How does React ensure that cleanup functions for effects are executed in the correct order during the Commit phase?**
**A:**
- During the Mutation phase, React runs all cleanups for unmounting components and updates (e.g., `useLayoutEffect` cleanups) in a top-down traversal.
- During the Layout phase, React executes all new callbacks (e.g., `useLayoutEffect` mounts) in a bottom-up traversal.
This ensures child component cleanups run before parent mounts, maintaining consistent lifecycles.

**Q14: Explain the difference in execution between `getSnapshotBeforeUpdate` and `componentDidUpdate`.**
**A:**
- `getSnapshotBeforeUpdate` runs in the "Before Mutation" phase, allowing you to read properties from the DOM (like scroll positions) before changes are applied.
- `componentDidUpdate` runs in the "Layout" phase, after the DOM has been updated. The value returned by `getSnapshotBeforeUpdate` is passed as the third argument to `componentDidUpdate`.

**Q15: How does concurrent React handle a new update while it is in the middle of processing a Render phase?**
**A:**
React compares the priority lane of the new update with the priority lane of the current rendering task.
- **Lower Priority:** React continues the current Render phase. The new update is queued to run in the next cycle.
- **Higher Priority:** React interrupts the current Render phase, discards the in-memory workInProgress tree, and starts a new Render phase to process the high-priority update.

---

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React unmounts a component and cleans up its DOM nodes and effects.**
**A:**
1. During the Render phase, React identifies that a component is missing from the new tree. It marks the component's Fiber node with a `Deletion` flag.
2. React adds the deleted Fiber to the parent Fiber's deletion list.
3. The Reconciler enters the Commit phase and executes `commitDeletion` during the Mutation phase.
4. React performs a DFS traversal of the deleted component's subtree:
   - For every component, it schedules passive cleanups (`useEffect` cleanup callbacks) to run post-paint.
   - It executes layout cleanups (`useLayoutEffect` cleanup callbacks) synchronously.
   - It detaches ref pointers by setting `ref.current = null`.
5. Once the subtree is processed, the renderer removes the component's parent DOM element using `parentDOM.removeChild(childDOM)`.
6. After the browser paints, the scheduler runs the queued `useEffect` cleanup callbacks.

---

**Q17: Explain step-by-step how React resolves a state update triggered inside `useLayoutEffect`.**
**A:**
1. A component finishes rendering, and React commits the updates to the DOM.
2. During the Layout phase of the commit, React runs the component's `useLayoutEffect` callback synchronously.
3. The callback executes a state updater: `setCount(prev => prev + 1)`.
4. React schedules a new update on the component's Fiber node at synchronous priority (`SyncLane`).
5. Because the Commit phase is synchronous and cannot be interrupted, the layout phase continues until all callbacks are executed.
6. Once the Commit phase completes, instead of returning control to the browser paint cycle, React checks its scheduler queue.
7. React notices a pending `SyncLane` update.
8. React bypasses the paint cycle and starts a new Render phase synchronously.
9. React renders the component with the updated state, calculates the diffs, and enters the Commit phase.
10. The renderer applies the new updates to the DOM.
11. Finally, React yields control to the browser, which performs a single paint cycle. The intermediate state update is never visible on screen.
