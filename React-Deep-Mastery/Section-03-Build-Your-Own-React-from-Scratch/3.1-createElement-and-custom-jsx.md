# 📌 3.1 — CreateElement & Custom JSX

---

## ① DEFINITION

**Plain English:**
When you write `<h1 className="title">Hello World</h1>` in a React component, the browser doesn't actually understand this syntax. It's not valid JavaScript.
Before the browser runs your code, a compiler (like Babel or Esbuild) translates this HTML-like syntax into a plain JavaScript function call: `React.createElement("h1", { className: "title" }, "Hello World")`.
This function call is just a factory. It takes the tag type, the properties, and any child elements, and outputs a simple, lightweight JavaScript object: `{ type: 'h1', props: { className: 'title', children: ['Hello World'] } }`.
This section explains how to write our own `createElement` function and configure the compiler to use it instead of React's built-in one.

**Technical Definition:**
`React.createElement` is a core factory function that instantiates a Virtual DOM React Element. It takes three parameters:
1. `type`: A string representing an HTML tag (e.g., `'div'`), or a function reference representing a React component.
2. `props`: An object containing the element's attributes (e.g., `id`, `className`, event handlers).
3. `...children`: A rest parameter containing all child elements, which are nested inside the resulting `props.children` array.

**One-line mental model:**
> *JSX is a visual shortcut that compiles into createElement calls, returning plain JavaScript representations of DOM nodes.*

**Where does this live?**
In our custom library, this function will be exported by our core package. In official React, it lives in the `react` package.

## ② DRILL DOWN

Let's look at the mapping between JSX and its compiled output.

### 🔹 JSX Compilation Mapping

Suppose we write a JSX structure with nested children and attributes:

```jsx
// 1. JSX Source Code:
const element = (
  <div id="container" className="main">
    <h1>Title</h1>
    <p>Paragraph content</p>
  </div>
);
```

The compiler transforms the tags into nested calls:

```javascript
// 2. Compiled Output:
const element = MyReact.createElement(
  "div",
  { id: "container", className: "main" },
  MyReact.createElement("h1", null, "Title"),
  MyReact.createElement("p", null, "Paragraph content")
);
```

When these function calls run, they evaluate to a nested JavaScript object representing our Virtual DOM tree:

```javascript
// 3. Evaluated Virtual DOM Element Object:
{
  type: "div",
  props: {
    id: "container",
    className: "main",
    children: [
      {
        type: "h1",
        props: {
          children: [
            {
              type: "TEXT_ELEMENT",
              props: { nodeValue: "Title", children: [] }
            }
          ]
        }
      },
      {
        type: "p",
        props: {
          children: [
            {
              type: "TEXT_ELEMENT",
              props: { nodeValue: "Paragraph content", children: [] }
            }
          ]
        }
      }
    ]
  }
}
```

## ③ WHY IT EXISTS

### What problem does it solve?

Writing raw JavaScript objects for a complex UI is tedious and error-prone. Imagine writing a table layout manually using JavaScript objects:

```javascript
// Hard to read and write:
const table = {
  type: "table",
  props: {
    children: [
      {
        type: "tr",
        props: {
          children: [
            { type: "th", props: { children: ["Name"] } },
            { type: "th", props: { children: ["Age"] } }
          ]
        }
      }
    ]
  }
};
```

JSX solves this by providing a clean, HTML-like syntax that is easy for developers to write and read. `createElement` acts as the bridge, converting this readable syntax into the structured objects that the reconciliation engine needs.

## ④ INTERNAL WORKING

How do we build our own `createElement` function?

To make diffing simple, we must ensure that every child element is an object, including primitive values like strings and numbers. If a child is a string or a number, we will wrap it in a special element of type `"TEXT_ELEMENT"`.

### 🔹 The Implementation of `createElement`

Here is the complete implementation of our custom `createElement` function:

```javascript
function createElement(type, props, ...children) {
  return {
    type,
    props: {
      // 1. Copy all properties
      ...props,
      // 2. Process children: wrap primitives in TEXT_ELEMENT nodes
      children: children.map(child =>
        typeof child === "object"
          ? child
          : createTextElement(child)
      ),
    },
  };
}

function createTextElement(text) {
  return {
    type: "TEXT_ELEMENT",
    props: {
      nodeValue: text,
      children: [],
    },
  };
}
```

---

## ⑤ ADVANTAGES

1. **Simpler Traversal:** Wrapping text and number primitives in a uniform object structure allows the reconciler to process all nodes without needing type checks (`typeof child === 'string'`) at every step.
2. **Readability:** Developers can write clean HTML-like code, while the engine works with predictable, structured JavaScript objects.
3. **Validation Ready:** Having a single entry point for element creation allows us to validate attributes (props) and implement security checks (like adding `$$typeof`) before rendering.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Memory Allocation:** Every call to `createElement` allocates a new JavaScript object and array. For large pages, this can trigger frequent garbage collection cycles, especially during high-frequency updates.
2. **Text Node Overhead:** Wrapping every string in a `"TEXT_ELEMENT"` object increases memory usage, but this is a necessary trade-off for reconciler simplicity.

### ⚠️ Trap: Accidentally Mutating Props
React elements are supposed to be immutable. If you mutate the `props` object directly, you can bypass React's optimization checks, leading to visual bugs.

```javascript
// ❌ BAD: Modifying the element object after creation
const element = MyReact.createElement("div", { id: "box" });
element.props.id = "new-box"; // Mutation! Breaks optimization checks

// ✅ GOOD: Use immutable updates or pass properties at creation time
const element = MyReact.createElement("div", { id: "new-box" });
```

## ⑦ PRODUCTION USAGE

### Step-by-Step Custom JSX Setup

To test our custom `createElement` implementation, we need to instruct the compiler (Babel) to use our custom function instead of React's default.

#### Option A: Using the JSX Pragma Comment (Per-file basis)
Add this comment to the very top of your file. This tells Babel to transpile JSX tags using `MyReact.createElement`:

```javascript
/** @jsx MyReact.createElement */
const element = (
  <div id="container">
    <h1>Hello custom React</h1>
  </div>
);
```

#### Option B: Babel Configuration (Project-wide basis)
In a production Webpack or Vite project, configure `.babelrc` to use your custom library:

```json
{
  "plugins": [
    [
      "@babel/plugin-transform-react-jsx",
      {
        "pragma": "MyReact.createElement",
        "pragmaFrag": "MyReact.Fragment"
      }
    ]
  ]
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What does `React.createElement` return?**
**A:** It returns a plain JavaScript object representing a Virtual DOM element (often referred to as a React Element). This object contains properties like `type`, `props`, and `props.children`.

**Q2: What is the purpose of the JSX compiler?**
**A:** The JSX compiler (like Babel) transpiles the XML-like JSX tags in your code into standard JavaScript function calls (`React.createElement` or `_jsx`), which browsers can execute.

**Q3: How does React differentiate between a native HTML element and a custom React component in JSX?**
**A:** By checking the capitalization of the tag name:
- Lowercase tags (e.g., `<div />`) are compiled as string literals: `createElement("div")`.
- Capitalized tags (e.g., `<Button />`) are compiled as identifier references: `createElement(Button)`.

**Q4: Why does our custom `createElement` convert strings and numbers into `"TEXT_ELEMENT"` objects?**
**A:** This standardizes our Virtual DOM tree, ensuring that every node (whether it is an HTML container or a text leaf node) has a `props` object and a `children` array. This avoids the need for special type checks inside the reconciler.

**Q5: What are the three parameters of the `createElement` function?**
**A:**
1. `type`: The tag name or component function.
2. `props`: An object containing attributes and properties.
3. `...children`: A rest parameter containing all nested child elements.

---

### 🟡 Intermediate (5)

**Q6: Explain what JSX pragma is and how it is used.**
**A:** A JSX pragma is a compiler directive (often written as a comment like `/** @jsx pragmaName */`) that tells the JSX transpiler which function to use when converting JSX tags. For example, `/** @jsx MyReact.createElement */` tells Babel to compile JSX into `MyReact.createElement(...)` instead of React's default.

**Q7: How does Babel transpile JSX attributes (like `className` or custom data attributes)?**
**A:** Babel groups all JSX attributes into a single JavaScript object and passes it as the second argument (`props`) to `createElement`. For example, `<div class="box" data-id="123" />` becomes `createElement("div", { className: "box", "data-id": "123" })`.

**Q8: Explain the difference between React 17's automatic JSX runtime and the classic JSX compilation runtime.**
**A:**
- **Classic Runtime:** Transpiles `<div />` into `React.createElement("div", null)`. This requires importing `React` in every file where you use JSX.
- **Automatic Runtime:** Transpiles `<div />` into `_jsx("div", null)` from the `react/jsx-runtime` package. The compiler imports this runtime function automatically, so you no longer need to import `React` at the top of every file.

**Q9: If `children` is passed as a rest parameter `...children` in `createElement(type, props, ...children)`, how does React handle cases where no children are provided?**
**A:** If no children are passed, the rest parameter `children` resolves to an empty array. In our implementation, `props.children` will be an empty array (`[]`). In standard React, `children` is omitted from `props` entirely or set to `undefined` if no arguments are passed.

**Q10: Why does transpiled JSX use nested function calls instead of a flat array of elements?**
**A:** Nested function calls naturally represent the tree structure of the UI. The evaluation of nested functions starts from the innermost children and bubbles up, allowing parent elements to receive their fully constructed child objects as inputs.

---

### 🔴 Advanced (5)

**Q11: Explain how the Rest Parameter syntax `...children` in our `createElement` function affects performance, and how we could optimize it.**
**A:** Using rest parameters (`...children`) causes the JavaScript engine to allocate a new array on every function call. In a large UI tree, this creates thousands of short-lived arrays that trigger frequent garbage collection cycles. We can optimize this by modifying the function signature to accept `children` as a property inside the `props` object directly, or by passing pre-allocated array indices to avoid destructuring.

**Q12: If JSX transpiles `<div children="Hello" />` and `<div>World</div>`, which child representation takes precedence in `createElement`?**
**A:**
- In `<div children="Hello" />`, the children property is passed as part of the `props` object (the second argument of `createElement`).
- In `<div>World</div>`, the child is passed as the third argument (`...children`).
According to the specification, child arguments passed as parameters (the third argument onwards) take precedence and will overwrite any `children` property defined inside the `props` object.

**Q13: How can you write a custom JSX transpiler configuration in Babel that supports both our custom `createElement` and custom Fragments (`<></>`)?**
**A:** To support both, you must specify the pragma for elements and the pragma for fragments.
In `.babelrc`:
```json
{
  "plugins": [
    [
      "@babel/plugin-transform-react-jsx",
      {
        "pragma": "MyReact.createElement",
        "pragmaFrag": "MyReact.Fragment"
      }
    ]
  ]
}
```
Then, you must define `MyReact.Fragment` in your library as a unique symbol or placeholder (e.g., `const Fragment = Symbol('react.fragment')`) that your reconciler knows to skip.

**Q14: Explain the difference between React Elements, React Nodes, and React Components.**
**A:**
- **React Component:** A template (either a class or a function) that takes props and returns a React Element tree.
- **React Element:** A lightweight, immutable JavaScript object representation of a UI node returned by `createElement` (e.g., `{ type: 'div', props: {...} }`).
- **React Node:** A broader type that includes React Elements, strings, numbers, arrays, booleans, and null/undefined values.

**Q15: Why are React Element objects frozen (`Object.freeze`) in production?**
**A:** React elements are meant to represent a snapshot of the UI at a single point in time. Freezing them in production prevents developers from mutating their properties (such as modifying `props.children` directly after creation). This ensures the Virtual DOM remains immutable, allowing the reconciler to make safe optimization assumptions.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how the expression `<MyComponent title="Hello"><span>World</span></MyComponent>` is transpiled and evaluated by React.**
**A:**
1. Babel parses the JSX syntax. It identifies `MyComponent` starts with a capital letter, so it treats it as an identifier reference rather than a string.
2. Babel identifies `span` starts with a lowercase letter, so it treats it as a string literal.
3. Babel transpiles the outer tag using the pragma function:
   `MyReact.createElement(MyComponent, { title: "Hello" }, MyReact.createElement("span", null, "World"))`
4. During execution, JavaScript evaluates the arguments first. It calls the inner element factory:
   `MyReact.createElement("span", null, "World")`
5. This inner call returns a child Virtual DOM object:
   `{ type: "span", props: { children: [ { type: "TEXT_ELEMENT", props: { nodeValue: "World", children: [] } } ] } }`
6. JavaScript then calls the outer factory, passing the child object:
   `MyReact.createElement(MyComponent, { title: "Hello" }, childElementObject)`
7. The outer call returns the final element representation:
   `{ type: MyComponent, props: { title: "Hello", children: [ childElementObject ] } }`
8. React receives this element tree and hands it to the reconciler to mount or update the component.

**Q17: Explain step-by-step how to write a script that validates our custom `createElement` output to ensure it matches React's structural standards.**
**A:**
1. Write a validation helper function:
   ```javascript
   function validateElement(element) {
     if (typeof element !== "object" || element === null) {
       throw new Error("Element must be a non-null object");
     }
     if (!element.type) {
       throw new Error("Element must have a type property");
     }
     if (typeof element.props !== "object" || element.props === null) {
       throw new Error("Element must have a props object");
     }
     if (!Array.isArray(element.props.children)) {
       throw new Error("Element children must be an array");
     }
   }
   ```
2. Test an HTML tag element:
   ```javascript
   const el = createElement("div", { id: "test" }, "Hello");
   validateElement(el); // Passes. Returns { type: "div", props: { id: "test", children: [...] } }
   ```
3. Test that children are converted to objects:
   ```javascript
   const child = el.props.children[0];
   if (child.type !== "TEXT_ELEMENT" || child.props.nodeValue !== "Hello") {
     throw new Error("Text child was not wrapped correctly");
   }
   ```
4. Verify nested elements:
   ```javascript
   const nested = createElement("div", null, createElement("p", null, "Nested"));
   validateElement(nested);
   validateElement(nested.props.children[0]); // Verifies child <p> element
   ```
