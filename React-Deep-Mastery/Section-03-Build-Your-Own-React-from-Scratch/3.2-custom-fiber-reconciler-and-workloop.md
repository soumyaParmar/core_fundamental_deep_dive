# 📌 3.2 — Custom Fiber Reconciler & Work Loop

---

## ① DEFINITION

**Plain English:**
Imagine building a custom engine for a car. In Section 3.1, we built the fuel injector (`createElement`), which turns raw fuel into spray (JSX to Virtual DOM elements).
Now, we need to build the actual engine block (the **Reconciler**) and the timing belt (the **Work Loop**).
This engine takes our Virtual DOM elements and builds a network of tasks (the Fiber Tree). It runs in the background using the browser's idle time, so it won't freeze the screen. Once the entire tree is computed, it writes all the changes to the screen at once (the Commit phase).
This section implements a fully functional Fiber Reconciler from scratch in pure JavaScript.

**Technical Definition:**
A **Fiber Reconciler** is a scheduling and diffing engine. It converts a tree of React Elements into a linked list of Fiber nodes, schedules rendering work across frames using cooperative scheduling (`requestIdleCallback`), performs element diffing, and commits mutations to the DOM in a single transaction.

**One-line mental model:**
> *An asynchronous loop that converts virtual elements to linked Fibers, matches them, and commits DOM writes in a single, atomic transaction.*

**Where does this live?**
In our custom library, this is the core engine file (`reconciler.js`). In official React, it is implemented in `react-reconciler`.

## ② DRILL DOWN

Let's look at the core structure of our custom reconciler. It is built around three global variables that track our progress:

```javascript
let nextUnitOfWork = null; // The Fiber node we are currently working on
let wipRoot = null;        // The root of the workInProgress tree
let currentRoot = null;    // The root of the current tree (visible in the DOM)
let deletions = null;      // An array of Fibers that need to be removed from the DOM
```

```
         └─────────────────────────────────────────────────────────────────────────────────┘
```

## ③ WHY IT EXISTS

### What problem does it solve?

Traditional virtual DOM libraries (like early versions of Mithril or Snabbdom) use recursive diffing. When state changes, they recursively walk the entire virtual tree to find changes.
If the tree contains thousands of elements, this recursive walk takes over 50ms, causing the browser to drop frames and lag.

Our custom reconciler solves this by:
1. Converting the recursive tree traversal into an **iterative loop** using Fiber pointers (`child`, `sibling`, `return`).
2. Splitting the process into small chunks, checking the deadline timer (`deadline.timeRemaining()`) after each node.
3. Batching all DOM updates into a final, synchronous commit phase, avoiding partial layout updates.

## ④ INTERNAL WORKING

Here is the complete, self-contained implementation of our custom Fiber Reconciler. You can run this directly in a browser or test it in Node.js with a mock DOM.

### 🔹 The Custom Reconciler Implementation

```javascript
// 1. DOM Helper: Create a real DOM node from a Fiber node
function createDom(fiber) {
  const dom =
    fiber.type === "TEXT_ELEMENT"
      ? document.createTextNode("")
      : document.createElement(fiber.type);

  updateDom(dom, {}, fiber.props);

  return dom;
}

// 2. DOM Helper: Update DOM node attributes and event listeners
const isEvent = key => key.startsWith("on");
const isProperty = key => key !== "children" && !isEvent(key);
const isNew = (prev, next) => key => prev[key] !== next[key];
const isGone = (prev, next) => key => !(key in next);

function updateDom(dom, prevProps, nextProps) {
  // Remove old event listeners
  Object.keys(prevProps)
    .filter(isEvent)
    .filter(key => isGone(prevProps, nextProps)(key) || isNew(prevProps, nextProps)(key))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2);
      dom.removeEventListener(eventType, prevProps[name]);
    });

  // Remove old properties
  Object.keys(prevProps)
    .filter(isProperty)
    .filter(isGone(prevProps, nextProps))
    .forEach(name => {
      dom[name] = "";
    });

  // Add new properties
  Object.keys(nextProps)
    .filter(isProperty)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      dom[name] = nextProps[name];
    });

  // Add new event listeners
  Object.keys(nextProps)
    .filter(isEvent)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2);
      dom.addEventListener(eventType, nextProps[name]);
    });
}

// 3. Commit Phase: Apply all side effects to the real DOM
function commitRoot() {
  deletions.forEach(commitWork);
  commitWork(wipRoot.child);
  currentRoot = wipRoot;
  wipRoot = null;
}

function commitWork(fiber) {
  if (!fiber) return;

  // Find the parent DOM node (handles components that don't render a DOM node directly)
  let domParentFiber = fiber.return;
  while (!domParentFiber.dom) {
    domParentFiber = domParentFiber.return;
  }
  const domParent = domParentFiber.dom;

  if (fiber.effectTag === "PLACEMENT" && fiber.dom != null) {
    domParent.appendChild(fiber.dom);
  } else if (fiber.effectTag === "UPDATE" && fiber.dom != null) {
    updateDom(fiber.dom, fiber.alternate.props, fiber.props);
  } else if (fiber.effectTag === "DELETION") {
    commitDeletion(fiber, domParent);
    return; // Stop traversal for deleted nodes
  }

  commitWork(fiber.child);
  commitWork(fiber.sibling);
}

function commitDeletion(fiber, domParent) {
  if (fiber.dom) {
    domParent.removeChild(fiber.dom);
  } else {
    // If the Fiber is a component node, recursively find its first child DOM node to remove
    commitDeletion(fiber.child, domParent);
  }
}

// 4. Render Entrypoint: Sets up the initial root Fiber
function render(element, container) {
  wipRoot = {
    dom: container,
    props: {
      children: [element],
    },
    alternate: currentRoot,
  };
  deletions = [];
  nextUnitOfWork = wipRoot;
}

// 5. The Scheduling Loop
function workLoop(deadline) {
  let shouldYield = false;
  while (nextUnitOfWork && !shouldYield) {
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork);
    shouldYield = deadline.timeRemaining() < 1; // Yield if less than 1ms remains
  }

  if (!nextUnitOfWork && wipRoot) {
    commitRoot();
  }

  requestIdleCallback(workLoop);
}

requestIdleCallback(workLoop);

// 6. Perform Unit of Work: Process a single Fiber node and traverse the tree
function performUnitOfWork(fiber) {
  // Create a real DOM node if it doesn't exist
  if (!fiber.dom) {
    fiber.dom = createDom(fiber);
  }

  const elements = fiber.props.children;
  reconcileChildren(fiber, elements);

  // Return the next unit of work using DFS traversal
  if (fiber.child) {
    return fiber.child;
  }
  let nextFiber = fiber;
  while (nextFiber) {
    if (nextFiber.sibling) {
      return nextFiber.sibling;
    }
    nextFiber = nextFiber.return;
  }
  return null;
}

// 7. Reconciliation: Compare the old Fiber children with the new elements
function reconcileChildren(wipFiber, elements) {
  let index = 0;
  let oldFiber = wipFiber.alternate && wipFiber.alternate.child;
  let prevSibling = null;

  while (index < elements.length || oldFiber != null) {
    const element = elements[index];
    let newFiber = null;

    const sameType = oldFiber && element && element.type === oldFiber.type;

    if (sameType) {
      // UPDATE: Reuse DOM node, update props
      newFiber = {
        type: oldFiber.type,
        props: element.props,
        dom: oldFiber.dom,
        return: wipFiber,
        alternate: oldFiber,
        effectTag: "UPDATE",
      };
    }
    if (element && !sameType) {
      // PLACEMENT: Create new DOM node
      newFiber = {
        type: element.type,
        props: element.props,
        dom: null,
        return: wipFiber,
        alternate: null,
        effectTag: "PLACEMENT",
      };
    }
    if (oldFiber && !sameType) {
      // DELETION: Mark old Fiber for removal
      oldFiber.effectTag = "DELETION";
      deletions.push(oldFiber);
    }

    if (oldFiber) {
      oldFiber = oldFiber.sibling;
    }

    if (index === 0) {
      wipFiber.child = newFiber;
    } else if (element) {
      prevSibling.sibling = newFiber;
    }

    prevSibling = newFiber;
    index++;
  }
}

// Export custom MyReact library
const MyReact = {
  createElement: (type, props, ...children) => ({
    type,
    props: {
      ...props,
      children: children.map(child =>
        typeof child === "object" ? child : {
          type: "TEXT_ELEMENT",
          props: { nodeValue: child, children: [] }
        }
      )
    }
  }),
  render
};
```

## ⑤ ADVANTAGES

1. **Responsive UI:** Because reconciliation is broken down into small units, heavy updates will not block the browser thread, preventing input lag and animation stutters.
2. **Atomic DOM Writes:** All DOM mutations are applied in a single, synchronous transaction (`commitRoot`), preventing the user from seeing layout shifts or broken UI states.
3. **Monomorphic Node Traversal:** The reconciler treats DOM elements and text elements as uniform Fiber objects, keeping tree traversal fast and predictable.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Lack of Priority Levels:** Our custom implementation treats all tasks with the same priority. In a production reconciler, we need a lane system to prioritize interactive updates (like typing) over background renders (like fetching data).
2. **Synchronous Deletions List:** Storing deleted nodes in a flat array (`deletions`) can consume extra memory during large unmounts, but this is a necessary trade-off for simplicity.

### ⚠️ Trap: Memory Leak from Retaining Alternates
If a component is unmounted but its parent holds a reference to its Fiber node, the `alternate` pointers will keep the entire subtree in memory, preventing garbage collection.

```javascript
// Ensure alternate references are cleared when a tree is completely torn down
function commitDeletion(fiber, domParent) {
  if (fiber.dom) {
    domParent.removeChild(fiber.dom);
  } else {
    commitDeletion(fiber.child, domParent);
  }
  // Clear alternate pointers to allow garbage collection
  fiber.alternate = null; 
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the two main phases of our custom reconciler?**
**A:**
1. **The Render Phase:** Traverses the Fiber tree, performs diffing, and collects update tags (e.g. `PLACEMENT`, `UPDATE`, `DELETION`). This phase is pauseable.
2. **The Commit Phase:** Applies all collected DOM updates synchronously. This phase cannot be paused.

**Q2: What is the purpose of `requestIdleCallback` in our work loop?**
**A:** It tells the browser to run our work loop during its idle periods, ensuring that the reconciler only executes JavaScript when the browser has finished painting and handling user input.

**Q3: How does the work loop know when to yield to the browser?**
**A:** It calls `deadline.timeRemaining() < 1`. If the browser has less than 1 millisecond left in its current frame, the work loop yields control back to the event loop.

**Q4: What is the purpose of the `effectTag` property on a Fiber node?**
**A:** The `effectTag` records what DOM mutation is required for the node (e.g. `PLACEMENT` to insert, `UPDATE` to modify attributes, or `DELETION` to remove), which is read by the commit phase.

**Q5: Why do we need the `wipRoot` variable?**
**A:** `wipRoot` holds the root node of the new tree we are constructing in memory. It prevents us from committing a partially completed tree to the DOM if we are interrupted.

---

### 🟡 Intermediate (5)

**Q6: Explain why `commitWork` must search up the tree for a parent DOM element using a `while` loop.**
**A:** If we render a component (e.g., `<MyComponent><p>Hello</p></MyComponent>`), the component Fiber itself does not have a DOM element (`fiber.dom = null`). To append the child `<p>` tag, we must search up the parent pointers (`return`) until we find a Fiber that represents a physical DOM element (like `div` or the root container).

**Q7: How does `reconcileChildren` reuse existing DOM nodes when updating?**
**A:** It compares the `type` of the old Fiber (`oldFiber.type`) with the `type` of the new React element. If they are identical, it copies the DOM node reference from the old Fiber to the new Fiber (`newFiber.dom = oldFiber.dom`) and marks the node with the `UPDATE` tag.

**Q8: Explain how deleting a DOM node is handled differently from an update in our reconciler.**
**A:** When a node type changes or an item is removed:
1. `reconcileChildren` flags the old Fiber with `DELETION` and adds it to the global `deletions` array.
2. During the Commit phase, React loops through the `deletions` array and calls `removeChild` on the parent DOM container.

**Q9: What happens if `requestIdleCallback` is triggered when there is no rendering work scheduled?**
**A:** If `nextUnitOfWork` is `null`, the work loop will simply skip processing and reschedule itself using `requestIdleCallback(workLoop)`, waiting for the next render call.

**Q10: Why does `performUnitOfWork` return a Fiber node instead of executing recursively?**
**A:** By returning the next Fiber node (the first child, next sibling, or parent's sibling), we can exit the function after processing each node. This allows the work loop to check the deadline timer between nodes and pause if needed.

---

### 🔴 Advanced (5)

**Q11: Explain how our custom reconciler's DFS traversal behaves when reaching a leaf node that has no sibling.**
**A:**
1. `performUnitOfWork` processes the leaf node and finds `fiber.child` is `null`.
2. It enters a `while` loop to find a sibling:
   - If the current node has a sibling (`nextFiber.sibling`), it returns that sibling.
   - If not, it moves up to the parent (`nextFiber = nextFiber.return`).
3. The loop continues up the tree until it finds a parent with a sibling or reaches the root (`HostRoot`), returning `null` to stop the loop.

**Q12: If `updateDom` is called during the Commit phase, how does it handle removing event listeners that are no longer present?**
**A:** It filters the keys of the previous props for event names (starting with "on"). It checks if the event is missing from the new props or has changed. If so, it removes the listener from the DOM node using `dom.removeEventListener(eventType, prevProps[name])`.

**Q13: How does the reconciler prevent memory leaks from retaining deleted DOM elements?**
**A:** In a production reconciler, when a Fiber is deleted, we must clear its pointers (`child`, `sibling`, `return`, `alternate`) and remove any attached event listeners. This allows the JavaScript engine to garbage collect the detached DOM node and its children.

**Q14: Explain the difference in execution between a browser environment using `requestIdleCallback` and a Node.js test runner running our reconciler.**
**A:** Node.js does not run a browser layout engine or support `requestIdleCallback`. To test our reconciler in Node, we must mock `requestIdleCallback` using `setTimeout(fn, 1)` or custom macro-tasks, and mock the DOM using a library like `jsdom`.

**Q15: How would we modify `reconcileChildren` to support key-based diffing for list items?**
**A:** Instead of doing a simple index-to-index comparison:
1. We would map the old children by their `key` attribute.
2. For each new child, we would check if a matching key exists in the map.
3. If found, we would reuse the DOM node and update its properties.
4. If the new index differs from the old index, we would mark the Fiber for movement.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how the reconciler processes and renders a single static element tree (`<div><h1>Hello</h1></div>`) inside a container.**
**A:**
1. We call `MyReact.render(element, container)`.
2. The render function instantiates `wipRoot` with the container, sets `deletions = []`, and points `nextUnitOfWork = wipRoot`.
3. The browser runs the scheduled `workLoop`.
4. **Step 1 (`wipRoot`):** `performUnitOfWork(wipRoot)` runs. It finds one child element (`div`). `reconcileChildren` creates a child Fiber node for `div` and links `wipRoot.child = divFiber`. It returns the `divFiber`.
5. **Step 2 (`divFiber`):** `performUnitOfWork(divFiber)` runs. It creates a physical `div` DOM element (`divFiber.dom`). It reconciles `div`'s children, creating a child Fiber node for `h1` and linking `divFiber.child = h1Fiber`. It returns the `h1Fiber`.
6. **Step 3 (`h1Fiber`):** `performUnitOfWork(h1Fiber)` runs. It creates an `h1` DOM element (`h1Fiber.dom`). It reconciles `h1`'s children, creating a child Fiber node for the text "Hello" (`textFiber`). It returns the `textFiber`.
7. **Step 4 (`textFiber`):** `performUnitOfWork(textFiber)` runs. It creates a text DOM node. Since there are no children, it returns `null` to finish.
8. The DFS traversal moves up: `textFiber` has no child/sibling, so it returns to `h1Fiber` (no sibling), which returns to `divFiber` (no sibling), which returns to `wipRoot`.
9. The work loop finishes (`nextUnitOfWork = null`), and calls `commitRoot()`.
10. `commitRoot` traverses the tree: it appends the text node to `h1`, appends `h1` to `div`, and appends `div` to the root container DOM.
11. Finally, it sets `currentRoot = wipRoot` and finishes rendering.

---

**Q17: Explain step-by-step what happens internally in our custom reconciler when an element type changes (e.g. `<div>` is replaced by `<span>`).**
**A:**
1. During reconciliation, `reconcileChildren` compares the new element `{ type: 'span' }` with the old Fiber node `{ type: 'div' }`.
2. It checks `element.type === oldFiber.type` (`'span' === 'div'`), which evaluates to `false`.
3. Since the types differ, React does not reuse the node.
4. It flags the new `span` Fiber node with `PLACEMENT` and sets `dom = null`.
5. It flags the old `div` Fiber node with `DELETION` and pushes it to the `deletions` array.
6. The work loop continues and builds the rest of the tree.
7. During the Commit phase, React processes deletions first: it calls `parentDOM.removeChild(divDOM)`, removing the `div` and its children from the screen.
8. React then processes placements: it creates a new `span` DOM element, appends its children, and appends the `span` to the parent container.
