# 📌 3.3 — Custom Hooks: useState & useRef

---

## ① DEFINITION

**Plain English:**
In the previous sections, we built the HTML rendering engine. It can take static layout descriptions and draw them on the screen.
But real apps are not static; they have **state** (data that changes over time, like a shopping cart total) and **persistent values** (data that we want to remember across screen refreshes without redraws, like an input focus).
To handle this, React introduced **Hooks**. Under the hood, a hook is just a memory slot attached to the component's Fiber node.
Because functional components are just functions that run from top to bottom, they don't have instances to store variables. Every time the component renders, the variables are recreated.
To remember values, React keeps a list of memory slots on the component's Fiber. When you call `useState` or `useRef`, React returns the value from the current slot and moves a pointer to the next slot. This is why hooks must always be called in the **exact same order** on every render.

**Technical Definition:**
In functional components, state is stored on the Fiber node's `memoizedState` property. When a component renders, the reconciler tracks the active Fiber in a global pointer (`wipFiber`) and keeps a sequential index (`hookIndex`).
- **`useState`:** Retrieves the hook state from the active Fiber, applies any queued updates, and returns the current state along with a dispatch function. Calling the dispatch function schedules a new update.
- **`useRef`:** Retrieves the hook state from the active Fiber. If it is the initial mount, it creates a mutable object `{ current: initialValue }` and saves it. On subsequent renders, it returns the exact same object reference.

**One-line mental model:**
> *Hooks are a sequential memory array attached to a Fiber node, accessed by an index that resets on every render.*

**Where does this live?**
In our custom library, this is the hooks module. In official React, hook execution is managed by dispatchers in `ReactFiberHooks.js`.

## ② DRILL DOWN

Let's look at how hooks are stored inside a component's Fiber node.

### 🔹 Hook Storage: The Sequential Array Pattern
In our custom reconciler, we represent the component's hook memory as a flat array on the Fiber node: `fiber.hooks = []`.

Suppose we have a component with three hook calls:

```javascript
function ProfileComponent() {
  const [count, setCount] = useState(0);    // Hook Index 0
  const [name, setName] = useState("Alex");  // Hook Index 1
  const clickRef = useRef(0);                // Hook Index 2
  
  // ...
}
```

When React renders this component, it builds the following memory structure on the Fiber node:

```
                  wipFiber (ProfileComponent Fiber)
                   ├── hooks: [
                   │     0: { state: 0, queue: [] }              <-- useState
                   │     1: { state: "Alex", queue: [] }         <-- useState
                   │     2: { memoizedState: { current: 0 } }    <-- useRef
                   │   ]
                   └── memoizedState ──► (Points to head of list)
```

During rendering:
1. **Hook 0 (`useState`):** Resolves index `0`. Returns the state `0`.
2. **Hook 1 (`useState`):** Resolves index `1`. Returns the state `"Alex"`.
3. **Hook 2 (`useRef`):** Resolves index `2`. Returns the object `{ current: 0 }`.

If you call hooks conditionally:

```javascript
// ❌ CRITICAL BUG: Condition breaks the hook sequence!
if (condition) {
  const [count, setCount] = useState(0);
}
const [name, setName] = useState("Alex");
```

If `condition` changes from `true` to `false`:
- On the first render, `useState(0)` is at index `0`, and `useState("Alex")` is at index `1`.
- On the second render, `useState(0)` is skipped.
- `useState("Alex")` is now called first, resolving to index `0` and reading the value `0` from memory.
- The state values are now mixed up, leading to bugs.

---

## ③ WHY IT EXISTS

### What problem does it solve?

Before Hooks were introduced in React 16.8, state could only be managed inside Class Components. Functional components were purely visual ("dumb") wrappers.

Class components had several design challenges:
1. **Hard to Share Logic:** Code sharing required complex patterns like Higher-Order Components (HOCs) or Render Props, which led to nested "wrapper hell" in the component tree.
2. **Scattered Side Effects:** A single action (like fetching data on mount and cleaning it up on unmount) required writing code across three different lifecycle methods: `componentDidMount`, `componentDidUpdate`, and `componentWillUnmount`.
3. **Complex JS Classes:** Developers had to understand JavaScript's prototype inheritance, bind event handlers manually, and manage the context of the `this` keyword.

**The Hooks Solution:**
Hooks allow developers to write stateful logic using pure functions, separating concerns into custom reusable hook functions without modifying the component hierarchy.

## ④ INTERNAL WORKING

Here is the complete, self-contained implementation of our custom React library, including the Fiber Reconciler from Section 3.2 updated to support **Functional Components**, **`useState`**, and **`useRef`**.

### 🔹 Custom Reconciler with Hooks Support

```javascript
// 1. Globals for Reconciler & Hooks tracking
let nextUnitOfWork = null;
let wipRoot = null;
let currentRoot = null;
let deletions = null;
let wipFiber = null;  // The Fiber node currently rendering
let hookIndex = null; // The index of the hook currently being processed

// 2. DOM Helper: Create a real DOM node from a Fiber node
function createDom(fiber) {
  const dom =
    fiber.type === "TEXT_ELEMENT"
      ? document.createTextNode("")
      : document.createElement(fiber.type);

  updateDom(dom, {}, fiber.props);
  return dom;
}

// 3. DOM Helper: Update DOM properties and listeners
const isEvent = key => key.startsWith("on");
const isProperty = key => key !== "children" && !isEvent(key);
const isNew = (prev, next) => key => prev[key] !== next[key];
const isGone = (prev, next) => key => !(key in next);

function updateDom(dom, prevProps, nextProps) {
  Object.keys(prevProps)
    .filter(isEvent)
    .filter(key => isGone(prevProps, nextProps)(key) || isNew(prevProps, nextProps)(key))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2);
      dom.removeEventListener(eventType, prevProps[name]);
    });

  Object.keys(prevProps)
    .filter(isProperty)
    .filter(isGone(prevProps, nextProps))
    .forEach(name => {
      dom[name] = "";
    });

  Object.keys(nextProps)
    .filter(isProperty)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      dom[name] = nextProps[name];
    });

  Object.keys(nextProps)
    .filter(isEvent)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2);
      dom.addEventListener(eventType, nextProps[name]);
    });
}

// 4. Commit Phase
function commitRoot() {
  deletions.forEach(commitWork);
  commitWork(wipRoot.child);
  currentRoot = wipRoot;
  wipRoot = null;
}

function commitWork(fiber) {
  if (!fiber) return;

  // Search up parent pointers to find a physical DOM container
  let domParentFiber = fiber.return;
  while (!domParentFiber.dom) {
    domParentFiber = domParentFiber.return;
  }
  const domParent = domParentFiber.dom;

  if (fiber.effectTag === "PLACEMENT" && fiber.dom != null) {
    domParent.appendChild(fiber.dom);
  } else if (fiber.effectTag === "UPDATE" && fiber.dom != null) {
    updateDom(fiber.dom, fiber.alternate.props, fiber.props);
  } else if (fiber.effectTag === "DELETION") {
    commitDeletion(fiber, domParent);
    return;
  }

  commitWork(fiber.child);
  commitWork(fiber.sibling);
}

function commitDeletion(fiber, domParent) {
  if (fiber.dom) {
    domParent.removeChild(fiber.dom);
  } else {
    // If it's a function component Fiber, recursively find child DOM nodes to delete
    commitDeletion(fiber.child, domParent);
  }
}

// 5. Render Entrypoint
function render(element, container) {
  wipRoot = {
    dom: container,
    props: {
      children: [element],
    },
    alternate: currentRoot,
  };
  deletions = [];
  nextUnitOfWork = wipRoot;
}

// 6. Work Loop
function workLoop(deadline) {
  let shouldYield = false;
  while (nextUnitOfWork && !shouldYield) {
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork);
    shouldYield = deadline.timeRemaining() < 1;
  }

  if (!nextUnitOfWork && wipRoot) {
    commitRoot();
  }

  requestIdleCallback(workLoop);
}

requestIdleCallback(workLoop);

// 7. Perform Unit of Work
function performUnitOfWork(fiber) {
  const isFunctionComponent = typeof fiber.type === "function";
  if (isFunctionComponent) {
    updateFunctionComponent(fiber);
  } else {
    updateHostComponent(fiber);
  }

  // Traversal (DFS)
  if (fiber.child) {
    return fiber.child;
  }
  let nextFiber = fiber;
  while (nextFiber) {
    if (nextFiber.sibling) {
      return nextFiber.sibling;
    }
    nextFiber = nextFiber.return;
  }
  return null;
}

// 7a. Processing Function Components (Evaluating Hooks)
function updateFunctionComponent(fiber) {
  // Initialize hook tracking variables for the current Fiber
  wipFiber = fiber;
  hookIndex = 0;
  wipFiber.hooks = [];

  // Run the component function to retrieve its children
  const children = [fiber.type(fiber.props)];
  reconcileChildren(fiber, children);
}

// 7b. Processing Host HTML Elements
function updateHostComponent(fiber) {
  if (!fiber.dom) {
    fiber.dom = createDom(fiber);
  }
  reconcileChildren(fiber, fiber.props.children);
}

// 8. Reconciler Diffing
function reconcileChildren(wipFiber, elements) {
  let index = 0;
  let oldFiber = wipFiber.alternate && wipFiber.alternate.child;
  let prevSibling = null;

  while (index < elements.length || oldFiber != null) {
    const element = elements[index];
    let newFiber = null;

    const sameType = oldFiber && element && element.type === oldFiber.type;

    if (sameType) {
      newFiber = {
        type: oldFiber.type,
        props: element.props,
        dom: oldFiber.dom,
        return: wipFiber,
        alternate: oldFiber,
        effectTag: "UPDATE",
      };
    }
    if (element && !sameType) {
      newFiber = {
        type: element.type,
        props: element.props,
        dom: null,
        return: wipFiber,
        alternate: null,
        effectTag: "PLACEMENT",
      };
    }
    if (oldFiber && !sameType) {
      oldFiber.effectTag = "DELETION";
      deletions.push(oldFiber);
    }

    if (oldFiber) {
      oldFiber = oldFiber.sibling;
    }

    if (index === 0) {
      wipFiber.child = newFiber;
    } else if (element) {
      prevSibling.sibling = newFiber;
    }

    prevSibling = newFiber;
    index++;
  }
}

// 9. Custom Hook: useState
function useState(initial) {
  // Check if a previous state exists in the alternate Fiber
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  const hook = {
    state: oldHook ? oldHook.state : initial,
    queue: [],
  };

  // Run pending state updates from the queue
  const actions = oldHook ? oldHook.queue : [];
  actions.forEach(action => {
    hook.state = typeof action === "function" ? action(hook.state) : action;
  });

  // State dispatcher function
  const setState = action => {
    hook.queue.push(action);
    // Schedule a new render from the root
    wipRoot = {
      dom: currentRoot.dom,
      props: currentRoot.props,
      alternate: currentRoot,
    };
    nextUnitOfWork = wipRoot;
    deletions = [];
  };

  wipFiber.hooks.push(hook);
  hookIndex++;
  return [hook.state, setState];
}

// 10. Custom Hook: useRef
function useRef(initial) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  const hook = {
    memoizedState: oldHook ? oldHook.memoizedState : { current: initial },
  };

  wipFiber.hooks.push(hook);
  hookIndex++;
  return hook.memoizedState;
}

// Export custom library
const MyReact = {
  createElement: (type, props, ...children) => ({
    type,
    props: {
      ...props,
      children: children.map(child =>
        typeof child === "object" ? child : {
          type: "TEXT_ELEMENT",
          props: { nodeValue: child, children: [] }
        }
      )
    }
  }),
  render,
  useState,
  useRef
};
```

## ⑤ ADVANTAGES

1. **Stateful Functional Components:** Functions can manage dynamic state and persist references without the complexity of ES6 classes.
2. **Batch Update Ready:** State updaters queue actions and schedule updates, allowing multiple actions to process in a single render cycle.
3. **No Wrapper Hell:** Logic can be extracted into custom helper functions, avoiding nested component wrappers.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Stale Closures Trap:** Since functional components capture variables during execution, asynchronous callbacks (like `setTimeout`) can capture old values unless updated references are stored in a `ref`.
2. **Hook Order Restriction:** Hooks cannot be called inside loops, conditions, or nested functions.

### ⚠️ Trap: Stale Closure inside Asynchronous Callback

```jsx
// ❌ BAD: Asynchronous callback captures stale state
function Counter() {
  const [count, setCount] = MyReact.useState(0);
  
  const handleAlert = () => {
    setTimeout(() => {
      // If count updates from 0 to 3, this alert will still show "0"
      // because it captured the state from the initial render cycle!
      alert(`Current count: ${count}`); 
    }, 3000);
  };
}

// ✅ GOOD: Use useRef to read the latest value
function Counter() {
  const [count, setCount] = MyReact.useState(0);
  const countRef = MyReact.useRef(0);
  
  // Keep the ref in sync
  countRef.current = count; 
  
  const handleAlert = () => {
    setTimeout(() => {
      alert(`Current count: ${countRef.current}`); // Always reads the latest value
    }, 3000);
  };
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: Where does a functional component store its state in React?**
**A:** In the component's Fiber node, specifically inside a singly linked list (or array in our custom implementation) attached to the `memoizedState` property.

**Q2: Why must React hooks be called at the top level of a component and not inside conditions?**
**A:** Because React matches hook values using their call index. If a hook call is skipped inside a conditional branch, all subsequent hooks will slide down one slot, resulting in mismatched values.

**Q3: What is the main difference between `useState` and `useRef`?**
**A:** 
- `useState` returns a state variable and a dispatch function. Calling the dispatcher updates the state and schedules a component re-render.
- `useRef` returns a mutable object `{ current: value }` that persists across renders. Mutating the `.current` property does not trigger a re-render.

**Q4: What does the `wipFiber` variable track in our custom implementation?**
**A:** It tracks the Fiber node of the component currently executing. This allows hooks (like `useState`) to access the active Fiber and save state or queues to it.

**Q5: How does the `setState` dispatch function trigger a re-render?**
**A:** It creates an update object, adds it to the hook's queue, instantiates a new `wipRoot` using the `currentRoot` properties, and sets `nextUnitOfWork = wipRoot` to trigger the scheduler.

---

### 🟡 Intermediate (5)

**Q6: Explain what happens to the hook state array when a component is unmounted and then remounted.**
**A:** When a component unmounts, its Fiber node is garbage collected, destroying the attached state array. When the component remounts, React creates a new Fiber node with an empty state array, initializing all hooks from scratch.

**Q7: How does our custom `useState` handle function-based updaters (e.g., `setCount(c => c + 1)`) vs direct values (e.g., `setCount(5)`)?**
**A:** It checks the type of the action. If the action is a function (`typeof action === 'function'`), it executes it by passing the current state: `action(currentState)`. If it is a primitive, it returns the value directly.

**Q8: Explain the "Stale Closures" problem in functional components.**
**A:** Functional components are JavaScript functions that run from scratch on every render, capturing variables (including props and state) in a closure. If a callback (like an asynchronous `setTimeout`) executes after a state update, it will refer to the stale variables captured during the older render cycle.

**Q9: Why does `updateFunctionComponent` reset `hookIndex` to `0`?**
**A:** Functional components run from scratch on every render. Resetting the index to `0` ensures that when the component calls its hooks again, they match the sequence of slots in the Fiber's state array.

**Q10: Why doesn't mutating `ref.current` trigger a component re-render?**
**A:** Because mutating the `.current` property of the ref object is a simple reference mutation. Unlike a state dispatcher (`setState`), it does not schedule updates or register a new `wipRoot` with the reconciler.

---

### 🔴 Advanced (5)

**Q11: Explain how React stores hooks internally using a Singly Linked List instead of an array.**
**A:** Standard React uses a linked list of hook objects where each node contains:
- `memoizedState`: The hook's state.
- `queue`: Pending updates.
- `next`: Pointer to the next hook in the component sequence.
The Fiber's `memoizedState` points to the head of this list. When rendering a component, React navigates the list using the `next` pointer, which matches the behavior of an array index pointer.

**Q12: If we trigger a state update with the same value (e.g. `setState(currentValue)`), how does React prevent unnecessary renders?**
**A:** This is called a **Bailout**. React compares the new state with the current state using `Object.is` (strict equality). If they are identical, React bails out of the update and avoids scheduling a new render cycle.

**Q13: How does React ensure that event listeners in functional components capture the latest state without causing memory leaks?**
**A:** During the Commit phase, React binds event listeners to the DOM node. When a component re-renders and generates new event handlers, React replaces the event listeners on the DOM node (via `removeEventListener` and `addEventListener`) to ensure the callbacks have access to the latest state.

**Q14: Explain the difference between hook storage in React's Concurrent Mode vs older synchronous rendering.**
**A:** In Concurrent Mode, multiple versions of the state can exist simultaneously (e.g. if an update is interrupted). React manages this by assigning priority lanes to updates in the hook's queue. During rendering, React only processes updates that match the priority of the current render lane, keeping other updates in the queue for a future cycle.

**Q15: How can you write a custom hook (e.g., `usePrevious(value)`) using `useRef`?**
**A:**
```javascript
function usePrevious(value) {
  const ref = useRef();
  // useEffect runs after the component renders.
  // The ref holds the value from the previous render cycle during the current execution.
  useEffect(() => {
    ref.current = value;
  }, [value]);
  return ref.current;
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React resolves two consecutive state updates (`setCount(c => c + 1)` and `setCount(c => c + 2)`) inside a click handler.**
**A:**
1. The user clicks a button, triggering the handler.
2. The handler calls `setCount(c => c + 1)`. The dispatcher pushes the updater function to the hook's queue (`[c => c + 1]`).
3. The dispatcher schedules an update by creating a new `wipRoot` and setting `nextUnitOfWork = wipRoot`.
4. The handler calls `setCount(c => c + 2)`. The dispatcher pushes this function to the same hook queue (`[c => c + 1, c => c + 2]`).
5. The handler finishes executing, and the main thread yields to the Scheduler.
6. The Scheduler executes the work loop.
7. During the Render phase, React traverses down to the counter component.
8. `updateFunctionComponent` runs. It resets `hookIndex = 0` and runs the component function.
9. Inside the component, `useState(0)` runs at index `0`:
   - It reads the current state `0` from the old Fiber's hook.
   - It loops through the hook's queue:
     - Applies `0 + 1 = 1`.
     - Applies `1 + 2 = 3`.
   - It saves the final state `3` to the new hook object.
   - It clears the queue.
10. The component finishes rendering with `count = 3` and updates the DOM in a single commit.

---

**Q17: Explain step-by-step how `useRef` maintains a stable reference to a DOM node throughout a component's lifecycle.**
**A:**
1. You call `const myRef = useRef(null)` in a component.
2. During the initial mount:
   - React creates a hook object at the current index.
   - It initializes `memoizedState` with a mutable reference object: `{ current: null }`.
   - It returns this object, which you pass to a JSX element: `<div ref={myRef} />`.
3. The reconciler processes the element, creates a DOM node, and flags the Fiber node with `Ref`.
4. During the Layout phase of the commit, React sees the `Ref` flag. It assigns the real DOM node to the ref object: `myRef.current = domNode`.
5. When the component re-renders:
   - React runs the component function again.
   - `useRef` resolves to the same hook index and reads the existing hook state from the old Fiber node.
   - It returns the exact same object reference (`myRef`).
6. Because the object reference is preserved, `myRef.current` points to the active DOM element across all subsequent render cycles.
