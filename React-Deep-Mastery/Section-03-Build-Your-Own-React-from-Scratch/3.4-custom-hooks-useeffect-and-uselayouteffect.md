# 📌 3.4 — Custom Hooks: useEffect & useLayoutEffect

---

## ① DEFINITION

**Plain English:**
Imagine you are setting up a smart TV.
When you plug it in, you want to immediately configure the physical stand and adjust the screen height so it sits correctly on your table (this represents **`useLayoutEffect`** — it runs synchronously, blocking the paint cycle until it is completed, ensuring there is no visual flicker).
Once the TV is positioned, you connect it to Wi-Fi, download updates, and set up streaming accounts in the background (this represents **`useEffect`** — it runs asynchronously after the TV is fully set up, without blocking your initial viewing experience).
When you replace the TV, you want to log out of your streaming accounts first (the **cleanup function**) before setting up the new TV.
In React, these lifecycle events are managed by effect hooks. They allow functional components to sync with external systems (like databases, APIs, or physical DOM structures) based on dependency changes.

**Technical Definition:**
- **`useLayoutEffect`:** Runs synchronously during the Layout phase of the commit process, after DOM mutations are applied but before the browser paints the screen.
- **`useEffect`:** Schedules a macro-task to execute its callback asynchronously after the commit phase completes and the browser has painted the screen.
Both hooks accept a callback function and a dependency array. The callback runs only if the dependency array has changed since the last render. If the callback returns a function, React treats it as a cleanup function and runs it before executing the next callback or when the component unmounts.

**One-line mental model:**
> *useLayoutEffect runs synchronously before the browser paints; useEffect runs asynchronously after the paint.*

**Where does this live?**
In our custom library, this is the effects runner. In official React, it is managed by the reconciler's hook manager in `ReactFiberHooks.js` and executed during the commit phase in `ReactFiberCommitWork.js`.

---

## ② DRILL DOWN

Let's look at the lifecycle timeline of these hooks relative to DOM mutations and the browser paint cycle:

```
[Commit Phase: Mutation]
  └── DOM updates are applied to the screen (Placement, Update, Deletion)
  └── Synchronous layout cleanup functions run (e.g. useLayoutEffect cleanup)
       │
       ▼
[Commit Phase: Layout]
  └── myRef.current is updated with DOM nodes
  └── useLayoutEffect callback functions run synchronously (Blocks browser paint!)
       │
       ▼
[Browser Reflow & Paint] (User sees the updated UI)
       │
       ▼
[Post-Paint Event (setTimeout / macro-task)]
  └── Asynchronous passive cleanup functions run (e.g. useEffect cleanup)
  └── Asynchronous passive callback functions run (e.g. useEffect callback)
```

## ③ WHY IT EXISTS

### What problem does it solve?

In class components, side effects were managed across multiple lifecycle methods:

```javascript
class Timer extends React.Component {
  componentDidMount() {
    this.timerId = setInterval(this.tick, 1000); // 1. Start timer
  }
  componentDidUpdate(prevProps) {
    if (this.props.userId !== prevProps.userId) {
      this.resetUserLogs(); // 2. Log changes
    }
  }
  componentWillUnmount() {
    clearInterval(this.timerId); // 3. Clean up timer
  }
}
```

This design created several issues:
1. **Scattered Logic:** The code to manage the timer was split between `componentDidMount` and `componentWillUnmount`.
2. **High Complexity:** If a component managed three different side-effects (e.g., API calls, timers, and event listeners), the code for all three was mixed together inside the same lifecycle methods, making it hard to maintain.

**The Effects Solution:**
`useEffect` groups side-effect code by concern. A single hook call can manage both setup and cleanup, keeping the logic self-contained.

## ④ INTERNAL WORKING

Here is the complete, self-contained implementation of our custom React library, combining the Fiber Reconciler and Hooks from previous sections with support for **`useEffect`** and **`useLayoutEffect`**.

### 🔹 Custom Reconciler with Hooks & Effects Support

```javascript
// 1. Globals for Reconciler, Hooks, and Effects tracking
let nextUnitOfWork = null;
let wipRoot = null;
let currentRoot = null;
let deletions = null;
let wipFiber = null;
let hookIndex = null;

// Queues to track effects that need to run
let effectQueue = [];       // Holds useEffect callbacks and cleanups
let layoutEffectQueue = []; // Holds useLayoutEffect callbacks and cleanups

// 2. DOM Helper: Create a real DOM node
function createDom(fiber) {
  const dom =
    fiber.type === "TEXT_ELEMENT"
      ? document.createTextNode("")
      : document.createElement(fiber.type);

  updateDom(dom, {}, fiber.props);
  return dom;
}

// 3. DOM Helper: Update properties and listeners
const isEvent = key => key.startsWith("on");
const isProperty = key => key !== "children" && !isEvent(key);
const isNew = (prev, next) => key => prev[key] !== next[key];
const isGone = (prev, next) => key => !(key in next);

function updateDom(dom, prevProps, nextProps) {
  Object.keys(prevProps)
    .filter(isEvent)
    .filter(key => isGone(prevProps, nextProps)(key) || isNew(prevProps, nextProps)(key))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2);
      dom.removeEventListener(eventType, prevProps[name]);
    });

  Object.keys(prevProps)
    .filter(isProperty)
    .filter(isGone(prevProps, nextProps))
    .forEach(name => {
      dom[name] = "";
    });

  Object.keys(nextProps)
    .filter(isProperty)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      dom[name] = nextProps[name];
    });

  Object.keys(nextProps)
    .filter(isEvent)
    .filter(isNew(prevProps, nextProps))
    .forEach(name => {
      const eventType = name.toLowerCase().substring(2);
      dom.addEventListener(eventType, nextProps[name]);
    });
}

// 4. Commit Phase
function commitRoot() {
  deletions.forEach(commitWork);
  commitWork(wipRoot.child);
  
  // A. Execute Layout Effects synchronously (pre-paint)
  runLayoutEffects();

  currentRoot = wipRoot;
  wipRoot = null;

  // B. Schedule Passive Effects (useEffect) asynchronously (post-paint)
  setTimeout(runEffects, 0);
}

function commitWork(fiber) {
  if (!fiber) return;

  let domParentFiber = fiber.return;
  while (!domParentFiber.dom) {
    domParentFiber = domParentFiber.return;
  }
  const domParent = domParentFiber.dom;

  if (fiber.effectTag === "PLACEMENT" && fiber.dom != null) {
    domParent.appendChild(fiber.dom);
  } else if (fiber.effectTag === "UPDATE" && fiber.dom != null) {
    updateDom(fiber.dom, fiber.alternate.props, fiber.props);
  } else if (fiber.effectTag === "DELETION") {
    commitDeletion(fiber, domParent);
    return;
  }

  commitWork(fiber.child);
  commitWork(fiber.sibling);
}

function commitDeletion(fiber, domParent) {
  if (fiber.dom) {
    domParent.removeChild(fiber.dom);
  } else {
    commitDeletion(fiber.child, domParent);
  }
  // Run cleanups for components that are being unmounted
  if (fiber.hooks) {
    fiber.hooks.forEach(hook => {
      if (hook.cleanup) {
        hook.cleanup();
      }
    });
  }
}

// 5. Run Effects Helpers
function runLayoutEffects() {
  // 1. Run cleanups synchronously first
  layoutEffectQueue.forEach(effect => {
    if (effect.cleanup) {
      effect.cleanup();
      effect.cleanup = null;
    }
  });

  // 2. Run layouts synchronously
  layoutEffectQueue.forEach(effect => {
    effect.cleanup = effect.callback();
  });

  layoutEffectQueue = []; // Clear queue
}

function runEffects() {
  // 1. Run cleanups asynchronously first
  effectQueue.forEach(effect => {
    if (effect.cleanup) {
      effect.cleanup();
      effect.cleanup = null;
    }
  });

  // 2. Run effects asynchronously
  effectQueue.forEach(effect => {
    effect.cleanup = effect.callback();
  });

  effectQueue = []; // Clear queue
}

// 6. Render Entrypoint
function render(element, container) {
  wipRoot = {
    dom: container,
    props: {
      children: [element],
    },
    alternate: currentRoot,
  };
  deletions = [];
  nextUnitOfWork = wipRoot;
}

// 7. Work Loop
function workLoop(deadline) {
  let shouldYield = false;
  while (nextUnitOfWork && !shouldYield) {
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork);
    shouldYield = deadline.timeRemaining() < 1;
  }

  if (!nextUnitOfWork && wipRoot) {
    commitRoot();
  }

  requestIdleCallback(workLoop);
}

requestIdleCallback(workLoop);

// 8. Perform Unit of Work
function performUnitOfWork(fiber) {
  const isFunctionComponent = typeof fiber.type === "function";
  if (isFunctionComponent) {
    updateFunctionComponent(fiber);
  } else {
    updateHostComponent(fiber);
  }

  if (fiber.child) {
    return fiber.child;
  }
  let nextFiber = fiber;
  while (nextFiber) {
    if (nextFiber.sibling) {
      return nextFiber.sibling;
    }
    nextFiber = nextFiber.return;
  }
  return null;
}

function updateFunctionComponent(fiber) {
  wipFiber = fiber;
  hookIndex = 0;
  wipFiber.hooks = [];

  const children = [fiber.type(fiber.props)];
  reconcileChildren(fiber, children);
}

function updateHostComponent(fiber) {
  if (!fiber.dom) {
    fiber.dom = createDom(fiber);
  }
  reconcileChildren(fiber, fiber.props.children);
}

// 9. Reconciler Diffing
function reconcileChildren(wipFiber, elements) {
  let index = 0;
  let oldFiber = wipFiber.alternate && wipFiber.alternate.child;
  let prevSibling = null;

  while (index < elements.length || oldFiber != null) {
    const element = elements[index];
    let newFiber = null;

    const sameType = oldFiber && element && element.type === oldFiber.type;

    if (sameType) {
      newFiber = {
        type: oldFiber.type,
        props: element.props,
        dom: oldFiber.dom,
        return: wipFiber,
        alternate: oldFiber,
        effectTag: "UPDATE",
      };
    }
    if (element && !sameType) {
      newFiber = {
        type: element.type,
        props: element.props,
        dom: null,
        return: wipFiber,
        alternate: null,
        effectTag: "PLACEMENT",
      };
    }
    if (oldFiber && !sameType) {
      oldFiber.effectTag = "DELETION";
      deletions.push(oldFiber);
    }

    if (oldFiber) {
      oldFiber = oldFiber.sibling;
    }

    if (index === 0) {
      wipFiber.child = newFiber;
    } else if (element) {
      prevSibling.sibling = newFiber;
    }

    prevSibling = newFiber;
    index++;
  }
}

// Helper: Check if dependencies have changed
function areDepsEqual(prevDeps, nextDeps) {
  if (!prevDeps || !nextDeps) return false;
  if (prevDeps.length !== nextDeps.length) return false;
  for (let i = 0; i < prevDeps.length; i++) {
    if (prevDeps[i] !== nextDeps[i]) return false;
  }
  return true;
}

// 10. Custom Hook: useState
function useState(initial) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  const hook = {
    state: oldHook ? oldHook.state : initial,
    queue: [],
  };

  const actions = oldHook ? oldHook.queue : [];
  actions.forEach(action => {
    hook.state = typeof action === "function" ? action(hook.state) : action;
  });

  const setState = action => {
    hook.queue.push(action);
    wipRoot = {
      dom: currentRoot.dom,
      props: currentRoot.props,
      alternate: currentRoot,
    };
    nextUnitOfWork = wipRoot;
    deletions = [];
  };

  wipFiber.hooks.push(hook);
  hookIndex++;
  return [hook.state, setState];
}

// 11. Custom Hook: useRef
function useRef(initial) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  const hook = {
    memoizedState: oldHook ? oldHook.memoizedState : { current: initial },
  };

  wipFiber.hooks.push(hook);
  hookIndex++;
  return hook.memoizedState;
}

// 12. Custom Hook: useEffect
function useEffect(callback, deps) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  const hasChanged = !oldHook || !areDepsEqual(oldHook.deps, deps);

  const hook = {
    deps,
    cleanup: oldHook ? oldHook.cleanup : null,
  };

  if (hasChanged) {
    // Queue the effect to run asynchronously post-commit
    effectQueue.push({
      callback,
      // Pass the cleanup function wrapper
      get cleanup() { return hook.cleanup; },
      set cleanup(val) { hook.cleanup = val; }
    });
  }

  wipFiber.hooks.push(hook);
  hookIndex++;
}

// 13. Custom Hook: useLayoutEffect
function useLayoutEffect(callback, deps) {
  const oldHook =
    wipFiber.alternate &&
    wipFiber.alternate.hooks &&
    wipFiber.alternate.hooks[hookIndex];

  const hasChanged = !oldHook || !areDepsEqual(oldHook.deps, deps);

  const hook = {
    deps,
    cleanup: oldHook ? oldHook.cleanup : null,
  };

  if (hasChanged) {
    // Queue the layout effect to run synchronously during commit
    layoutEffectQueue.push({
      callback,
      get cleanup() { return hook.cleanup; },
      set cleanup(val) { hook.cleanup = val; }
    });
  }

  wipFiber.hooks.push(hook);
  hookIndex++;
}

// Export custom library
const MyReact = {
  createElement: (type, props, ...children) => ({
    type,
    props: {
      ...props,
      children: children.map(child =>
        typeof child === "object" ? child : {
          type: "TEXT_ELEMENT",
          props: { nodeValue: child, children: [] }
        }
      )
    }
  }),
  render,
  useState,
  useRef,
  useEffect,
  useLayoutEffect
};
```

## ⑤ ADVANTAGES

1. **Simplified Concern Groups:** Related side-effect code (setup and cleanup) is kept in a single hook, rather than split across class lifecycles.
2. **Prevent Paint Lag:** `useEffect` runs asynchronously post-paint, preventing network fetches or heavy computations from delaying screen updates.
3. **Flicker-Free Updates:** `useLayoutEffect` runs synchronously before paint, allowing you to read layout dimensions and update positions without creating visual flickering.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Infinite Re-render loops:** If you update state inside `useEffect` or `useLayoutEffect` without providing a dependency array (or passing dependencies that change on every render), the component will enter an infinite loop.
2. **Synchronous Performance Bottleneck:** Since `useLayoutEffect` executes synchronously before paint, running heavy computations inside it will block the main thread and cause UI lag.

### ⚠️ Trap: Infinite Re-render Loop

```jsx
// ❌ BAD: State update inside effect without a dependency array
function BadApp() {
  const [data, setData] = MyReact.useState([]);
  
  MyReact.useEffect(() => {
    // This updates state, which triggers a re-render.
    // Since there is no dependency array, the effect runs again,
    // causing an infinite loop!
    fetch('/api/data')
      .then(res => res.json())
      .then(res => setData(res)); 
  }); 
}

// ✅ GOOD: Pass a dependency array to control when the effect runs
function GoodApp() {
  const [data, setData] = MyReact.useState([]);
  
  MyReact.useEffect(() => {
    fetch('/api/data')
      .then(res => res.json())
      .then(res => setData(res));
  }, []); // Empty array ensures this runs only once on mount
}
```

## ⑦ PRODUCTION USAGE

### Building a Clock Component with Our Library

Here is how you can use our library to build a component that manages a timer and cleans it up when unmounted.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Custom Effects Demo</title>
</head>
<body>
  <div id="root"></div>

  <script src="./reconciler-effects.js"></script>
  <script>
    /** @jsx MyReact.createElement */

    function TimerComponent() {
      const [seconds, setSeconds] = MyReact.useState(0);

      MyReact.useEffect(() => {
        // 1. Setup: Start the interval timer
        const timerId = setInterval(() => {
          setSeconds(s => s + 1);
        }, 1000);

        // 2. Cleanup: Clear the interval timer when unmounting
        return () => {
          clearInterval(timerId);
        };
      }, []); // Run only on mount

      return (
        <div style="font-family: sans-serif; padding: 20px;">
          <h1>Time Elapsed: {seconds}s</h1>
          <p>This timer updates every second and cleans up correctly when unmounted.</p>
        </div>
      );
    }

    MyReact.render(<TimerComponent />, document.getElementById("root"));
  </script>
</body>
</html>
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is the main difference in execution timing between `useEffect` and `useLayoutEffect`?**
**A:**
- `useLayoutEffect` executes synchronously during the layout sub-phase of the Commit phase, before the browser paints the screen.
- `useEffect` executes asynchronously after the Commit phase is complete and the browser has painted the screen.

**Q2: What is the purpose of the cleanup function returned by `useEffect`?**
**A:** It cleans up resources allocated by the effect (like clearing timers, unsubscribing from event listeners, or canceling network requests) to prevent memory leaks before the next effect runs or when the component unmounts.

**Q3: What happens if you omit the dependency array in `useEffect`?**
**A:** The effect's callback function will run after every single render of the component, which can lead to performance issues or infinite loops if state is updated inside the callback.

**Q4: How does an empty dependency array `[]` affect the execution of `useEffect`?**
**A:** It tells React that the effect does not depend on any state or props, so the callback runs only once when the component mounts, and the cleanup runs only once when the component unmounts.

**Q5: What are the two queues we introduced in our custom reconciler to handle effects?**
**A:**
1. `effectQueue`: Stores `useEffect` callbacks to execute asynchronously post-paint.
2. `layoutEffectQueue`: Stores `useLayoutEffect` callbacks to execute synchronously during the Commit phase.

---

### 🟡 Intermediate (5)

**Q6: Explain why running cleanups first is necessary before running new effects.**
**A:** Running cleanups first ensures that old resources (like active subscriptions or open database connections) are closed before new ones are created, preventing resource duplication, connection leaks, and state conflicts.

**Q7: Explain the bug that occurs if you read state variables inside `useEffect` without including them in the dependency array.**
**A:** The effect's closure will only capture the state values from the render cycle in which it was created. If you update the state later, the effect will continue to read the stale, older values from that initial closure, leading to incorrect calculations.

**Q8: Why does `useLayoutEffect` block browser painting?**
**A:** Because it runs synchronously inside the Commit phase on the same thread that handles browser rendering. The browser cannot perform paint or layout calculations until all synchronous JavaScript execution in the Commit phase has finished.

**Q9: How does our custom `areDepsEqual` helper determine if dependencies have changed?**
**A:** It loops through the values of both dependency arrays at the same index and performs a shallow equality check using the strict equality operator (`!==`). If any pair of values differs, it returns `false`, indicating that the dependencies have changed.

**Q10: Why does `commitDeletion` run the cleanups of unmounting components?**
**A:** When a component is unmounted, its DOM elements are removed. If we didn't run its cleanups, resources like active intervals or event listeners bound to global objects would remain active in memory, causing memory leaks.

---

### 🔴 Advanced (5)

**Q11: Explain how React avoids stale closures inside the cleanup functions of `useEffect`.**
**A:** When a component renders, it generates a new cleanup function. React saves this reference in the Fiber node's hook object. Before executing a new effect callback, React retrieves the cleanup function from the *previous* hook state, ensuring the cleanup closure captures the correct variables from that older render cycle.

**Q12: Why is `useLayoutEffect` preferred over `useEffect` when mutating DOM nodes directly to adjust visual properties?**
**A:**
- If you use `useEffect`, the callback runs after the browser paints. The user will see the initial layout first, followed immediately by the adjusted layout, resulting in a visual flicker.
- If you use `useLayoutEffect`, the callback runs before paint. The DOM adjustments are computed and painted in a single cycle, ensuring a smooth, flicker-free update.

**Q13: How does React schedule `useEffect` calls in a browser environment to ensure they run after paint without blocking layout?**
**A:** Standard React uses its `Scheduler` package. During the Commit phase, it registers a macro-task callback using the browser's event loop (via `MessageChannel`). When the macro-task executes, it runs the passive effects queue, ensuring the paint cycle completes before the effects run.

**Q14: Explain the difference in execution order of cleanups and callbacks between nested parent and child components.**
**A:**
- **Cleanups:** Executed in a top-down order (parent cleanups run before child cleanups).
- **Callbacks (Mounts):** Executed in a bottom-up order (child effects run before parent effects).
This ensures that when a parent component mounts, all its children have finished mounting and setting up their resources.

**Q15: How can you implement a custom `useInsertionEffect` in our reconciler, and what is its use case?**
**A:**
`useInsertionEffect` runs synchronously *before* layout effects, before DOM nodes are read. It is used by CSS-in-JS libraries (like styled-components) to inject `<style>` tags into the DOM before layout calculations run, avoiding recalculations during reflows.
In our reconciler, we would add an `insertionEffectQueue` and run it at the very beginning of the commit phase, before processing the layout queue.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React handles mounting a component that has a `useEffect` call with an empty dependency array `[]`.**
**A:**
1. During the Render phase, React processes the component's Fiber node and runs its function.
2. `useEffect(callback, [])` is called. React notices it is the initial mount, creates a hook object, and stores the dependency array `[]` and the callback.
3. Because the dependencies changed (from undefined on mount), React marks the hook for execution.
4. The work loop completes and enters the Commit phase.
5. The renderer commits the new DOM nodes to the screen.
6. React queues the effect callback in `effectQueue` and completes the commit.
7. React yields control to the browser, which paints the screen.
8. The browser finishes painting and triggers the scheduled macro-task callback (`runEffects`).
9. `runEffects` retrieves the callback from `effectQueue`, executes it, and saves the returned cleanup function in the Fiber's hook state.

---

**Q17: Explain step-by-step how React handles a dependency update in a component with a `useEffect` call.**
**A:**
1. A component re-renders because of a state change, and React runs its function.
2. `useEffect(callback, [count])` runs.
3. React compares the new dependency `[count]` with the old dependency `[prevCount]` from the existing hook state using `areDepsEqual`.
4. If they differ, React updates the hook's `deps` property with the new array `[count]`.
5. React queues the effect to run in the `effectQueue`.
6. The Commit phase completes and updates the DOM.
7. After the browser paints, the scheduler executes the `runEffects` macro-task.
8. React first runs the old cleanup function saved in the hook state (`oldHook.cleanup()`), which runs within the closure of the previous render cycle.
9. React then runs the new callback function (`newHook.callback()`), which runs within the closure of the current render cycle.
10. The new cleanup function returned by the callback is saved back in the hook state for the next update cycle.
