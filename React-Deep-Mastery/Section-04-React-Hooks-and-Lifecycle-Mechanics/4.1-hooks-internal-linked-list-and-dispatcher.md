# 📌 4.1 — Hooks Internal Linked List & Dispatcher

---

## ① DEFINITION

**Plain English:**
Imagine checking in to a hotel.
On your first visit (mount phase), the receptionist assigns you a set of drawers (the Fiber's `memoizedState`) and places a key card in the first drawer (Hook 1). To store another item, they create a label pointing to the next drawer (Hook 2), and so on.
When you return to get your items (update phase), you must access the drawers in the **exact same order** because the key card for drawer 2 is stored inside drawer 1. If you skip drawer 1, you can't open drawer 2.
React uses this sequential pattern to manage hooks. Rather than storing hooks in a key-value map (like `{ count: 0 }`), React stores them as a **Singly Linked List** attached to the component's Fiber node.
To control what happens during mounting vs updating, React swaps the receptionist (the **Dispatcher**). When mounting, it uses the mount dispatcher; when updating, it swaps to the update dispatcher, which automatically hooks into the existing list.

**Technical Definition:**
In functional components, state is stored on the Fiber node's `memoizedState` field as a singly linked list of `Hook` objects.
The **Dispatcher** is an internal object (`ReactCurrentDispatcher.current`) that contains references to all hook implementations. React swaps dispatchers depending on the component's phase:
- `HooksDispatcherOnMount`: Instantiates and links hook nodes for the initial render.
- `HooksDispatcherOnUpdate`: Traverses and updates existing hook nodes for subsequent renders.
- `ContextOnlyDispatcher`: Throws errors if hooks are called outside of a component execution cycle.

**One-line mental model:**
> *Hooks are nodes in a singly linked list navigated by a dispatcher that swaps behavior between mounts and updates.*

**Where does this live?**
Hook structures are defined in `ReactFiberHooks.js` and dispatchers are managed in `ReactCurrentDispatcher.js` inside the core reconciler.

## ② DRILL DOWN

Let's look at the data structure of a single Hook node in React's source code:

```javascript
// ReactFiberHooks.js (Simplified Internal Hook Object)
const Hook = {
  memoizedState: any, // The hook's resolved state (e.g., current value for useState, or ref object)
  baseState: any,     // The base state used when computing updates
  baseQueue: any,     // The update queue that was bypassed during lower-priority renders
  queue: any,         // The update queue containing pending actions (e.g., circular linked list of updates)
  next: null | Hook,  // Pointer to the next hook in the component's list
};
```

### 🔹 The Singly Linked List Structure

Unlike our custom array-based reconciler in Section 3, official React links hook nodes sequentially.

Consider this component:

```javascript
function Dashboard() {
  const [theme, setTheme] = useState("dark"); // Hook 1
  const [user, setUser] = useState("Guest");  // Hook 2
  const activeRef = useRef(true);             // Hook 3
}
```

During rendering, React builds and traverses this linked list:

```
        Fiber Node (Dashboard)
         └── memoizedState ──► Hook 1 (Theme)
                                ├── memoizedState: "dark"
                                └── next ──► Hook 2 (User)
                                              ├── memoizedState: "Guest"
                                              └── next ──► Hook 3 (ActiveRef)
                                                            ├── memoizedState: { current: true }
                                                            └── next: null
```

If you call a hook conditionally:

```javascript
if (isAdmin) {
  useState("AdminPanel"); // Skipped if isAdmin is false
}
```

If `isAdmin` changes, the linked list structure breaks. The pointers will resolve to the wrong hook nodes, causing state mismatch bugs and errors.

---

## ③ WHY IT EXISTS

### What problem does it solve?

Why didn't React design hooks using a key-value map, which would allow calling hooks inside conditions?

```javascript
// Hypothetical map-based API (NOT React):
const [count, setCount] = useState("countState", 0); // Requires a unique string key
```

If React used string keys:
1. **Name Collisions:** If you imported two custom hooks that internally used the same key `"loading"`, they would overwrite each other's state.
2. **Boilerplate:** Developers would have to manually declare and maintain unique string keys for every hook call.
3. **No Automatic Garbage Collection:** Managing dynamic keys makes cleaning up unused state when components unmount highly complex.

By using a linked list with sequential execution, React eliminates keys entirely, guaranteeing that state is isolated to each specific component instance.

## ④ INTERNAL WORKING

React controls hook behavior using the **Dispatcher Pattern**.

### 🔹 The Dispatcher Architecture

```
[Hook Call: useState(0)]
            │
            ▼
┌───────────────────────────────┐
│ ReactCurrentDispatcher.current│ ◄── Swapped automatically by the Reconciler
└───────────┬───────────────────┘
            │
            ├─────► [Mount Phase] ──► HooksDispatcherOnMount.useState(0)
            │
            ├─────► [Update Phase] ──► HooksDispatcherOnUpdate.useState(0)
            │
            └─────► [Error State] ──► ContextOnlyDispatcher.useState(0) (Throws Error)
```

During a render cycle, the reconciler updates the dispatcher:

#### 1. Mounting Phase
When rendering a component for the first time, React sets:
`ReactCurrentDispatcher.current = HooksDispatcherOnMount;`

The mount dispatcher implements `mountState`:
```javascript
function mountState(initialState) {
  // 1. Create a new hook node
  const hook = mountWorkInProgressHook();
  
  // 2. Set the initial state
  hook.memoizedState = hook.baseState = initialState;
  
  // 3. Setup update queue
  const queue = (hook.queue = {
    pending: null,
    dispatch: null,
    lastRenderedReducer: basicStateReducer,
    lastRenderedState: initialState,
  });
  
  // 4. Bind the dispatch function to the Fiber node
  const dispatch = (queue.dispatch = dispatchSetState.bind(null, currentlyRenderingFiber, queue));
  
  return [hook.memoizedState, dispatch];
}
```

#### 2. Update Phase
When re-rendering a component, React sets:
`ReactCurrentDispatcher.current = HooksDispatcherOnUpdate;`

The update dispatcher implements `updateState`:
```javascript
function updateState(initialState) {
  // 1. Move to the next hook node in the existing list
  const hook = updateWorkInProgressHook();
  
  const queue = hook.queue;
  const pendingQueue = queue.pending;
  
  // 2. Process all updates in the queue
  if (pendingQueue !== null) {
    let first = pendingQueue.next;
    let state = hook.memoizedState;
    let update = first;
    
    do {
      const action = update.action;
      state = typeof action === "function" ? action(state) : action;
      update = update.next;
    } while (update !== first);
    
    queue.pending = null; // Clear queue
    hook.memoizedState = state; // Save updated state
  }
  
  return [hook.memoizedState, queue.dispatch];
}
```

#### 3. Execution Guard
Outside of a component's render execution cycle (e.g., calling a hook inside an event listener), React sets:
`ReactCurrentDispatcher.current = ContextOnlyDispatcher;`

All hook methods on the `ContextOnlyDispatcher` throw an error:
`Error: Invalid hook call. Hooks can only be called inside of the body of a function component.`

## ⑤ ADVANTAGES

1. **No Key Conflicts:** Because hooks are resolved by execution order, naming collisions between different custom hooks are impossible.
2. **Clean Syntax:** No string keys or boilerplate declarations are required.
3. **Optimized Swaps:** The dispatcher pattern allows React to bypass mount checks during updates, keeping hook execution fast.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Rigid Execution Order:** You cannot call hooks inside loops, conditions, or nested functions.
2. **Stale Dispatchers:** If you accidentally hold a reference to a dispatcher function across different components, updates can trigger on the wrong Fiber node.

### ⚠️ Trap: Calling Hooks Conditionally

```jsx
// ❌ BAD: Condition skips hook, breaking the linked list pointers
function UserProfile({ id }) {
  if (!id) {
    return <p>Anonymous User</p>;
  }
  
  // If id is null, this hook is skipped, mismatching all subsequent hook pointers!
  const [details, setDetails] = useState(null); 
  
  return <div>{details.name}</div>;
}

// ✅ GOOD: Call hooks at the top level. Handle conditions inside return logic
function UserProfile({ id }) {
  const [details, setDetails] = useState(null);
  
  if (!id) {
    return <p>Anonymous User</p>;
  }
  
  return <div>{details?.name}</div>;
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: Where does React store a functional component's hooks?**
**A:** In the component's Fiber node, inside a singly linked list linked to the `memoizedState` property.

**Q2: Why must React hooks be called at the top level of a component?**
**A:** Because React matches hook values using their call index. If a hook is called conditionally, the linked list order changes, causing React to read the wrong state for subsequent hooks.

**Q3: What is a Dispatcher in React?**
**A:** A dispatcher is an internal object (`ReactCurrentDispatcher.current`) that contains the active implementation of React's hooks. React swaps dispatchers depending on the component's lifecycle phase.

**Q4: Which dispatcher is active during a component's initial render?**
**A:** `HooksDispatcherOnMount`. It initializes hook nodes and links them together to build the singly linked list.

**Q5: What happens if you call a hook outside of a component function?**
**A:** The inactive dispatcher `ContextOnlyDispatcher` will intercept the call and throw an error: `Invalid hook call. Hooks can only be called inside of the body of a function component.`

---

### 🟡 Intermediate (5)

**Q6: Explain why React uses a Singly Linked List instead of a Key-Value Map to store hook state.**
**A:** Using a linked list avoids the need for developers to manually define and maintain unique string keys for every hook call. It also prevents key collisions when using third-party custom hooks.

**Q7: Explain the difference between `memoizedState` on a Fiber node and `memoizedState` on a Hook object.**
**A:** 
- The Fiber node's `memoizedState` points to the **head node** of the hook linked list.
- The Hook object's `memoizedState` holds the **resolved state value** for that specific hook (e.g., a state value, a ref, or an effect object).

**Q8: Explain what the `updateWorkInProgressHook` function does during a re-render.**
**A:** It moves the active pointer to the next hook node in the existing list. It clones the hook from the current Fiber, hooks it into the workInProgress tree, and returns it to be updated.

**Q9: Why does the mount dispatcher set up a `dispatch` function using `bind`?**
**A:** It binds the dispatch function to the active Fiber node and the hook's update queue. This ensures that when you call `setState` later, React knows exactly which Fiber node and hook queue to apply the update to.

**Q10: What is the `baseState` property on a hook object used for?**
**A:** `baseState` represents the state value before any pending updates are applied. If an update is skipped due to priority lanes, React uses `baseState` as the starting point to compute the state in the next cycle.

---

### 🔴 Advanced (5)

**Q11: Explain how React's dispatcher pattern helps optimize production builds.**
**A:** By separating mount and update logic into two different dispatcher objects, React avoids checking if a component is mounting or updating on every single hook call. This reduces runtime conditional checks, keeping hook execution fast.

**Q12: If a custom hook internally calls three other hooks, how many nodes are added to the Fiber's hook linked list?**
**A:** Three nodes. A custom hook does not create its own sub-list; it simply executes its internal hooks in order, adding them to the component Fiber's flat, singly linked list.

**Q13: How does React's hooks reconciler handle a component that throws an error midway through its render phase?**
**A:** React catches the error, stops execution, and discards the workInProgress Fiber tree. The incomplete hook linked list is garbage collected, and the current tree's hook list remains active and unchanged on screen.

**Q14: Explain why using dynamic conditions (e.g. `if (props.isLoaded)`) to call hooks is unsafe, but setting state conditionally (e.g. `if (props.isLoaded) setState(val)`) is completely safe.**
**A:** 
- Calling hooks conditionally changes the structure of the linked list during rendering, breaking the reconciler's pointer sequence.
- Calling state dispatchers (`setState`) conditionally is safe because the dispatcher function reference is stable and does not change the sequence of hooks in the linked list.

**Q15: How can a custom hook detect if it is running during the initial mount or an update cycle using the dispatcher pattern?**
**A:** While you cannot access the dispatcher directly, you can track the lifecycle using a `useRef` hook:
```javascript
function useIsMount() {
  const isMountRef = useRef(true);
  if (isMountRef.current) {
    isMountRef.current = false;
    return true;
  }
  return false;
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React builds the hook linked list during the mounting phase of a component with 2 hooks.**
**A:**
1. React begins the Render phase for the component and sets `ReactCurrentDispatcher.current = HooksDispatcherOnMount`.
2. The component function begins executing.
3. **Hook 1 (`useState(0)`):** `mountState(0)` is invoked.
4. `mountWorkInProgressHook` creates the first Hook node: `{ memoizedState: 0, next: null }`.
5. React attaches this node to the Fiber: `currentlyRenderingFiber.memoizedState = firstHook`.
6. React returns the state `0` and the bound dispatch function.
7. **Hook 2 (`useState("Alex")`):** `mountState("Alex")` is invoked.
8. `mountWorkInProgressHook` creates a second Hook node: `{ memoizedState: "Alex", next: null }`.
9. React links the first node to the second node: `firstHook.next = secondHook`.
10. React returns the state `"Alex"` and the bound dispatch function.
11. The component function finishes executing.

---

**Q17: Explain step-by-step how React traverses and updates the hook linked list during the update phase.**
**A:**
1. A state update triggers a re-render. React sets `ReactCurrentDispatcher.current = HooksDispatcherOnUpdate`.
2. The component function begins executing.
3. **Hook 1 (`useState(0)`):** `updateState(0)` is invoked.
4. `updateWorkInProgressHook` runs. It reads the head of the hook list from the current Fiber node, clones it, and attaches the clone to the new workInProgress Fiber node.
5. It processes any pending updates in the hook's queue, updates `memoizedState` with the new value, and returns the updated state and the dispatcher.
6. **Hook 2 (`useState("Alex")`):** `updateState("Alex")` is invoked.
7. `updateWorkInProgressHook` runs again. It follows the `next` pointer of the current hook node to find the second node, clones it, links it to the first cloned node, and returns it.
8. It processes the queue for the second hook and returns the state and dispatcher.
9. The component function finishes executing.
