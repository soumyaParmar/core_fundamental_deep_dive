# 📌 4.2 — State Hooks (useState & useReducer)

---

## ① DEFINITION

**Plain English:**
Imagine a bank drive-thru.
When customers arrive to deposit money, they don't walk directly to the bank vault to change the numbers. Instead, they write their request on a slip (an **Update**) and place it in a pneumatic tube (the **Update Queue**). The teller collects these slips in a circular tray. When the bank processes updates, it reads the slips one by one, calculates the new balance, and saves it in the system.
In React, `useState` and `useReducer` work like this drive-thru. Calling `setState` or `dispatch` doesn't change the component's state instantly. It creates an update object, places it in a circular queue on the Hook, and schedules rendering work.
Under the hood, `useState` is just a wrapper around `useReducer` with a pre-configured reducer.
To optimize updates, React uses **Eager Bailout**: if the calculated state is identical to the current state, it doesn't trigger a re-render.

**Technical Definition:**
`useState` and `useReducer` are React's primary state hooks. In React's source code, `useState` is implemented directly using `useReducer` with a default reducer called `basicStateReducer`.
Updates are queued on the Hook node using a circular singly linked list called the `UpdateQueue`.
- **Eager Bailout:** An optimization where React calculates the next state eagerly at dispatch time. If the next state is equal to the current state (using `Object.is`), React skips scheduling rendering work.
- **Render-Phase Updates:** State updates triggered during the execution of a component function. React processes these updates immediately within the same render pass, capped at 25 iterations.

**One-line mental model:**
> *useState is a simplified useReducer that queues updates in a circular list and performs eager bailouts if the state is unchanged.*

**Where does this live?**
These structures and update loops live inside `react-reconciler/src/ReactFiberHooks.js` in the React source code.

## ② DRILL DOWN

Let's look at how the update queue is structured inside a Hook node.

### 🔹 The Circular Singly Linked List Queue

The `queue` property on a Hook node points to a circular linked list of updates. The `queue.pending` pointer always points to the **last update** added. The last update's `next` pointer points back to the **first update**, forming a loop.

```
       Queue Structure (queue.pending points to Last Update):
       
       ┌────────────────────────────────────────────────────────┐
       │                                                        │
       ▼                                                        │
┌──────────────┐   next   ┌──────────────┐   next   ┌───────────┴──┐
│ Update 1     ├─────────►│ Update 2     ├─────────►│ Update 3     │ (queue.pending)
│ action: +1   │          │ action: +2   │          │ action: +5   │
└──────────────┘          └──────────────┘          └──────────────┘
```

#### Why use a circular linked list?
1. **O(1) Insertion:** To append a new update, React inserts it after `queue.pending` and updates the `queue.pending` pointer.
2. **O(1) Access to First and Last:** `queue.pending` gives direct access to the last update, and `queue.pending.next` gives direct access to the first update. No array traversal is needed.

---

### 🔹 `useState` as a Wrapper Around `useReducer`

In React's source code, `useState` is defined as:

```javascript
// The default reducer used by useState
function basicStateReducer(state, action) {
  return typeof action === "function" ? action(state) : action;
}

// How useState is resolved internally:
function mountState(initialState) {
  // useState calls useReducer with the basicStateReducer
  return mountReducer(basicStateReducer, initialState);
}
```

This means `useState(x)` is equivalent to `useReducer(basicStateReducer, x)`.

## ③ WHY IT EXISTS

### What problem does it solve?

Why does React use an update queue instead of mutating variables directly?

```javascript
// Purely hypothetical mutable state (NOT React):
let state = 0;
function onClick() {
  state = state + 1; // Mutates directly
}
```

If React used direct mutation:
1. **Loss of Batching:** React wouldn't know when to redraw the screen, resulting in multiple expensive DOM updates for every click.
2. **Inconsistent Renders:** In Concurrent Mode, React can pause rendering. If state was mutated directly, components would read inconsistent data if rendering resumed after a pause.
3. **No Time Travel Debugging:** Tools like Redux DevTools rely on immutable updates and queues to log and replay actions.

## ④ INTERNAL WORKING

Let's trace how React processes updates and handles eager bailouts.

### 🔹 The Eager Bailout Mechanism

When you call `setState(newValue)`, React tries to avoid scheduling work on the main thread.

```
[Call setState(newValue)]
          │
          ▼
[Is Reconciler idle?] ──► No: Queue update & schedule normal render.
          │
          ▼ Yes
[Eagerly calculate next state] (Run reducer using currentValue)
          │
          ▼
[Object.is(eagerState, currentState)?]
          │
          ├─────► Yes: [Bailout] ──► Save state to queue, skip scheduling rendering work!
          │
          └─────► No: Queue update & schedule rendering work.
```

If the reconciler is idle, React runs the reducer eagerly at dispatch time. If the computed `eagerState` is identical to the current state, React records the value but **skips scheduling work**, avoiding a component render cycle.

---

### 🔹 Render-Phase Updates: The Loop Guard

If you update state inside the component's render body (outside of an effect or event handler), React enters a render-phase update loop:

```javascript
function Counter() {
  const [count, setCount] = useState(0);
  
  // Render-phase update!
  setCount(count + 1); 
  
  return <div>{count}</div>;
}
```

#### How React processes this:
1. During `beginWork`, React executes the component function and encounters `setCount(count + 1)`.
2. The dispatcher detects that this update occurred during the rendering cycle (`currentlyRenderingFiber` is active).
3. Instead of scheduling work with the global scheduler, React adds the update to the render-phase update queue.
4. Once the component function finishes, React notices the render-phase queue is not empty.
5. React immediately re-runs the component function with the new state *during the same render pass*.
6. **The Loop Guard:** If this re-running occurs recursively, React increments a counter. If the counter reaches **25**, React throws an error:
   `Error: Too many re-renders. React limits the number of renders to prevent an infinite loop.`

---

## ⑤ ADVANTAGES

1. **State Consistency:** Queueing updates ensures state is computed predictably, even if rendering is paused or interrupted.
2. **Bailout Efficiency:** Eager bailouts skip rendering work entirely for redundant state updates.
3. **Render-Phase Flexibility:** Allows components to adjust state dynamically based on prop changes during rendering without waiting for a commit and repaint cycle.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Re-rendering Child Components:** A state change that passes the bailout check will trigger a recursive render of all child components, even if those children don't read the updated state.
2. **Object Identity Traps:** Because React uses `Object.is` for comparison, updating an array or object in place will bypass the bailout check, resulting in no re-render.

### ⚠️ Trap: In-place Object Mutation

```javascript
// ❌ BAD: Mutating state object directly
const [user, setUser] = useState({ name: 'Alex', age: 25 });
const birthday = () => {
  user.age = 26; // Mutates the object in place
  setUser(user); // Object.is sees the same reference, bails out, and doesn't update the screen!
};

// ✅ GOOD: Pass a new object reference
const [user, setUser] = useState({ name: 'Alex', age: 25 });
const birthday = () => {
  setUser({
    ...user,
    age: 26 // Creates a new object reference, triggering render
  });
};
```

## ⑦ PRODUCTION USAGE

### Real Production Pattern: State Reducer Pattern
When state changes are complex or depend on multiple actions, use `useReducer` to centralize state transitions, making the code easier to test and maintain than multiple `useState` calls.

```jsx
import { useReducer } from 'react';

const initialState = { count: 0, history: [] };

function reducer(state, action) {
  switch (action.type) {
    case 'increment':
      return {
        count: state.count + 1,
        history: [...state.history, 'increment']
      };
    case 'decrement':
      return {
        count: state.count - 1,
        history: [...state.history, 'decrement']
      };
    case 'reset':
      return initialState;
    default:
      throw new Error(`Unhandled action type: ${action.type}`);
  }
}

export default function CounterApp() {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <div>
      <h2>Count: {state.count}</h2>
      <button onClick={() => dispatch({ type: 'increment' })}>Add</button>
      <button onClick={() => dispatch({ type: 'decrement' })}>Sub</button>
      <button onClick={() => dispatch({ type: 'reset' })}>Reset</button>
    </div>
  );
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: How is `useState` implemented in React's source code?**
**A:** `useState` is implemented as a wrapper around `useReducer` using a default reducer function (`basicStateReducer`) that returns the new value directly or executes it if it is an updater function.

**Q2: What type of data structure does React use to store state updates?**
**A:** A circular singly linked list called the `UpdateQueue`.

**Q3: What is the purpose of Eager Bailout in React?**
**A:** It is an optimization where React calculates the next state eagerly at dispatch time. If the new state is equal to the current state, React skips scheduling rendering work, avoiding unnecessary renders.

**Q4: What comparison function does React use to check if state has changed?**
**A:** React uses `Object.is` (strict equality comparison).

**Q5: What error does React throw if a component updates state inside its render body continuously?**
**A:** `Error: Too many re-renders. React limits the number of renders to prevent an infinite loop.` This occurs if the loop repeats more than 25 times.

---

### 🟡 Intermediate (5)

**Q6: Why does React use a circular linked list for the update queue instead of a standard array?**
**A:** A circular linked list allows appending new updates in $O(1)$ complexity without traversing the list. It also provides direct, $O(1)$ access to both the first update (`queue.pending.next`) and the last update (`queue.pending`).

**Q7: Explain the execution flow when you pass an updater function (e.g. `setCount(c => c + 1)`) vs a direct value (e.g. `setCount(5)`).**
**A:**
- If you pass a value, the reducer returns the value directly.
- If you pass a function, the reducer executes the function, passing the current state as the argument: `action(state)`.
This execution occurs during the render phase when the update queue is processed.

**Q8: Explain what happens to the update queue of a hook once the component finishes its render and commit phase.**
**A:** Once React renders the component successfully, it saves the calculated state as `memoizedState` on the hook node, and clears the pending updates list by setting `queue.pending = null`.

**Q9: When does the Eager Bailout optimization skip running?**
**A:** It skips running if the reconciler is currently busy rendering. If a render cycle is active, React cannot safely perform eager calculations, so it skips the bailout check, adds the update to the queue, and schedules a render.

**Q10: Why does updating state using `setCount(count + 1)` multiple times in a single event handler result in the count only increasing by 1?**
**A:** Because `count` is a constant captured in the event handler's closure during the current render cycle. If `count` is `0`, calling `setCount(0 + 1)` three times queues three updates with the action payload `1`. When processed, the state is set to `1` three times. To add them sequentially, you must use an updater function: `setCount(c => c + 1)`.

---

### 🔴 Advanced (5)

**Q11: Explain the difference in hook state calculation between the initial render, a normal render, and a render-phase update.**
**A:**
- **Initial Render:** React reads the initial value, mounts the hook node, and returns the initial state.
- **Normal Render:** React checks the `queue.pending` queue, applies the updates to the old state, saves the result to `memoizedState`, and clears the queue.
- **Render-Phase Update:** React detects that updates are dispatched during rendering. It immediately re-runs the component function in the same cycle, applying the new updates to the state until the queue is empty.

**Q12: Detail how React handles state updates in Concurrent Mode when some updates are deferred due to low priority.**
**A:** In Concurrent Mode, each update has a priority level (lane). When processing the queue, React only applies updates that match the current rendering lane. Any deferred updates are kept in the queue (`baseQueue`). React saves the state calculated before the first skipped update as `baseState`, and uses it to recompute all updates (including the deferred ones) in the next render cycle.

**Q13: How does React's circular linked list insert a new update node in the queue? Write a code snippet that demonstrates this insertion.**
**A:**
```javascript
function enqueueUpdate(fiber, queue, update) {
  const pending = queue.pending;
  if (pending === null) {
    // First update: point to itself to form a circle
    update.next = update;
  } else {
    // Insert after the last update
    update.next = pending.next;
    pending.next = update;
  }
  // Make this update the new pending (last) node
  queue.pending = update; 
}
```

**Q14: Explain the difference in reconciliation behavior when a component bails out early during an eager bailout vs a bailout during the render phase.**
**A:**
- **Eager Bailout (Dispatch Phase):** React determines the state won't change before starting work. It skips scheduling work entirely, avoiding any rendering of the component.
- **Render-Phase Bailout:** React starts rendering the component and processes its update queue. It calculates the new state and notices it is equal to the old state. Since rendering has already started, React cannot skip the component itself, but it bails out of rendering its children, reusing the existing child Fiber tree.

**Q15: Why is the render-phase update loop capped at exactly 25 iterations?**
**A:** This is a safety threshold. Render-phase updates run synchronously within the same render pass. If there was no limit, a component that unconditionally updates state during rendering would run forever, freezing the browser thread. The cap of 25 loops detects and stops these infinite loops, throwing an error to recover.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step what happens internally when `setCount(c => c + 1)` is called while React is idle.**
**A:**
1. The user click event fires `setCount(c => c + 1)`.
2. React instantiates an `Update` object containing the updater action.
3. React checks if the reconciler is currently idle. It is.
4. **Eager calculation:** React retrieves the current state (e.g. `0`) and executes the updater: `action(0) = 1`.
5. React compares `1` with `0` using `Object.is(1, 0)`, which evaluates to `false`.
6. Since the state changed, React adds the update node to the hook's circular queue.
7. React schedules rendering work on the Fiber node using the scheduler.
8. The scheduler executes the work loop.
9. During the Render phase, React processes the hook's queue, calculates the final state `1`, saves it to the Fiber's `memoizedState`, and clears the queue.
10. The renderer commits `1` to the DOM.

---

**Q17: Explain step-by-step how React catches and resolves a render-phase update (e.g., calling `setCount(1)` in the component body).**
**A:**
1. During `beginWork`, React executes the component function: `Counter()`.
2. Inside the function, the code executes `setCount(1)`.
3. The dispatcher checks if a render cycle is active. It is (`currentlyRenderingFiber !== null`).
4. Instead of scheduling work with the global scheduler, React adds the update to the render-phase queue.
5. The component function finishes rendering and returns its JSX.
6. The reconciler checks if there are pending render-phase updates. Yes, the queue is not empty.
7. React immediately re-runs the component function: `Counter()` in the same render pass.
8. Inside the component, `useState` runs. It processes the render-phase queue, updates the state to `1`, and returns it.
9. If the component body executes `setCount(1)` again:
   - React calculates the new state `1`.
   - Compares it with the current state `1` using `Object.is(1, 1)`, which evaluates to `true`.
   - React bails out, does not queue another update, and the loop terminates.
10. If the state continues to change unconditionally, React increments its loop counter. If it reaches 25, it throws an error.
