# 📌 4.3 — Effect Hooks (useEffect, useLayoutEffect, useInsertionEffect)

---

## ① DEFINITION

**Plain English:**
Imagine building a house and styling it.
First, you inject the paint recipe and blueprint rules into the wall materials before they are assembled (this represents **`useInsertionEffect`** — it runs before any DOM elements are measured, injecting global CSS style rules).
Second, once the walls are erected, you measure the rooms synchronously to make sure the furniture fits perfectly before anyone enters (this represents **`useLayoutEffect`** — it runs synchronously after the DOM is updated but before the screen is painted, allowing you to measure layout dimensions without causing visual flickering).
Finally, once the house is completed and painted, you move in, set up Wi-Fi, and configure the smart devices in the background (this represents **`useEffect`** — it runs asynchronously after the paint cycle, ensuring it doesn't block the initial visual rendering).
React provides these three distinct effect hooks to manage side-effects at different stages of the render-commit-paint pipeline.

**Technical Definition:**
- **`useInsertionEffect`:** Executes synchronously during the Commit phase *before* DOM mutations are applied (pre-mutation). It is designed to inject global CSS stylesheet rules into the DOM before layout calculations run.
- **`useLayoutEffect`:** Executes synchronously during the layout sub-phase of the Commit phase, *after* DOM mutations are applied but *before* the browser paints the screen.
- **`useEffect`:** Schedules a callback to execute asynchronously as a macro-task *after* the Commit phase is complete and the browser has painted the screen.

**One-line mental model:**
> *useInsertionEffect injects styles before DOM writes; useLayoutEffect measures nodes before paint; useEffect syncs data after paint.*

**Where does this live?**
In React's source code, effect queues are created in `ReactFiberHooks.js` and processed during the commit phase in `ReactFiberCommitWork.js`.

---

## ② DRILL DOWN

Let's look at the execution timeline of these three hooks relative to browser style calculations, layouts, and paint cycles:

```
[Render Phase]
  └── Run component function to compute Virtual DOM
       │
       ▼
[Commit Phase: Before Mutation]
  └── useInsertionEffect cleanups run (synchronous)
  └── useInsertionEffect callbacks run (synchronous)  ◄── Injects <style> tags
       │
       ▼
[Commit Phase: Mutation]
  └── DOM mutations are applied (Placement, Update, Deletion)
  └── useLayoutEffect cleanups run (synchronous)
       │
       ▼
[Commit Phase: Layout]
  └── Refs are bound (myRef.current = DOMNode)
  └── useLayoutEffect callbacks run (synchronous)     ◄── Measures DOM dimensions
       │
       ▼
[Browser Style Calculation, Layout & Paint]           ◄── Browser updates screen
       │
       ▼
[Post-Paint Macro-task Callback]
  └── useEffect cleanups run (asynchronous)
  └── useEffect callbacks run (asynchronous)          ◄── Runs network fetches, subscriptions
```

## ③ WHY IT EXISTS

### What problem does it solve?

Why does React separate effects into three different hooks?

#### 1. The useInsertionEffect Problem (CSS-in-JS Recalculations)
CSS-in-JS libraries (like styled-components or Emotion) need to inject `<style>` tags into the document head dynamically based on component states.
- If they use `useEffect`, styles are injected after the paint. The browser must recalculate all styles and redraw the screen, causing a noticeable flash of unstyled content (FOUC).
- If they use `useLayoutEffect`, styles are injected during the layout phase. This forces the browser to recalculate styles *during* layout, causing a performance bottleneck known as **Layout Thrashing**.
`useInsertionEffect` solves this by executing *before* layout calculations start, allowing styles to inject once without triggering style recalculations during rendering.

#### 2. The useLayoutEffect Problem (Visual Flickers)
If you measure a DOM node's dimensions and adjust its position inside `useEffect`, the browser will paint the initial layout, run the effect, adjust the position, and paint again. The user will see a visual jump or flicker.
`useLayoutEffect` solves this by executing synchronously before paint, applying all visual adjustments in a single rendering transaction.

#### 3. The useEffect Problem (Performance Blocks)
Most side-effects (like API calls, logging, or setting up subscriptions) do not affect the visual layout of the page. If we ran these side-effects synchronously, they would block the browser from painting the screen, causing the page to feel slow and laggy.
`useEffect` solves this by deferring these tasks to run asynchronously after the paint cycle, keeping the UI responsive.

## ④ INTERNAL WORKING

React represents all effects as a linked list of effect objects attached to the Fiber's `updateQueue`.

### 🔹 The Effect Object Structure

Every time you call an effect hook, React creates an `Effect` node:

```javascript
// ReactFiberHooks.js (Simplified Internal Effect Structure)
const Effect = {
  tag: HookFlags,     // Flags distinguishing the type (e.g., Passive, Layout, Insertion)
  create: Function,   // The effect's callback function (e.g. () => { ... })
  destroy: Function,  // The cleanup function returned by the callback (or undefined)
  deps: Array,        // The dependency array
  next: Effect,       // Pointer to the next effect in the circular list
};
```

### 🔹 Hook Flag Bitmasks

React uses bitwise logic to identify which effects need to run during the commit phase:

```javascript
// ReactHookEffectTags.js
export const NoFlags =   0b0000;
export const HasEffect = 0b0001; // The dependencies changed; the effect must execute

// Effect Hook Types
export const Insertion = 0b0010; // useInsertionEffect
export const Layout =    0b0100; // useLayoutEffect
export const Passive =   0b1000; // useEffect
```

#### How the Reconciler filters effects:
- During the **Before Mutation** phase, React walks the queue and executes effects that match:
  `(effect.tag & Insertion) !== NoFlags && (effect.tag & HasEffect) !== NoFlags`
- During the **Layout** phase, React executes effects that match:
  `(effect.tag & Layout) !== NoFlags && (effect.tag & HasEffect) !== NoFlags`
- Post-paint, the Scheduler executes passive effects that match:
  `(effect.tag & Passive) !== NoFlags && (effect.tag & HasEffect) !== NoFlags`

---

## ⑤ ADVANTAGES

1. **Flicker-Free Layout Mutations:** `useLayoutEffect` runs synchronously before paint, ensuring layout adjustments are computed in a single visual frame.
2. **Dynamic Styling without Jank:** `useInsertionEffect` injects CSS styles before layout, avoiding browser reflow overhead and style recalculations.
3. **Unblocked Paint Cycle:** `useEffect` runs asynchronously, keeping animations and interactions smooth.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Server-Side Rendering (SSR) Warning:** `useLayoutEffect` and `useInsertionEffect` do not run on the server. If you use them in a server-rendered component, React will log a warning because the server-rendered HTML cannot match the layout updates made before the browser paint.
2. **Blocking the Main Thread:** Running heavy calculations or synchronous loops inside `useLayoutEffect` will delay browser painting, causing the page to feel laggy.

### ⚠️ Trap: Using useLayoutEffect in Server-Side Rendering (SSR)

```jsx
// ❌ BAD: useLayoutEffect will throw warnings on the server
function ClientComponent() {
  const [width, setWidth] = useState(0);
  const ref = useRef(null);

  useLayoutEffect(() => {
    // Throws warning on server because the DOM doesn't exist yet!
    if (ref.current) setWidth(ref.current.getBoundingClientRect().width);
  }, []);
  
  return <div ref={ref}>Width: {width}</div>;
}

// ✅ GOOD: Defer effect to mount time or use isomorphic alternatives
function ClientComponent() {
  const [width, setWidth] = useState(0);
  const ref = useRef(null);

  useEffect(() => {
    // Safe: runs only on the client after mounting
    if (ref.current) setWidth(ref.current.getBoundingClientRect().width);
  }, []);
  
  return <div ref={ref}>Width: {width}</div>;
}
```

## ⑦ PRODUCTION USAGE

### Real Production Pattern: Isomorphic Layout Effect
In shared library codebases that run in both SSR (Node.js) and CSR (Browser) environments, define a helper to use the appropriate hook dynamically to avoid console warnings.

```javascript
import { useEffect, useLayoutEffect } from 'react';

// Detect if we are running in a browser environment
const isBrowser = typeof window !== 'undefined' && window.document?.createElement;

// Use useLayoutEffect in browser, fallback to useEffect on server
export const useIsomorphicLayoutEffect = isBrowser ? useLayoutEffect : useEffect;
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the three effect hooks available in React?**
**A:**
1. `useInsertionEffect`: Injects styles before DOM mutations.
2. `useLayoutEffect`: Runs synchronously after DOM mutations but before paint.
3. `useEffect`: Runs asynchronously after the paint cycle.

**Q2: Which hook executes first during the Commit phase?**
**A:** `useInsertionEffect` executes first, running during the "Before Mutation" phase before DOM nodes are created or modified.

**Q3: Why doesn't `useEffect` run synchronously?**
**A:** If it ran synchronously, it would block the browser's single thread from painting and handling user inputs, causing the page to freeze during network requests or heavy tasks.

**Q4: When should you use `useLayoutEffect` instead of `useEffect`?**
**A:** You should use `useLayoutEffect` when you need to measure DOM node dimensions (like height or width) and modify layout properties based on those measurements before the screen paints to prevent visual flickering.

**Q5: Can you run CSS-in-JS library injections inside `useEffect` safely?**
**A:** No. If styles are injected after paint inside `useEffect`, the browser will be forced to recalculate styles and redraw, causing a noticeable flash of unstyled content (FOUC).

---

### 🟡 Intermediate (5)

**Q6: Why is `useInsertionEffect` preferred over `useLayoutEffect` for dynamic style injections?**
**A:** If you inject styles inside `useLayoutEffect`, the browser is forced to perform style recalculations *during* the layout phase of the commit, which can cause significant rendering delays (Layout Thrashing). `useInsertionEffect` runs before layout, avoiding this issue.

**Q7: Explain the execution order of cleanup functions and callbacks for `useEffect` and `useLayoutEffect` during a component update.**
**A:**
1. **Mutation Phase:** React executes all pending `useLayoutEffect` cleanup functions.
2. **Layout Phase:** React executes all new `useLayoutEffect` callback functions.
3. **Browser Paint:** The browser paints the screen.
4. **Post-Paint:** React executes all pending `useEffect` cleanup functions.
5. **Post-Paint:** React executes all new `useEffect` callback functions.

**Q8: Why does React log a warning if you use `useLayoutEffect` in a server-rendered component?**
**A:** Because `useLayoutEffect` requires a browser DOM to run. Since Node.js does not have a DOM, the hook is skipped on the server. When the browser receives the server-rendered HTML, it will not match the client-side layout updates, throwing hydration mismatch warnings.

**Q9: Explain the role of the `HasEffect` flag in an effect's tag bitmask.**
**A:** The `HasEffect` flag (`0b0001`) indicates that the hook's dependencies have changed during the current render cycle. The commit phase will only execute effects that have this flag enabled.

**Q10: Can you call `useInsertionEffect` inside a custom hook?**
**A:** Yes. Custom hooks can compose any built-in hooks, including `useInsertionEffect`. However, it must still follow the rules of hooks, such as executing at the top level of the custom hook function.

---

### 🔴 Advanced (5)

**Q11: Explain how React uses bitwise operations to filter and execute different types of effects from the Fiber's `updateQueue`.**
**A:**
React stores all effects in a single queue. During the commit phase, it checks the bitmask flags:
- For insertion effects, it filters: `(effect.tag & Insertion) !== 0 && (effect.tag & HasEffect) !== 0`.
- For layout effects, it filters: `(effect.tag & Layout) !== 0 && (effect.tag & HasEffect) !== 0`.
- For passive effects, it filters: `(effect.tag & Passive) !== 0 && (effect.tag & HasEffect) !== 0`.
This allows React to manage and run different effect categories using a single queue.

**Q12: Why is it unsafe to read DOM layout properties (like `getBoundingClientRect()`) inside `useInsertionEffect`?**
**A:** Because `useInsertionEffect` executes before DOM mutations are applied. The DOM elements at this stage do not reflect the updated state or layout, so reading their dimensions will return stale data.

**Q13: How does React ensure that asynchronous `useEffect` callbacks do not overlap or execute out of order if state changes occur rapidly?**
**A:** React's scheduler assigns priority lanes to updates. When a state change occurs, the scheduler queues passive effects as macro-tasks. If a new update commits before the previous effects have run, the scheduler forces the execution of the pending effects first, ensuring callbacks run in chronological order.

**Q14: Explain the difference in execution order between a parent component's `useLayoutEffect` and its child component's `useLayoutEffect`.**
**A:** Child component effects execute **before** parent component effects. During the upward traversal of the Commit phase (`completeWork` / layout phase), React completes child nodes before parent nodes, running child layout effects first.

**Q15: How would you design a custom hook that runs a callback only after the browser has completed its layout and paint cycles?**
**A:** You can use a double `requestAnimationFrame` call or the native `useEffect` hook, which is already optimized to run post-paint:
```javascript
function usePostPaint(callback) {
  useEffect(() => {
    // Already guaranteed to run post-paint by React's Scheduler
    callback(); 
  }, []);
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React executes `useInsertionEffect`, `useLayoutEffect`, and `useEffect` during a component's mount phase.**
**A:**
1. React processes the component's Fiber node and compiles the Virtual DOM tree.
2. The component registers three hooks, appending them to the circular `updateQueue`.
3. The reconciler enters the Commit phase.
4. **Before Mutation Phase:** React checks the root container's `subtreeFlags`. It detects pending insertion effects. It traverses to the component and runs the `useInsertionEffect` callback, injecting styling into the DOM.
5. **Mutation Phase:** React applies the computed layout updates to the browser DOM.
6. **Layout Phase:** React detects pending layout effects. It runs the `useLayoutEffect` callback synchronously, allowing it to measure DOM dimensions.
7. React completes the Commit phase and yields control to the browser.
8. The browser calculates styles, layout, and paints the screen.
9. **Post-Paint:** The Scheduler processes macro-tasks and triggers the passive effects queue, executing the `useEffect` callback asynchronously.

---

**Q17: Explain step-by-step what happens internally in React when a component unmounts, in relation to its effect cleanups.**
**A:**
1. During the Render phase, React identifies that a component has been removed from the new tree. It flags the component's Fiber node with `Deletion`.
2. The reconciler enters the Commit phase and processes deletions first during the Mutation phase.
3. React performs a top-down DFS traversal of the deleted subtree:
   - For every component, it finds the hook linked list.
   - For `useLayoutEffect` hooks, it executes their cleanup functions (`hook.destroy()`) synchronously, closing active layout operations.
   - For `useEffect` hooks, it schedules their cleanups to run asynchronously by adding them to the post-paint scheduler queue.
   - For `useInsertionEffect` hooks, it executes their cleanups synchronously.
4. React calls `removeChild` on the parent DOM node to remove the component's DOM elements from the screen.
5. React yields control to the browser, which paints the updated page.
6. Post-paint, the Scheduler runs macro-tasks, executing the queued `useEffect` cleanup callbacks.
7. The Fiber node and its hook list are garbage collected.
