# 📌 4.4 — Memoization Hooks (useMemo, useCallback, useRef)

---

## ① DEFINITION

**Plain English:**
Imagine you are a mathematician working on a complex math puzzle.
If someone asks you to calculate $354 \times 892$, you write down the steps, work out the answer ($315,768$), and write it on a post-it note (this represents **`useMemo`** — caching the result of an expensive calculation so you don't have to calculate it again unless the input numbers change).
If someone asks you to write down a recipe for baking a cake, you write the instructions on a card and reuse that same card every time you bake, rather than rewriting the recipe from scratch (this represents **`useCallback`** — caching the function reference itself so it remains identical across render cycles).
Finally, if you need to keep track of how many times you've opened the kitchen cabinets, you write the count on a chalkboard. You can change this count anytime without needing to clean or rebuild the kitchen (this represents **`useRef`** — keeping a mutable value that persists across renders without triggering a redraw of the UI).

**Technical Definition:**
- **`useMemo`:** Memoizes (caches) the value returned by a computation. It only recalculates the value during rendering if one of the dependencies in its dependency array has changed.
- **`useCallback`:** Memoizes the function reference itself. It returns the same function instance across renders unless its dependencies change, preventing child components that rely on reference equality checks from re-rendering.
- **`useRef`:** Returns a persistent, mutable object `{ current: initialValue }`. Its reference remains stable across all render cycles, and mutating its `.current` property does not trigger component re-renders.

**One-line mental model:**
> *useMemo caches a computed value; useCallback caches a function reference; useRef persists a mutable object reference without triggering renders.*

**Where does this live?**
In React's source code, these hooks are managed by the mount and update dispatchers inside `ReactFiberHooks.js`.

## ② DRILL DOWN

Let's look at the storage representation of these hooks inside the Fiber node's hook linked list.

### 🔹 Hook Memory Storage Comparison

```
                     wipFiber Hook Linked List Nodes
                     
  [useMemo Hook Node]               [useCallback Hook Node]           [useRef Hook Node]
  ┌────────────────────────┐        ┌────────────────────────┐        ┌────────────────────────┐
  │ memoizedState: [       │        │ memoizedState: [       │        │ memoizedState: {       │
  │   resolvedValue,       │        │   functionReference,   │        │   current: value       │
  │   [dependencyArray]    │        │   [dependencyArray]    │        │ }                      │
  │ ]                      │        │ ]                      │        └────────────────────────┘
  └────────────────────────┘        └────────────────────────┘
```

#### 1. `useMemo` Storage
When `useMemo(computeFn, [a, b])` runs:
- **Mount Phase:** React runs `computeFn()`, saves the result and dependencies in the hook state: `hook.memoizedState = [value, deps]`, and returns `value`.
- **Update Phase:** React compares the new dependency array with the saved dependencies.
  - **No Change:** Returns the saved `value` directly, skipping execution of `computeFn()`.
  - **Dependencies Changed:** Re-runs `computeFn()`, saves the new value and dependencies: `hook.memoizedState = [newValue, newDeps]`, and returns `newValue`.

#### 2. `useCallback` Storage
`useCallback(fn, deps)` is syntactically identical to `useMemo(() => fn, deps)`. Instead of executing the function, `useCallback` stores the function reference directly:
- **Mount Phase:** Saves the function and dependencies: `hook.memoizedState = [fn, deps]`, and returns `fn`.
- **Update Phase:** If dependencies are unchanged, it returns the saved `fn`. If they changed, it saves and returns the new function reference.

---

## ③ WHY IT EXISTS

### What problem does it solve?

Functional components are pure functions that run from top to bottom on every render. This creates two main challenges:

#### 1. Expensive Re-computations
If a component contains code that filters a list of 10,000 items, that code will execute on *every keypress* or state update, causing performance lag.
`useMemo` solves this by caching the filtered list, recalculating it only when the search query changes.

#### 2. Reference Equality Mismatches (Infinite Loops and Re-renders)
In JavaScript, functions and objects are compared by reference:
```javascript
const fn1 = () => {};
const fn2 = () => {};
console.log(fn1 === fn2); // false
```
Every time a component re-renders, it recreates all functions declared inside it. If you pass one of these functions as a prop to a child component, the child will see a new reference and re-render by default, even if the prop values are identical. This can also cause infinite loops if the function is used as a dependency in a child's `useEffect`.
`useCallback` solves this by keeping the function reference stable across renders.

## ④ INTERNAL WORKING

Let's look at the internal implementation of `useMemo`, `useCallback`, and `useRef` from React's source code.

### 🔹 Mount Phase Implementations

During the initial mount, React initializes the hook state:

```javascript
// ReactFiberHooks.js (Simplified Mount Implementations)

function mountRef(initialValue) {
  const hook = mountWorkInProgressHook();
  // 1. Create a stable, mutable ref object
  const ref = { current: initialValue };
  hook.memoizedState = ref;
  return ref;
}

function mountCallback(callback, deps) {
  const hook = mountWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  // 2. Store the callback function and dependencies directly
  hook.memoizedState = [callback, nextDeps];
  return callback;
}

function mountMemo(nextCreate, deps) {
  const hook = mountWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  // 3. Execute the function and store the result
  const nextValue = nextCreate();
  hook.memoizedState = [nextValue, nextDeps];
  return nextValue;
}
```

---

### 🔹 Update Phase Implementations

During subsequent renders, React checks dependencies to determine whether to reuse or recalculate values:

```javascript
// ReactFiberHooks.js (Simplified Update Implementations)

function updateRef(initialValue) {
  const hook = updateWorkInProgressHook();
  // Simply returns the existing stable ref object
  return hook.memoizedState; 
}

function updateCallback(callback, deps) {
  const hook = updateWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  const prevState = hook.memoizedState;
  
  if (prevState !== null) {
    if (nextDeps !== null) {
      const prevDeps = prevState[1];
      // Check if dependencies are equal
      if (areHookInputsEqual(nextDeps, prevDeps)) {
        return prevState[0]; // Return the cached function reference
      }
    }
  }
  
  // If dependencies changed, store and return the new function reference
  hook.memoizedState = [callback, nextDeps];
  return callback;
}

function updateMemo(nextCreate, deps) {
  const hook = updateWorkInProgressHook();
  const nextDeps = deps === undefined ? null : deps;
  const prevState = hook.memoizedState;
  
  if (prevState !== null) {
    if (nextDeps !== null) {
      const prevDeps = prevState[1];
      if (areHookInputsEqual(nextDeps, prevDeps)) {
        return prevState[0]; // Return the cached computed value
      }
    }
  }
  
  // If dependencies changed, execute the function, save, and return the new result
  const nextValue = nextCreate();
  hook.memoizedState = [nextValue, nextDeps];
  return nextValue;
}
```

## ⑤ ADVANTAGES

1. **Calculations are Cached:** Expensive operations are skipped on unrelated state updates, saving CPU cycles.
2. **Stable References:** Prevents child components from re-rendering due to reference mismatches, and prevents infinite loops in `useEffect` dependency chains.
3. **Persistent Mutable State:** `useRef` allows storing layout metrics, timer IDs, and event values across renders without triggering screen updates.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Shallow Comparison Limits:** Dependency arrays are compared using shallow equality. If you pass nested arrays or object literals, React will treat them as different on every render, bypassing the cache.
2. **Premature Memoization Overhead:** Every call to `useMemo` and `useCallback` requires allocating a dependency array and checking dependencies at runtime. Memoizing cheap operations (like simple string concatenations) can actually be slower than recalculating them.

### ⚠️ Trap: Over-memoizing Inline Functions

```jsx
// ❌ BAD: Memoizing a simple, non-shared inline function
function ActionButton() {
  // The cost of allocating the dependency array and running the check
  // is higher than simply recreating this cheap function on every render!
  const handleClick = useCallback(() => {
    console.log("Clicked");
  }, []); 

  return <button onClick={handleClick}>Click Me</button>;
}

// ✅ GOOD: Use standard inline functions for cheap event listeners
function ActionButton() {
  const handleClick = () => {
    console.log("Clicked");
  };

  return <button onClick={handleClick}>Click Me</button>;
}
```

## ⑦ PRODUCTION USAGE

### Real Production Pattern: Memoizing List Items with React.memo
Use `useCallback` in the parent and wrap the child component in `React.memo` to skip child re-renders when parent state updates.

```jsx
import React, { useState, useCallback } from 'react';

// Wrap the child component in React.memo to enable shallow prop checks
const ListItem = React.memo(function ListItem({ item, onDelete }) {
  console.log(`Rendered item: ${item.text}`);
  return (
    <li>
      {item.text}
      <button onClick={() => onDelete(item.id)}>Delete</button>
    </li>
  );
});

export default function ListApp() {
  const [items, setItems] = useState([
    { id: 1, text: 'Buy groceries' },
    { id: 2, text: 'Write React Guide' }
  ]);
  const [inputText, setInputText] = useState('');

  // useCallback keeps the function reference stable across renders.
  // Without it, ListItem would re-render on every character typed into the input!
  const handleDelete = useCallback((id) => {
    setItems((prev) => prev.filter((item) => item.id !== id));
  }, []); // Stable reference (no dependencies)

  return (
    <div>
      <input value={inputText} onChange={(e) => setInputText(e.target.value)} />
      <ul>
        {items.map(item => (
          <ListItem key={item.id} item={item} onDelete={handleDelete} />
        ))}
      </ul>
    </div>
  );
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is the purpose of `useMemo`?**
**A:** `useMemo` is used to cache the result of an expensive calculation, recalculating it only when one of its dependencies changes.

**Q2: What is the difference between `useMemo` and `useCallback`?**
**A:**
- `useMemo` executes the callback and caches the **returned value**.
- `useCallback` does not execute the callback; it caches the **function reference itself**.

**Q3: Why doesn't mutating `ref.current` trigger a component re-render?**
**A:** Because mutating `ref.current` is a simple object property mutation. Unlike `useState` state updaters, it does not create a new `wipRoot` or schedule work with the reconciler.

**Q4: How does a component Fiber store references to `useRef` hooks?**
**A:** Inside the hook's `memoizedState` property on the linked list, which holds a reference to a mutable JavaScript object: `{ current: initialValue }`.

**Q5: What comparison check is used to evaluate dependency array changes?**
**A:** React uses a shallow equality check based on `Object.is` for each element in the dependency array.

---

### 🟡 Intermediate (5)

**Q6: Explain why `useCallback(fn, deps)` is syntactically equivalent to `useMemo(() => fn, deps)`.**
**A:** Because `useMemo` returns the value returned by its callback. If `useMemo` receives a callback that returns another function `() => fn`, it will cache and return that function reference, matching the behavior of `useCallback`.

**Q7: Explain the performance trap of passing object literals (e.g. `style={{ color: 'red' }}`) as props to a component wrapped in `React.memo`.**
**A:** Object literals create a brand-new object reference in memory on every render. When the parent component re-renders, `React.memo` performs a shallow comparison of props. It detects that the style object reference has changed, bypassing the cache and forcing the child component to re-render.

**Q8: When does `useMemo` run: during the Render phase or the Commit phase?**
**A:** `useMemo` runs during the **Render phase**. Since the render phase must be pure and side-effect free, you should never trigger side-effects (like API calls or mutations) inside a `useMemo` callback.

**Q9: Explain how `useRef` can be used to track if a component is mounting for the first time vs updating.**
**A:** You can use a boolean ref value:
```javascript
const isMounted = useRef(false);
useEffect(() => {
  isMounted.current = true; // Set to true after initial mount
}, []);
```
During rendering, you can check `isMounted.current` to determine the component's lifecycle phase.

**Q10: Why does memoizing a cheap operation (like string concatenation) sometimes decrease performance?**
**A:** Because memoization has an overhead: React must allocate memory for the dependency array on every render, iterate through the dependencies, and perform shallow equality checks. For cheap calculations, this check takes more time than recalculating the value.

### 🔴 Advanced (5)

**Q11: Explain how React avoids stale closures inside a memoized function generated by `useCallback`.**
**A:** React does not prevent stale closures automatically. If you omit a dependency from the dependency array in `useCallback`, the cached function closure will continue to read the variables captured during the older render cycle in which it was created. To prevent stale closures, you must list all variables read inside the function in the dependency array.

**Q12: Write a custom hook `useMemoizedObject` that ensures an object reference remains stable across renders even if its property values are recreated.**
**A:**
```javascript
function useMemoizedObject(obj) {
  const ref = useRef(obj);
  // Compare properties shallowly
  if (!shallowCompare(ref.current, obj)) {
    ref.current = obj; // Update ref if properties changed
  }
  return ref.current;
}
```

**Q13: How does React's concurrent mode handle `useMemo` when rendering is paused and discarded?**
**A:** If rendering is paused or discarded, the workInProgress Fiber tree is destroyed, along with any temporary values calculated by `useMemo` during that interrupted pass. The old values in the current tree remain active and unchanged on screen.

**Q14: Explain the difference in memory layout between storing references using global variables outside of a component vs using `useRef` inside a component.**
**A:**
- A global variable is shared across all instances of the component. If you mount multiple instances, they will overwrite each other's global reference.
- `useRef` attaches the reference to the specific component instance's Fiber node, ensuring that each mounted instance has its own isolated, persistent reference in memory.

**Q15: How would you implement a custom `useMemo` hook using only `useState` and `useRef`?**
**A:**
```javascript
function useCustomMemo(computeFn, deps) {
  const memoRef = useRef(null); // Stores [value, deps]
  
  const hasChanged = !memoRef.current || !areDepsEqual(memoRef.current[1], deps);
  
  if (hasChanged) {
    const value = computeFn();
    memoRef.current = [value, deps];
  }
  
  return memoRef.current[0];
}
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React resolves `useMemo` during an update cycle when dependencies do not change.**
**A:**
1. A component state update triggers a re-render.
2. The component function runs, and executes `useMemo(expensiveFn, [count])`.
3. The Reconciler moves to the corresponding hook node in the linked list.
4. React reads the saved state from the previous render: `prevState = [value, [prevCount]]`.
5. React compares the new dependency array `[count]` with the saved dependency `[prevCount]` using a shallow comparison.
6. The values are equal (e.g. `5 === 5`).
7. React skips running `expensiveFn()`.
8. React returns the cached computed value `value` directly.

---

**Q17: Explain step-by-step how React resolves `useCallback` when a dependency changes.**
**A:**
1. A component re-renders because of a dependency change, and executes `useCallback(newClickFn, [userId])`.
2. The Reconciler moves to the corresponding hook node.
3. React reads the saved state from the previous render: `prevState = [oldClickFn, [prevUserId]]`.
4. React compares the dependencies: `Object.is(userId, prevUserId)` (e.g. `123 !== 456`), which evaluates to `false`.
5. Since the dependencies changed, React discards the cached callback function.
6. React saves the new function reference and dependencies: `hook.memoizedState = [newClickFn, [userId]]`.
7. React returns the new function reference `newClickFn`.
