# 📌 4.5 — Concurrent Hooks (useTransition, useDeferredValue)

---

## ① DEFINITION

**Plain English:**
Imagine typing in a search box.
When you type the letter "A", you expect to see "A" appear in the input box instantly (this is an **urgent update**). However, the page also has to update a heavy list of 1,000 search results. If the list updates synchronously, it will freeze the input box, causing the typing to lag.
React's concurrent hooks solve this by splitting updates:
- **`useTransition`** allows you to mark the heavy list update as "non-urgent". React will render the input box instantly, and then render the list in the background. If you type another letter before the list finishes rendering, React will stop the background render and start over with the new letter.
- **`useDeferredValue`** works similarly for values. It takes a fast-changing value (like the input text) and returns a "deferred" version of it. The deferred version only updates once the main thread is idle, keeping the interface responsive.

**Technical Definition:**
- **`useTransition`:** Returns a pending indicator (`isPending`) and a transition function (`startTransition`). Calling updates inside `startTransition` flags them as non-urgent, scheduling them at a lower priority lane (`TransitionLane`) that can be interrupted by higher-priority events.
- **`useDeferredValue`:** Takes a value and returns a deferred version of that value. If an update is active, React will render the component first with the old value, and then schedule a background render with the new value.

**One-line mental model:**
> *Concurrent hooks mark rendering tasks as interruptible background work, allowing the browser to prioritize input events.*

**Where does this live?**
In React's source code, these hooks are managed by the scheduler and reconciler lane manager in `ReactFiberHooks.js` and `ReactFiberWorkLoop.js`.

---

## ② DRILL DOWN

Let's compare urgent updates with concurrent transition updates.

### 🔹 Urgent Updates vs. Transitions

| Attribute | Urgent Updates (Default) | Transition Updates (`startTransition`) |
|:---|:---|:---|
| **Priority Lane** | `SyncLane` or `InputContinuousLane` | `TransitionLane` |
| **Interruptible?** | No (must render to completion) | Yes (can be paused or discarded) |
| **UI Updates** | Updates immediately in one frame | Renders in background; UI updates when ready |
| **Pending State?** | No | Yes (provides `isPending` state) |
| **Typical Use Case** | Text inputs, clicks, drag and drop | Searching, filtering lists, tab switching |

---

## ③ WHY IT EXISTS

### What problem does it solve?

Before Concurrent React (React 18), updates were synchronous. Once an update started, the browser could not paint or process inputs until rendering was fully complete.

Developers tried to solve this using two main techniques:
1. **Debouncing:** Delaying the update until the user stops typing for 300ms.
   - *Problem:* There is still a 300ms delay even if the computer is fast, and the page will still freeze once the search runs.
2. **Throttling:** Limiting the update to run once every 100ms.
   - *Problem:* The page will still freeze at regular intervals.

**The Concurrent Solution:**
Concurrent React solves this by making rendering **pauseable**. Instead of artificial timers, React works on the background render in 5ms chunks. If the user types a new character, the browser intercepts the event, and React discards the background render, starting over with the new input value immediately.

## ④ INTERNAL WORKING

React manages concurrent updates using the **Lane System**.

### 🔹 The Reconciler Lane Hierarchy

React assigns updates to one of 32 binary lanes:

```javascript
// ReactFiberLane.js (Simplified Lane Hierarchy)
export const SyncLane =               0b0000000000000000000000000000001; // Urgent (Typing)
export const InputContinuousLane =   0b0000000000000000000000000000010; // Hover, Drag
export const DefaultLane =           0b0000000000000000000000000010000; // API responses
export const TransitionLane =        0b0000000000000000000000001000000; // useTransition
```

### 🔹 Step-by-Step Transition Processing:
1. You call `startTransition(() => setSearchQuery(text))`.
2. React sets a global flag indicating we are in a transition context: `isTransition = true`.
3. The state updater `setSearchQuery` is called. React creates an update node and assigns it to the `TransitionLane` instead of the urgent `SyncLane`.
4. React schedules rendering work.
5. **Urgent Render Pass:** React renders the UI. Since `setSearchQuery` has `TransitionLane` priority, React skips calculating its new value during this urgent pass. The input box updates instantly, and `isPending` is set to `true`.
6. **Concurrent Render Pass:** Once the urgent pass is committed, React starts a background render for `TransitionLane`.
7. **Interruption Check:** After processing each Fiber node, the work loop checks if a new `SyncLane` task is pending.
   - If you type a new character, a new `SyncLane` task is queued.
   - React interrupts the work loop, discards the partially completed `TransitionLane` tree, and runs the `SyncLane` update to show the new character.
8. **Completion:** When the user stops typing and the background render completes, React commits the new search results to the DOM and sets `isPending` to `false`.

---

## ⑤ ADVANTAGES

1. **Responsive Inputs:** Interactive elements (like typing or buttons) remain responsive even during heavy background renders.
2. **Visual States:** `isPending` allows you to display a loading indicator or dim the old content while the next page is rendering in the background.
3. **No Stale Frames:** If a user types "A", "B", "C" rapidly, React will discard the renders for "A" and "B" midway, calculating and painting only the final results for "C".

---

## ⑥ DISADVANTAGES / TRAPS

1. **Increased Memory Usage:** Running background renders requires React to maintain multiple Fiber tree branches in memory at the same time.
2. **Double Render Passes:** Marking an update as a transition causes React to run two render cycles: one urgent pass to update the pending state, and one background pass to calculate the results.

### ⚠️ Trap: Putting Fast-Changing Inputs in a Transition

```jsx
// ❌ BAD: Putting the text input state itself in a transition
function SearchApp() {
  const [text, setText] = useState('');
  const [isPending, startTransition] = useTransition();
  
  const onChange = (e) => {
    // If the input state is in a transition, the input box
    // will feel laggy and drop keystrokes!
    startTransition(() => {
      setText(e.target.value); 
    });
  };
  
  return <input value={text} onChange={onChange} />;
}

// ✅ GOOD: Keep the input state urgent. Transition the heavy calculation
function SearchApp() {
  const [text, setText] = useState('');
  const [results, setResults] = useState([]);
  const [isPending, startTransition] = useTransition();
  
  const onChange = (e) => {
    const val = e.target.value;
    setText(val); // Urgent: updates input instantly
    
    startTransition(() => {
      setResults(heavySearch(val)); // Transition: runs in background
    });
  };
}
```

## ⑦ PRODUCTION USAGE

### Real Production Pattern: Deferring Heavy Lists with useDeferredValue
Use `useDeferredValue` to split updates when parent components pass fast-changing inputs to heavy child components that you cannot modify directly.

```jsx
import { useState, useDeferredValue, useMemo } from 'react';

// A heavy component that takes time to render
function HeavyProductList({ query }) {
  const filtered = useMemo(() => {
    console.log(`Filtering products for query: ${query}`);
    return heavyFilterProducts(query);
  }, [query]);

  return (
    <ul>
      {filtered.map(product => <li key={product.id}>{product.name}</li>)}
    </ul>
  );
}

export default function CatalogApp() {
  const [query, setQuery] = useState('');
  
  // Defer the query value
  const deferredQuery = useDeferredValue(query);

  return (
    <div>
      <input value={query} onChange={e => setQuery(e.target.value)} />
      
      {/* HeavyProductList receives the deferred query value.
          Typing remains smooth because the list renders at low priority. */}
      <HeavyProductList query={deferredQuery} />
    </div>
  );
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is the purpose of `useTransition` in React?**
**A:** `useTransition` is used to mark state updates as non-urgent, allowing React to render them in the background without blocking the main browser thread.

**Q2: What are the two return values of the `useTransition` hook?**
**A:**
1. `isPending`: A boolean that is `true` while the background transition is rendering.
2. `startTransition`: A function used to wrap non-urgent state updates.

**Q3: What is the difference between `useTransition` and `useDeferredValue`?**
**A:**
- `useTransition` is used to wrap **state update functions** (e.g. `startTransition(() => setState(val))`).
- `useDeferredValue` is used to wrap **primitive values** (e.g. `const deferred = useDeferredValue(value)`).

**Q4: How does a component behave while a deferred value is updating?**
**A:** React renders the component using the old value first, and then performs a background render with the new value, swapping the values once rendering is complete.

**Q5: What are lanes in React's Concurrent Reconciler?**
**A:** Lanes are 32-bit binary integers used to represent update priorities. This allows React to filter, group, and pause updates using bitwise math.

---

### 🟡 Intermediate (5)

**Q6: Explain how Concurrent React handles a new keystroke while a low-priority transition is rendering.**
**A:** When the keystroke is registered, React schedules a high-priority task. The work loop checks for pending tasks after processing each Fiber node. It detects the high-priority task, pauses the work loop, discards the in-memory transition tree, and renders the keystroke update immediately.

**Q7: Why does `useTransition` cause a component to render twice?**
**A:**
- The first render is urgent: it sets `isPending` to `true` to display a loading indicator.
- The second render runs in the background at low priority to calculate the state change wrapped in `startTransition`.

**Q8: Explain the difference between Debouncing and React Transitions in terms of user experience.**
**A:**
- **Debouncing** adds a fixed delay (e.g. 300ms) before starting the update. Once started, the render blocks the main thread, causing lag.
- **Transitions** start immediately in the background without a delay, and run in non-blocking chunks, keeping the UI responsive.

**Q9: What happens if an API fetch completes and updates state inside a transition?**
**A:** The state update is processed at `TransitionLane` priority. React will render the results in the background and commit them once completed.

**Q10: Why is it unsafe to wrap a text input's `onChange` state update in `startTransition`?**
**A:** Because typing in a text box requires immediate visual feedback. If you wrap the input state in a transition, the update is deprioritized, which can cause characters to render out of order or keystrokes to drop.

---

### 🔴 Advanced (5)

**Q11: Explain how the Reconciler uses bitwise operations to check if a pending update lane has higher priority than the active rendering lane.**
**A:**
React defines lanes as binary numbers where smaller bits represent higher priority.
- `SyncLane = 0b0001`
- `TransitionLane = 0b0100`
React checks for higher priority updates using bitwise comparisons:
`const hasHigherPriority = (pendingLanes & ~currentRenderLanes) < currentRenderLanes;`
If this check evaluates to `true`, React pauses the current rendering task.

**Q12: How does `useDeferredValue` determine when to update its returned value?**
**A:** During rendering, React compares the new value with the saved value using `Object.is`. If they differ, React registers a low-priority task (`TransitionLane`) to update the value. It returns the old value for the current frame, and returns the new value once the low-priority render completes.

**Q13: Explain how React prevents "tearing" (visual state inconsistencies) when components read from external state stores during a transition.**
**A:** If an external store changes while a transition is rendering, different components might read different values. React prevents this using `useSyncExternalStore`. This hook detects if the store changed during rendering and immediately restarts the render cycle at synchronous priority, preventing tearing.

**Q14: Explain the difference in execution between `startTransition` and `React.startTransition`.**
**A:**
- `useTransition` is a hook that provides an `isPending` state bound to a specific component.
- `React.startTransition` is a global utility function that does not track `isPending` state. It is used to trigger transitions outside of components (e.g. in utility libraries).

**Q15: How can you write a custom hook that debounces state updates using `useDeferredValue`?**
**A:**
`useDeferredValue` already implements an adaptive debounce. Unlike time-based debouncing, it starts rendering immediately in the background and yields if the user interacts with the page, rendering the output as soon as the main thread is free.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React coordinates rendering when a user switches tabs using a transition.**
**A:**
1. The user clicks a tab button, executing `startTransition(() => setActiveTab('Settings'))`.
2. React sets `isTransition = true` and updates the active tab state at `TransitionLane` priority.
3. React schedules rendering work.
4. **Urgent Pass:** React renders the parent component. It sets `isPending = true` (to display a loading indicator) and returns the current active tab. This update commits and paints.
5. **Background Pass:** React starts a background render for the `Settings` tab at `TransitionLane` priority.
6. The work loop processes the `Settings` tab in 5ms chunks.
7. If the user clicks back to the home tab before the `Settings` tab finishes rendering:
   - React processes the click event at `SyncLane` priority.
   - It pauses the work loop and discards the unfinished `Settings` tab tree.
   - React renders and mounts the home tab immediately, keeping the interface responsive.

---

**Q17: Explain step-by-step how `useDeferredValue` prevents page freezing when filtering a long list of items.**
**A:**
1. A user types a character into an input field: `setQuery('a')`.
2. The component renders. The input updates instantly to display 'a'.
3. `useDeferredValue('a')` runs:
   - It notices the value changed from `""` to `"a"`.
   - It schedules a low-priority background task to update the deferred value.
   - It returns the old value `""` for the current render frame.
4. The list renders using the old deferred value `""`, skipping recalculations and committing the frame instantly.
5. Once the frame is painted, React starts a background render with the new deferred value `"a"`.
6. The list recalculates and filters the items using `"a"`.
7. Once the background render completes, React commits the filtered list to the DOM. The user sees the filtered list update smoothly.
