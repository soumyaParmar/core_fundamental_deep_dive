# 📌 5.1 — Redux Core Architecture

---

## ① DEFINITION

**Plain English:**
Imagine a large city-wide banking system.
If each branch manager recorded deposits and withdrawals in their own local notebooks (local component state), keeping accounts in sync would be a nightmare.
To solve this, the bank uses a single, centralized vault (the **Redux Store**). The store is read-only. If you want to change your balance, you cannot walk into the vault and update the numbers yourself. You must submit a deposit slip (an **Action**) to a bank teller (the **Dispatcher**). The teller reviews the slip and follows the bank's rules (the **Reducer**) to update the ledger.
If you need to log transactions or verify checks, you can hire security guards (the **Middleware**) to inspect the slips before they reach the teller.
Redux is this centralized pattern. It extracts all state management out of React and organizes it into a single, predictable state container.

**Technical Definition:**
Redux is a predictable state container for JavaScript applications based on three core principles:
1. **Single Source of Truth:** The state of your whole application is stored in an object tree inside a single store.
2. **State is Read-Only:** The only way to change the state is to emit an action, an object describing what happened.
3. **Changes are made with Pure Functions:** To specify how the state tree is transformed by actions, you write reducers (pure functions that take the current state and an action, and return a new state object).

**One-line mental model:**
> *Centralized, immutable state updated by dispatching actions through pure reducers and middleware pipelines.*

**Where does this live?**
The core state container is independent of UI frameworks and lives in the `redux` package. The integration with React lives in the `react-redux` package.

## ② DRILL DOWN

Let's look at the lifecycle flow of data in a Redux application:

```
                  ┌────────────────────────────────────────┐
                  │                 STORE                  │
                  │  ┌───────────┐          ┌───────────┐  │
                  │  │   State   ├─────────►│ Selectors │  │
                  │  └─────▲─────┘          └─────┬─────┘  │
                  └───┼───────────────────────────┼────────┘
                      │                           │
                   Reducers                    Subscribes
                      │                           │
                      │                           ▼
               ┌──────┴──────┐             ┌────────────┐
               │ Dispatcher  │             │ React View │
               └──────▲──────┘             └─────┬──────┘
                      │                           │
                  Middleware                   Dispatches
                      │                           │
                      ▼                           ▼
                  [ Action ] ◄────────────────────┘
```

### 🔹 The Redux Lifecycle Steps:
1. **Action Dispatched:** The UI dispatches an action object: `dispatch({ type: 'counter/increment', payload: 1 })`.
2. **Middleware Pipeline:** The action passes through any registered middleware (for logging, analytics, or async tasks).
3. **Reducer Execution:** The store passes the current state and the action to the root reducer.
4. **State Update:** The reducer computes a new state object. The store saves the new state.
5. **Subscription Notify:** The store notifies all active subscribers (e.g. React components) that the state changed.
6. **UI Re-render:** Subscribed components select their data, check if it changed, and re-render if necessary.

## ③ WHY IT EXISTS

### What problem does it solve?

Why should you use Redux instead of React's built-in **Context API**?

#### 1. Context API Re-render Issues (Performance)
The Context API is designed for simple, infrequently updated data (like themes or user sessions).
- When a context value changes, **every component** that calls `useContext(MyContext)` will re-render, even if they only read a small, unchanged property of that context.
- Context lacks selectors, meaning you cannot easily prevent re-renders on components that only care about specific fields of the state.

#### 2. Lack of Side-Effect Management
Context does not provide a built-in pipeline to handle side-effects (like logging, telemetry, or async API fetches). Developers have to write custom hooks or trigger API calls inside components, dispersing logic.

#### 3. Unpredictable State Transitions
In a large application with multiple contexts, tracking *why* a particular state variable changed is difficult. Redux enforces unidirectional data flow, ensuring that every state change is triggered by a dispatched action that can be logged and inspected.

## ④ INTERNAL WORKING

Let's look at how Redux executes middleware pipelines using **Currying**.

### 🔹 Middleware Pipelines & Currying

A Redux middleware is a curried function with three nesting levels:

```javascript
const loggerMiddleware = store => next => action => {
  console.log('Prev State:', store.getState());
  console.log('Action:', action);
  
  // Pass the action to the next middleware in the pipeline
  const result = next(action); 
  
  console.log('Next State:', store.getState());
  return result;
};
```

#### Why use this nested structure?
1. **`store` (Outer Level):** Provides the middleware with access to `store.getState()` and `store.dispatch()`.
2. **`next` (Middle Level):** A reference to the next middleware in the pipeline, or the store's dispatch function if it is the last middleware.
3. **`action` (Inner Level):** The actual action object being processed.

When you configure the store, Redux chains these middleware functions together using composition:
`dispatch = middleware1(middleware2(store.dispatch))`

This allows actions to flow through all middleware before executing the reducer, making it easy to implement features like asynchronous fetches (Redux Thunk) or logging.

## ⑤ ADVANTAGES

1. **Predictable State:** Pure reducers and unidirectional data flow ensure state transitions are consistent and predictable.
2. **Centralized Logic:** State management and side-effects are extracted out of UI components, making the codebase easier to test.
3. **Subscription Optimization:** React-Redux uses internal selectors to ensure components only re-render if their selected state properties actually change.
4. **Powerful Debugging:** Enforcing actions and state immutability enables features like Time-Travel Debugging (using Redux DevTools).

---

## ⑥ DISADVANTAGES / TRAPS

1. **Boilerplate Code:** Redux requires writing actions, action creators, reducers, and store configurations, which can be overkill for simple apps.
2. **Immutability Overhead:** Creating copy objects (via spread operators) on every state update can increase memory allocation and garbage collection overhead.

### ⚠️ Trap: Accidentally Mutating State in Reducers

Reducers must be pure. Mutating the state object directly inside a reducer will bypass reference change checks, preventing subscribers (React components) from detecting updates.

```javascript
// ❌ BAD: Mutating state inside the reducer
function todoReducer(state = [], action) {
  if (action.type === 'ADD_TODO') {
    state.push(action.payload); // Mutates array in place!
    return state; // Returns same array reference; React won't re-render
  }
  return state;
}

// ✅ GOOD: Return a new array reference
function todoReducer(state = [], action) {
  if (action.type === 'ADD_TODO') {
    return [...state, action.payload]; // Returns a new array reference
  }
  return state;
}
```

## ⑦ PRODUCTION USAGE

### Modern Redux: Redux Toolkit (RTK)
In production, developers avoid writing raw Redux boilerplate. Instead, they use **Redux Toolkit (RTK)**, which uses **Immer** internally to allow writing mutative syntax that is automatically converted to immutable updates.

```javascript
import { createSlice, configureStore } from '@reduxjs/toolkit';

// 1. Create a Slice (Combined actions and reducers)
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => {
      // Safe: Immer converts this direct mutation to an immutable copy!
      state.value += 1; 
    },
    decrement: (state) => {
      state.value -= 1;
    }
  }
});

export const { increment, decrement } = counterSlice.actions;

// 2. Configure the Store
export const store = configureStore({
  reducer: {
    counter: counterSlice.reducer
  }
});
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What are the three core principles of Redux?**
**A:**
1. **Single Source of Truth:** State is stored in a single object tree inside a single store.
2. **State is Read-Only:** State can only be updated by dispatching actions.
3. **Changes are made with Pure Functions:** Reducers must be pure functions.

**Q2: What is an Action in Redux?**
**A:** An action is a plain JavaScript object that describes a state change. It must have a `type` property (typically a string) and can optionally carry a `payload` containing data.

**Q3: What is a Reducer in Redux?**
**A:** A reducer is a pure function that takes the current state and an action as arguments, and returns a new state object. It must not perform side-effects or mutate the state directly.

**Q4: What is the difference between `redux` and `react-redux`?**
**A:**
- `redux` is the core, platform-agnostic state management library.
- `react-redux` is the integration package that binds the Redux store to React components using hooks (like `useSelector` and `useDispatch`) and context.

**Q5: What is the purpose of the Store in Redux?**
**A:** The store is the central container that holds the application state, executes dispatch functions, runs reducers, and notifies subscribers when state updates.

---

### 🟡 Intermediate (5)

**Q6: Why is Redux more performant than the Context API for frequently updating state?**
**A:** The Context API forces all components that consume the context to re-render when any property of the context value changes. React-Redux uses internal selectors (`useSelector`) to perform strict equality checks. Components only re-render if the specific state properties they select actually change, avoiding redundant renders.

**Q7: Explain the signature of a Redux Middleware. Why does it use three curried functions?**
**A:** The signature is: `store => next => action => { ... }`.
- The outer function accepts the `store` reference.
- The middle function accepts `next`, a pointer to the next middleware or the core dispatch function.
- The inner function accepts the `action` object.
This curried structure allows Redux to chain middlewares during store configuration using functional composition.

**Q8: What is a Thunk in Redux?**
**A:** A thunk is a function returned by an action creator that can execute asynchronous logic. The Thunk middleware checks if a dispatched action is a function instead of a plain object. If so, it executes the function, passing `dispatch` and `getState` as arguments.

**Q9: Why must Redux reducers be pure functions?**
**A:** Because Redux compares state objects by reference to detect changes. If a reducer mutates state in place, the old and new state references will be identical, preventing Redux from detecting updates and notifying React components to re-render.

**Q10: What is the role of `combineReducers` in Redux?**
**A:** It is a helper function that combines multiple slice reducers into a single root reducer. It maps keys in the state object to their corresponding slice reducers and manages delegation.

### 🔴 Advanced (5)

**Q11: Explain functional composition in Redux's middleware pipeline. Write a code snippet that demonstrates how middlewares are chained.**
**A:**
Redux chains middlewares by nesting them from right to left using functional composition.
```javascript
// Simple compose helper
function compose(...funcs) {
  if (funcs.length === 0) return arg => arg;
  if (funcs.length === 1) return funcs[0];
  return funcs.reduce((a, b) => (...args) => a(b(...args)));
}

// Chaining middleware
const chain = middlewares.map(middleware => middleware(middlewareAPI));
const dispatch = compose(...chain)(store.dispatch);
```
This wraps the core `store.dispatch` in a chain of middleware functions, allowing actions to flow through each one in sequence.

**Q12: How does `useSelector` prevent re-renders when other state properties change?**
**A:**
When a component calls `useSelector(selector)`, React-Redux subscribes the component to the store.
When state updates, React-Redux runs the `selector` function using the new state. It then performs a shallow comparison (by default, strict reference check `===`) between the new selected value and the previous value. If the values are identical, React-Redux skips triggering a re-render for that component.

**Q13: How does Redux Toolkit's integration of Immer allow writing code like `state.todos.push(todo)` safely?**
**A:** Redux Toolkit wraps all reducers inside Immer's `produce` function. Immer uses ES6 Proxies to track all property mutations made to the `state` draft object. It then automatically translates these mutations into a new, immutable state copy, allowing developers to write simple, mutative code while maintaining purity.

**Q14: Explain the difference between React-Redux's `connect` (Higher-Order Component) and hook-based integrations (`useSelector` / `useDispatch`) in terms of performance and bundle optimization.**
**A:**
- `connect` wraps components in a Higher-Order Component that implements optimization checks (`shouldComponentUpdate`). This isolates child components from parent updates and can be highly performant. However, it increases component depth and is harder to type in TypeScript.
- Hooks (`useSelector`) do not add wrapper components, keeping the element tree flat. However, if a parent component updates, child components using hooks will re-render unless optimized using `React.memo` or selectors.

**Q15: How would you configure a Redux store to handle custom logging, crash reporting, and analytics telemetry in a production application?**
**A:** You can write custom middleware to handle each task:
```javascript
const crashReporter = store => next => action => {
  try {
    return next(action);
  } catch (err) {
    sendErrorTelemetry(err, action); // Send to Sentry
    throw err;
  }
};

const analyticsMiddleware = store => next => action => {
  if (action.meta?.track) {
    sendAnalyticsEvent(action.type, action.payload);
  }
  return next(action);
};
```
You then register these middlewares in your store configuration to intercept and process actions globally.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how a Thunk action flows through a Redux store configured with Thunk and Logger middlewares.**
**A:**
1. You dispatch an async action: `dispatch(fetchUser(id))`.
2. The action reaches the **Thunk Middleware** first.
3. The Thunk middleware checks the type of the action:
   - `typeof action === 'function'` is `true`.
4. The Thunk middleware interrupts the pipeline and executes the function: `action(dispatch, getState)`.
5. Inside the async function, an API call is made. Once it completes, it dispatches a plain object action: `dispatch({ type: 'FETCH_SUCCESS', payload: data })`.
6. This new action starts at the beginning of the middleware pipeline.
7. It reaches the Thunk middleware. Since the action is now a plain object, Thunk calls `next(action)`, passing it to the **Logger Middleware**.
8. The Logger middleware logs the previous state, calls `next(action)` to execute the reducer, and logs the next state.
9. The action reaches the reducer, which updates the state.

---

**Q17: Explain step-by-step how React-Redux notifies components of state changes using the store subscription mechanism.**
**A:**
1. A state change occurs, and the Redux store updates its internal state reference.
2. The store iterates through its list of active subscriptions and executes their callback functions.
3. React-Redux intercept callbacks are triggered for all mounted components.
4. For each component:
   - React-Redux executes the component's `selector` function using the updated state: `newValue = selector(state)`.
   - It performs a reference comparison: `Object.is(newValue, oldValue)`.
5. If the comparison returns `true`, React-Redux skips updating the component.
6. If it returns `false`, React-Redux schedules a local state update on the component's Fiber node, forcing a re-render to display the updated data.
