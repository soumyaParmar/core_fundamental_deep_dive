## ① DEFINITION

**Plain English:**
In Section 5.1, we explored the blueprint and rules of Redux (centralized store, dispatch pipeline, and subscriber notifications).
Now, we are going to build this entire state management system from scratch in pure JavaScript.
We will write the core Redux function (`createStore`), add support for dividing store logic (`combineReducers`), implement a pipeline to intercept transactions (`applyMiddleware`), and construct the integration hooks (`Provider`, `useSelector`, and `useDispatch`) to bind our store to React.
This section provides a fully complete, executable implementation of Redux and React-Redux.

**Technical Definition:**
This custom library implements Redux's core state container and React-Redux context bindings:
- **`createStore`:** Manages internal state, triggers reducer functions, registers listeners, and notifies subscribers.
- **`combineReducers`:** Combines multiple reducers into a single root reducer.
- **`applyMiddleware`:** Chains and composes middleware functions to intercept dispatches.
- **`useSelector`:** Subscribes components to store updates, running selectors and triggering re-renders only when the selected value changes.

**One-line mental model:**
> *A self-contained state container that manages state updates, chains middlewares, and notifies React components via custom context hooks.*

**Where does this live?**
In our custom library, this is our state management module (`custom-redux.js`).

## ③ WHY IT EXISTS

### What problem does it solve?

Writing a custom state management system from scratch helps us understand:
1. **Purity Requirements:** Why mutating state directly inside reducers breaks selector reference checks and prevents React components from updating.
2. **Middleware Chains:** How middleware functions are composed to create a single, continuous dispatch pipeline.
3. **Selector Optimization:** How React-Redux avoids the Context API's re-render bottleneck by managing subscriptions and change checks at the component level.

## ④ INTERNAL WORKING

Here is the complete, self-contained implementation of our custom Redux and React-Redux library.

### 🔹 Custom Redux & React-Redux Implementation

```javascript
// ==========================================
// 1. REDUX CORE LIBRARY
// ==========================================

// Helper: Compose functions from right to left
function compose(...funcs) {
  if (funcs.length === 0) return arg => arg;
  if (funcs.length === 1) return funcs[0];
  return funcs.reduce((a, b) => (...args) => a(b(...args)));
}

// createStore: Central state container
function createStore(reducer, preloadedState, enhancer) {
  // If an enhancer (middleware) is provided, delegate store creation to it
  if (typeof preloadedState === "function" && typeof enhancer === "undefined") {
    enhancer = preloadedState;
    preloadedState = undefined;
  }

  if (typeof enhancer === "function") {
    return enhancer(createStore)(reducer, preloadedState);
  }

  let currentState = preloadedState;
  let currentListeners = [];

  function getState() {
    return currentState;
  }

  function subscribe(listener) {
    currentListeners.push(listener);
    // Return unsubscribe function
    return function unsubscribe() {
      currentListeners = currentListeners.filter(l => l !== listener);
    };
  }

  function dispatch(action) {
    if (typeof action.type === "undefined") {
      throw new Error("Actions must have a type property.");
    }
    // Calculate new state using the reducer
    currentState = reducer(currentState, action);
    
    // Notify all subscribers
    currentListeners.forEach(listener => listener());
    return action;
  }

  // Dispatch initial action to populate state
  dispatch({ type: "@@redux/INIT" });

  return {
    getState,
    dispatch,
    subscribe,
  };
}

// combineReducers: Combines multiple reducers
function combineReducers(reducers) {
  const reducerKeys = Object.keys(reducers);
  
  return function combination(state = {}, action) {
    let hasChanged = false;
    const nextState = {};
    
    for (let i = 0; i < reducerKeys.length; i++) {
      const key = reducerKeys[i];
      const reducer = reducers[key];
      const previousStateForKey = state[key];
      // Execute the slice reducer
      const nextStateForKey = reducer(previousStateForKey, action);
      
      nextState[key] = nextStateForKey;
      hasChanged = hasChanged || nextStateForKey !== previousStateForKey;
    }
    
    return hasChanged ? nextState : state;
  };
}

// applyMiddleware: Configures middleware chains
function applyMiddleware(...middlewares) {
  return (createStore) => (reducer, preloadedState) => {
    const store = createStore(reducer, preloadedState);
    let dispatch = () => {
      throw new Error("Dispatching while constructing middleware is not allowed.");
    };

    const middlewareAPI = {
      getState: store.getState,
      dispatch: (...args) => dispatch(...args)
    };

    // Chain middlewares: store => next => action
    const chain = middlewares.map(middleware => middleware(middlewareAPI));
    dispatch = compose(...chain)(store.dispatch);

    return {
      ...store,
      dispatch
    };
  };
}

// ==========================================
// 2. REACT-REDUX BINDINGS (CONTEXT)
// ==========================================

// Create global context to distribute the store
const ReduxContext = React.createContext(null);

// Provider Component
function Provider({ store, children }) {
  return React.createElement(
    ReduxContext.Provider,
    { value: store },
    children
  );
}

// Custom Hook: useSelector
function useSelector(selector) {
  const store = React.useContext(ReduxContext);
  if (!store) {
    throw new Error("useSelector must be used inside a Provider");
  }

  // Force component re-renders when state updates
  const [, forceUpdate] = React.useReducer(x => x + 1, 0);
  
  const latestSelectedState = React.useRef(null);

  // Calculate selected state
  const selectedState = selector(store.getState());

  React.useLayoutEffect(() => {
    latestSelectedState.current = selectedState;
  });

  React.useEffect(() => {
    function checkForUpdates() {
      try {
        const newSelectedState = selector(store.getState());
        // Shallow comparison: check if selected state reference changed
        if (newSelectedState === latestSelectedState.current) {
          return; // Bail out if values are identical
        }
        latestSelectedState.current = newSelectedState;
        forceUpdate(); // Trigger re-render
      } catch (err) {
        forceUpdate();
      }
    }

    // Subscribe to store updates
    const unsubscribe = store.subscribe(checkForUpdates);
    return unsubscribe;
  }, [store, selector]);

  return selectedState;
}

// Custom Hook: useDispatch
function useDispatch() {
  const store = React.useContext(ReduxContext);
  if (!store) {
    throw new Error("useDispatch must be used inside a Provider");
  }
  return store.dispatch;
}

// Export custom library
const CustomRedux = {
  createStore,
  combineReducers,
  applyMiddleware,
  Provider,
  useSelector,
  useDispatch
};
```

---

## ⑤ ADVANTAGES

1. **Strict Performance Optimization:** Custom `useSelector` filters updates, ensuring components only re-render if their selected values change.
2. **Decoupled Architecture:** State management is fully decoupled from React's component lifecycles.
3. **Chainable Extensions:** `applyMiddleware` allows extending the store dispatch pipeline with custom actions (like logging or error monitoring).

---

## ⑥ DISADVANTAGES / TRAPS

1. **Context Hydration Lag:** When using React Server Components, context-based state containers cannot easily share state across server-client boundaries without explicit serialization.
2. **Recursive Subscription Bottleneck:** If thousands of components subscribe to the store simultaneously, executing selector comparisons on every state change can introduce CPU overhead.

### ⚠️ Trap: Creating New Object References in Selectors

If a selector returns a new object reference on every execution (e.g. using `.map()` or `.filter()`), the strict equality check `newSelectedState === latestSelectedState.current` will always evaluate to `false`, causing the component to re-render on every state change.

```javascript
// ❌ BAD: Selector returns a new array reference on every run
const items = useSelector(state => state.items.filter(i => i.active));

// ✅ GOOD: Perform filtering in useMemo, keep selector reference stable
const rawItems = useSelector(state => state.items);
const activeItems = useMemo(() => rawItems.filter(i => i.active), [rawItems]);
```

## ⑦ PRODUCTION USAGE

### Integrating Custom Redux in an Application

Here is a complete example of how you can build a counter application using our custom `CustomRedux` library:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Custom Redux Demo</title>
</head>
<body>
  <div id="root"></div>

  <!-- Load React first, then our custom Redux library -->
  <script src="https://unpkg.com/react@18/umd/react.development.js"></script>
  <script src="https://unpkg.com/react-dom@18/umd/react-dom.development.js"></script>
  <script src="./custom-redux.js"></script>

  <script>
    // 1. Define a Reducer
    const initialState = { count: 0 };
    function counterReducer(state = initialState, action) {
      switch (action.type) {
        case 'INCREMENT':
          return { count: state.count + 1 };
        case 'DECREMENT':
          return { count: state.count - 1 };
        default:
          return state;
      }
    }

    // 2. Logger Middleware
    const logger = store => next => action => {
      console.log('Action dispatched:', action);
      const result = next(action);
      console.log('Next state:', store.getState());
      return result;
    };

    // 3. Create the Store with Middleware
    const store = CustomRedux.createStore(
      counterReducer,
      CustomRedux.applyMiddleware(logger)
    );

    // 4. Counter Component
    function Counter() {
      // Select count state property
      const count = CustomRedux.useSelector(state => state.count);
      const dispatch = CustomRedux.useDispatch();

      return React.createElement('div', { style: { padding: '20px', fontFamily: 'sans-serif' } },
        React.createElement('h2', null, `Count: ${count}`),
        React.createElement('button', { onClick: () => dispatch({ type: 'INCREMENT' }) }, 'Increment'),
        React.createElement('button', { onClick: () => dispatch({ type: 'DECREMENT' }) }, 'Decrement')
      );
    }

    // 5. Mount App with Provider
    const root = ReactDOM.createRoot(document.getElementById('root'));
    root.render(
      React.createElement(CustomRedux.Provider, { store: store },
        React.createElement(Counter)
      )
    );
  </script>
</body>
</html>
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What does `createStore` return in Redux?**
**A:** It returns a store object containing three core methods:
1. `getState()`: Returns the current state tree.
2. `dispatch(action)`: Triggers state updates by running the reducer.
3. `subscribe(listener)`: Registers callbacks to run when the state changes.

**Q2: What is the purpose of the `Provider` component?**
**A:** `Provider` wraps the React component tree and passes the Redux store down using React Context, making it available to hooks like `useSelector` and `useDispatch`.

**Q3: How does `useSelector` retrieve the store's dispatch function?**
**A:** It accesses the store reference passed down by the `Provider` using React's `useContext` hook.

**Q4: How does `combineReducers` detect if state has changed?**
**A:** It executes each slice reducer and checks if the returned state reference differs from the previous reference: `nextStateForKey !== previousStateForKey`. If any slice returns a new reference, it returns a new combined state object.

**Q5: What is the purpose of the initial dispatch `dispatch({ type: "@@redux/INIT" })` inside `createStore`?**
**A:** It runs the reducer with an undefined state and a placeholder action, forcing the reducer to return and populate the store with the default initial state values.

---

### 🟡 Intermediate (5)

**Q6: Explain why `useSelector` uses a `forceUpdate` function internally.**
**A:** Redux state is stored outside of React. When the Redux store updates, React components do not re-render automatically. `useSelector` uses a dummy `forceUpdate` function (implemented via `useReducer`) to force a component re-render when the selected state changes.

**Q7: Explain the bug that occurs if you call `subscribe` inside a component without returning an unsubscribe function.**
**A:** Every time the component mounts or updates, a new listener is added to the store. When the component unmounts, the old listener remains active in the store's registry, resulting in memory leaks and attempts to trigger updates on unmounted components.

**Q8: Explain the functional composition helper `compose(...funcs)` and how it works.**
**A:** `compose` chains functions from right to left: `compose(f, g, h)(x)` is equivalent to `f(g(h(x)))`. It uses `Array.prototype.reduce` to wrap each function around the next one, creating a nested execution chain.

**Q9: Why does `useSelector` use `useLayoutEffect` to save the latest selected state reference?**
**A:** `useLayoutEffect` runs synchronously before paint, ensuring `latestSelectedState.current` is updated with the latest value before the browser paints. This prevents rendering inconsistencies if state updates occur during layout.

**Q10: Why must `dispatch` throw an error if the action is missing a `type` property?**
**A:** Redux reducers evaluate updates using `action.type` (typically inside `switch` cases). If an action lacks a type, the reducer will skip processing, making state updates unpredictable.

---

### 🔴 Advanced (5)

**Q11: Explain how `useSelector` avoids unnecessary re-renders when other state properties change.**
**A:**
1. When state changes, the store executes the registered `checkForUpdates` callback.
2. `checkForUpdates` runs the selector function with the new state: `newSelectedState = selector(state)`.
3. It performs a reference check: `newSelectedState === latestSelectedState.current`.
4. If the references match, the function bails out, avoiding the `forceUpdate` call. The component does not re-render.

**Q12: Detail the execution flow of `applyMiddleware` and explain how it intercepts the store's dispatch function.**
**A:**
1. `applyMiddleware` takes a list of middlewares and returns a store enhancer function.
2. The enhancer executes `createStore` to build the base store.
3. It maps the middlewares to pass the `getState` and `dispatch` references.
4. It composes the middleware chain: `dispatch = compose(...chain)(store.dispatch)`.
5. It returns the store object with the wrapped `dispatch` function, ensuring all dispatched actions pass through the middleware chain.

**Q13: Why is it unsafe to dispatch actions inside a Redux reducer?**
**A:** Reducers are pure functions designed to compute the next state snapshot. If you dispatch an action inside a reducer, you trigger a recursive update cycle while the store is still computing the previous update, resulting in inconsistent state and call stack overflows.

**Q14: Explain the difference in behavior when a selector returns a primitive value (like a string or number) vs an object literal.**
**A:**
- **Primitive Value:** Strict equality (`===`) checks pass if the value is unchanged, preventing unnecessary re-renders.
- **Object Literal:** A new object reference is created on every run, so the comparison `newSelectedState === latestSelectedState.current` always evaluates to `false`, forcing a re-render on every state change even if the property values are identical.

**Q15: How would you implement a custom middleware `redux-thunk` in our custom library?**
**A:**
```javascript
const thunk = ({ dispatch, getState }) => next => action => {
  if (typeof action === 'function') {
    // If action is a function, execute it and pass dispatch/getState
    return action(dispatch, getState); 
  }
  return next(action); // Otherwise pass the plain object action
};
```

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how an action is processed by a logger middleware before reaching the reducer.**
**A:**
1. A component dispatches an action: `dispatch({ type: 'ADD' })`.
2. The action enters the logging middleware: `action => { ... }`.
3. The logger calls `store.getState()` to read and log the current state.
4. The logger calls `next(action)` to pass the action to the next handler in the chain.
5. Since this is the last middleware, `next` points to the base `store.dispatch` function.
6. The base dispatch function runs the reducer, computes the new state, and updates the store.
7. The base dispatch completes and returns control to the logger middleware.
8. The logger calls `store.getState()` to read and log the updated state.
9. The logger returns the action object to complete execution.

---

**Q17: Explain step-by-step how a component subscribing via `useSelector` is notified and updated when state changes.**
**A:**
1. A component renders and calls `useSelector(state => state.count)`.
2. React runs the selector, retrieves `0`, and registers a listener callback: `checkForUpdates`.
3. An action updates `count` to `1` in the store.
4. The store executes all registered listener callbacks.
5. The component's `checkForUpdates` listener executes.
6. It runs the selector: `newSelected = selector(store.getState())` (resolving to `1`).
7. It compares the new value with the saved value: `1 === 0`, which evaluates to `false`.
8. React-Redux calls `forceUpdate()`, scheduling a local state update on the component's Fiber node.
9. The scheduler runs the work loop and re-renders the component.
10. The component runs `useSelector` again, receives the updated value `1`, and paints the screen.
