# 📌 6.1 — Webpack from Scratch (Without CRA)

---

## ① DEFINITION

**Plain English:**
Imagine writing a book with hundreds of chapters, but each chapter is written in a different dialect and scattered across different folders on your computer.
To publish the book, you need a compiler and editor (a **Bundler** like **Webpack**) to collect all the chapters, translate them into a single standard language (Babel translating modern JS/JSX into standard JavaScript), and bind them into a single, organized volume (a **Bundle**).
Webpack is this publication machinery. It reads your code, tracks how files import one another (building a **Dependency Graph**), processes assets (loaders for CSS, images, and JSX), and outputs a optimized set of static files (`main.js`, `index.html`) that any browser can read.
This section explains how to configure Webpack and Babel manually from absolute scratch without relying on pre-packaged tools like Create React App (CRA).

**Technical Definition:**
Webpack is a static module bundler for modern JavaScript applications. It recursively builds a dependency graph that maps every module your project needs, and generates one or more bundles.
- **Loaders:** Out-of-the-box, Webpack only understands JavaScript and JSON. Loaders allow Webpack to process other file types (e.g. `babel-loader` for JSX, `css-loader` for stylesheets) and convert them into valid modules.
- **Plugins:** Extend Webpack's capabilities to perform tasks like bundle optimization, asset management, and environment variable injection (e.g. `HtmlWebpackPlugin`).

**One-line mental model:**
> *Webpack builds a dependency graph from entry files, processes modules through loaders, and outputs optimized, browser-ready bundles.*

**Where does this live?**
In a project, this is configured inside the `webpack.config.js` file and executed via the `webpack-cli` command in terminal build tasks.

## ② DRILL DOWN

Let's look at the flow of assets through the Webpack bundling pipeline:

```
                          Webpack Bundling Pipeline:
                          
  [ Source Files ]
    ├── index.js (Entry) ──► [ Build Dependency Graph ]
    ├── App.jsx                   │
    ├── styles.css                ▼
    └── logo.svg             [ Run Loaders ]
                                  ├── babel-loader  (JSX ──► JS)
                                  ├── css-loader    (CSS ──► JS module)
                                  └── svg-inline-loader
                                  │
                                  ▼
                             [ Apply Plugins ]
                                  └── HtmlWebpackPlugin (Injects JS into HTML)
                                  │
                                  ▼
                             [ Optimization ]
                                  └── SplitChunks & Minification
                                  │
                                  ▼
                            [ Output Bundle ]
                                  └── dist/main.bundle.js
```

### 🔹 Webpack Core Concepts

A Webpack configuration is structured around five core concepts:

| Concept | Purpose |
|:---|:---|
| **Entry** | The starting point Webpack uses to build its internal dependency graph (typically `src/index.js`). |
| **Output** | Tells Webpack where to write the compiled bundles and how to name them (typically `dist/`). |
| **Loaders** | Transforms source files during compilation (e.g., converting JSX or TypeScript to JavaScript). |
| **Plugins** | Performs broader build tasks, asset optimization, and file injection. |
| **Mode** | Enables built-in optimizations for the environment (`development` vs `production`). |

## ③ WHY IT EXISTS

### What problem does it solve?

Before bundlers existed, developers imported scripts manually in HTML files:

```html
<!-- Hard to manage and maintain: -->
<script src="./src/utils.js"></script>
<script src="./src/api.js"></script>
<script src="./src/components/Button.js"></script>
<script src="./src/index.js"></script>
```

This approach had major drawbacks:
1. **Global Scope Pollution:** Every script declared variables on the global `window` object, leading to naming collisions.
2. **Network Overhead:** Loading dozens of small files required multiple round-trip HTTP requests, slowing down page loads.
3. **Execution Order Mismatches:** If you imported `index.js` before `utils.js`, the application would crash due to missing dependencies.

Webpack solves these issues by treating every file as a modular unit (using ES Modules `import/export` or CommonJS `require`), resolving the dependencies automatically, and outputting optimized bundles.

## ④ INTERNAL WORKING

Let's write a complete, production-ready Webpack and Babel configuration from scratch.

### 🔹 1. The Babel Configuration (`.babelrc`)

Babel is the transpiler that translates modern ECMAScript and JSX into standard, backward-compatible JavaScript.

```json
{
  "presets": [
    // 1. Compiles modern JavaScript features to ES5 based on target browsers
    ["@babel/preset-env", {
      "targets": {
        "browsers": [">0.25%", "not dead"]
      },
      "useBuiltIns": "usage", // Polyfill missing features automatically
      "corejs": 3
    }],
    // 2. Transpiles React JSX tags
    ["@babel/preset-react", {
      "runtime": "automatic" // Injects react/jsx-runtime automatically (React 17+)
    }]
  ]
}
```

---

### 🔹 2. The Webpack Configuration (`webpack.config.js`)

Here is the complete configuration file showing loaders, plugins, and optimization settings:

```javascript
const path = require('path');
const HtmlWebpackPlugin = require('html-webpack-plugin');
const MiniCssExtractPlugin = require('mini-css-extract-plugin');
const TerserPlugin = require('terser-webpack-plugin');

module.exports = (env, argv) => {
  const isProduction = argv.mode === 'production';

  return {
    // 1. Entry point
    entry: './src/index.js',

    // 2. Output configuration
    output: {
      path: path.resolve(__dirname, 'dist'),
      filename: isProduction ? '[name].[contenthash].js' : '[name].js',
      clean: true, // Clean output directory before every build
      publicPath: '/'
    },

    // 3. Module loaders
    module: {
      rules: [
        // Rule A: Transpile JS & JSX
        {
          test: /\.(js|jsx)$/,
          exclude: /node_modules/,
          use: {
            loader: 'babel-loader'
          }
        },
        // Rule B: Process CSS
        {
          test: /\.css$/,
          use: [
            // Extract CSS into separate files in production, inject into style tag in development
            isProduction ? MiniCssExtractPlugin.loader : 'style-loader',
            'css-loader'
          ]
        },
        // Rule C: Process Assets (images, fonts)
        {
          test: /\.(png|svg|jpg|jpeg|gif)$/i,
          type: 'asset/resource'
        }
      ]
    },

    // 4. Resolve file extensions automatically
    resolve: {
      extensions: ['.js', '.jsx']
    },

    // 5. Plugins
    plugins: [
      // Injects bundled JS files into index.html template
      new HtmlWebpackPlugin({
        template: './public/index.html',
        minify: isProduction ? {
          collapseWhitespace: true,
          removeComments: true
        } : false
      }),
      // Extracts CSS into a separate file in production
      isProduction && new MiniCssExtractPlugin({
        filename: '[name].[contenthash].css'
      })
    ].filter(Boolean), // Filter out false values in development

    // 6. DevServer: fast local development configurations
    devServer: {
      port: 3000,
      hot: true, // Enable Hot Module Replacement (HMR)
      historyApiFallback: true, // Redirect 404s to index.html for SPA routing
      open: true
    },

    // 7. Optimization configurations
    optimization: {
      minimize: isProduction,
      minimizer: [new TerserPlugin()],
      splitChunks: {
        chunks: 'all', // Code split third-party libraries (node_modules) automatically
        cacheGroups: {
          vendor: {
            test: /[\\/]node_modules[\\/]/,
            name: 'vendors',
            chunks: 'all'
          }
        }
      }
    },

    // 8. Generate Source Maps for easy debugging
    devtool: isProduction ? 'source-map' : 'eval-cheap-module-source-map'
  };
};
```

## ⑤ ADVANTAGES

1. **Granular Control:** You can configure loaders, plugins, and optimizations exactly as your project requires, without the constraints of Create React App.
2. **Optimized Bundles:** Automated code splitting (`splitChunks`) and minification (Terser) reduce bundle size and speed up page load times.
3. **Isomorphic Loaders:** Webpack can import CSS, images, SVGs, and fonts directly inside JavaScript files, treating all assets as modules.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Configuration Complexity:** Webpack configuration is notoriously verbose and hard to learn. Getting it wrong can lead to build errors and bloated bundle sizes.
2. **Slow Build Times:** Webpack builds a full in-memory dependency graph. For large applications, cold starts and production builds can take several minutes.

### ⚠️ Trap: Forgetting to Cache-Bust in Production

If you output files with static names (like `main.js`), browsers will cache the file. When you push updates, users won't see the changes because their browsers will continue to read the cached `main.js`.

```javascript
// ❌ BAD: No cache busting
output: {
  filename: 'main.js' 
}

// ✅ GOOD: Use content hashes in production
output: {
  filename: '[name].[contenthash].js' // Generates main.a8f2c3d9.js
}
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What is Webpack?**
**A:** Webpack is a static module bundler for modern JavaScript applications. It recursively builds a dependency graph mapping all project modules and generates optimized bundles that browsers can execute.

**Q2: What are Loaders in Webpack?**
**A:** Loaders are functions that process and transform non-JavaScript files (like CSS, JSX, TypeScript, or images) into valid modules during the bundling process.

**Q3: What is the difference between Loaders and Plugins in Webpack?**
**A:**
- **Loaders** operate at the individual file level during compilation (e.g. converting a JSX file to JS).
- **Plugins** operate on the entire bundling lifecycle and can perform tasks like bundle optimization, asset extraction, and environment variables injection.

**Q4: What is the entry point in a Webpack configuration?**
**A:** The entry point is the file Webpack uses as the starting node to build its internal dependency graph (typically `src/index.js` in React projects).

**Q5: Why do we use Babel in a React project?**
**A:** Because browsers cannot execute JSX or modern ECMAScript syntax directly. Babel transpiles these files into standard, backward-compatible ES5 JavaScript that is widely supported by browsers.

---

### 🟡 Intermediate (5)

**Q6: Explain what `HtmlWebpackPlugin` does in a Webpack configuration.**
**A:** It generates an `index.html` file in the output directory using a template. It automatically inserts `<script>` tags for all bundled JavaScript files, including hashed filenames, ensuring correct asset paths.

**Q7: Explain the difference between `style-loader` and `MiniCssExtractPlugin.loader`.**
**A:**
- `style-loader` injects CSS code directly into a `<style>` tag in the HTML head dynamically during runtime. It is used in development for fast updates.
- `MiniCssExtractPlugin.loader` extracts CSS into separate physical files in the build directory. It is preferred in production to leverage browser styling caches and allow parallel asset downloads.

**Q8: What is Cache-Busting and how is it implemented in Webpack?**
**A:** Cache-Busting is a technique to force browsers to fetch the latest version of a file instead of using their local cache. In Webpack, it is implemented by adding a content-based hash to the output filename: `[name].[contenthash].js`. If the file contents change, Webpack generates a new hash name.

**Q9: Why does Webpack's `devServer` need the `historyApiFallback: true` configuration in SPA applications?**
**A:** Single Page Applications (SPAs) use client-side routing. If a user visits `/profile` directly, the browser requests that route from the Webpack dev server, which will return a 404 because the file `/profile` doesn't exist. Setting `historyApiFallback: true` redirects all 404 requests to `index.html`, allowing the client-side router to parse the URL and render the correct view.

**Q10: What does `clean: true` do in the Webpack output settings?**
**A:** It tells Webpack to clean and empty the build output directory (`dist/`) before every build run, preventing old, unused hashed bundle files from accumulating.

### 🔴 Advanced (5)

**Q11: Explain how Webpack builds its internal Dependency Graph during the build process.**
**A:**
1. Webpack starts at the specified `entry` file.
2. It parses the file and searches for import directives (e.g. `import`, `require`, `import()`).
3. For every import found, Webpack resolves the file path, adds it as a dependency node, and repeats the process recursively for the imported file.
4. If a loader rule matches the file extension, Webpack processes the file contents before parsing.
5. This recursive traversal continues until all files are resolved, forming a complete dependency graph.

**Q12: Detail the mechanics of Code Splitting using `splitChunks` in Webpack. How does it optimize page performance?**
**A:** `splitChunks` extracts shared modules (such as heavy third-party libraries inside `node_modules`) into separate bundle files. If your app bundle is `2MB` and contains React, Webpack splits it:
- `vendors.bundle.js`: Contains React and dependencies (`1.5MB`).
- `main.bundle.js`: Contains your component code (`500KB`).
When you update component code, the browser only re-downloads the `500KB` file, reading the unchanged `vendors.bundle.js` from its local cache, saving bandwidth and speeding up load times.

**Q13: Explain how Webpack handles Tree Shaking and what configurations are required to enable it.**
**A:** Tree Shaking is the process of removing dead (unused) code from the final bundle.
To enable it:
1. Use ES Modules (`import`/`export`) syntax, as its static structure allows Webpack to analyze imports at build time.
2. Set `mode: 'production'` to enable Terser minification, which strips unused functions.
3. Configure `sideEffects: false` in `package.json` to tell Webpack that modules do not run global side-effects (like modifying window objects), allowing it to safely remove unused imports.

**Q14: Explain the difference in compilation performance between `eval-source-map` and `source-map` in Webpack.**
**A:**
- `eval-source-map` wraps modules in `eval()` functions and generates source maps inline. It is very fast to rebuild, making it ideal for development, but it results in large files that are unsuited for production.
- `source-map` generates full, high-quality mapping files saved as separate physical `.map` files. It has a slower build time, but it keeps the production bundles clean and optimized while allowing production debugging.

**Q15: How would you configure Webpack to compile and support TypeScript alongside React from scratch?**
**A:**
1. Install TypeScript and loaders: `npm install typescript ts-loader`.
2. Create a `tsconfig.json` configuring compiler targets.
3. Add a loader rule in `webpack.config.js` to process `.ts` and `.tsx` files:
   ```javascript
   {
     test: /\.(ts|tsx)$/,
     exclude: /node_modules/,
     use: 'ts-loader'
   }
   ```
4. Add `.ts` and `.tsx` to Webpack's resolution extensions list:
   `resolve: { extensions: ['.ts', '.tsx', '.js', '.jsx'] }`.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how Webpack processes a CSS import (`import './styles.css'`) inside a JavaScript file in development.**
**A:**
1. Webpack encounters the import statement during dependency graph construction.
2. It matches the file extension with the CSS loader rule: `test: /\.css$/`.
3. It passes the CSS file content to `css-loader` first.
4. `css-loader` compiles the CSS text into a JavaScript string module: `module.exports = ".box { color: red; }"`.
5. It then passes this module to `style-loader`.
6. `style-loader` wraps the CSS string in a helper script that creates a `<style>` DOM element at runtime.
7. Webpack includes the compiled JavaScript module inside the main bundle file.
8. When the browser executes the bundle, `style-loader` runs, creates a `<style>` tag, injects the CSS content, and appends the tag to the HTML head.

---

**Q17: Explain step-by-step how Babel transpiles a React JSX element into a standard JavaScript object.**
**A:**
1. Babel reads a component file containing: `<h1 className="active">Title</h1>`.
2. It parses the syntax into an Abstract Syntax Tree (AST).
3. The parser matches the JSX element node.
4. The `@babel/preset-react` plugin intercepts the node and updates the AST branch.
5. In React 17+ automatic runtime:
   - Babel replaces the JSX tag with a transpiled call to the compiler runtime: `_jsx("h1", { className: "active", children: "Title" })`.
   - It inserts the runtime import statement at the top of the file: `import { jsx as _jsx } from "react/jsx-runtime"`.
6. Babel's generator walks the updated AST and writes the final JavaScript string.
7. During execution, the browser calls the runtime function, returning the plain React element object.
