# 📌 7.1 — React Below 16 vs. 16 vs. 17

---

## ① DEFINITION

**Plain English:**
Think of React as a car company evolving its engine over the years.
- **Below React 16:** The engine was a basic manual transmission (the Stack Reconciler). It was simple, but if you drove up a steep hill (heavy rendering), the engine would choke and freeze, unable to change gears mid-way. You had to drive using complex levers (Class components, lifecycle methods, and Mixins).
- **React 16:** The company completely rebuilt the engine block (introducing the Fiber Reconciler) and added an automatic transmission (Hooks in v16.8). Rendering became pauseable, and developers could manage state using simple buttons (Functional components with hooks).
- **React 17:** This was a "tune-up" release. The company didn't add new features, but they rewired the dashboard (moving event listeners from the document root to the specific app root) to allow running multiple versions of React on the same page.
This section traces the architectural and API changes from React's early days through version 17.

**Technical Definition:**
- **React Below 16:** Built around the **Stack Reconciler**, which evaluated updates using synchronous recursion. State and lifecycles were managed using ES5 `React.createClass` or ES6 Class components, and code reuse was achieved via **Mixins**.
- **React 16 (2017):** Introduced the **Fiber Reconciler**, enabling asynchronous rendering. It introduced **Hooks** (v16.8), **Error Boundaries**, **Portals**, and **React.lazy / Suspense**.
- **React 17 (2020):** Known as the "No New Features" release. It changed the event delegation target from `document` to the application's root container, aligned effect cleanup timings to run asynchronously, and removed the performance-focused **Event Pooling** mechanism.

**One-line mental model:**
> *Below 16 was recursive and class-heavy; 16 introduced Fiber, Hooks, and portals; 17 rewired event delegation to the root.*

**Where does this live?**
These changes represent updates to React's core packages (`react` and `react-dom`) and reconciliation engines over successive releases.

## ② DRILL DOWN

Let's compare the features and architectural structures across these versions:

### 🔹 Feature Comparison Table

| Feature / Architecture | Below React 16 | React 16 | React 17 |
|:---|:---|:---|:---|
| **Reconciler Engine** | Stack Reconciler (Recursive) | Fiber Reconciler (Iterative DFS) | Fiber Reconciler (Iterative DFS) |
| **Primary Component Type** | Class Component / `createClass` | Class + Functional Components | Class + Functional Components |
| **State hooks?** | No | Yes (v16.8+) | Yes |

## ③ WHY IT EXISTS

### What problem does it solve?

Each version solved critical limitations of its predecessor:

#### 1. Why React 16 Replaced Below 16
The Stack Reconciler was too rigid.
- If a component threw an error during rendering, the native call stack unwound completely, leaving the DOM in a corrupted state and crashing the app. React 16 introduced **Error Boundaries** (`componentDidCatch`) to catch rendering errors and show a fallback UI.
- Class components forced writing duplicate logic across lifecycle methods, which led React 16.8 to introduce **Hooks** to allow functional components to manage local state and reuse logic cleanly.

#### 2. Why React 17 Rewired Event Delegation
In React 16 and below, React listened to events by binding a single listener to the browser's `document` object.
- If you mounted two different React applications on the same page (e.g. embedding a React v15 widget inside a React v16 app), events would bubble up to `document`, causing event handlers in both apps to trigger and conflict.
- React 17 solved this by binding event listeners to the specific **DOM root container** (e.g. `<div id="root">`), making it safe to run multiple nested React applications on a single page.

## ④ INTERNAL WORKING

Let's look at the changes in Event Delegation and Event Pooling.

### 🔹 Event Delegation: React 16 vs. React 17

```
  React 16 Event Delegation (Delegates to Document):
  
  [ User Click ] ──► [ Bubbles up through DOM ] ──► [ document ] (React Listener catches)
                                                         │
                                                         ▼
                                              Executes all handlers
  ==================================================================================================
  
  React 17 Event Delegation (Delegates to App Root):
  
  [ User Click ] ──► [ Bubbles up ] ──► [ App Root div#root ] (React Listener catches)
                                               │
                                               ▼
                                      Executes local handlers
```

#### Why Event Pooling was removed in React 17:
In React 16, to keep memory usage low, React recycled `SyntheticEvent` objects.
If you tried to read an event asynchronously inside a callback, the event properties were cleared to `null` by the pool, resulting in errors unless you called `e.persist()`.

```javascript
// React 16 Event Pooling Trap:
function handleClick(e) {
  setTimeout(() => {
    // CRASH: e.target is null because the event object was recycled!
    console.log(e.target.value); 
  }, 1000);
}
```

Since modern browser engines have optimized garbage collection, this recycling overhead is no longer necessary. React 17 removed event pooling entirely, allowing event properties to persist inside asynchronous closures.

## ⑤ ADVANTAGES

1. **Fiber Resiliency:** React 16's Fiber architecture allows pausing rendering, rendering offscreen tabs, and catching errors gracefully.
2. **Logic Reusability:** Hooks in v16.8 removed the boilerplate of class components and solved "wrapper hell".
3. **Safe Multi-Version Mounts:** React 17's root event delegation allows deploying nested applications running different React versions safely on a single page.

---

## ⑥ DISADVANTAGES / TRAPS

1. **Deprecated Lifecycles:** Switching to React 16/17 deprecated older Render-phase lifecycles (`componentWillMount`, `componentWillReceiveProps`, `componentWillUpdate`), requiring developers to refactor older code.
2. **Uncaught Hydration Mismatches:** React 16/17 became stricter during SSR hydration, throwing errors if the client-rendered output differed from the server-rendered HTML.

### ⚠️ Trap: Using Deprecated Lifecycles in Concurrent React

```javascript
// ❌ BAD: Using unsafe lifecycles that can execute multiple times
class OldComponent extends React.Component {
  componentWillReceiveProps(nextProps) {
    // In concurrent React, this can execute multiple times per render,
    // leading to duplicate API calls or state updates!
    if (nextProps.id !== this.props.id) {
      this.fetchData(nextProps.id); 
    }
  }
}

// ✅ GOOD: Use componentDidUpdate (runs safely in the Commit phase)
class NewComponent extends React.Component {
  componentDidUpdate(prevProps) {
    if (this.props.id !== prevProps.id) {
      this.fetchData(this.props.id); // Runs once, safely after DOM is updated
    }
  }
}
```

## ⑦ PRODUCTION USAGE

### Incremental Migration Strategy
React 17's event delegation target enables incremental migrations in large production codebases. You can build and deploy new pages using React 17, while leaving older pages running on React 16 inside the same container.

```jsx
// index.js (Mounting a React 17 sub-application inside a React 16 host)
import React from 'react';
import ReactDOM from 'react-dom';
import ModernWidget from './ModernWidget'; // Compiled with React 17

// In React 17, events are bound to #modern-widget-root,
// preventing conflicts with the host application's v16 listeners on the document.
ReactDOM.render(
  <React.StrictMode>
    <ModernWidget />
  </React.StrictMode>,
  document.getElementById('modern-widget-root')
);
```

## ⑧ INTERVIEW QUESTIONS

### 🟢 Basic (5)

**Q1: What reconciler was used in React before version 16?**
**A:** The **Stack Reconciler**, which traversed the component tree using synchronous recursion and could not be paused.

**Q2: In what version of React were Hooks introduced?**
**A:** Hooks were introduced in **React 16.8**.

**Q3: What are Error Boundaries in React 16?**
**A:** Error Boundaries are class components that implement `static getDerivedStateFromError` or `componentDidCatch` to catch JavaScript errors in their child component tree, log errors, and render a fallback UI instead of crashing the app.

**Q4: How did React 17 change Event Delegation?**
**A:** React 17 moved the event delegation target from the browser's `document` object to the specific **DOM root container** of the application (e.g. `div#root`).

**Q5: What is Event Pooling and why was it removed in React 17?**
**A:** Event Pooling was a mechanism where React recycled `SyntheticEvent` objects to reduce memory usage. It was removed in React 17 because modern browser engines manage garbage collection efficiently, making pooling unnecessary and eliminating the need to call `e.persist()`.

---

### 🟡 Intermediate (5)

**Q6: Why are lifecycles like `componentWillMount` deprecated in React 16/17?**
**A:** Because they execute during the Render phase. In concurrent React, the Render phase can be paused, discarded, or restarted, meaning these lifecycles could execute multiple times for a single update, leading to duplicate side effects.

**Q7: Explain the difference between React Portals and standard rendering.**
**A:** Standard components are mounted as children of their closest parent DOM node. React Portals (`ReactDOM.createPortal`) allow rendering children into a completely different DOM node located outside the parent component's tree, while preserving React context and event bubble propagation.

**Q8: Explain the change in `useEffect` cleanup execution timing between React 16 and React 17.**
**A:** 
- In **React 16**, the cleanup function of `useEffect` ran synchronously before the next effect executed, which could block browser painting.
- In **React 17**, effect cleanups execute asynchronously after paint, keeping the UI responsive.

**Q9: Why does `React.StrictMode` run component renders twice in development?**
**A:** It runs them twice to help developers detect side-effects in the render phase. Since the render phase is supposed to be pure, running it twice will highlight incorrect side-effects (like global mutations or variable increments) during development.

**Q10: What is the purpose of `forwardRef` introduced in React 16?**
**A:** `forwardRef` is a utility that allows a component to receive a `ref` from its parent and pass ("forward") it down to an underlying HTML child element, allowing parent components to access the child's DOM node directly.

---

### 🔴 Advanced (5)

**Q11: Explain how React 17's change in event delegation target enables running multiple versions of React on the same page without conflicts.**
**A:** In React 16 and below, all React instances bound event listeners to the global `document` object. If two apps ran on different versions, an event bubbling up to the document would trigger event listeners in both apps, leading to conflicts. By binding listeners to the app's root element in React 17, events are scoped to their respective containers, isolating event handling for each app.

**Q12: Detail the architectural reason why Error Boundaries must be Class Components and cannot be written as Functional Components with hooks.**
**A:** Error boundaries catch rendering exceptions during the Fiber reconciliation work loop. They rely on the lifecycle methods `componentDidCatch` and `getDerivedStateFromError` to intercept these exceptions. As of React 19, there are no hook equivalents (like `useErrorBoundary`) that map to these low-level reconciler lifecycle methods, so class components are required.

**Q13: How does React's synthetic event bubble propagation behave when passing through a Portal boundary in React 16/17?**
**A:** Although a Portal renders its DOM nodes into a different physical container in the document, React preserves the component hierarchy. An event triggered inside a portal child will still bubble up through the virtual React parent tree, triggering handlers on React parent components even if they are physically located elsewhere in the DOM.

**Q14: Explain the difference in execution timing between `componentWillUnmount` and the cleanup function of `useEffect`.**
**A:**
- `componentWillUnmount` runs synchronously during the Mutation phase of the commit, before DOM elements are removed.
- `useEffect` cleanup runs asynchronously after the Commit phase is complete and the browser has painted, freeing up thread time.

**Q15: How would you migrate a legacy codebase that uses Mixins to modern functional components?**
**A:** Mixins were used to share utility functions and state across components.
To migrate them:
1. Extract the shared state and logic into custom hook functions.
2. Replace class components with functional components.
3. Import and call the custom hooks inside the functional components, mapping the return values to props.

### 🧠 "Explain What Happens Internally" (2)

**Q16: Explain step-by-step how React catches a rendering crash inside a component and delegate it to an Error Boundary.**
**A:**
1. During `beginWork`, a component function runs and throws a JavaScript error (e.g. `undefined.map()`).
2. The work loop catches the exception and stops the DFS traversal.
3. React enters a recovery pass: it walks up the `return` pointers of the virtual Fiber stack to search for a Fiber of type `ClassComponent` that implements `getDerivedStateFromError` or `componentDidCatch`.
4. If it finds one:
   - React updates the boundary's state with the error details.
   - It marks the boundary Fiber with the `Callback` and `Update` flags.
   - It discards the crashed child subtree.
5. In the Commit phase, React unmounts the crashed children and executes the boundary's `componentDidCatch` callback.
6. React schedules a new render, rendering the boundary's `fallback` UI instead of the crashed children.

---

**Q17: Explain step-by-step what happens internally when a user clicks a button inside a Portal, and how React bubbles the event to a parent component.**
**A:**
1. The user clicks a button rendered inside a Portal.
2. The browser registers a native click event on the button DOM element.
3. The event bubbles up the physical DOM tree.
4. React's event listener (bound to the application's root container) intercepts the native event.
5. React wraps the native event in a `SyntheticEvent` object.
6. Instead of using the physical DOM hierarchy, React reads the event target's Fiber node pointer and follows the `return` pointers of the virtual Fiber tree.
7. The traversal moves up through the Portal's virtual parent component Fiber nodes.
8. If React finds an `onClick` listener on a virtual parent component, it executes the handler.
9. This virtual bubbling continues up to the root container, bypassing the physical DOM structure.
